# 3.8 Agent 框架对比与选型

> **核心问题**：LangGraph、CrewAI、OpenAI Agents SDK、Claude Agent SDK、Google ADK、Microsoft Agent Framework、Pydantic AI、LlamaIndex Workflows……框架多到眼花，各自强在哪、代价是什么？以及一个更根本的问题——**你到底该不该用框架，还是直接调 API 手搓？** 具体项目怎么选？
>
> **目标**：
> - **先破除"必须用框架"的迷信**——理解 Anthropic 官方为什么建议"从直接调 API 开始"
> - 建立一张 8 大主流框架的全景定位图（编排抽象、成熟度、一句话定位）
> - **掌握本节的核心方法论：不按"功能清单"选框架，而按"你的主约束"选**
> - 吃透关键取舍维度：控制力 vs 上手速度、抽象税、model-agnostic vs vendor-lock、状态/持久化、可观测、学习曲线
> - 理解框架的演进与迁移策略（原型用轻的，生产降抽象；CrewAI→LangGraph 的典型迁移）
>
> **边界**：多 Agent 的编排"模式"（Orchestrator-Worker/Swarm 等抽象概念）属于 **3.7**，本节讲的是**实现这些模式的具体框架工具**；框架背后的协议（MCP/A2A）属于 **3.3**；通用的部署/CI/监控工程属于 **第 7 章**，本节只讲框架自带的可观测能力。

---

## 3.8.0 一句先声明：这一节的知识会过期，但方法论不会

框架是本章**变化最快**的领域。就在写作的这段时间里（2026 年年中），LangGraph 刚到 1.0、Microsoft 把 Semantic Kernel 和 AutoGen 合并成 Agent Framework 1.0、Pydantic AI V2 和 LlamaIndex Workflows 1.0 在同一个 48 小时窗口内先后 stable……**任何"版本号 + 功能清单"的对比，几个月后就是过时的。**

所以本节的写法刻意不同：**具体框架的功能会列，但重点是教你一套"以不变应万变"的选型方法论——按主约束选、认清抽象税、原型与生产分开考虑。** 记住方法，框架换代你也不慌。

---

## 3.8.1 选框架前，先问：你真的需要框架吗？

这是本节最重要、也最反直觉的一条，必须放在最前面。**很多人一上来就在纠结"选 LangGraph 还是 CrewAI"，却跳过了那个更该先问的问题：我需要框架吗？**

Anthropic 在《Building Effective Agents》里给了一个非常克制、也非常诚实的官方建议（这和他们"能简单就别复杂"的一贯立场一致）：

> **开发者应该从直接使用 LLM API 开始——很多模式用几行代码就能实现。** 框架常常引入额外的抽象层，**掩盖底层的 prompt 和响应，让调试更难**，并且**诱使你在本可以更简单的地方增加复杂度。**

这段话点破了框架的**根本代价——"抽象税"**：框架为了给你便利，在你和"真正发生的事"（发给模型的 prompt、模型返回的 raw response、工具调用的实际参数）之间加了一层。这层抽象在原型期帮你省事，但在生产期排障时，**它挡在你和真相之间**。

一个可验证的事实：**你可以用不到 200 行 Python，直接基于 API 手搓一个具备工具调用、记忆、多步推理的生产可用 Agent。** 因为回忆 3.1——Agent 的本质就是"LLM + 工具 + 循环"，这个循环真的很简单。框架并没有什么魔法，它只是把这个循环和一些周边（状态管理、重试、追踪）替你包好了。

**所以正确的决策顺序是：**

1. **默认起点：不用框架，直接调 API。** 尤其是简单 Agent（单 Agent、几个工具、线性循环），直接 API 给你**最大的控制力、最小的依赖、以及对每个组件"到底在干什么"的完全理解**。
2. **考虑用框架，当：** 你需要的周边能力（复杂状态图、检查点持久化、多 Agent 编排、可视化调试、团队协作抽象）自己搓成本太高，且框架提供的正是这些。
3. **即使用了框架，也要能降抽象。** Anthropic 的原话："框架能帮你快速起步，但当你走向生产时，**不要犹豫去减少抽象层、用基础组件构建。**"

> **给你的实操判断**：如果你的 Agent 是"单 Agent + 少量工具 + 基本线性"，**优先直接 API**（或厂商轻量 SDK）。只有当复杂度真的顶到"我需要状态化的图、需要检查点、需要多 Agent 编排"时，才引入重框架。**别为你还没遇到的复杂度，先交抽象税。** 这条和 3.6（能不加规划就不加）、3.7（能单不多）是同一种工程审美——**克制。**

## 3.8.2 八大主流框架全景图

确实需要框架时，先建立全景。下面是 2026 年中的 8 个主流框架，**重点看"编排抽象"这一列**——它是每个框架的灵魂，也是最不容易过时的特征。

| 框架 | 编排抽象（灵魂） | 一句话定位 | 成熟度信号 |
|---|---|---|---|
| **LangGraph** | **有向图 + 条件边**（节点/边/状态） | 复杂**状态化**工作流的生产标杆 | 1.0 stable，~39M/月 PyPI 下载 |
| **CrewAI** | **角色制 crew**（researcher/writer/…）+ process | **最快搭出多 Agent 原型** | 1.14，~46K GitHub stars |
| **OpenAI Agents SDK** | **显式 handoff** + guardrails | GPT-native 单 Agent 最快路径 | 2025.3 发布，LiteLLM 接 100+ 模型（beta） |
| **Claude Agent SDK** | **tool-use 链 + 子 Agent** | Claude 生态，预置 bash/file/web + MCP | 支持层级化子 Agent spawn |
| **Google ADK** | **层级 agent 树**（root 委派 sub） | 深绑 Vertex AI/Gemini/GCP，**原生 A2A** | 2025.4 发布 |
| **Microsoft Agent Framework** | **对话式 GroupChat**（AutoGen 传统） | 微软/企业生态，SK+AutoGen 合并体 | 1.0（2026.4，合并 Semantic Kernel + AutoGen） |
| **Pydantic AI** | **类型安全** + harness-first | Python 团队的类型安全之选 | V2（2026.6） |
| **LlamaIndex Workflows** | **事件驱动 workflow** | RAG 中心的 Agent 工作流 | Workflows 1.0（2026.6） |

**读这张表的正确方式**：不要试图记住每个的功能列表（会过期），而是记住**它们的编排抽象各不相同**，这决定了它们各自"顺手"的场景：

- 要**精细控制流**（循环、分支、审批关卡、回溯）→ 图抽象（LangGraph）最贴合。
- 要**按角色分工**快速拼多 Agent → 角色制（CrewAI）最省心。
- 要**A 转 B 的分流**→ handoff（OpenAI SDK）最直接。
- 要**层级委派**→ agent 树（Google ADK）最自然。
- 要**多 Agent 对话协商**→ GroupChat（Microsoft）最对路。

这就自然引出本节的核心方法论。

## 3.8.3 核心方法论：按"主约束"选，不按"功能清单"选

**新手选框架的最大错误，是打开一堆对比表格，逐行比功能，然后选那个"打勾最多"的。** 这是错的——因为几乎所有主流框架，在功能清单上都趋同了（都支持工具、记忆、多 Agent、MCP、流式……）。功能对比区分不出它们。

**正确的方法：先想清楚你这个项目的"主约束"（the one thing that matters most），然后选那个为这个约束而生的框架。** 主约束是那个"如果它不满足，别的再好都白搭"的硬要求。一个项目通常只有一个主约束。

先明确你的主约束是什么，再看下一节的映射。常见的主约束有这么几类：

- **控制力与状态**：需要复杂的、可检查点恢复的、能回溯调试的状态化控制流。
- **上手速度**：要在最短时间内出一个能跑的（多 Agent）原型。
- **厂商绑定**：已经押注了某家的模型/云，要最短路径接入它的生态。
- **工程规范**：团队重视类型安全、可测试、可维护。
- **互操作**：要和别的团队/别的框架的 Agent 互通。
- **最大控制/最小依赖**：不想要任何抽象税，要完全掌控。

## 3.8.4 按主约束的选型决策

把上面的主约束映射到具体选择。**找到你的主约束那一行，那就是你的答案。**

| 你的主约束（最在乎的那件事） | 选它 | 为什么 |
|---|---|---|
| **复杂状态化控制流**：循环、条件分支、人工审批关卡、检查点恢复、回溯调试 | **LangGraph** | 图抽象天生适合表达复杂控制流；内置 checkpointing、time-travel debug、durable state；model-agnostic；企业合规（HIPAA/SOC2）。代价：1-2 周学习曲线 |
| **最快出多 Agent 原型**，且任务天然拆成角色（研究员/写手/审校） | **CrewAI** | 角色制 DSL，~20 行起步，学习曲线最低；MCP 一等支持。代价：控制流灵活度有天花板 |
| **已押注 OpenAI 模型**，要最快搭单 Agent / handoff 链 | **OpenAI Agents SDK** | handoff 一等原语 + guardrails，<100 行；追踪开箱即用。代价：偏 vendor-lock（虽有 LiteLLM beta 接多模型） |
| **已押注 Claude**，要预置工具（bash/file/web）+ MCP | **Claude Agent SDK** | 厂商 SDK 无抽象税，直连 Claude 能力 + MCP + 层级子 Agent。代价：绑 Claude |
| **深度绑定 Google Cloud/Gemini**，且要**跨框架互通** | **Google ADK** | 深集成 Vertex AI/Gemini/GCP；**原生 A2A**，能发现并调用 LangGraph/CrewAI 建的 Agent（回顾 3.3） | 
| **微软/企业生态**，要多 Agent 对话协商 | **Microsoft Agent Framework** | 承接 Semantic Kernel 的企业集成 + AutoGen 的 GroupChat 对话式多 Agent |
| **类型安全、可测试、工程规范**（严肃 Python 团队） | **Pydantic AI** | 建立在 Pydantic 之上，类型安全、结构化输出可靠，harness-first 设计 |
| **RAG 是核心**，Agent 围着检索转 | **LlamaIndex Workflows** | 事件驱动 workflow + LlamaIndex 最强的检索/索引生态无缝衔接 |
| **最大控制力、最小依赖、要完全理解每个组件** | **不用框架，直接 API** | 见 3.8.1——单 Agent/简单场景的最佳选择，<200 行搞定，无抽象税 |

> **注意**：这张表的"选它"会随版本演进变化，但**"按主约束映射"这个动作永远有效**。用的时候，先诚实地问自己"这个项目真正的硬约束是哪一个"，别贪多——你不可能同时优化所有维度。

## 3.8.5 关键取舍维度详解

主约束帮你快速定位，但落地前要理解几个**贯穿所有框架的取舍轴**。这些是你评估任何框架（包括未来新出的）的通用标尺。

### 一、控制力 vs 上手速度（最核心的一根轴）

这是框架世界的第一性张力。**上手越快的框架，通常越"帮你做决定"，也就越难在它不合你意时掰回来。**

- **CrewAI / OpenAI SDK**：高层抽象，几十行起步，但当你要的控制流超出它的预设模式，就顶到天花板。
- **LangGraph**：低层图控制，什么都能精细表达，但要多花 1-2 周学它的心智模型。
- **直接 API**：完全控制，零抽象，但周边全得自己搓。

**没有"更好"，只有"匹配你项目在控制力光谱上的位置"。** 原型偏速度，复杂生产偏控制。

### 二、抽象税——框架帮你的越多，挡你的越多

重申 3.8.1 的核心：抽象是**双刃**。它在顺境（功能正好被框架覆盖）帮你飞快，在逆境（要调试、要定制、要理解 raw prompt）挡你视线。评估一个框架时，专门问一句：**"当它出问题时，我能不能轻松看到底层发生了什么、并绕过它？"** 能，抽象税可控；不能，慎入。

### 三、model-agnostic vs vendor-lock

- **model-agnostic**（LangGraph、CrewAI、Pydantic AI）：随时换模型，能独立于框架优化 LLM 成本——**这在成本敏感、或想跟随最优模型的场景很关键**（回顾 3.7：升级模型常比堆 token 更划算，前提是你能自由换模型）。
- **vendor SDK**（OpenAI Agents SDK、Claude Agent SDK、Google ADK）：绑定厂商，换模型难，但**换来的是零抽象税、第一时间用上厂商最新能力**。押注一家生态时这是优势，想保持灵活时这是枷锁。

### 四、状态与持久化

长时运行 / 可恢复的 Agent（回顾 3.7 的 durable execution、3.5 的记忆），**检查点（checkpointing）能力是硬指标**。LangGraph 在这块最强（内置 checkpoint、time-travel）。如果你的 Agent 会跑很久、需要中断恢复，这一维度权重要拉满。

### 五、可观测性

生产 Agent 必须能看到"每一步在干什么"（回顾 3.7 Anthropic "像你的 Agent 一样思考"、连着 3.12 可观测）。有的框架自带追踪（LangGraph 配 LangSmith、OpenAI SDK 内置 tracing），有的要自己接。**生产前必须确认可观测方案**——一个看不见内部的 Agent 是不可维护的。

### 六、学习曲线与团队

别忽略**团队现实**：LangGraph 强但要学 1-2 周；CrewAI 一下午就能上手。如果团队要快速验证想法、或成员不熟 Agent，先用低曲线的框架把价值验证出来，比硬啃 LangGraph 更务实。

## 3.8.6 演进与迁移策略

框架选择不是"一次定终身"，而是**随项目成熟度演进**的。有两条经过验证的路径：

### 路径一：原型用轻的，生产降抽象

呼应 Anthropic 的建议，一条健康的演进曲线是：

1. **验证期**：用低曲线框架（CrewAI / 厂商 SDK）或直接 API，最快跑出能证明价值的原型。
2. **成熟期**：当复杂度上升、需要精细控制和可观测时，要么升级到 LangGraph 这类可控框架，要么**反向减少抽象**、用基础组件重写关键路径。

关键心态：**框架是脚手架，不是地基。** 别把"当初选了哪个框架"当成不可变的架构承诺。

### 路径二：CrewAI → LangGraph（最常见的迁移）

业界最常见的迁移路径就是 **CrewAI → LangGraph**：团队先用 CrewAI 快速做原型、验证概念，然后**撞到 CrewAI 的控制流天花板**（需要循环、并行分支、审批关卡这些角色制 DSL 表达不了的东西），于是迁移。迁移方法是机械的：**把每个 CrewAI agent 映射成一个 LangGraph 节点，把顺序/层级 process 转成显式的图的边，把共享上下文搬进 LangGraph 的 state 对象。**

**这条迁移路径本身就是一条选型情报**：如果你**已经预见**到项目会需要复杂控制流，可以跳过 CrewAI 直接上 LangGraph，省一次迁移；如果前景不明，先用 CrewAI 快速验证也完全合理——**过早上 LangGraph 也是一种过度工程。**

## 3.8.7 常见误区与小结

### 常见误区

| 误区 | 为什么错 | 正确做法 |
|---|---|---|
| **"做 Agent 必须先选个框架"** | 简单 Agent 直接 API <200 行搞定，框架的抽象税可能得不偿失 | 默认从直接 API 起步，复杂度顶上来再引框架 |
| **"打勾最多的框架最好"** | 主流框架功能已趋同，功能清单区分不出它们 | 按你项目的**主约束**选，不按功能清单 |
| **"LangGraph 最强，直接上它"** | 它学习曲线 1-2 周，简单项目上它是过度工程 | 匹配项目在"控制力↔速度"光谱上的位置 |
| **"框架帮我做得越多越好"** | 抽象税双刃——顺境省事，逆境挡住底层真相 | 评估"出问题时能否看到底层并绕过" |
| **"选了框架就定终身"** | 框架是脚手架不是地基，项目会演进 | 原型用轻的，生产降抽象；预见复杂就直接上可控的 |
| **"这篇对比表能用很久"** | 框架是变化最快的领域，版本/功能几个月就过时 | 记方法论（主约束+取舍轴），别背功能清单 |

### 小结

- **先问该不该用框架**：Anthropic 建议从直接 API 起步——很多模式几行代码就够，框架的抽象税会掩盖 prompt、增加调试难度、诱导过度复杂。**简单 Agent 优先直接 API（<200 行）。**
- **八大框架看灵魂（编排抽象）**：LangGraph（图）、CrewAI（角色）、OpenAI SDK（handoff）、Claude SDK（tool 链+子 Agent）、Google ADK（agent 树+A2A）、Microsoft（GroupChat）、Pydantic AI（类型安全）、LlamaIndex（事件驱动/RAG）。
- **核心方法论：按主约束选，不按功能清单选**。诚实找出项目那个"不满足就白搭"的硬约束，映射到对应框架。
- **六根取舍轴**：控制力↔速度、抽象税、model-agnostic↔vendor-lock、状态/持久化、可观测、学习曲线/团队。这是评估任何框架（含未来新出的）的通用标尺。
- **演进而非终身**：原型用轻的、生产降抽象；CrewAI→LangGraph 是最常见迁移，预见复杂可直接上 LangGraph、前景不明先 CrewAI 验证。
- **一句话纪律**：**能不用框架就先别用，要用就按主约束选，选了也别当终身承诺。**
