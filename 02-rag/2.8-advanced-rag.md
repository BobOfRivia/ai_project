# 2.8 高级 RAG 架构

> **核心问题**：Self-RAG / CRAG / RAPTOR / GraphRAG / Adaptive RAG / Agentic RAG / Long RAG 各自的适用场景与权衡？
> **目标**：能识别这些架构的适用场景，知道工程代价。

---

## 一、先立一条铁律：别在检索不牢时上高级架构

2.2–2.7 走通了 Naive/Advanced RAG 主链路。本节的高级架构是**在主链路顶不住某类问题时的针对性升级**，不是"更高级所以更好"。

> **2026 年最贵、最常见的错误：在底层检索还不扎实时，就急着上 agentic 编排。**"一个在弱检索器上反复循环的 agent，只是花更多钱、更精致地给出错误答案。"

所以进本节前先确认：**混合检索 + 重排 + 好切片（2.2/2.4/2.6）已经压榨到位了吗？** 是，再往下看。

---

## 二、认知地图：三条主线 + 一个特例

七个架构不要死记，按**它们解决的问题**归成三类：

| 主线 | 解决什么问题 | 架构 |
|---|---|---|
| **① 反思纠错** | 检索到的东西不对 / 答案没证据支撑 | Self-RAG、CRAG |
| **② 结构化组织** | 单块检索答不了"跨文档关系"和"全局汇总" | RAPTOR、GraphRAG |
| **③ 路由调度** | 不同复杂度的问题该走不同管线 | Adaptive RAG、Agentic RAG |
| （特例） | 上下文够长，干脆少切多塞 | Long RAG |

---

## 三、主线一：反思纠错

核心思想：**给 RAG 加一个"自我检查"环节**，不盲目相信一次检索的结果。

### 3.1 CRAG（Corrective RAG，纠错式）—— 最实用

- **怎么做**：加一个**评估器模型**给检索结果打分。分数太低 → 不信内部库，**触发 web 搜索或改写查询**去别处找。
- **为什么最实用**：**换掉底座 LLM 依然有效**（评估器和主模型解耦），是**最贴近企业落地**的自纠错方案。
- **成熟度**：✅ **LangGraph 的 CRAG 教程是 2025 年被复现最多的 RAG 模式**，工程资料丰富，可放心上。
- **适合**：知识库覆盖不全、需要"库里没有就去外部找"的场景。

### 3.2 Self-RAG（自我反思）

- **怎么做**：模型在生成过程中自己发信号——**要不要检索、检索到的相不相关、自己的答案有没有被证据支撑**。
- ⚠️ **成熟度提醒**：**原版 Self-RAG 需要用特殊反思 token 微调模型**（偏研究、落地门槛高）。生产里的做法是**用现成 LLM 近似**：把每步包成显式评估 prompt（"这段相关吗？"），用**结构化输出**（JSON `is_relevant: true/false`）来 gate 流程。LlamaIndex 有 `SelfRAGQueryEngine` 干净地实现了这个模式。
- **适合**：对答案可靠性要求高、愿意多花推理成本换质量的场景。

> 一句话对比：**CRAG 更工程化、即插即用；Self-RAG 思想更完整但原版偏研究，生产用近似版**。多数团队先上 CRAG。

---

## 四、主线二：结构化组织

核心思想：**改变"知识的组织方式"，让检索能回答单块层面答不了的问题**（跨文档关系、全局汇总）。

### 4.1 RAPTOR（递归树摘要）

- **怎么做**：对文档块**递归地聚类 + 摘要**，建成一棵树——叶子是原始块，父节点是子节点的摘要，一路汇总到根。检索时可**多粒度**命中（要细节走叶子、要概览走上层摘要）。
- **收益**：QuALITY 基准 **+20 个百分点**（GPT-4）。
- **适合**：长文档、需要"既能答细节又能答全局概览"的场景。比 GraphRAG 轻，是"想要层次化能力但不想上知识图谱"的中间选择。

### 4.2 GraphRAG（知识图谱 RAG）—— 强大但要算清成本

- **怎么做**：不再只切块 embedding，而是**抽取实体和关系建成知识图谱**，再对图做社区摘要 / 图遍历检索。
- **收益（关系型/多跳问题）**：
  - 多跳任务准确率 **86%，而向量 RAG 只有 32%**。
  - 查询涉及实体 **超过 5 个时向量 RAG 准确率趋近 0，GraphRAG 在 10+ 实体下仍稳定**。
- ⚠️ **成本诚实**（这是关键决策点）：
  - **建索引比向量 RAG 贵 10–40 倍**：同一批语料向量库 <$5，GraphRAG 实体抽取 + 社区摘要要 $50–200；1 万文档的知识库，跑第一条查询前就是四位数的索引成本。
  - 好消息：成本在快速下降。**LightRAG**（扁平图、简化抽取）能以约 **1/100 的索引成本拿到 GraphRAG 70–90% 的质量**；微软还出了更省的 **LazyGraphRAG**。
- **什么时候值得**：**只有当答案依赖"实体/事件之间的关系"、而非单纯词面匹配时**。甜区大致是 **>1k 文档、<1M tokens**；再大 LightRAG 更划算。
- **落地现实**：**多数团队生产里仍跑向量 RAG，切到 GraphRAG 的都是有明确关系推理需求**。别默认升级。

## 五、主线三：路由调度

核心思想：**不是所有问题都走同一条管线**，按复杂度动态调度，平衡成本与质量。

### 5.1 Adaptive RAG（复杂度分级路由）

- 已在 **2.5 第三节**展开：小分类器把 query 分为直答 / 单跳 / 多跳，路由到不同成本的管线（延迟 -35%、成本 -28%、准确率 +8%）。
- 这里定位：它是**连接"简单管线"和下面"agentic 重管线"的调度层**——简单问题别浪费 agent。

### 5.2 Agentic RAG（Agent 驱动的检索）

- **怎么做**：让一个 Agent 来驱动检索——自己决定检索几次、怎么改写、要不要多源、够不够停。**五种生产模式**覆盖绝大多数系统：**Router（路由）、ReAct（推理+行动）、Plan-and-Execute（先规划后执行）、Multi-agent retrieval（多 Agent 检索）、Self-RAG（自反思）**。
- **成熟度**：✅ 已有真实生产部署（如 ServiceNow 的 P1 事件管理、Workday HR）。但——
- ⚠️ **重申铁律**：**别在检索不牢时上它**（见第一节）。Agentic 是"复杂多跳、多源、需要多轮推理"时的答案，不是默认。
- **与第 3、4 章的边界**：Agentic RAG 的"Agent 本体"（ReAct、Plan-Execute、多 Agent 协作、工具调用）属于**第 3 章 Agent** 和 **第 4 章工作流编排**；本节只讲"把 Agent 用在检索上"这个切面，机制细节交叉引用那两章。

---

## 六、特例：Long RAG（大块 + 长上下文）

- **怎么做**：上下文窗口够大时，**少切、切大块**，直接把大段内容塞给长上下文模型，减少切碎带来的语义割裂。
- **权衡**：省了精细切片的麻烦，但要注意 **lost-in-the-middle（2.7）** 和成本；本质是"用模型的长上下文能力换检索精细度"。
- **和 2.1 的呼应**：这是"RAG vs 长上下文"光谱上偏长上下文的一端，语料能塞下时可考虑。

---

## 七、选型决策：什么时候上哪个

按"先便宜后贵、先简单后复杂"的顺序判断：

| 症状 | 优先考虑 | 别急着上 |
|---|---|---|
| 主链路够用 | **不上高级架构** | 所有 |
| 库里常没有、要去外部找 | **CRAG** | GraphRAG |
| 要答案强可靠、可接受慢 | Self-RAG（近似版） | Agentic |
| 长文档要"细节+全局概览" | **RAPTOR** | GraphRAG |
| 答案依赖实体间关系/多跳、且预算够 | GraphRAG（或先试 LightRAG） | — |
| 问题复杂度差异大、要控成本 | **Adaptive 路由** | — |
| 复杂多跳/多源、需多轮推理 | Agentic RAG（且检索已牢） | — |
| 语料塞得下、想省切片 | Long RAG | — |

**贯穿原则**：**每上一层都要能说出它修复的具体问题和愿付的代价**（成本/延迟/复杂度）——这正是我们知识整理的"成熟度分级、权重匹配"原则在架构选型上的应用。

---

## 八、本节小结

1. **铁律**：检索不牢别上高级架构，尤其别过早上 agentic。
2. **三条主线**：反思纠错（CRAG/Self-RAG）、结构化组织（RAPTOR/GraphRAG）、路由调度（Adaptive/Agentic）。
3. **最实用起点**：**CRAG**（LangGraph 教程成熟、换底座仍有效）。
4. **GraphRAG 算清成本**：贵 10–40 倍，只为关系/多跳问题；先评估 LightRAG。
5. **RAPTOR 是轻量层次化选择**：想要全局概览但不想上图谱。
6. **Agentic 是终点不是起点**：有真实部署，但门槛是"检索已牢 + 问题确实复杂"。
7. **Agent 机制细节**看第 3/4 章，本节只讲"用在检索上"的切面。
