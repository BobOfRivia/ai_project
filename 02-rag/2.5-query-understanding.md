# 2.5 查询理解与改写

> **核心问题**：怎么让用户的问题更适合被检索？
> **目标**：能针对"用户提问与文档表达不一致"的核心痛点做工程化优化。

---

## 一、要解决的根本矛盾：查询与文档的"表达鸿沟"

2.4 把检索层搭好了，但有个前提问题：**用户怎么问，和文档怎么写，往往对不上**。检索器再强，喂给它一个糟糕的 query，也召回不了好结果——**Garbage in, garbage out**。

查询侧的典型病症：

| 病症 | 例子 | 后果 |
|---|---|---|
| **措辞不一致** | 用户："电脑连不上网"；文档："网络适配器故障排查" | 语义相关但词面差远，稀疏检索失效 |
| **过于简略** | 用户："退款"（两个字） | 信息太少，向量太宽泛，命中一大堆无关 |
| **口语/有指代** | "那个上次说的功能怎么开"（多轮对话里） | 缺上下文，检索不到 |
| **一句多意图** | "对比 A 和 B 的价格和售后" | 单次检索覆盖不全，多跳需求 |
| **多知识源** | 有的答案在 FAQ、有的在数据库、有的在最新公告 | 单一检索源不够，需路由 |

本节就是一组**在检索之前对 query 动手术**的技术，让它更适合被检索。

---

## 二、查询改写技术家族

按"从轻到重"排列，每个都对应一类具体病症：

### 2.1 Query Rewriting（改写）

把口语化、有指代、不规范的 query 改写成规范、自包含的检索式。

- **典型场景**：多轮对话里把"那它多少钱"改写成"iPhone 15 Pro 的价格"（消解指代）；把口语改成书面术语。
- **最基础也最常用**，多轮对话 RAG 几乎必配。

### 2.2 Query Expansion（扩展）

给原 query 补上同义词、相关词、上位词，扩大召回面。

- **典型场景**："笔记本发热" → 补充"过热 / 温度高 / 散热"。
- 补稀疏检索"换个说法就丢"的短板。

### 2.3 Multi-Query（多查询）

用 LLM 把一个 query 扩成**多个不同角度的 query**，各自检索，再合并去重。

- **典型场景**：一个模糊问题从多个措辞/子角度并行捞，提升召回率。
- ⚠️ 落地要点：**保持小、确定、可审计**——生成 3–6 个改写就够，别无限扩，否则延迟和噪声都上来。

### 2.4 HyDE（Hypothetical Document Embeddings，假设文档嵌入）

反直觉但有效：**先让 LLM 编一个"假设答案"，拿这个假答案去做 embedding 检索**（而不是用原始问题）。

- **为什么有效**：假答案在措辞和篇幅上更像**目标文档段落**，比一个短问题更容易向量命中。
- **什么时候用**：语料是**密集、描述性的长文**（技术文档、长指南）时效果好；短问题 vs 段落级内容的鸿沟正好被它填上。
- ⚠️ 代价：多一次 LLM 调用；假答案可能带偏，需评估。

### 2.5 Step-back Prompting（后退提问）

让模型先"退一步"，把具体问题抽象成更一般的原理性问题，先检索背景原理，再回答具体问题。

- **典型场景**：具体问题"为什么这段代码在 X 情况下报错" → 后退到"X 机制的工作原理"，先捞原理再答。
- 适合需要背景知识支撑的推理型问题。

### 2.6 Query Decomposition（查询分解）

把一句多意图/多跳问题拆成若干子问题，分别检索再综合。

- **典型场景**："对比 A 和 B 的价格和售后" → 拆成 4 个子查询分别检索。
- 是**多跳推理**的基础，和 2.8 的 Agentic RAG 衔接。

## 三、Routing：把 query 送对地方

改写解决"怎么问"，路由解决"去哪问 / 用多重的管线"。

### 3.1 多源路由（Source Routing）

有多个知识源时，先判断该查哪个：FAQ 库？产品文档？结构化数据库（走 Text-to-SQL）？实时接口？

- 由 LLM 或小分类器判断意图 → 路由到对应检索源。
- 呼应 2.4：**结构化精确查询路由到 SQL，非结构化路由到向量检索**。

### 3.2 Adaptive RAG：按查询复杂度分级（2026 主流最佳实践）

不是所有问题都值得走重管线。用一个**小分类器**判断 query 复杂度，路由到不同成本的管线：

| 复杂度 | 例子 | 走的管线 |
|---|---|---|
| **简单事实（single-step）** | "公司成立于哪年" | 直答 / 单次检索 |
| **推理（multi-hop）** | "A 和 B 的政策差异导致了什么结果" | 分解 + 多跳检索 |
| **汇总（summarization）** | "整个语料对 X 的总体看法" | 语料级聚合（可能走 GraphRAG，2.8） |

**为什么值得做（硬数据）**：

- 复杂度分类用小模型（Phi-2、Llama-2-7B）即可达 **85–92% 准确率，每 query <1ms、成本极低**。
- 生产实测：**P50 延迟 -35%、LLM 成本 -28%、准确率 +8%**。

**落地要点**：用**非对称阈值**——单跳需 85%+ 置信才走快管线，模糊区间（50–85%）**默认走多跳**（保守，宁可多花也别答错）。

> **查询侧 vs 架构侧的分工**：本节讲的是"如何在查询入口分类并路由"；分类之后各条管线（Self-RAG 反思、多跳迭代、GraphRAG 聚合）的**完整架构**在 2.8 展开。

## 四、一条贯穿本节的关键原则：别为了技术而堆技术

2026 年的共识警示：

> **查询变换只在它修复某个具体的检索失败时才值得加。在你能说出"它修复的是哪个失败模式、你愿意为它付多少延迟"之前，不要再加一层变换。**

每加一步改写/路由都意味着**多一次 LLM 调用（延迟+成本）**和**多一个出错点**。正确姿势：

1. 先用 2.10 的评估找出**具体失败的 query 类型**。
2. 对症下药选**一种**变换（措辞不一致→改写/扩展；短问题配长文档→HyDE；多意图→分解）。
3. 上线后再测，确认真的修复了、且延迟可接受，再考虑下一层。

---

## 五、落地建议：一个小而稳的查询处理层

- **多轮对话**：必配 Query Rewriting（消解指代、补全上下文）。
- **召回不足**：Multi-Query（3–6 个改写）或 Query Expansion。
- **短问题 + 描述性长语料**：评估 HyDE。
- **多意图 / 多跳**：Query Decomposition。
- **多知识源 / 成本敏感**：Routing + Adaptive 复杂度分级。
- **整体**：让这一层**小、确定、可审计**——能日志追踪每次改写成了什么，方便排查。

---

## 六、本节小结：查询处理决策清单

1. **先诊断再动手**：用评估定位具体失败的 query 类型，别盲目堆变换。
2. **多轮必配改写**：指代消解 + 上下文补全。
3. **对症选一种**：措辞→扩展/改写；短问长文→HyDE；多意图→分解；召回低→Multi-Query。
4. **上 Adaptive 路由**（性价比高）：小分类器分级，快问走快管线，难问走多跳（延迟 -35%、成本 -28%）。
5. **多源路由**：结构化→SQL，非结构化→向量检索。
6. **守住可审计**：每步改写可日志追踪，别做成黑箱。

> 记住：这一节是"锦上添花"层——**地基是 2.4 的混合检索**。先把检索本身做好，再用查询变换补最后那几个百分点。
