# 3.12 · 深挖：Harness 的沙箱（Sandbox）

> **本文定位**：这是 3.12「Agent 工程化」第一支柱 **Harness** 的一篇**组件深挖**。在 [3.12（一）Harness 工程](./3.12-harness-engineering.md) 里，沙箱只是 harness 8 大组成中的第 3 块，一句话带过。但沙箱是那 8 块里**最"基础设施"、也最容易被应用工程师当成黑盒**的一块——一旦你的 agent 要执行代码、跑 shell、装依赖、访问文件系统，沙箱就从"可选项"变成"地基"。这篇把它讲透。
>
> **核心问题**：沙箱到底是什么？它解决的到底是**哪个问题**（很多人以为只是"安全"，其实更重要的是"让 agent 敢自主")？隔离有几种做法、强弱差在哪、代价是什么？以及最落地的一问——**2026 年市面上的沙箱产品长什么形态，我该给我的 agent 配哪一种？**
>
> **边界**：**沙箱作为"四层护栏之一"、在跨工具攻击链里挡哪一环**（致命三要素的"对外通道"闸门、软护栏 vs 硬护栏），属于 [3.11 Agent 安全与护栏](./3.11-agent-security.md)，本文不重复那套威胁模型，只在必要处引用。本文讲的是**沙箱这个基础设施本身**：它是什么技术、隔离怎么实现、产品里什么形态、怎么选。通用容器/K8s 运维、部署工程属于 **ch7**，本文只讲"为 agent 跑代码"这个特定用途。

## 一、什么是沙箱——先把定义钉死

一句话定义：

> **沙箱（sandbox）= 一个被刻意"削弱了能力"的执行环境。你在里面跑代码/命令，但它默认碰不到沙箱外的东西——文件、网络、别的进程、宿主机——除非你显式地开一道口子。**

注意这个定义的**两个关键词**，它们决定了沙箱和"普通运行环境"的根本区别：

1. **默认拒绝（deny by default）**：沙箱里的进程一出生就是"什么都不能碰"，然后你**按需授予**（allowlist 白名单）几项能力——能读这个目录、能连这几个域名。这和普通进程"默认拥有当前用户的全部权限"正好相反。这套思路有个正式名字叫 **capability-based security（基于能力的安全）**：不问"你是谁、可不可信"，只问"这个环境被授予了哪几项能力"。

2. **限制环境，而不是检测行为**：沙箱不去"判断这段代码是好是坏"——它根本不判断，它只是**把危险动作物理上变得不可能**。想删 `/etc`？目录根本不在沙箱里，删不到。想把 SSH 私钥发到外网？外网连不上,发不出去。**这是一道确定性的硬边界，不是一个概率性的检测器。** （对应 3.11 的"硬护栏 vs 软护栏"：沙箱是最典型的**硬护栏**——不靠模型自觉,靠环境兜底。）

一个好用的比喻：**沙箱之于 agent，就像小孩玩的那个"沙坑（sandbox）"**——名字就是这么来的。沙坑四周有围栏,小孩在里面怎么折腾、把沙子扬得多高都行,**乱只会乱在坑里,弄不脏坑外的客厅**。你不需要盯着小孩的每一个动作(审批),你只需要确保围栏够高。

## 二、沙箱解决什么问题——不止是"安全"

大多数人对沙箱的第一反应是"防黑客/防注入"。这没错,但**把沙箱仅仅理解成安全工具,会错过它一半的价值**。沙箱真正解决的是**三个递进的问题**,安全只是第一个。

### 问题一(基础层):执行不可信代码而不炸掉真实世界

Agent 时代出现了一类前所未有的东西:**LLM 生成的、你没读过就要运行的代码**。

- 一个 data-analysis agent 现写一段 Python 去处理你的 CSV;
- 一个 coding agent 现写一段 shell 去装依赖、跑测试;
- 一个被**间接提示注入**(3.11)攻陷的 agent,可能现写一段 `curl` 把你的 `~/.ssh/id_rsa` 发出去。

这段代码**可能有 bug(误删文件)、可能被攻击者操纵(恶意 exfiltration)**。你不可能逐行审。沙箱给的保证是:**无论这段代码写了什么,它的破坏半径被锁死在沙箱内**——这就是安全领域说的 **blast radius(爆炸半径)**。删文件只删得到沙箱里的临时文件;连外网只连得到白名单上的域名。**代码可以任性,后果被围栏兜住。**

### 问题二(关键层):打破"审批疲劳 vs YOLO"的两难——沙箱是自主性的前提 ⭐

这才是沙箱**最被低估的价值**,也是 Anthropic 给 Claude Code 上沙箱时的**头号动机**。

没有沙箱时,一个能执行动作的 agent 只有两个选择,都很糟:

| 模式 | 做法 | 问题 |
|---|---|---|
| **逐动作审批** | 每跑一条命令都弹窗问你 "允许吗?" | Anthropic 原话:"不停点 approve 拖慢开发,还导致**审批疲劳(approval fatigue)**——用户懒得细看就点允许,反而更不安全" |
| **YOLO 模式** | 一把梭 `--dangerously-skip-permissions`,全放开 | 快,但等于把裸机交给一个可能被注入的 agent,一旦出事就是宿主机级灾难 |

**两难的本质是:安全和自主在"没有围栏"时是对立的**——要安全就得频繁打断(不自主),要自主就得放弃防线(不安全)。

**沙箱把这个对立解开了**。Anthropic 的说法很精准:

> "Sandboxing creates pre-defined boundaries within which Claude can work more freely, instead of asking for permission for each action."
> (沙箱**划定一个预设边界,让 Claude 在边界内更自由地干活**,而不是每个动作都来问你。)

有了围栏,**边界内的动作不用再逐个审批**——反正跑不出围栏。人的注意力只需要花在"要不要开一道新口子"(比如访问一个白名单外的新域名)这种**真正越界**的少数时刻。结果 Anthropic 实测:

> **沙箱安全地把权限弹窗减少了 84%。**

这句话要反过来读才懂它的分量:**沙箱不是"限制 agent",而是"解放 agent"**。它是让 agent 敢于**长时间自主运行**的前提。没有围栏,你不敢让 agent 自己跑 20 分钟;有了围栏,你才敢。**沙箱是 harness 里"自主性"这个属性的物理地基**——这一点直接呼应 [3.12 Harness 工程](./3.12-harness-engineering.md) 讲的"durable execution 让 agent 能长跑",沙箱则让这种长跑**安全到你敢放手**。

### 问题三(纵深层):即使被攻陷,也止损——凭证隔离与嵌套围栏

最后一层,是"假设围栏也会被突破"的纵深防御。Claude Code 的做法给了一个漂亮的样板:

> **敏感凭证(git 凭证、签名密钥)从来不放进沙箱里。** 它们待在沙箱外的代理(proxy)进程里。这样**即使沙箱内的代码被完全攻陷,攻击者也拿不到凭证**,伤害到沙箱边界为止。

这是**嵌套围栏(nested sandbox / 分层 blast radius)**的思想:agent 干活的沙箱是第一层围栏;凭证被隔在第二层围栏(沙箱外)。攻破第一层,还有第二层。**"假设会被攻破,那就让攻破的代价尽量小"**——这是成熟 harness 的标志。

> **小结这一节**:沙箱解决的三个问题层层递进——① 让不可信代码的破坏半径可控(安全底座)、② 解开"审批疲劳 vs YOLO"的死结让 agent 敢自主(**最被低估、最重要**)、③ 即使被攻陷也能靠嵌套围栏止损(纵深)。**记住第二点:沙箱首先是"自主性的赋能器",其次才是"安全工具"。**

## 三、隔离的两个维度:文件系统 + 网络(为什么缺一不可)

"把环境削弱"具体削哪些?Anthropic 给出的答案是**两个维度,而且强调两个必须同时上**。这是理解一切沙箱产品的骨架。

### 维度一:文件系统隔离

**控制"沙箱里的进程能读写哪些目录"**。典型策略:只允许读写当前工作目录(cwd),cwd 之外一律不可改。

- 实现:靠 OS 级原语。**Linux 用 [bubblewrap](https://github.com/containers/bubblewrap)(bwrap,Flatpak 用的那个无特权沙箱工具),macOS 用 seatbelt(苹果 App Store 应用的隔离框架)**。
- 挡住什么:被注入的 agent **改不了系统敏感文件**(比如往 `~/.bashrc` 里塞后门、改 `/etc/hosts`)。

### 维度二:网络隔离

**控制"沙箱里的进程能连哪些网络目的地"**。典型策略:默认断网,所有出站流量走一个**沙箱外的代理(proxy)**,代理按**域名白名单**放行,遇到白名单外的新域名时才弹窗问人。

- 实现:沙箱内进程只能通过一个 **unix domain socket** 连到外面的代理,代理决定放不放行。
- 挡住什么:被注入的 agent **发不出数据**(SSH 私钥、`.env` 里的密钥),也**下载不了恶意载荷**。

### 为什么两个必须一起上——defense in depth

Anthropic 特意强调,**只上一个维度等于没上**,因为两者会互相"补锅":

> - **只有文件系统隔离、没有网络隔离**:被攻陷的 agent 读不到系统文件,但能**把它当前目录里的敏感数据(SSH key、token)直接发到外网**——照样泄露。
> - **只有网络隔离、没有文件系统隔离**:被攻陷的 agent 发不出数据,但能**改写文件系统里的脚本/配置来逃逸沙箱**,一旦逃逸就重新拿到网络——围栏形同虚设。

> **"It's by using both techniques that we can provide a safer and faster agentic experience."** 两个维度是一对咬合的齿轮,拆开任何一个,另一个的防线都会被绕过。**这就是纵深防御(defense in depth)在沙箱上的具体形态。**

这个"文件 + 网络双隔离"的心智模型,是你评估任何一款沙箱产品的**第一把尺子**:先问它这两个维度各怎么做、能不能都做。

## 四、隔离技术光谱:从进程沙箱到 microVM

上面讲的是"隔离哪些维度",这一节讲"用什么技术实现隔离"。这里有一条从**弱而快**到**强而重**的光谱,理解它你才看得懂各家产品在拿什么做底座。**核心权衡只有一句话:隔离强度 ↔ 启动速度/开销,鱼与熊掌。**

| 技术 | 怎么隔离 | 隔离强度 | 冷启动 | 开销 | 适合 |
|---|---|---|---|---|---|
| **OS 进程沙箱**<br>(bubblewrap/seatbelt) | 用 OS 原语(namespace/seccomp)给**单个进程**套限制,**不起容器/VM** | 中(仍共享宿主内核) | 近乎零 | 极低 | **在你本机上跑的 agent**(Claude Code 式) |
| **容器**<br>(Docker/namespaces) | Linux namespace + cgroups 做进程隔离,**多容器共享宿主内核** | **弱**(内核漏洞→逃逸,内核每年 300+ CVE) | ~50ms | ~10MB | 只跑**可信代码**的内部工具 |
| **gVisor**<br>(用户态内核) | Go 写的 Sentry **在用户态拦截 syscall**,不让代码直接碰宿主内核 | 中(大幅缩小内核攻击面) | +20~50% 开销 | ~30MB | 多租户 SaaS 跑**半可信**代码(Modal 选它) |
| **Firecracker**<br>(microVM) | KVM + VT-x 硬件虚拟化,**每个沙箱自带独立内核** | **最强**(逃了 guest 也碰不到 host) | ~125–150ms | ~5MB/VM,3MB 二进制 | 跑**不可信 LLM 代码**、合规场景(AWS Lambda 用它) |
| **WebAssembly**<br>(WASM/WASI) | 运行时级能力隔离,**无环境权限**,每项能力显式授予 | 强(但作用域窄) | **微秒级** | 几 MB | 无状态边缘函数;**无持久文件系统**,不适合重 OS 集成 |

读这张表的三个要点:

1. **"容器不是沙箱"**——这是 2026 年安全圈反复强调的一句话。Docker 容器**共享宿主内核**,一个内核提权漏洞就能逃逸并波及同宿主的所有租户。跑**你自己写的可信代码**没问题;跑 **LLM 生成的、可能被对抗性构造的代码**,容器的隔离是不够的。这类场景业界共识是上 **microVM(Firecracker/Kata)**。

2. **microVM 打破了"VM 很慢"的旧印象**。传统虚拟机启动要 1–2 秒,而 **Firecracker 把一个自带内核的轻量 VM 压到 ~125ms 启动、~5MB 内存**——这才让"每个 agent 会话一个独立 VM 级沙箱"在经济上可行。它是很多云沙箱产品的底座。

3. **gVisor 是"折中派"**:不用起完整 VM,靠用户态内核拦 syscall 把攻击面砍小,代价是 I/O 密集负载有 20~50% 性能损耗。安全强于容器、弱于 microVM,启动和开销介于两者之间。

> **一句话选型直觉**:可信代码 → 容器够用;半可信 / 重性能 → gVisor;**不可信的 LLM 生成代码 / 合规硬要求 → microVM(Firecracker)**;你本机上的 agent → OS 进程沙箱最轻。

## 五、沙箱在产品里长什么形态

理论落到产品,2026 年沙箱主要以**三种形态**出现。搞清楚你的 agent 属于哪种场景,直接决定你配哪一类。

### 形态 A:本地 OS 进程沙箱——给"在你机器上跑的 agent"

**代表:Claude Code 的 sandboxing(2025 年 10 月上线)、开源的 [anthropic-experimental/sandbox-runtime](https://github.com/anthropic-experimental/sandbox-runtime)**。

- **形态**:不起容器、不起 VM,直接用 bubblewrap/seatbelt 给 agent 进程本身套一层 OS 级限制(文件 + 网络双隔离)。轻到几乎零开销,启动即用。
- **适合**:运行在**开发者本机 / CI**上的 coding agent、本地 assistant。你不想为跑个 agent 起一整套云基础设施,只想给它一副"手铐",让它在你的项目目录里自由跑、但碰不到目录外和乱连网。
- **要点**:Anthropic 已**开源**这套 runtime 并公开呼吁其他 agent 采用——它是"给本地 agent 加围栏"目前最现成的方案。凭证隔离(git 凭证不进沙箱)也在这套里。

### 形态 B:云端代码执行沙箱 as a Service——给"跑 LLM 生成代码"的 agent

**代表:E2B / Daytona / Modal / Cloudflare Sandbox / Vercel Sandbox**。

这是增长最猛的一类:你的 agent(可能跑在你自己的服务器上)需要**执行一段现生成的代码**,于是通过 SDK 调一个**云上的隔离沙箱**,把代码丢进去跑、拿回结果。底座多为 microVM 或 gVisor。选型时看四个维度:**冷启动、持久化、隔离底座、语言/GPU**。

| 产品 | 隔离底座 | 冷启动 | 持久化 | 一句话最擅长 |
|---|---|---|---|---|
| **E2B** | VM 级(为 agent 而生) | ~150ms;暂停恢复 ~1s | 有状态,pause/resume + 快照 | **agent 专用**,SDK 顺手,与 LangChain/CrewAI/OpenAI Agents SDK 集成成熟 |
| **Daytona** | 每沙箱独立内核/文件/网络栈 | **~90ms(最快)** | **默认持久**(像 Codespace,删才没) | 冷启动最快 + 持久工作区 + 语言最全 + 支持 GPU/虚拟桌面(computer-use) |
| **Modal** | gVisor | 亚秒级调度 | 分布式卷 + 文件快照(30 天) | **GPU 负载 / RL 训练 / 超大规模批量**(记录过 10 万+ 并发沙箱) |
| **Cloudflare Sandbox** | Workers + Durable Objects + 容器 | 未公开 | 默认临时,需挂 R2 桶持久 | 已用 Cloudflare 栈的团队;**凭证代理式安全模型**对 agent 友好 |
| **Vercel Sandbox** | Firecracker microVM | **毫秒级** | 默认持久,自动快照 | AI 编码 agent / "vibe coding" 平台;与 Vercel 部署栈贴合 |

选型直觉:

- **要"给 agent 一个能长期住进去的工作区"**(多轮、跨会话攒状态)→ **Daytona / E2B**(默认持久);
- **要跑 GPU / 大规模并发** → **Modal**;
- **本来就在某个云栈里**(Cloudflare / Vercel)→ 用它家的,省集成;
- **要 agent 框架开箱集成 + 成熟生态** → **E2B**(它"从第一行代码就是为 agent 造的",这是它的定位)。

### 形态 C:框架内建的轻量沙箱——给"只想安全跑段 Python"的场景

有时你不需要一整个云沙箱,只想让 agent **安全地算一算、跑一小段纯计算 Python**。这时框架内建的轻沙箱就够:

- **Pydantic AI 的 code mode** 用 **[Monty](https://github.com/pydantic/pydantic-ai)** 做沙箱化 Python 执行(见 3.12 harness 工程);
- **WASM 系**(Pyodide 等)把 Python 编进 WebAssembly,靠 WASM"无环境权限"的特性做隔离,微秒级启动——**代价是没有持久文件系统、装不了任意原生依赖**,只适合纯计算 / 无状态场景。

这类是"够用就好"的选择:**计算密集但不需要真实文件系统和网络**时最轻便;一旦 agent 需要装依赖、读写文件、跑真实工具链,还是得回到形态 A/B。

> **三形态一句话对照**:**A(本地 OS 进程沙箱)**给本机 agent 套手铐;**B(云沙箱服务)**给"跑 LLM 代码"的 agent 一个隔离的远程执行环境;**C(框架轻沙箱)**给"只想安全算一段"的场景最轻的围栏。**先定位你的 agent 在哪个场景,再看表选具体产品。**

## 六、落地清单与常见误区

**给你的 agent 配沙箱,按这几条走:**

1. **先定位场景**(本机 / 云执行 / 轻计算)→ 对应形态 A / B / C。别拿云沙箱去解本地 agent 的问题,反之亦然。
2. **文件 + 网络双隔离都要**(第三节那把尺子)。只做一个等于没做。
3. **跑不可信的 LLM 生成代码 → 别用裸容器,上 microVM/gVisor**(第四节)。合规场景(HIPAA/PCI)几乎必须 microVM。
4. **敏感凭证隔在沙箱外**(纵深防御)。git 凭证、API key、签名密钥——永远不进沙箱。
5. **网络走白名单代理,不是全开或全关**。默认断网 + 白名单放行 + 新域名弹窗,是审批疲劳和 YOLO 之间的正解。
6. **把沙箱当"自主性开关"用**:配好围栏后,**放宽边界内的逐动作审批**,让 agent 长跑——这才吃到沙箱 84% 那个红利。配了沙箱却还逐条审批,等于白配。

**常见误区表:**

| 误区 | 纠正 |
|---|---|
| "容器 = 沙箱,Docker 起一个就安全了" | 容器**共享宿主内核**,跑不可信代码隔离不足;要 microVM/gVisor |
| "沙箱主要是为了安全" | 更大的价值是**解开审批疲劳 vs YOLO 的死结、让 agent 敢自主**(问题二) |
| "只做网络隔离/只做文件隔离就行" | 两者会互相补锅,**必须同时上**,否则都能被绕过 |
| "沙箱能识别恶意代码" | 沙箱**不判断好坏**,它只是**让危险动作物理上做不到**(硬护栏,不是检测器) |
| "配了沙箱就万无一失" | 沙箱是**一层**,不是全部。它和 3.11 的模型层/检测层/人机层护栏是**纵深关系**,假设它会被突破、再叠凭证隔离 |
| "沙箱只挡外部攻击" | 它同样挡**自己 agent 的 bug**(误删、跑飞的脚本)——blast radius 对 bug 和攻击一视同仁 |

---

## 七、小结

- **沙箱 = 一个"默认拒绝、按需授权"的削弱版执行环境**。两个内核词:**capability-based(默认无权限,显式授予)** 和 **限制环境而非检测行为(硬护栏)**。
- **它解决三个递进的问题**:① 让不可信代码的**破坏半径(blast radius)**可控;② **解开"审批疲劳 vs YOLO"的死结,让 agent 敢自主**(最被低估、Anthropic 头号动机,实测减少 84% 弹窗);③ 即使被攻陷,靠**凭证隔离/嵌套围栏**止损。**记住第二点:沙箱首先是自主性的赋能器。**
- **隔离两维度,缺一不可**:**文件系统**(bubblewrap/seatbelt,限制读写目录)+ **网络**(白名单代理,限制出站)。只做一个会被另一个维度绕过——**defense in depth**。
- **隔离技术光谱**(隔离强度 ↔ 速度的权衡):OS 进程沙箱 < 容器(弱,共享内核) < gVisor(用户态内核) < **Firecracker microVM(最强,自带内核)**;WASM 微秒启动但无持久 FS。**跑不可信 LLM 代码 → microVM。**
- **产品三形态**:**A 本地 OS 进程沙箱**(Claude Code / sandbox-runtime,给本机 agent)、**B 云代码执行沙箱**(E2B/Daytona/Modal/Cloudflare/Vercel,给跑 LLM 代码的 agent,按冷启动/持久化/GPU 选)、**C 框架轻沙箱**(Pydantic Monty / WASM,给纯计算)。**先定位场景,再选产品。**
