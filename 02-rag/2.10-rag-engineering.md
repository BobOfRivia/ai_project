# 2.10 RAG 工程化

> **核心问题**：怎么让 RAG 系统在生产环境稳定运行、可观测、可演进？
> **目标**：增量更新、数据版本、监控、A/B、Latency / Cost 优化。

---

## 一、从 Demo 到 24/7 生产的鸿沟

前面九节能搭出一个"能跑"的 RAG。但**能 demo ≠ 能上线**。生产要多解决四件事：**数据会变（增量更新）、质量会漂（评估+监控）、要能改进（A/B）、会被攻击（安全）**。本节就是这四件事。

> **边界说明**（承接我们的归属原则）：
> - **通用评估体系**（LLM-as-Judge、Tracing 平台、回归测试）→ **第 6 章**；本节只讲 **RAG 特有指标**。
> - **通用推理优化/成本控制**（vLLM、模型路由、缓存机制）→ **第 7 章**；本节只讲 **RAG 特有的延迟/成本点**。
> - **通用注入/越狱防护** → **第 8 章**；本节只给 **RAG 特有的数据投毒**一个锚点。

---

## 二、增量更新与索引维护

文档会新增、修改、删除。**每次全量重建索引又慢又贵**，生产必须增量。

### 2.1 增量而非全量

- **Delta 更新 / CDC（变更数据捕获）**：只对**变化的文档**重新切片、重新 embedding，用事件触发，别全量重跑。
- **Streaming reindexing（流式重索引）**：文档一变就重嵌那一篇——比"每晚全量重建"新鲜得多，且 embedding API 成本只是零头。

### 2.2 最隐蔽的坑：部分更新（Partial Update）

> **RAG 最被低估的失败模式**：重索引跑到一半崩了，索引里留下**版本不一致**的文档——有的新有的旧，检索出来自相矛盾。

**安全模式：别名原子切换（Alias-based deployment）**

```
1. 完整构建一个新索引（不动线上）
2. 在新索引上跑评估验证
3. 验证通过 → 原子地把 alias 从旧索引切到新索引
4. 出问题 → 一键切回旧索引（回滚）
```

这样线上永远指向一个**完整、验证过**的索引，杜绝"半新半旧"。

### 2.3 数据版本管理

- 给每个索引/语料打**版本号（corpus version）**。
- **把 corpus version 写进 trace 属性**——这样监控发现质量下降时，能直接关联到"是哪次重索引引起的"（见第四节）。

## 三、RAG 特有评估（通用 Eval → 第 6 章）

不评估就没法改进。RAG 评估的精髓是**把"检索失败"和"生成失败"分开测**（呼应全章"73% 失败在检索"）。

### 3.1 检索质量指标

| 指标 | 测什么 |
|---|---|
| **Recall@K** | 相关文档有没有被召回进 top-K（召回率，最关键） |
| **MRR** | 第一个相关结果排多靠前（平均倒数排名） |
| **NDCG@K** | 综合考虑相关性和排序位置（重排效果常看它） |

### 3.2 生成质量指标：RAGAS 四件套

**RAGAS 是四指标模式的开源标准参考**，正好拆开检索与生成：

| 指标 | 测什么 | 归属 |
|---|---|---|
| **Context Precision** | 召回的上下文里有多少是相关的（噪声多不多） | 检索 |
| **Context Recall** | 回答所需的信息有没有被全部召回 | 检索 |
| **Faithfulness** | 答案是否**真由上下文支撑**（防幻觉，防 2.7 的 post-hoc 贴证据） | 生成 |
| **Answer Relevance** | 答案切不切题 | 生成 |

> **哪两个掉了，就知道该修哪一半**：Context 指标差 → 修检索（2.2–2.6）；Faithfulness 差 → 修 prompt/grounding（2.7）。

### 3.3 离线 + 在线

- **离线**：准备 **100–500 条人工核对的 golden 测试集**，用 RAGAS/DeepEval 跑自动指标。改任何东西（模型、chunk size、prompt）前后都用它比。
- **在线**：用 Phoenix 等做**生产 tracing + 实时监控**，看真实流量表现。

---

## 四、监控与漂移检测

上线不是终点。RAG 会随时间**漂移**：

- **监控 embedding 向量分布 和 检索分数分布**——分布变了，说明数据或用户行为变了。
- **关联 corpus version**：把语料版本写进 trace，就能把"分数骤降"和"某次重索引"对上号，快速定位。
- **业务指标**：检索命中率、无答案率、引用覆盖率、用户反馈（点赞/点踩）。

> 核心：**RAG 的质量是会衰减的**（新文档进来、用户问法变化），监控是为了在用户投诉前发现衰减。

---

## 五、A/B 实验

RAG 有太多可调项，靠"感觉"改不行，要用 A/B 对照真实流量验证：

- **常见变量**：embedding 模型、chunk size/overlap、检索策略、Top-K、reranker、prompt 模板。
- **纪律**：一次只改一个变量，用第三节的指标 + 业务指标判定，别凭直觉上线。

## 六、RAG 特有的 Latency / Cost（通用优化 → 第 7 章）

### 6.1 成本量级参考

不同复杂度每 query 成本差两个数量级（回看 2.1 帮决策）：

| 形态 | 每 query 成本 |
|---|---|
| Naive RAG | ~$0.001 |
| Hybrid + Rerank | ~$0.005 |
| Agentic RAG | $0.02–0.10 |

按 10 万 query/月估算，$100 到 $1 万不等——**架构复杂度直接换算成账单**，这也是 2.8"别过早上 agentic"的成本理由。

### 6.2 RAG 特有的省钱/提速点

- **缓存高频 query 的检索结果**（同问不必重检）。
- **Embedding 复用**：增量更新只嵌变化文档（第二节）。
- **分级路由**：简单问走便宜管线（Adaptive RAG，2.5）。
- **精排控候选量**：候选越少，重排和生成的 token 越省（2.6/2.7）。
- （通用的 KV cache / prompt caching / 模型路由等 → 第 7 章）

## 七、RAG 特有安全：数据投毒（纵深防护 → 第 8 章）

RAG 引入了一个裸 LLM 没有的攻击面：**检索内容被默认信任**。

### 7.1 攻击：PoisonedRAG / 语料投毒

> **PoisonedRAG（2024）**：在**百万级语料里只塞 5 篇**精心构造的恶意文档，针对特定触发问题，就能让 AI **90% 概率输出攻击者想要的错误答案**。

机制：**语料投毒 = 间接注入的投递手段**——把优化过的少量恶意文档注入大语料，就能高概率把恶意内容顶进检索 top 结果，再由检索喂给模型执行其中的指令。

### 7.2 这不是学术空谈

- **Slack AI（2024.8）**：间接注入 + RAG 式检索，嵌入的指令诱导 AI 构造钓鱼链接、泄露用户对话上下文里的数据。
- ChatGPT 记忆投毒等事件也已发生。

### 7.3 防御方向

- **防线在摄取（ingestion），不在输出**：入库前对来源做信任分级、清洗、过滤不可信内容——尤其是**来自互联网/用户上传**的语料。
- **权限过滤在检索层强制**（回看 2.3/2.4）。
- **输出侧仍需 grounding + faithfulness 校验**（2.7/本节评估）作为兜底。
- 纵深的注入防护、内容审查、红队 → **第 8 章**。

## 八、生产上线清单

一套 RAG 上生产前，逐条自查：

1. **增量更新**：CDC/delta，别全量重建。
2. **原子切换**：alias 部署，杜绝半新半旧索引。
3. **数据版本**：corpus version 打进 trace。
4. **离线评估**：100–500 golden 集 + RAGAS 四件套，检索/生成分开测。
5. **在线监控**：embedding/分数漂移 + 业务指标，关联版本。
6. **A/B 纪律**：一次一变量，指标说话。
7. **成本核算**：按架构估每 query 成本，别为不需要的复杂度买单。
8. **安全**：ingestion 层做来源信任 + 清洗，防语料投毒；权限在检索层强制。

---

## 九、全章收尾

第 2 章到此完整。一条主线贯穿始终：

> **RAG 的胜负在检索（73% 失败在这里）**——所以篇幅重心在 2.2–2.6（切片、向量、检索、查询、重排），2.7 让模型用好检索结果并可溯源，2.8/2.9 是复杂场景的针对性升级，2.10 让它在生产里稳、可观测、可演进。

选型的总原则也一致：**先做扎实基础（Naive + 混合检索 + 重排 + 好切片解决 80% 问题），再按具体失败对症升级，每加一层都要能说出它修复的问题和代价。**
