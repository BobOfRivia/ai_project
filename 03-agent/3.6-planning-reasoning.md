# 3.6 Planning & Reasoning 模式

> **核心问题**：一个 Agent 面对"帮我调研三家供应商并写一份对比报告"这种多步任务时，它怎么决定"先做什么、再做什么、第一步现在就干什么"？ReAct / Plan-and-Execute / Reflexion / ReWOO 这些工程模式各自怎么工作、代价是什么？以及——**当 o3、Claude 的 extended thinking 这类推理模型内置了强大的规划能力后，我们还需要手写这些规划逻辑吗？**
>
> **目标**：
> - 理解 Planning 为什么是 Agent 的核心难点（两个根本张力）
> - 吃透五种经典规划模式（ReAct / Plan-and-Execute / ReWOO / LLMCompiler / Reflexion）的原理、流程、代价与失败模式
> - 掌握"按你能承受的失败模式来选"的选型逻辑，以及生产中的组合用法
> - **想清楚推理模型崛起后，规划工程发生了什么变化——哪些下沉进了模型，哪些仍然必须由工程师显式实现**
> - 能根据任务复杂度、latency/cost 预算、容错要求做出规划模式决策
>
> **边界**：单次 prompt 里的 Chain-of-Thought 技巧属于 **1.3 提示工程**，本节讲的是**多步任务的编排结构**；多个 Agent 之间的分工协作（谁规划、谁执行、怎么通信）属于 **3.7 Multi-Agent**，本节聚焦**单 Agent 内部**的规划；上下文膨胀的治理手段属于 **3.4 上下文工程**，本节只讲它如何影响规划质量。

---

## 3.6.1 什么是 Planning，为什么它是 Agent 的难点

先给一个精准的定义。LangChain 的 Harrison Chase 把 Agent 的规划能力描述为一句很实在的话：

> 智能体必须"评估所有可用信息，然后决定：**我需要采取哪一系列步骤，以及现在应该迈出的第一步是什么？**"（evaluate all available information and then decide: what are the series of steps that I need to take, and which is the first one I should take right now?）

注意这句话里藏着规划的两个层次：**既要有对整条路径的全局判断（the series of steps），又要落到当下这一步的具体动作（the first one right now）。** 这两者的矛盾，正是规划之所以难的根源。

### 规划的两个根本张力

**张力一：长期目标 vs 近期动作。** Agent 必须一边盯着"最终要交付什么"，一边决定"这一拍具体调哪个工具、传什么参数"。盯着远处容易在近处走错（规划很宏大但第一步就调错工具），盯着近处容易忘了远方（每一步局部最优，累积起来偏离目标）。人类靠"在脑子里同时保持大局和细节"来平衡，而 LLM 每次只能看到你塞进上下文的东西——这个平衡要靠工程结构来支撑。

**张力二：上下文膨胀导致的规划退化。** 这一点极其关键，直接连着 3.4。Agent 每多走一步，就往上下文里追加一轮"思考 + 动作 + 观察结果"。步数一多，上下文迅速膨胀，触发 3.4.2 讲过的 **Context Rot** 和 **Lost-in-the-Middle**——模型对早期定下的目标和中间的关键约束"记不清了"，规划质量随步数增加而衰减。**这就是为什么"多步任务的可靠性会指数级下降"（回忆 3.1 的 0.95¹⁰≈60%）——每一步的规划都在一个越来越浑浊的上下文里做出。**

理解了这两个张力，你就理解了后面所有规划模式在优化什么：**它们本质上都是在用不同的方式，管理"全局 vs 局部"的注意力分配，以及对抗"步数增长带来的上下文退化"。**

### 一个必须先建立的判断：规划不是免费的

初学者容易觉得"让 Agent 先做个计划总是好的"。错。**规划本身是有成本的**——它要花 LLM 调用、花 token、花延迟，而且一个错误的计划可能比没有计划更糟（Agent 会固执地沿着错误路径执行到底）。所以规划模式的选择，本质是在**"规划的收益"和"规划的成本"之间找平衡**。简单任务硬套复杂规划框架，是本节最常见的过度工程。

## 3.6.2 五种经典规划模式

下面五种是单 Agent 规划的经典武器库。**不要把它们当成"哪个更先进"的排序**——它们是针对不同失败模式的不同取舍。每一种都讲透：怎么工作、代价、什么时候用、会怎么坏。

### 模式一：ReAct（Reason + Act）—— 边想边做

**原理**：最基础、最广为使用的模式。一个循环：**思考（Reason）→ 行动（Act，调工具）→ 观察（Observe，看结果）→ 再思考……** 每一步一次 LLM 调用，模型基于最新的观察结果决定下一步。

```
Thought: 我需要先查一下这家公司的营收
Action: search("Acme Corp 2025 revenue")
Observation: Acme Corp 2025 营收 12 亿美元
Thought: 好，现在查它的员工数
Action: search("Acme Corp employee count")
Observation: ...
（循环直到得出答案）
```

**优点**：
- **极其灵活**：每一步都能根据上一步的真实结果即时调整，非常适合探索性、路径不确定的任务。
- **实现简单、可解释**：思考过程显式可见，好调试。

**缺点/失败模式**：
- **贵**：每一步都要一次完整的（通常是大模型的）LLM 调用，长链条上 token 和延迟快速累积。
- **短视**：本质上只规划"下一步"，缺乏全局视野，容易在长任务里绕路、重复、走进死胡同。
- **上下文膨胀最严重**：每一步都把 thought+action+observation 全塞进上下文，是张力二的重灾区。

**什么时候用**：简单、直接、路径不确定、需要根据中间结果灵活应变的任务。**它是默认起点**——大多数 Agent 从 ReAct 开始，撞到成本或可靠性的墙才升级。

### 模式二：Plan-and-Execute —— 先规划整条路，再执行

**原理**：把"规划"和"执行"彻底分离。**先让一个 planner（通常是强模型）一次性生成完整的多步计划，然后 executor 逐步执行**，执行子步骤时可以用更便宜的小模型甚至不用 LLM。全部执行完，再回到 planner 判断：是给出最终答案，还是需要重新规划（replan）。

```
Planner（强模型，一次）：
  1. 查 Acme 营收
  2. 查 Acme 员工数
  3. 查竞品 Beta 的同样数据
  4. 生成对比表
Executor（逐步执行，可用小模型）：执行 1→2→3→4
Replan（如有必要）：结果不全就补充计划
```

**优点**（LangChain 官方博客明确列出）：
- **更快**：执行子步骤不用每次都请示大模型，大模型只在"规划"和"重规划"时被调用。
- **更省**：子任务可以下放给更小、更专的模型；大模型调用次数大幅减少。
- **质量可能更好**：强制 planner"想清楚完成整个任务需要的所有步骤"，这种全局思考本身能提升完成率和质量。
- **可审查**：计划在执行前就可见，能人工检查甚至加审批关卡（对安全敏感场景很重要）。

**缺点/失败模式**：
- **僵化**：计划一旦定死，对执行中冒出的意外适应性差——**所以"重规划机制"是它的必备组件**，没有 replan 的 Plan-and-Execute 很脆。
- **串行**：子步骤顺序执行，无法利用步骤间的并行机会。
- 每个子任务仍需一次 LLM 调用（步骤之间不能传递变量——这正是 ReWOO 要解决的）。

**什么时候用**：步骤相对**明确、可预先确定、有依赖顺序**的多步任务，且"规划清晰度"比"执行中的灵活应变"更重要。

### 模式三：ReWOO（Reasoning Without Observation）—— 规划时用占位符，减少 LLM 调用

**原理**：Plan-and-Execute 的进化。核心创新是**变量赋值**——planner 生成计划时，用占位符（`#E1`、`#E2`）引用前序步骤的输出，这样后续步骤能直接引用前面的结果，**不必每一步都回调 LLM 去"观察"**。

```
Plan: 先查 Acme 营收
E1 = Search["Acme Corp 2025 revenue"]
Plan: 用营收数据去查行业排名
E2 = LLM["根据 #E1 判断 Acme 在行业的位置"]
Plan: 查竞品
E3 = Search["Acme 的主要竞争对手"]
（最后 Solver 把 #E1 #E2 #E3 汇总成答案）
```

关键区别：ReAct 是"做一步→看结果→再想下一步"，**ReWOO 是"一次性把整个推理链和工具调用规划好（带占位符），再执行"**——它在规划阶段就"无需观察"地把推理做完了，这也是名字的由来。

**优点**：
- **token 效率高**：官方数据显示比 ReAct 减少 30-50% token 消耗，只需约 2 次 LLM 调用（一次规划 + 一次汇总）。
- 前序输出能作为后续输入，上下文传递更精准（只传相关数据，不是把全部 observation 堆进去）。

**缺点/失败模式**：
- **脆**：因为它在"没看到真实结果"的情况下就规划好了一切，**一旦某个工具返回了意料之外的东西，整条链就崩了**——它没有 ReAct 那种"看到结果再调整"的机会。
- 仍是串行执行。
- 变量引用管理增加了实现复杂度。

**什么时候用**：多步工作流中，**任务间有明确的输入-输出依赖、且每步结果相对可预测**（研究链、数据处理管线）。任务结果不可预测、需要临场应变的场景不适合。

### 模式四：LLMCompiler —— 把任务编译成 DAG 并行执行

**原理**：把任务执行重构成**有向无环图（DAG）**。planner 流式输出一个任务 DAG，每个任务包含"工具 + 参数 + 依赖列表"；一个 task-fetching 单元**一旦某任务的依赖就绪就立即调度它**，从而实现**并行执行**。任务参数可以引用其他任务的变量（`search("${1}")`）。

**优点**：
- **最快**：官方称通过 DAG 并行可达 **3.6x 加速**，甚至比 OpenAI 的"并行工具调用"更快（因为它流式解析计划、依赖一满足就调度）。
- 支持动态重规划（joiner 根据完整图的执行历史决定收尾还是重规划）。

**缺点/失败模式**：
- **架构最复杂**：需要健壮的依赖追踪和调度逻辑。
- **难调试**：并行执行让问题定位更难。
- 上手成本最高。

**什么时候用**：需要**低延迟**、且有**多个相互独立或弱依赖的工具调用**的场景（实时信息聚合、多 API 编排、复杂研究）。这是 ReWOO"串行"痛点的并行化答案。

> **注意 LLMCompiler 与 3.2.11（并行工具调用 / Code-as-Orchestrator）的关系**：LLMCompiler 是"用 DAG 做并行编排"这一思想在规划层的体现，和 3.2 讲的并行工具调用是同一件事的不同抽象层。生产中，很多时候直接用"代码编排 + 并行 tool call"就够了，不必上完整的 LLMCompiler 框架。

### 模式五：Reflexion —— 失败后自我批判再重试

**原理**：和前四种正交——前四种解决"怎么规划路径"，Reflexion 解决"**失败了怎么从错误中学习**"。机制：Agent 每完成一次尝试，用一个 evaluator 判断结果好坏，然后**对自己的输出做语言化的自我批判（self-reflection），把这段批判存进（情景）记忆，带着它重试**。

```
尝试 1：写了个函数 → 测试失败
Self-reflection：我没考虑空数组的边界情况，下次要先处理它
（把这条反思存入记忆）
尝试 2：带着上面的反思重写 → 测试通过
```

这本质上是**用自然语言反馈做"强化"**——不改模型权重，而是把"上次错在哪"作为上下文喂回去。它和 3.5 的**情景记忆**天然耦合（反思就是一种从经历中提炼的情景记忆）。

**优点**：
- 显著提升需要**试错迭代**的任务（写代码、解题）的成功率。
- 不需要训练，纯 prompt/上下文层面实现。

**缺点/失败模式**：
- **增加延迟和成本**：每次失败都要多跑"批判 + 重试"的循环。
- 依赖一个**靠谱的 evaluator**——如果无法自动判断成败（没有测试、没有 ground truth），自我批判可能是空中楼阁甚至越改越错。
- 有陷入"反复重试仍不得其解"的风险，需要设重试上限。

**什么时候用**：有**明确成败信号**（测试通过与否、能否编译、是否满足约束）的迭代型任务。没有客观 evaluator 的场景慎用。

## 3.6.3 五种模式横向对比

| 模式 | LLM 调用 | 执行方式 | 核心优势 | 主要失败模式 | 最适合 |
|---|---|---|---|---|---|
| **ReAct** | 高（每步一次） | 串行、边走边定 | 灵活、可解释 | 贵、短视、上下文膨胀 | 简单、探索性、路径不确定 |
| **Plan-and-Execute** | 中（规划+执行） | 串行、先定后行 | 快、省、可审查 | 僵化（需 replan 兜底） | 步骤明确、有依赖顺序 |
| **ReWOO** | 低（~2 次） | 串行、占位符 | token 效率最高 | 脆（意外结果即崩） | 依赖明确且结果可预测 |
| **LLMCompiler** | 低（流式+并行） | **并行 DAG** | 最快（~3.6x） | 复杂、难调试 | 低延迟、多独立工具调用 |
| **Reflexion** | 高（含重试） | 迭代重试 | 从失败中学习 | 慢、依赖 evaluator | 有明确成败信号的试错任务 |

**选型的正确心法（记住这一句）**——不要问"哪个最好"，要问 **"我能承受哪种失败模式"**：

> - 能承受**浪费 token** → ReAct
> - 能承受**僵化**（愿意加 replan） → Plan-and-Execute
> - 能承受**脆**（结果可预测） → ReWOO
> - 能承受**实现复杂度** → LLMCompiler
> - 能承受**延迟**（换取成功率） → Reflexion

### 生产中很少单用一种：组合模式

真实系统里，这些模式常常**分层组合**（这是社区总结的最常见生产搭配）：

- **Plan-and-Execute 做外层骨架 + ReAct 子 Agent 处理每个步骤**：外层保证全局有计划、可审查，内层保证每步能灵活应变。这是复杂 Agent 最常见的结构。
- **ReAct 做执行层 + Reflexion 做工作流层**：底层灵活执行，上层在失败时触发反思重试。
- **Plan-and-Execute + Reflexion 的 replan**：把"重规划"实现成一种反思——执行失败后批判原计划，生成改进的新计划。

**记住：模式是积木，不是单选题。** 一个成熟 Agent 的规划结构，往往是两三种模式的嵌套。

## 3.6.4 推理模型如何改变规划工程

这是本节最重要、也最"活"的部分——**推理模型（Reasoning Models）的崛起，正在重写规划工程的剧本。** 这里必须讲清楚：什么变了，什么没变。

### 先搞清楚：推理模型是什么，它内置了什么

o3/o4-mini、Claude 的 extended thinking、Gemini Deep Think、DeepSeek-R1 这类模型，和普通模型的关键区别是：**它们在给出答案前，会先自主地生成一大段"思考过程"（内部推理），把问题拆解、试探、自我纠错**——而这一切是模型**内置**的，不需要你在 prompt 里教它"让我们一步步思考"。

更进一步，**Claude 4 支持 interleaved thinking（交错思考）**——它能在**工具调用之间**思考，把多次工具调用和中间的推理步骤串起来。这一点直接冲击了 ReAct 的地基：**ReAct 的"Reason"步骤，正在被模型内化成它自己的能力。** 下面把它讲透，因为这里有个极易混淆的点。

#### interleaved thinking 到底"多"了什么

先破一个误解：interleaved thinking **不是**"单次模型调用内部把推理内化"。它发生在 **agent 循环层面**——真正的增量只有一句话：**普通 extended thinking 只在"选工具前"想一次；interleaved thinking 让模型在"工具结果回来之后"再想一次（而且每次结果回来都能再想）。**

同一条工具循环，两种模式的差别：

```
普通 extended thinking（只在开头想一次）
  [thinking] 分析任务，决定先调 A
  [tool_use: A]
    ↓ A 的结果回来
  [tool_use: B]          ← 没有 thinking，闷头调下一个
    ↓ B 的结果回来
  [text] 直接给答案       ← 也没有 thinking

interleaved thinking（每次结果回来后再想）
  [thinking] 分析任务，决定先调 A
  [tool_use: A]
    ↓ A 的结果回来
  [thinking] A 返回了 X，说明……那下一步该调 B   ← 关键：基于结果再推理
  [tool_use: B]
    ↓ B 的结果回来
  [thinking] X 与 Y 对比，得出……               ← 又想一次
  [text] 给出结论
```

官方原话：interleaved thinking 让 Claude 能 "**reason about the results of a tool call before deciding what to do next**"——即"在决定下一步之前，先对上一个工具的结果做推理"。

#### 两个关键边界（别把它误当成"更聪明的单次调用"）

1. **"再思考"发生在下一次 API 往返里，不是同一次生成内部。** 执行工具必须先把控制权交还给 harness，结果才拿得到；所以模型不可能在一次生成里真正 think→act→observe→think。真实时序是"往返 N 生成 `[thinking][tool_use]` 后停下 → harness 执行 → 往返 N+1 再生成 `[thinking][…]`"。**交错是在循环里跨多次往返涌现的**，这也正是它没有取代"需要 harness 驱动循环"这件事的原因。

2. **它比经典 ReAct 更"重"，不是同一个东西。** 经典 ReAct 的 "Thought:" 是明文文本 token，框架可以随意裁剪重排；interleaved thinking 的思考块是模型内建、带签名的结构化 reasoning artifact，**必须原样回传**（"you must pass `thinking` blocks back to the API... include the complete unmodified block"），改动或丢弃会报错。而且 Anthropic 在模型视角里把**整条工具循环视为"一个还没结束的 assistant turn"**，不是经典 ReAct 那样平铺的多个回合。

一句话定位：**interleaved thinking 把经典 ReAct 手写的 "Reason" 步骤，从框架层的明文文本，下沉成了模型内建、跨往返保持、必须原样回传的签名推理块。它不改变"要 harness 驱动循环"这一事实，改变的是——每次往返，模型是否被允许"先想再动"。**（延伸阅读见本节末 Anthropic extended thinking 文档。）

### 变化一：显式的 CoT prompting 收益正在归零

一个关键的、可验证的发现（Wharton 2025 Prompting Science Report）：

> **对已经会自己一步步思考的模型，chain-of-thought 提示带来的收益微乎其微。**（chain-of-thought prompting adds negligible benefit on models that already think step-by-step.）

原因很直白：**你想通过 prompt "催生"出来的那个推理过程，模型自己已经在做了。** 在 o3 上写"让我们一步步思考"，约等于对一个正在心算的人喊"记得要用脑子算"——没用，因为它本来就在算。

这条原理可以推广到本节：**很多我们过去手写的"轻量规划脚手架"，在推理模型上正在变成冗余甚至干扰。**

### 变化二：但显式规划**不会消失**——它会分层

这是最容易被误读的地方。有人从"推理模型很强"推出"以后不用写规划逻辑了"——**这是错的**。Harrison Chase 给了一个很有分量的论断：

> **"无论模型变得多强大，你永远都需要以某种形式，告诉智能体它应该做什么。"**（no matter how powerful the model becomes, you will always need to communicate to the agent, in some form, what it should do.）

关键在于**"以某种形式"**。规划责任正在发生一场**分层迁移**：

| | 通用推理（"怎么想"） | 任务特定的认知架构（"这个业务该怎么办"） |
|---|---|---|
| **趋势** | **下沉到模型层** | **仍然留在工程师手里** |
| **含义** | "把问题拆成子步骤""发现自己错了要回头"这类通用推理，模型越来越能自己搞定，不用你写 | "先跑测试再写代码再迭代"这类**领域特定的工作流**，仍需你显式编码 |
| **例子** | 你不再需要手写 ReAct 的思考模板 | AlphaCodium 把"tests → code → iteration"的流程**硬编码**下来 |

Chase 用 AlphaCodium 的例子把这层意思点透了：与其让 LLM 自己去规划怎么写竞赛代码，开发者**直接把工作流写死**（先生成测试 → 写代码 → 用测试迭代），相当于对模型说：**"规划的事你别操心了，LLM，我替你规划好了（don't worry about planning, LLM, I'll do it for you!）。"** 这种**领域特定认知架构（domain-specific cognitive architecture）**，是推理模型再强也替代不了的——因为它编码的是**业务知识和约束**，不是通用智能。

### 变化三：规划模式没有被淘汰，而是"变轻"和"混合"

推理模型时代，经典规划模式仍在用，但用法变了。一个可验证的例子：一个 **ReAct&Plan 混合 Agent**，用 GPT-4o 跑 30 轮 ReAct，只在第 12 轮前插入**一次**规划步骤，就多解决了 4 个任务、达到 95% 成功率。这说明：

- **不必再给推理模型套厚重的规划框架**——模型自己的推理能力已经承担了大部分"想"的工作。
- **但一个恰当位置的、轻量的显式规划步骤，依然能提升效果**——因为它提供的是**结构和方向**，不是"教模型思考"。

### 给工程师的实操结论

把上面三条变化落成可操作的判断：

1. **用推理模型时，先删掉冗余的规划脚手架**。别再在 o3/Claude-thinking 上堆砌"一步步思考""先列计划再执行"的 prompt 模板——先看模型裸奔能做到什么，很可能已经够了。
2. **保留并投资"领域特定认知架构"**。你的独特价值不在于教模型怎么推理（那是模型厂商的事），而在于把**你的业务流程、约束、成败判据**显式编码进 Agent。这部分推理模型永远替代不了。
3. **规划模式当"轻量补丁"用**。需要全局审查时加一个 Plan-and-Execute 的规划步；需要并行提速时用 DAG 编排；需要从失败学习时加 Reflexion。**按需、单点地加，而不是默认套一整套框架。**
4. **警惕成本和延迟**。推理模型的"思考"是要花 token 和时间的。extended thinking 开得越猛越慢越贵。把它用在真正需要深度推理的环节，简单步骤用普通模型。

## 3.6.5 选型决策框架

综合经典模式和推理模型的影响，规划模式的选型可以收敛成一条决策路径：

**第 0 步（推理模型时代的新起点）：先让推理模型裸跑。**
拿一个推理模型（Claude extended thinking / o-series），不加任何规划框架，直接给任务。**很多中等复杂度的任务，模型内置推理已经够用**——这一步能帮你避免一大半的过度工程。

**第 1 步：裸跑不够，先加最轻的——ReAct。**
需要调工具、需要根据中间结果应变，就上 ReAct。它是灵活性最高、实现最简单的默认选择。

**第 2 步：ReAct 太贵/太慢/太短视，按瓶颈升级：**
- 瓶颈是**成本/延迟**、且步骤可预先规划 → **Plan-and-Execute**（大模型只在规划时用）。
- 瓶颈是**成本**、且步骤依赖明确且结果可预测 → **ReWOO**。
- 瓶颈是**延迟**、且有多个可并行的工具调用 → **LLMCompiler / 并行工具调用**。
- 瓶颈是**成功率**、且有客观成败信号 → 叠加 **Reflexion**。

**第 3 步：编码你的领域认知架构。**
不管用哪种模式，把**你的业务特定工作流**（该先做什么、什么情况下必须停下来问人、什么是完成的标准）显式写进 Agent。这是你的核心资产。

**第 4 步：组合。**
复杂系统用分层组合（Plan-Execute 外层 + ReAct 内层 + Reflexion 兜底）。

> **一以贯之的纪律**：**能不加规划就不加，能用轻的就不用重的。** 规划是成本，不是免费的好处。推理模型时代，这条纪律比以往更重要——因为模型已经替你做掉了很多规划。

## 3.6.6 常见误区与小结

### 常见误区

| 误区 | 为什么错 | 正确做法 |
|---|---|---|
| **"先让 Agent 做个计划总没错"** | 规划要花调用/token/延迟，错误的计划比没计划更糟（会固执执行到底） | 规划是成本-收益权衡；简单任务别套复杂框架 |
| **"推理模型这么强，不用写规划逻辑了"** | 通用推理下沉到模型，但**领域特定认知架构**（业务流程/约束/成败判据）永远要工程师写 | 删冗余脚手架，但保留并投资领域认知架构 |
| **"给 o3 也写'一步步思考'的 prompt"** | 模型已内置 step-by-step，CoT 提示收益趋零，甚至干扰 | 推理模型上先裸跑，别堆规划模板 |
| **"ReWOO/LLMCompiler 更新更快，就用它们"** | ReWOO 脆（意外结果即崩）、LLMCompiler 复杂难调试，各有硬伤 | 按"能承受哪种失败模式"选，不按"新旧" |
| **"Reflexion 能自我改进，加上准没错"** | 没有客观 evaluator 时，自我批判可能空转甚至越改越错 | 只在有明确成败信号时用，且设重试上限 |
| **"选一种模式用到底"** | 生产系统几乎都是分层组合 | 模式是积木，按层嵌套组合 |

### 小结

- **规划难在两个张力**：全局目标 vs 当下动作；步数增长导致的上下文退化（连着 3.4 的 Context Rot，也是 3.1 多步可靠性衰减的根因）。
- **五种经典模式各有取舍**：ReAct（灵活但贵/短视）、Plan-and-Execute（快/省/可审但僵化）、ReWOO（token 最省但脆）、LLMCompiler（最快但复杂）、Reflexion（能从失败学习但慢/依赖 evaluator）。**按"能承受哪种失败模式"来选**，生产中分层组合。
- **推理模型改变了什么**：① 显式 CoT prompting 收益趋零；② **但显式规划不会消失，而是分层**——通用推理下沉进模型，领域特定认知架构仍留在工程师手里（AlphaCodium 是范例）；③ 经典模式没被淘汰，而是"变轻""按需单点加"。
- **选型纪律**：推理模型裸跑 → 不够加 ReAct → 按瓶颈升级到 Plan-Execute/ReWOO/LLMCompiler/Reflexion → 始终编码你的领域认知架构 → 复杂场景组合。**能不加就不加，能用轻的不用重的。**

> **成熟度标注**：ReAct、Plan-and-Execute——生产广泛验证的成熟模式，放心用。ReWOO、LLMCompiler——可用但相对小众，落地前确认你的场景匹配其失败模式（结果可预测 / 值得为并行付复杂度）。Reflexion——思想已被广泛借鉴（尤其编码 Agent），但完整实现偏重、强依赖 evaluator，属"谨慎落地"。**推理模型对规划的影响是快速演进的领域**——本节结论（CoT 收益趋零、规划分层、模式变轻）方向明确且有一手佐证，但具体边界会随模型迭代变化，需持续跟踪厂商官方文档。
