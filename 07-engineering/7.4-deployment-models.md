# 7.4 部署形态决策（自托管 / Serverless / 云 API）

> **核心问题**：自托管（vLLM / Ollama）vs Serverless GPU（Modal / RunPod）vs 云 API（OpenAI / Anthropic / Bedrock）什么时候选谁？什么时候做 hybrid？breakeven 在哪？
> **目标**：能根据"流量、合规、模型需求、团队能力"画出 hybrid 部署架构，并预估自托管的真实总拥有成本。

**展开重点**：
- **三大形态对比**：
  - **云 API**（OpenAI / Anthropic / Gemini / Bedrock）：零运维、按 token 付费；frontier 模型唯一通路；流量峰谷自适应
  - **Serverless GPU**（Modal / RunPod / Replicate / Bedrock Custom Model Import）：按秒付费、自己 model artifact、冷启动是核心代价
  - **自托管**（vLLM on K8s / EC2 / 自建 GPU）：固定成本、最高吞吐、最低单 token 成本（量大才划算）
- **2026 共识：hybrid 是默认**（非二选一）：
  - **Baseline 流量**（可预测、高频、延迟敏感）→ 自托管 vLLM
  - **Overflow 流量**（峰值溢出）→ 云 API
  - **Frontier 任务**（GPT-5 / Claude Opus 推理）→ 云 API
  - **PII / 合规敏感**（金融、医疗、政府）→ 自托管
- **Breakeven 经济学**：
  - **门槛量**：约 **1B token/月** + **60-70%+ GPU 利用率**
  - 过此点自托管胜出，industrial scale 可省 **~5x**
  - 真实案例：一家 fintech 月支出 **$47K → $8K（-83%）**：可预测 workload 切自托管，frontier 保留云 API
- **选型决策矩阵**（2026）：
  | 场景 | 推荐 |
  |---|---|
  | < 50M token/月 + 多模型探索 | **云 API**（无脑） |
  | 单模型、稳定流量、要 SLO | **自托管 vLLM**（或托管 vLLM） |
  | 流量 burst、CI/批处理 | **Serverless GPU** |
  | PII / GDPR / HIPAA / 国家合规 | **自托管**（或专用云区域 / VPC） |
  | 需要 frontier 推理 | **云 API**（无替代品） |
  | 部分模型已开源、可替代 | **Hybrid（自托管 + 云 API）** |
- **自托管 TCO 真账**（容易低估的）：
  - GPU 实例（H100 月租 $25-40K）
  - 工程成本：1-2 个 SRE / ML infra 工程师
  - 模型管理：升级、量化、回归测试
  - 容灾：多副本 / 多区
  - GPU 利用率 < 50% → 经济上不如云 API
- **Serverless GPU 关键参数**（详 → 7.9）：
  - Modal：冷启动 <2s（GPU snapshot 加持），延迟敏感首选
  - RunPod：最快上手
  - Replicate：模型市场体验最好
  - Bedrock Custom Model Import：26B 模型冷启动几分钟，需 keepalive
- **常见反模式**：
  - "省钱"立刻自托管：低于 breakeven，反而更贵
  - "云 API 锁定"恐惧 → 整套自托管：忽视了 frontier 模型才是核心价值
  - 忽视合规：直接把客户 PII 送到第三方 API
