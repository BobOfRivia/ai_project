# 8.7 数据驻留 / Zero Retention / 合规部署架构

> **核心问题**：监管行业（医疗 / 金融 / 政府）怎么用 LLM？BAA / HIPAA / GDPR 区域驻留怎么落到部署架构？ZDR 是不是"安全"？
> **目标**：能根据"数据敏感度 × 合规要求 × 成本"画出 5 档部署架构：公共 API → ZDR 企业合同 → 区域化云 API → VPC 自托管 → 完全离线。

**展开重点**：
- **2026 五档部署架构**（敏感度从低到高）：
  | 档位 | 形态 | 数据流 | 合规适用 |
  |---|---|---|---|
  | **L1** | 公共 API (OpenAI / Anthropic 默认) | Prompt → 全球任意 region | 一般 SaaS、无敏感数据 |
  | **L2** | 同上 + **Zero Data Retention** 企业合同 | 不存训练、不留日志 | 中度敏感、需企业合同 |
  | **L3** | **AWS Bedrock / Azure OpenAI**（同账户 region） | 数据不出客户 VPC region | HIPAA + BAA、欧盟驻留 |
  | **L4** | **自托管开源模型**（vLLM in VPC / on-prem） | 完全本地、可断网 | 高敏感（PHI、机密） |
  | **L5** | **物理隔离**（air-gapped）+ 自托管 | 无 Internet | 国防 / 国安 / 极敏感 |
- **BAA 与 HIPAA**（美国医疗）：
  - 任何接触 PHI 的"业务伙伴"必须签 **Business Associate Agreement**
  - **AWS Bedrock / Azure OpenAI 提供 BAA 选项**，但默认公共 API 不在内
  - BAA-eligible ≠ HIPAA-compliant：还要配置（区域、加密、日志、访问控制）
  - 高敏感场景：自托管开源模型 + VPC 隔离往往更稳
- **Zero Data Retention (ZDR)**：
  - 内容不存储、不用于训练、不留日志（除审计必需）
  - OpenAI / Anthropic 都提供 ZDR，**需企业合同**（不是默认）
  - ZDR ≠ 数据不出境：传输仍可能跨 region
  - ZDR ≠ 满足 GDPR：仍需法律基础（合同 / 合法利益）
- **区域化 / 数据驻留要求**：
  - **欧盟**：GDPR + 跨境传输（充分性决定 / SCC / BCR）
  - **中国**：PIPL，重要数据出境需安全评估
  - **加拿大 / 巴西 / 印度**：本地化趋势
  - 解法：用云厂区域版本（Bedrock EU、Azure OpenAI EU、Vertex AI EU）
- **架构决策矩阵**：
  - **能不能不用 LLM**？最敏感字段（SSN、医保卡号）可能根本不该发
  - **能不能脱敏后用**？很多场景 PII tokenize 后可用 L1/L2（→ 8.6）
  - **必须原文？** → L3+
  - **断网要求？** → L4/L5 + 自托管
- **自托管开源模型的合规优势 / 劣势**：
  - 优势：数据完全本地、审计完整、可定制
  - 劣势：质量天花板低（vs GPT-5/Claude）、运维成本高
  - 2026 折中：**Hybrid**——脱敏后用 frontier API + 全文走本地小模型
- **加密与密钥**：
  - 传输 TLS 1.3、存储 AES-256、**per-tenant CMK**（不是单一密钥）
  - 推理过程中的 GPU 内存——目前仍是明文，Confidential Computing（H100 CC / Intel TDX）2026 在企业级展开
- **审计与可追溯**：
  - 每次 LLM 调用：who / when / model / input hash / output hash / region
  - **WORM** 日志，监管要求保留 6 个月-7 年不等
  - PII 不能写进日志（→ 8.6）
- **常见反模式**：
  - "我们用 ZDR 就合规了"——ZDR 是 retention 维度，不解决传输 / 法律基础
  - 公共 API 处理 PHI 没签 BAA：直接违法
  - 自托管小模型质量太差被用户绕过——通过自带浏览器去公共 API
  - 区域配置错（"我以为 Azure OpenAI EU 就 EU 了"，结果用错 endpoint）
