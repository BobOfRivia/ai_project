# 2.3 向量化与向量数据库

> **核心问题**：怎么存、用什么存（单向量 vs 多向量）？
> **目标**：能为业务量级、预算、部署形态选对向量库与 Embedding 模型。

---

## 一、向量化在管线里的位置

2.2 把文档切成了块。**向量化（Embedding）就是把每个文本块变成一串数字（向量），让"语义相近"变成"空间距离相近"**，从而可以用数学方法快速找相似内容。

```
文本块 ──Embedding模型──▶ [0.12, -0.8, 0.3, ...]  ──写入──▶ 向量数据库
                          (一个 N 维向量,               (存向量 + 建 ANN 索引,
                           N 通常 384~3072)              支持快速近邻搜索)
```

本节两个决策：**用什么 Embedding 模型**（决定语义表达质量）+ **用什么向量库**（决定存取性能与运维成本）。这两个选型基本锁定了检索层的地基。

---

## 二、Embedding 模型选型

### 2.1 选型看五个维度

| 维度 | 说明 |
|---|---|
| **检索质量** | 参考 MTEB 榜，但**别只看榜**（见下方警示） |
| **语言覆盖** | 中文/多语言场景必须选支持中文的模型 |
| **维度** | 影响存储与检索速度，越大不一定越好 |
| **成本** | API 按 token 计费 vs 自托管的 GPU 成本 |
| **部署形态** | 托管 API（省心）vs 开源自托管（可控、数据不出域） |

### 2.2 主流模型对比（2026）

| 模型 | 类型 | 维度 | 亮点 | 适合 |
|---|---|---|---|---|
| **OpenAI text-embedding-3-large** | API | 3072（可截断） | 稳、省心的默认托管选择 | 不想运维、要稳的生产 |
| **Cohere embed-v4** | API | 可变 | MTEB 领先，**唯一成熟的多模态** embedding | 高精度 / 多模态、想省 OCR |
| **Voyage voyage-3-large** | API | 可变 | MTEB 顶尖，检索质量强 | 追求最高检索精度 |
| **BGE-M3** | 开源(MIT) | 1024 | **100+ 语言**，一个模型同出 dense/sparse/多向量 | 自托管主力，中文友好 |
| **Qwen3-Embedding-8B** | 开源 | 可变 | MTEB 约 70.6，**超过 OpenAI(64.6)/Google(68.3)** | 高精度开源，Q4 量化约 5GB 显存 |
| **Jina-embeddings-v3** | API/开源 | 可变 | 性价比突出（~$0.02/1M tokens） | 预算敏感 |

> 2026 年**自托管栈的事实默认**：**BGE-M3 + BGE-reranker-v2**（reranker 见 2.6）。中文场景这套组合尤其常见。

### 2.3 维度与成本权衡

- **维度不是越大越好**：维度越高，存储越大、检索越慢。
- **Matryoshka（套娃）截断**：OpenAI large 出 3072 维，但**可以只取前 256 维，质量损失极小**——省存储、加速检索。很多新模型（OpenAI text-embedding-3、Gemini Embedding、Nomic、部分 BGE 等）都支持这种"按需截断"。原理见下方专栏。
- **成本量级**：OpenAI 3-small 约 $0.02/1M tokens，Qwen3-8B（OpenRouter）约 $0.01/1M。API 便宜到一定程度，自托管的意义更多在**数据不出域**和**大批量摊薄成本**，而非省小钱。

> 🔬 **想深入原理**：embedding 模型是怎么在 LLM 基础上训练出来的（池化/双向化/对比学习/多阶段管线/hard negatives）见 → [`2.3.1-embedding-model-training.md`](./2.3.1-embedding-model-training.md)（深入选读）。

#### 📌 专栏：Matryoshka 截断原理（含损失函数）

**核心认知：截断能力是"训练时烤进去的"，不是输出时临时压缩。**

普通 embedding 的信息**均匀摊在所有维度**上，砍掉后半段等于随机丢信息，向量直接失去意义。**MRL（Matryoshka Representation Learning）**改的是训练目标，让信息**由粗到细（coarse-to-fine）前置到最靠前的维度**——像套娃，小的那个也是完整可用的表示。

**怎么做到的：多截断点的联合损失**

训练时预先选一组嵌套维度 $M = \{8, 16, 32, \dots, d\}$。对每个维度 $m$，取表示向量的**前 $m$ 维前缀**单独算一次任务损失，再加权求和：

$$L_{\text{MRL}} = \sum_{m \in M} c_m \cdot L_{\text{task}}\big(f_m(x),\, y\big)$$

- $f_m(x)$：前 $m$ 维前缀经过对应输出头得到的结果；
- $c_m$：第 $m$ 个粒度的权重（常用**全部取 1** 的无权重版）；
- $L_{\text{task}}$：**按任务而定**——分类模型用交叉熵（CE）；**检索/嵌入模型用对比损失（InfoNCE）**，即在截断并归一化后的前缀上算 query-正样本的对比损失。

这个"每个前缀都要独立把任务做好"的约束，逼模型把最关键的语义塞进前几维。

**权重共享（MRL-E / weight-tying）**：各粒度的输出头不各建一套，而是共享同一个大矩阵 $W$、各粒度用它的前 $m$ 列 $W^{(m)} = W_{1:m}$，进一步省参数、保证嵌套一致性。

**使用时只做两步**（无额外压缩计算）：

1. 取前 $k$ 维：`v = full[:256]`
2. **重新 L2 归一化**：`v = v / ‖v‖`（截断后模长不再为 1，余弦相似度会失真，必须重算）。
   > OpenAI 传 `dimensions=256` 时，截断 + 归一化都在**服务端**做好，返回的直接是归一化后的 256 维向量。

**效果与边界**：

- MRL 的 128 维常能追平独立训练的普通 512 维——约 **4× 存储/计算节省**、质量几乎不掉。
- ⚠️ **只有 MRL 训练过的模型能这么截**；拿非 MRL 的老模型硬截前 $k$ 维质量会明显崩。选型时确认模型声明支持 Matryoshka / 可变维度。

### 2.4 一条重要警示：MTEB 榜 ≠ 你的真实表现

MTEB 是行业标准基准，但**榜单排名不总能预测真实 RAG 表现**。原因：你的语料领域、语言、query 分布都和基准不同。

> **落地做法**：用 MTEB 圈定 2–3 个候选，然后**在你自己的数据上跑一次小规模评估（见 2.10）再定**。这一步比信榜单重要得多。

### 2.5 中文 / 多语言

- 首选 **BGE-M3**（100+ 语言、中文强、开源）或 **Qwen3-Embedding**（中文场景表现优秀）。
- 托管要中文好又省心，可评估 Cohere / Voyage 的多语言版本。
- ⚠️ 避免用纯英文模型硬跑中文——检索质量会明显掉。

## 三、向量数据库选型

### 3.1 向量库到底干什么

存向量只是表面，核心价值是**在千万/十亿级向量里毫秒级找近邻**——靠的是 **ANN（近似最近邻）索引**（HNSW、IVF 等）。此外它还要提供：**元数据过滤、混合检索、多租户、持久化与扩缩容**。

### 3.2 主流向量库对比（2026）

| 向量库 | 形态 | 亮点 | 最适合 |
|---|---|---|---|
| **Pinecone** | 全托管 | Serverless 免容量规划、十亿级、亚 100ms、自带重排 | 企业级、负载不可预测、不想运维 |
| **Qdrant** | 开源(Rust) | **开源里最快**（10M 向量 p99 ~12ms），过滤强 | 自托管、要性能与丰富过滤 |
| **Weaviate** | 开源 | **原生混合检索最强**（vector+BM25+过滤），多租户好 | 要开箱即用的混合检索 |
| **Milvus** | 开源 | 十亿级、多索引、多模态；运维偏重 | 超大规模自托管 |
| **pgvector** | Postgres 扩展 | 复用现有 Postgres、事务一致性 | 已经在用 Postgres 的团队 |
| **Chroma** | 轻量/嵌入式 | Python 优先、上手最快 | 本地开发、原型 |

### 3.3 按规模分层选择（最实用的一刀）

| 向量量级 | 选择 |
|---|---|
| **< 10M** | 随便选，都能扛。优先 DX 好的（Chroma 开发、pgvector 顺手、Qdrant 性能） |
| **10M – 1B** | 收窄：Pinecone（托管）/ Qdrant、Weaviate、Milvus（自托管）/ Vespa |
| **> 1B** | Milvus 或 Vespa 的分布式部署 |

### 3.4 元数据过滤与混合检索（选型硬指标）

- **纯向量检索在多数生产负载上打不过混合检索**（vector + BM25 + 元数据过滤）——因为要精确命中专有名词、版本号、ID，同时又要语义匹配。详见 2.4。
- **原生支持混合检索的**：Weaviate、Vespa、Qdrant（Pinecone 后加了；pgvector 需手动拼）。
- 选库时**把"是否原生混合检索 + 过滤性能"当硬指标**，不是可选项。

### 3.5 多租户与权限

SaaS / 企业内多部门场景，必须在向量库层面隔离数据：

- **多租户**：Weaviate 原生多租户较强；其他库可用 namespace / collection / 元数据分区实现。
- **权限过滤**：把权限（部门、角色、可见范围）写进元数据，检索时用过滤强制施加——**权限过滤要在检索层生效，不能只靠生成时提示**（安全见第 8 章）。

## 四、单向量 vs 多向量表征（为 2.4 铺垫）

前面讲的都是**单向量**：一个块 → 一个向量。还有一条正在生产落地的路线：**多向量（Multi-vector）**。

| | 单向量（Dense） | 多向量（Multi-vector / Late Interaction） |
|---|---|---|
| 表征 | 整块压成 1 个向量 | 每个 token 一个向量（如 ColBERT） |
| 打分 | 向量点积/余弦 | **MaxSim**：query 每 token 找块内最相似 token 再求和 |
| 精度 | 好 | **更高**（细粒度匹配，接近 cross-encoder） |
| 代价 | 存储小、快 | **存储大**（每块上千向量）、索引复杂 |

- 代表：**ColBERT / ColBERTv2**；工程上用 **PLAID** 引擎加速、**MUVERA** 等做压缩来缓解存储爆炸。
- 多模态版是 **ColPali**（把 PDF 页当图像做多向量检索，见 2.9）。

> 这里先建立"还有多向量这条路"的认知，**具体怎么用在检索里、什么时候值得上，2.4 的 Late Interaction 检索会展开**。

---

## 五、本节小结：向量化决策清单

1. **选 Embedding**：托管省心 → OpenAI 3-large；追精度 → Voyage / Cohere；自托管/中文 → **BGE-M3**（或 Qwen3-Embedding）。
2. **别信榜单信自己的数据**：MTEB 圈候选，自有数据小评估定胜负。
3. **维度按需截断**：Matryoshka 模型可截短省存储（如 3072→256）。
4. **选向量库看规模**：<10M 随意、10M–1B 收窄、>1B 上 Milvus/Vespa。
5. **混合检索 + 过滤当硬指标**：优先原生支持（Weaviate/Qdrant/Vespa）；已用 Postgres 就 pgvector。
6. **权限写进元数据**、在检索层强制过滤。
7. **知道多向量这条路**：高精度需求可评估 ColBERT 式方案（2.4 展开）。
