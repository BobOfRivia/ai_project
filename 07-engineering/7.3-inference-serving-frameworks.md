# 7.3 推理服务框架选型

> **核心问题**：vLLM / SGLang / TensorRT-LLM / Ollama / TGI 在 2026 各占什么生态位？什么场景该选谁？
> **目标**：能根据"用户场景 + 硬件 + 延迟/吞吐偏好"挑出合适的推理服务，不被"benchmark 第一名"带偏。

**展开重点**：
- **2026 格局速读**：
  - **vLLM**：**默认**，最大开源社区，模型支持最广，HuggingFace Inference Endpoints 默认
  - **SGLang**：**共享前缀场景胜出**（agent / 多轮 / RAG），**RadixAttention** 显著吞吐增益
  - **TensorRT-LLM**：**H100 极致延迟**（+15-30% vs vLLM），代价是 ~28 min 编译启动
  - **Ollama**：**开发 / 个人 / 隔离环境**首选；单请求方便，**生产多并发吞吐落后 vLLM ~10-16x**
  - **TGI**：**2025-12 起仅接 bug fix**，HuggingFace 自己已切默认到 vLLM
- **选型决策树**：
  1. **多用户高并发生产 API** → vLLM（默认）
  2. **Agent / RAG / 多轮共享前缀** → SGLang
  3. **NVIDIA H100/H200 + 严格延迟 SLO + 模型固定** → TensorRT-LLM
  4. **本机 / 开发 / air-gapped** → Ollama（生产请勿）
  5. **极小规模 / 单租户** → Ollama 或 LM Studio 也可
- **关键能力对照**：
  - **PagedAttention**：vLLM / SGLang / TensorRT-LLM 都有
  - **Continuous batching**：全部支持
  - **FP8**：vLLM ✓ / SGLang ✓ / TensorRT-LLM ✓
  - **Speculative decoding**：TensorRT-LLM 最成熟、vLLM 跟进、SGLang 部分支持
  - **RadixAttention（前缀共享缓存）**：SGLang 独有
  - **多硬件支持**（AMD / Intel / TPU）：vLLM 最广
  - **OpenAI 兼容 API**：vLLM / SGLang / Ollama 都内置
- **吞吐对比（参考量级，2026 公开 benchmark）**：
  - vLLM @ Llama 3.1 8B / 10 并发：**~485 tok/s** 总
  - Ollama 同配置：**~30 tok/s** 总（差 **~16x**）
  - SGLang @ 多轮共享前缀：vLLM 之上再 **+30-50%**
  - TensorRT-LLM @ 单请求延迟：vLLM 基础上首 token 再 **-15-30%**
- **常见反模式**：
  - 生产用 Ollama 接客户流量
  - 因为"听说 TensorRT-LLM 最快"就用，模型一升级就要重新编译
  - 自己手搓 batching / paging（vLLM 已经做完了）
- **2026 趋势**：
  - vLLM 0.17+：cloud model offloading、web search 内置、multimodal、streaming tool calls
  - SGLang 进 RAG / agent 主流栈
  - 推理服务越来越像"标准化基础设施"，应用工程师只配不写
