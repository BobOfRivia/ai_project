# 5.10 多模态应用工程

> **核心问题**：多模态调用的延迟预算、token 成本、流式 UI、缓存、降级、可观测怎么做？为什么这层"工程"在多模态比纯文本更关键？
> **目标**：能给一个多模态产品列出延迟预算表、成本表、降级链与监控指标，并在主线模型故障 / 高峰 / 涨价时不影响业务。
>
> **与 ch7 的关系**：通用工程化（路由 / 部署 / cache 元方法）放 ch7；本节聚焦**多模态特有**的工程问题。

**展开重点**：
- **延迟预算（latency budget）**：
  - 多模态调用普遍 1 秒级以上，需要明确预算分摊
  - 拆解：上传/下采样 → 模型 TTFT/TTFA → 生成完成 → 后处理
  - 语音对话 turn 预算（用户感知）：< 1 秒优秀、1–2 秒可接受、> 3 秒明显糟糕
- **成本结构（远比纯文本复杂）**：
  - 图像理解 / 生成：按图 / 按 token / 按分辨率
  - 视频理解：按秒 / 按帧 / 按 token；长视频几美元起
  - 视频生成：按秒，几秒至几十秒即美元级
  - 音频：按分钟（STT/TTS）或 token（Realtime）
  - **多模态 token 普遍贵 5–20×**，业务量起来后是首要成本项
- **流式 UI 与异步**：
  - 文本流式（SSE / WebSocket）熟门熟路
  - 图像 / 视频：**异步 + webhook + 进度反馈**；用户感知不焦虑
  - 语音：流式音频包；端点 / 打断
- **缓存策略**：
  - **内容 hash → 解析结果**（OCR、图像描述、视频摘要）—— 多模态最大省钱杠杆
  - prompt + image-hash → 生成结果（图像 / 视频生成的 idempotency）
  - CDN 缓存最终媒体资源
- **降级链 / 多模型路由**：
  - Premium 模型故障 → 同类备用（Veo 故障 → Kling）
  - 质量档降级（高分辨率 → 标清；Realtime → Pipeline）
  - 模态降级（视频不可生成 → 静图 + 文字摘要）
- **可观测**：
  - 多模态调用的 trace 要包含**输入摘要 / 输出哈希 / 模型 / 资源 ID**
  - latency 分位（P50/P90/P99）按模态分桶
  - 成本看板按"模态 × 模型 × 业务线"维度
- **失败模式**：
  - safety filter 拒绝（图像 / 视频生成常见）→ 需要可观测 + 反馈机制
  - 内容审核违规（用户上传素材）
  - 模型 deprecation（Sora 2 是 2026 头号案例）—— 需要版本告警
  - 延迟尾巴（长视频 / 复杂 prompt）
