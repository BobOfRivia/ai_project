# 6.5 Agent 评估（轨迹与工具）

> **核心问题**：Agent 评估为什么比 LLM 评估难？怎么评"轨迹"（trajectory）而不仅是"最终答案"？BFCL / GAIA / TRAJECT-Bench 怎么用？
> **目标**：能为一个 agent 系统建立**两层评估** —— 轨迹级（工具是否调对、是否高效、是否回环）+ outcome 级（最终目标是否达成），并理解"哪一层是 solved，哪一层仍未 solved"。
>
> **与 3.12 的关系**：3.12 在 agent 视角讲 eval & obs；本节从评估视角讲，互相 see-also。

**展开重点**：
- **Agent 评估为何难**：
  - 多步 + 非确定（planner / tool selector / retriever / LLM 重试）
  - 同一个目标可能有多条"对的轨迹"，传统精确匹配不适用
  - 失败有多种形态：调错工具、调对工具传错参、回环、超时、放弃
- **2026 共识：两层评估**：
  - **Step-level / trajectory tracing**（已解决）：每一步都能记录、回放、断言
  - **Outcome scoring**（未解决）：最终是否达到"领域专家会认可"的状态；不能从 trace 回放得出
- **轨迹级一等公民指标**：
  - **Tool-call accuracy**：调用了正确的工具吗？参数正确吗？
  - **Trajectory efficiency**：步数 / 成本 / 时间 是否合理？有无回环？
  - **Retrieval quality**：每次取数据是否相关（agent 内部 RAG）
  - **Plan quality**：planning 阶段的步骤拆分是否合理
  - **Loop detection**：同一动作反复执行
  - **Step-level latency**：定位慢的环节
- **Outcome 级指标**：
  - **Task success rate**：人定义的"任务完成"
  - **Goal-level judge**：让强模型从用户角度判"这次有没有解决问题"
  - **Cost-per-task**：跑一次平均花多少
- **公开 benchmark**：
  - **BFCL（Berkeley Function Calling Leaderboard）**：3 个 track —— AST（语法）+ Executable（真跑）+ Irrelevance detection；function calling 黄金 benchmark
  - **GAIA**：真实助手任务，多跳推理、工具、多模态；评 final exact answer
  - **TRAJECT-Bench**：轨迹感知的工具使用 benchmark
  - **AgentBench / SWE-bench / WebArena** 等垂域
- **框架与平台**：
  - **Inspect AI**：Anthropic 系，agent / capability eval 标杆
  - **LangSmith / Langfuse**：trace 原生，最适合轨迹分析
  - **Latitude**：从 agent session 为分析单位设计
  - **DeepEval / Confident AI**：带轨迹评估指标
- **可观测要点**：
  - trace 必须分 step（OTel GenAI `gen_ai.agent.*` + `gen_ai.tool.*`）
  - 输入 / 输出 / 工具调用要可重放
  - PII / 数据合规：生产 trace 默认脱敏，仅 sample 完整保留
