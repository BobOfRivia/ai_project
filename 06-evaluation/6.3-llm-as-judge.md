# 6.3 LLM-as-Judge 深入与偏差治理

> **核心问题**：LLM-as-Judge 在 2026 已知 5 大偏差怎么治？G-Eval / pairwise / 评分式哪种更稳？什么样的 judge 算"对齐了人评"？
> **目标**：能搭一套"可信的 judge 系统" —— prompt 设计、偏差缓解、跨家族交叉、月度人评对齐，让 judge 结果可作为生产决策依据。

**展开重点**：
- **5 大已知偏差**（2026 全行业公认）：
  - **Position bias**：成对比较里 judge 偏向前 / 后位；**GPT-4 上 ~40% 不一致**；缓解：(A,B) 和 (B,A) 双跑、只算两次一致
  - **Verbosity bias**：长答案更易得高分；**~15% 膨胀**；缓解：1–4 量表 + 显式奖励 conciseness、prompt 里讲清不要因长惩短
  - **Self-preference / family bias**：judge 偏好同家族模型；缓解：**跨家族 judge**（用 Claude 评 GPT 输出，反之亦然）
  - **Format bias**：结构化 / Markdown 答案被高估；缓解：把 format 评分单列、内容分纯文本
  - **Calibration drift**：judge 模型版本升级后分数漂移；缓解：每次升级 judge 跑历史 benchmark 看是否一致
- **三种 judge 模式**：
  - **Direct scoring**（1–5 / 1–10）：简单、可累加，但 calibration 难；适合大规模 sweep
  - **Pairwise comparison**：稳定性高于评分；但 N 例需 O(N²) 或 tournament；位置偏差必须治
  - **Rubric / chain-of-thought（G-Eval 风格）**：让 judge 先 reasoning 再打分；提高与人评相关性；贵
- **G-Eval 框架要点**：
  - 提供任务定义 → 评分准则 → 让 judge 生成 CoT → 输出分数
  - 论文显示与人评 Spearman 相关性显著高于传统 reference-based
- **人评对齐 SOP（2026 标配）**：
  - 至少 50–100 例标注集
  - 月度跑：judge vs human 的相关系数 / 一致率
  - judge 更换或升级 = eval-suite 迁移，不是 config 变更
- **prompt 设计要点**：
  - 角色 + 任务 + 评分准则 + 输出格式 + 一致性要求
  - 明确说"不要因长惩短"、"不要因 Markdown 偏好"
  - 给少量示例（few-shot）显著提分
  - 让 judge 输出 reasoning，便于人 review
- **成本 / 选型**：
  - 强 judge（GPT-5 / Claude Opus）贵但稳；用于关键节点
  - 中等 judge（Sonnet / Haiku / GPT-5-mini）做大量 sweep
  - 自家小模型做 judge —— 仅在已与人评对齐、稳定后
