# 3.10（深拆二）从零实现一个 Computer / Browser Use Agent：技术细节

>
> **配套阅读**：路线与选型总纲见 [3.10 主文件](./3.10-computer-browser-use.md)；市面头部产品拆解见 [3.10 产品深拆](./3.10-products-deep-dive.md)。

一个可靠的 Browser/Computer Use Agent，由**九个构件**拼成。下面逐块讲清"做什么、怎么做、有哪些坑与取舍"。最后给一份把它们缝在一起的完整参考骨架。

```
①界面表征 → ②Grounding定位 → ③动作空间 → ④Agent Loop与步数预算
     ↑                                              ↓
⑨评估可观测 ← ⑧安全内建 ← ⑦运行时沙箱 ← ⑥错误恢复 ← ⑤上下文工程
```

## 构件 1：界面表征层——模型"看"什么

**这是最重要的架构决策**，直接决定可靠性/成本的上限。四选一（或混合）：

| 表征 | 怎么拿 | 优点 | 缺点 | 何时用 |
|---|---|---|---|---|
| **无障碍树（AX Tree）** | CDP / 浏览器 API | 去噪、带语义角色、省 token、定位准 | 依赖网页无障碍实现质量；桌面/canvas 无 | **网页首选** |
| **DOM 快照** | Playwright/CDP | 信息全 | 太吵（上万节点、样式脚本噪音）、烧 token | 无障碍树拿不到时的退路 |
| **截图（视觉）** | 屏幕/页面截图 | 通用（任何画面） | 慢、贵、定位靠坐标易偏 | 结构够不着（canvas/图片/桌面） |
| **混合** | 结构为主 + 截图为辅 | 兼顾定位准与视觉判断 | 实现复杂 | **强方案的常见选择** |

**混合是很多生产方案的实际形态**：用无障碍树定位元素（准），遇到"这个图标是什么、布局对不对"这类视觉判断时才补一张截图。

> **决策要点**：网页项目默认从**无障碍树**起步；只有当目标是 canvas 应用、图片驱动 UI、桌面软件、或反爬盾挡住结构解析时，才引入截图/视觉。**别一上来就用视觉**——那是把可靠性和成本主动往下拉。

## 构件 2：Grounding——把"意图"落到"具体元素/坐标"

Agent 决定"点登录按钮"，怎么变成一次真实点击？这一步叫 **grounding（定位）**，是**视觉路线错误的最大来源**，也是最值得投入的地方。

### 三种 grounding 方式

1. **结构 grounding（选择器/节点 id）**——结构路线用。直接拿无障碍节点 id 或 CSS/XPath 选择器定位。**确定、稳**，是可靠性最高的方式。
2. **坐标 grounding（模型直接输出 `[x,y]`）**——纯视觉路线用。模型看截图、吐坐标。**易偏移、受分辨率影响大**，长流程里点错是常态。
3. **Set-of-Marks（SoM）标记法**——视觉路线的关键增强，见下。

### Set-of-Marks（SoM）：把"猜坐标"变成"选编号"

**做法**：在把截图喂给模型前，**先在图上给每个可交互元素叠加编号标签**（框 + 数字），然后让模型只需回答"点 ⑦ 号"，而不是猜 `[x,y]`。执行时你用编号反查该元素的真实坐标/选择器去点。

- **为什么有效**：把开放的坐标回归问题，变成封闭的分类选择问题——模型准得多。SoM 最早（Yang et al., 2023）就是为解锁 GPT-4V 的视觉 grounding 提出的，此后被 GUI Agent 广泛采用。
- **怎么拿到"可交互元素"来打标**：通常来自无障碍树或 DOM（所以 SoM 本质是**视觉 + 结构的融合**）。
- **局限**：依赖能拿到元素信息/分割；无障碍树质量参差（取决于站点开发者），纯 canvas 场景打不了标。

### 专用 grounding 模型（进阶）

2025–2026 的研究方向是**训练专门的 grounding 模型/视觉基础模型**（如 UGround、Aria-UI、Magma 等）来做"给定指令 → 定位可点区域 + 预测原子动作"，用强化学习、注意力图迭代精炼等提升定位精度。**落地建议**：绝大多数团队用"无障碍树选择器 + SoM"就够了；只有当你的场景高度视觉化（游戏、设计工具、无无障碍支持的老应用）且量大到值得投入时，才考虑专用 grounding 模型。

## 构件 3：动作空间设计（action space）

给模型一套**固定、正交、原子**的动作词汇，别让它自由造动作。

- **典型动作集**：`click / type / scroll / key / navigate / go_back / hover / drag_and_drop / wait / extract`。参考 Gemini 的 13 动作或 Anthropic 的键鼠词汇。
- **粒度权衡**：
  - 太细（逐像素移动）→ 步数爆炸、慢、易错；
  - 太粗（"一句话填完整个表单"）→ 不可控、难调试。
  - 无障碍树路线可给较**高层**的语义动作（`act("填写登录表单")`，内部拆细）；视觉路线通常较**低层**（逐个点击输入）。
- **必备两个动作**：
  - **显式 `wait`**：页面异步加载是头号坑，模型要能"等一下再看"；
  - **"确认/验证"动作**：让模型能主动截图核对上一步是否成功（配合构件 4 的每步自检）。
- **并行动作**：安全时允许一轮提议多个原子动作（Gemini 支持 parallel function calling），省往返、降延迟——但**只对无副作用或明确独立的动作开**，有依赖/有副作用的动作必须串行。

## 构件 4：Agent Loop 与步数预算

内核是 3.6 讲的 ReAct 循环，落到 computer use 有几个必做项。

### 循环骨架

```python
def agent_loop(task, max_iterations=30):
    context = init_context(task)                 # 系统prompt(保持静态!) + 任务
    for step in range(max_iterations):           # ① 必设步数上限，防死循环烧钱
        observation = capture()                  # ② 观察：无障碍树 / (+截图，可打SoM标)
        context.append(observation)
        decision = model.decide(context)         # ③ 决策：交错思考→选动作(可并行)
        if decision.is_done:                      #    终止：模型判定完成
            return decision.result
        results = execute(decision.actions)      # ④ 执行：在沙箱/浏览器里做动作
        context.append(results)                  # ⑤ 回灌结果(含成功/失败)，形成闭环
        # ⑥ 每步自检：done就截图核对(见下)
    raise StepBudgetExceeded                      # 触及上限：升级求助，别无限重试
```

### 四个必做项

1. **必设 `max_iterations`**（官方示例默认 10 量级，复杂任务 30+）——防死循环导致意外账单。
2. **每步回灌观察**形成闭环——这就是 ReAct 的"观察"，让模型基于**真实的**新状态决策，而非想象。
3. **强制每步自检**（Anthropic 官方 prompt，能显著提升可靠性）：
   > "After each step, take a screenshot and carefully evaluate if you achieved the right outcome. Explicitly show your thinking... Only when you confirm a step was correct should you move on."
   
   因为模型**常假设动作成功而不核对**——强制"做完就验"能打断误差累积。
4. **交错思考（interleaved thinking）**：让模型在动作之间思考（Think→Act→Think），比开头一次性规划全流程更稳，尤其长任务（3.6）。

### 步数预算的经济学
每步 = 一次多模态模型调用 + 一次浏览器动作（视觉路线每步 2–5 秒）。步数直接决定延迟和成本。**降步数的手段**：结构路线（天然步数少）、`extract` 一次拿全（别一步步点）、并行动作、把稳定子流程固化成脚本。

## 构件 5：上下文工程（长时运行 Agent 的命门）

这是长任务能否跑通的**决定性构件**，也是 Manus《Context Engineering for AI Agents》六条经验的精华。一个 computer-use 任务动辄 50 次工具调用，上下文管理不好，要么烧钱、要么"鬼打墙"、要么忘了目标。逐条讲透。

### 5.1 KV-Cache 命中率——"生产级 Agent 最重要的单一指标"

Agent 不断把"动作+观察"追加进上下文，每轮都要重新处理。**KV-cache 复用相同前缀，能把延迟和成本砍一个数量级**（Claude Sonnet：缓存 token $0.30 vs 未缓存 $3 每 MTok，**10 倍差**）。

**什么会毁掉缓存**（务必避免）：
- **前缀不稳定**：系统 prompt 里放精确到秒的时间戳——**一个 token 变化就让整段缓存失效**；
- **非确定性序列化**：JSON key 顺序每次不一样；
- **中途增删工具定义**：让前缀变了、还可能让历史引用到"已不存在的工具"。

**保命做法**：系统 prompt **完全静态**、上下文**只追加（append-only）**、序列化**确定性**（key 排序固定）、框架层开 prefix caching（如 vLLM）、用 session id 把请求路由到同一 worker。

### 5.2 Mask，别删——用 logit 掩码管理工具可用性

工具变多会让模型选错，但**中途删工具会毁缓存、还制造"历史引用不存在工具"的矛盾**。Manus 的做法：**不改工具定义，而在解码时对 token logits 做掩码**，用一个**上下文感知的状态机**控制当前哪些动作可选：
- 三种约束模式：**Auto**（自由选）/ **Required**（必须调某工具）/ **Specified**（只能从子集选）；
- 给工具名设计**一致前缀**（`browser_` / `shell_`），方便按状态整组屏蔽。

效果：既控制了动作选择，又保住了稳定上下文与缓存。

### 5.3 文件系统即记忆——无限、持久的"外部上下文"

网页、PDF 等观察动辄爆窗口，激进压缩又丢信息。Manus 把**文件系统当成终极上下文：容量无限、天然持久**：
- 大观察（整页 HTML、长文档）**存成文件**，上下文里只留**可恢复的引用**（URL、路径）；
- 模型学会**按需读写文件**；
- 压缩要**可恢复**（例："丢掉页面正文但保留 URL"，需要时再拉回），而非不可逆截断。

这与 3.5 Memory、3.4 Context Engineering 是同一套思想：**上下文是 RAM，文件系统是磁盘**。

### 5.4 Recitation（复诵）——用 todo.md 把目标拉回注意力

长循环（~50 步）里模型容易"忘了最初要干嘛"（lost-in-the-middle）。Manus 的招：**持续重写一个 `todo.md`，勾掉已完成项**——**把全局目标不断挪到上下文最近端**，用自然语言复诵，不改架构就把注意力偏置回任务目标。极其实用，任何长任务 Agent 都该抄。

### 5.5 保留错误——别把失败从上下文里擦掉

出错时的本能是"重试/重置/清掉错误痕迹"。但 Manus 反其道：**把失败的动作和它引发的报错/堆栈留在上下文里**。因为"**擦掉失败就擦掉了证据，没有证据模型无法适应**"——留着错误，模型会隐式更新信念、避免重犯。**错误恢复能力才是真正 agentic 的标志**，不是只看成功率。（这与 3.9 "失败可读"、3.10 每步自检一脉相承。）

### 5.6 避免 Few-Shot 窠臼——给上下文加一点结构化噪音

上下文里若全是整齐划一的历史，模型会**机械模仿这个节奏**，批量任务里陷入重复、幻觉（例：简历审查 Agent 无视内容差异、按惯性重复同样动作）。破法：**在动作/观察里引入少量结构化变化**——不同的序列化模板、换个措辞、加一点微噪音，打破模式模仿、提升鲁棒性。

> **一句话总纲**：computer-use Agent 的可靠性，一半在"看得准"（构件 1-2），一半在"记得住、不跑偏、能从错误里学"（构件 5）。上下文工程不是锦上添花，是长任务的生死线。

## 构件 6：错误恢复与自愈

误差沿长流程累积（单步 95%、20 步连乘只剩 36%），所以恢复机制是可靠性的核心。

- **每步验证**（构件 4 的自检）：做完就核对，早发现早纠正；
- **失败重试 + 换策略**：同一按钮点不到，换定位方式（选择器→SoM→坐标）、先滚动到可见、等加载完；
- **自愈选择器（self-healing）**：页面小改动导致选择器失效时，用无障碍语义/多重定位线索回退匹配，而非硬崩；
- **卡住就升级求助**（human-in-the-loop，3.9）：设"连续 N 步无进展"熔断，交还人类，别无限重试烧钱；
- **对照六类失败模式**逐一设兜底：
  1. **定位错**（点错元素）→ SoM + 结构定位；
  2. **幻觉动作**（做了没意义的操作）→ 每步自检 + 明确动作空间；
  3. **异步竞态**（页面还没加载就操作）→ 显式 wait + 状态检查；
  4. **状态漂移**（以为在 A 页其实在 B 页）→ 每步回灌真实观察 + 校验 URL；
  5. **注入劫持**（执行了页面里的恶意指令）→ 构件 8 安全；
  6. **无限循环**（反复试同一失败动作）→ 步数上限 + 保留错误 + 熔断升级。

## 构件 7：运行时与沙箱

Agent 逻辑之外，真正"跑浏览器/桌面"的那层。

### 自托管 vs 托管

| | 自托管 Chromium/VM | 托管运行时（Browserbase 等） |
|---|---|---|
| 成本 | 便宜（自己的机器） | 按分钟计费 |
| 运维 | 自己扛反爬、验证码、代理、扩容、录制 | 内建这些 |
| 适合 | 量小、可控环境 | 量大、要反爬/代理/录制 |

**经验阈值：超过约 100 浏览器小时/月，自托管的运维成本通常就该换托管了。**

### 沙箱隔离（安全 + 稳定的基础）
- Anthropic 参考实现：**Docker 容器** + 虚拟显示器（Xvfb）+ 端口映射；
- Manus：**每会话一个隔离的 Linux 沙箱**；
- Atlas：**out-of-process Chromium**（进程级隔离，见产品深拆）。

隔离的意义是双重的：**稳定**（浏览器崩了不拖垮主程序）和**安全**（把 Agent 与你的真实系统/数据隔开，限制被劫持后的破坏半径）。

## 构件 8：安全内建——不是事后补的补丁

**铁律：永远假设 Agent 会被页面内容劫持（间接提示注入）。** 安全必须是**架构约束**，从一开始就内建。多层防御（可直接抄）：

1. **模型层**：用训练过抗注入的模型；
2. **裁决层**：
   - 学 Anthropic——**注入分类器**扫描截图，疑似注入就停下请求确认；
   - 学 Gemini——**把 `safety_decision` 做进每步返回值**（`allowed / require_confirmation / blocked`），审批不再是外挂而是每个动作自带的元数据；
3. **环境层**：**沙箱隔离** + **最小权限**（别给不必要的账号权限）+ **凭证隔离**（登录信息用 `<robot_credentials>` 之类隔离，且清楚这会放大注入风险）；架构上可学 Atlas 的 **out-of-process 隔离**、输入事件直达 renderer 不经特权层；
4. **动作边界**：禁高危动作（执行代码、下载文件、装扩展、访问文件系统）、**敏感站点（金融等）暂停交还人类**、**会话隔离**（结束丢弃 cookie/数据）；
5. **人机层**：不可逆/高风险动作**强制人工审批**（3.9 review），可逆性兜底。

> **教训**：即便架构做到 out-of-process（Atlas），仍出现过"Tainted Memories"——恶意页面把隐藏指令注入到**记忆系统**并跨会话持续。所以**记忆写入也要审查**，别让 Agent 无条件把页面内容写进长期记忆（呼应 3.5 记忆投毒 memory poisoning）。完整攻击链与四层护栏见 **3.11**。

## 构件 9：评估与可观测

不能评估的 Agent 无法上生产。

- **核心指标是 `cost per completed task`（每完成一个任务的成本），不是每 token 价格**——失败重试、多余步数才是成本大头；
- **轨迹回放**：完整记录每步（观察→思考→动作→结果），出问题能回放定位（也是 3.9 show-correct-resume 的底座）；
- **用你自己的任务集自测**：WebVoyager/WebArena/OSWorld 只建立量级直觉，**不代表你的场景**；建一个覆盖你真实流程的评测集，才是选型和回归的依据；
- **监控六类失败模式的分布**：知道错在哪一类，才知道该补哪个构件；
- 详细的 Agent 评估与可观测方法论见 **3.12**。

## 完整参考骨架（把九个构件缝起来）

```python
class BrowserAgent:
    def __init__(self):
        self.system_prompt = STATIC_SYSTEM_PROMPT      # 构件5: 保持静态,护住KV-cache
        self.tools = ALL_TOOLS                          # 构件3: 固定动作空间(前缀命名)
        self.fs = FileStore()                           # 构件5.3: 文件系统即记忆
        self.runtime = SandboxedBrowser()               # 构件7: 沙箱运行时
        self.safety = SafetyService()                   # 构件8: per-step裁决

    def run(self, task, max_iters=30):                  # 构件4: 步数预算
        ctx = Context(self.system_prompt, task)
        ctx.append(TodoMd(task))                        # 构件5.4: recitation
        for step in range(max_iters):
            obs = self.runtime.observe()                # 构件1: 无障碍树(+截图打SoM标)
            obs = mark_interactive_elements(obs)        # 构件2: Set-of-Marks
            ctx.append_recoverable(obs, self.fs)        # 构件5.3: 大观察落文件,留引用
            mask = self.state_machine.allowed_logits()  # 构件5.2: mask不删tool
            decision = model.decide(ctx, logit_mask=mask)  # 构件4: 交错思考+选动作
            if decision.done: return decision.result
            for action in decision.actions:             # 构件3: 可并行(仅无副作用)
                verdict = self.safety.check(action)     # 构件8: allowed/confirm/blocked
                if verdict == "require_confirmation":
                    if not human_approve(action): continue   # 构件8/3.9: 审批
                if verdict == "blocked": continue
                result = self.runtime.execute(action)
                ctx.append(result)                      # 构件5.5: 成功失败都留,别擦错误
            ctx.append(self_check_screenshot())         # 构件4: 每步自检
            if no_progress(step): escalate_to_human()   # 构件6: 熔断升级
        raise StepBudgetExceeded                        # 构件6: 别无限重试
```

---

## 小结：九个构件的优先级

如果时间有限，按这个顺序投入：

1. **界面表征选对**（构件 1，无障碍树优先）——地基，选错全盘皆输；
2. **上下文工程**（构件 5，尤其 KV-cache + 保留错误 + todo.md）——长任务生死线；
3. **每步自检 + 步数预算**（构件 4）——打断误差累积；
4. **Grounding**（构件 2，SoM）——视觉路线的可靠性命门；
5. **安全内建**（构件 8）——上生产前必须，别等出事；
6. 其余（动作空间/恢复/运行时/评估）随规模补齐。
