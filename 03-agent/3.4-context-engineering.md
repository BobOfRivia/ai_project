# 3.4 Context Engineering

> **核心问题**：Agent 的推理质量完全取决于它"看到了什么"——如何在有限的 token 预算内，把最有价值的信息放进 context，把噪音挡在外面？
>
> **目标**：
> - 理解 Context Engineering 的本质：为什么不是"塞得越多越好"
> - 掌握 5 层上下文结构模型与 token 预算分配框架
> - 能在长时运行 Agent 中管理 context 衰减（context rot）
> - 掌握压缩、记笔记、动态检索三类策略的适用场景与工程实现

---

## 3.4.1 什么是 Context Engineering

**一句话定义**：Context Engineering 是"在推理时，策划和维护最优 token 集合"的系统化方法。

它是 Prompt Engineering 的自然进化，但解决的是不同维度的问题：

| | Prompt Engineering | Context Engineering |
|---|---|---|
| **关注点** | 如何"问"（语言形式） | 放什么、压什么、何时取、何时丢 |
| **作用时机** | 写 prompt 时（静态） | 每次推理前（动态、循环） |
| **适用范围** | 单轮或短对话 | 多步 Agent、长时运行任务 |
| **核心约束** | 表达清晰 | token 预算有限 |

Gartner 在 2025 年宣告"Prompt Engineering out，Context Engineering in"，不是说 prompt 不重要，而是在 agent 场景下，光写好提示词远远不够——你还需要系统性地管理模型在每一步推理时能看到什么信息。

**最核心的认知转变**：

> 不要把 context window 当成"存储空间"，要把它当成"稀缺的注意力资源"。

同样的信息，放在 context 里不同的位置、不同的密度，对模型推理质量的影响差异可以超过 30%。

---

## 3.4.2 Context 的本质：有限且衰减的注意力

在动手优化之前，必须先理解两个会持续影响工程决策的物理现实。

### 现实一：Context Rot（上下文腐烂）

Context Rot 是指随着 context 窗口填充，模型推理质量系统性下降的现象。

Chroma 在 2025 年对 18 个主流模型做了测试，发现**每个模型的准确率都随 context 长度增长而下降**——没有例外。更关键的是：

> **有效利用率只有标称窗口的 60-70%**。
> 一个"200K token 上下文"的模型，实际可靠工作范围大约是 120K-140K token。

这意味着"买了 200K 窗口就能用满"是一个幻觉。超过有效容量后，模型不是报错，而是悄悄地开始忽略信息、产生幻觉。

**根本原因（了解机制，帮助决策）**：

Transformer 的注意力机制中，每个 token 需要与其他所有 token 建立关联。100K token 的 context 意味着约 100 亿对 token 关系。随着序列变长，信号-噪声比下降——有用信息被大量无关 token "稀释"。RoPE 位置编码的长程衰减特性进一步加剧这个问题：远距离 token 之间的注意力权重会系统性降低。

**量化影响**：

一个有实际工程价值的数据：Context 结构问题导致 GPT-4 的准确率从 98.1% 直接跌到 64.1%。这不是模型能力问题，是工程问题。

### 现实二：Lost-in-the-Middle 效应

即使在 context 窗口容量之内，信息放在哪里也至关重要。

研究在多个模型（GPT-3.5、GPT-4、Claude 等）上复现了相同现象：

```
准确率随信息位置的变化：

高 ▲
   │  ██                              ██
   │  ██                              ██
   │  ██        ↓↓↓                   ██
   │  ██      中间位置准确率            ██
   │          下降 30%+               ██
   │
低 └────────────────────────────────────►
   开头                              结尾
         ← Context 窗口 →
```

**实践含义**：

- 最重要的信息放**开头**（系统提示、当前任务）
- 其次重要的信息放**结尾**（最新的工具结果、当前用户消息）
- 中间段是"注意力黑洞"——长的历史记录、冗余的工具输出塞在中间，会被模型部分忽略

这两个现实决定了 Context Engineering 的所有策略方向：**不是塞更多，而是放更好。**

## 3.4.3 五层上下文结构模型

一个 Agent 在每次推理时的 context，由 5 层叠加组成。理解每层的特性，才能做好整体预算分配和管理。

```
┌─────────────────────────────────────────────────────┐
│  Layer 5: 外部检索内容（动态，每步按需加载）          │
│  工具结果 / RAG 片段 / 文件内容 / API 响应            │
├─────────────────────────────────────────────────────┤
│  Layer 4: 消息历史（动态，随对话增长）               │
│  过去的用户消息 / 模型回复 / 工具调用记录             │
├─────────────────────────────────────────────────────┤
│  Layer 3: 示例（半静态，任务启动时设置）             │
│  Few-shot 示例 / 期望的输出格式演示                  │
├─────────────────────────────────────────────────────┤
│  Layer 2: 工具定义（半静态，支持动态加载）            │
│  工具 schema / 参数描述 / 使用说明                   │
├─────────────────────────────────────────────────────┤
│  Layer 1: 系统提示（静态，会话固定）                 │
│  角色定义 / 行为约束 / 输出格式 / 全局原则            │
└─────────────────────────────────────────────────────┘
```

### Layer 1：系统提示（System Prompt）

系统提示是 Agent 行为的"宪法"——会话开始时加载，全程存在。

**最常犯的错误**：把系统提示当"什么都写"的地方，结果越来越臃肿。

Anthropic 的工程建议：找到"恰当的高度"——

```python
# 不好：过度编码硬性逻辑
"""
如果用户问的是关于订单的问题，先查询 order_db，
如果返回状态是 PENDING，回复"正在处理"，
如果是 SHIPPED，回复"已发货"...
"""
# 问题：脆性大，新情况一出现就失控

# 不好：过于模糊
"""
你是一个有帮助的客服助手，尽力帮助用户。
"""
# 问题：没给 Agent 足够的行为约束

# 好：在"足够具体"和"足够灵活"之间找平衡
"""
## 角色
你是 XX 公司的客服 Agent，处理订单查询和退换货请求。

## 核心原则
- 用工具查询后再回答，不要凭猜测回复
- 涉及退款超过 500 元时，需要转人工确认
- 用简洁友好的语言，每次回复不超过 150 字

## 工具使用指南
- 查订单状态：用 search_order(order_id)
- 发起退款：用 create_refund(order_id, reason)，仅在用户明确要求时使用
"""
```

**系统提示的推荐结构**（按优先级排列）：

| 位置 | 内容 | 说明 |
|---|---|---|
| 最开头 | 角色定义 + 核心使命 | 最重要，放开头确保被充分注意 |
| 中间 | 行为原则 + 约束 | 具体但不要过度枚举 |
| 靠后 | 工具使用指南 | 紧接着工具定义 |
| 最后 | 输出格式要求 | 离最终生成最近 |

### Layer 2：工具定义（Tools）

工具定义的 token 消耗常被低估。一个工具的完整 schema（名称 + 描述 + 参数说明）通常需要 200-500 token。如果有 30 个工具，光工具定义就消耗 6,000-15,000 token——这还没算任何对话内容。

**工具定义在 context 里的工程目标**：

- 只加载当前步骤相关的工具（动态加载，见 3.2.5）
- 每个工具描述尽量精简，但不能以牺牲准确性为代价
- 工具描述要与系统提示配套，不要重复写相同的行为约束

### Layer 3：示例（Few-shot Examples）

示例是"一图胜千言"——与其用 300 字解释期望的输出格式，不如给两个示例，通常效果更好且 token 更省。

**选例原则**：

- 选**多样化的规范示例**，覆盖不同类型的典型情况
- **不要**穷举所有边界情况——边界情况应该在系统提示的原则层面处理，不是靠示例穷举
- 3-5 个示例通常已经足够；超过 8 个边际收益迅速下降

**示例的 token 性价比**：Few-shot 示例每个消耗 100-500 token，但能节省系统提示里大量的规则说明，整体通常是 token 正收益的。

### Layer 4：消息历史（Message History）

消息历史是 context 里**唯一会无限增长**的部分。如果不主动管理，一个长对话的历史记录会挤占其他层的空间，导致工具定义被截断、检索结果放不下。

**消息历史的管理策略见 3.4.5。**

### Layer 5：外部检索内容（External Data）

每次工具调用的返回结果、RAG 检索到的文档片段、读取的文件内容……这些是 context 里**最大的单次 token 消耗**来源。

一个设计不好的工具可能单次返回 10,000+ token 的原始 API 响应（相关内容可能只有 200 token）。

**工程原则**：这层的内容在进入 context 之前必须过滤。在工具层过滤（见 3.2.6），而不是把过滤任务留给模型。

## 3.4.4 Token 预算分配框架

有了 5 层模型，下一个问题是：各层各分多少？

### 基础分配比例

来自生产实践的参考分配（基于主流模型的有效容量，而非标称容量）：

| 层 | 推荐比例 | 说明 |
|---|---|---|
| **系统提示** | 10-15% | 静态，可以精确控制 |
| **工具定义** | 15-20% | 动态加载可以大幅降低 |
| **检索内容** | 30-40% | 最大的单次消耗，需要精细过滤 |
| **消息历史** | 20-30% | 需要主动压缩管理 |
| **输出缓冲** | 10-15% | 为模型输出预留，不能压缩 |

**以 200K token 窗口为例的具体数字**（按 60-70% 有效利用率，约 130K 有效 token）：

| 层 | token 数 |
|---|---|
| 系统提示 | ~15,000 |
| 工具定义 | ~20,000 |
| 检索内容 | ~50,000 |
| 消息历史 | ~30,000 |
| 输出缓冲 | ~15,000 |

### 按 Agent 类型调整

不同类型的 Agent 对各层的需求差异显著：

| Agent 类型 | 系统提示 | 工具 | 检索 | 历史 | 缓冲 |
|---|---|---|---|---|---|
| **编程 Agent** | 10% | 20% | 45% | 15% | 10% |
| **客服 Agent** | 12% | 12% | 25% | 40% | 11% |
| **研究 Agent** | 10% | 25% | 35% | 20% | 10% |
| **单轮摘要** | 10% | 0% | 70% | 0% | 20% |

- 编程 Agent 需要大量代码上下文（检索比例高）
- 客服 Agent 需要保留长对话历史（历史比例高）
- 研究 Agent 需要大量工具组合（工具比例高）

### 五条花钱纪律

知道了预算分配，还需要知道怎么"省钱"：

**1. 缓存静态组件**

系统提示和工具定义在一次会话里几乎不变。使用 **Prompt Cache**（Anthropic、OpenAI 均支持），缓存 token 的成本是普通 token 的 1/10。对高频 Agent，系统提示缓存可以带来 50-70% 的成本降低。

**2. 历史压缩**

历史消息达到预算上限时，压缩旧轮次，保留最近 K 轮完整对话。目标压缩比：3:1 到 5:1。

**3. 按需检索**

不要在会话开始就把所有可能相关的文档塞进 context（pre-load），而是在每步推理前根据当前任务按需拉取（just-in-time）。

**4. 工具动态加载**

不要把所有工具定义都放进 context，按当前任务动态选择相关工具子集（见 3.2.5）。

**5. 预留输出缓冲**

对使用推理模型（o3、Claude-3.7 等）的 Agent，输出 token 可能很长。预留至少 15% 的缓冲，否则输出会被截断。

## 3.4.5 消息历史管理：四种策略

消息历史是 context 里最需要主动管理的部分，也是导致 context rot 最常见的原因。以下四种策略从简单到复杂，对应不同场景。

### 策略一：滑动窗口（Sliding Window）

**做法**：只保留最近 N 轮对话，超出的直接丢弃。

```python
def trim_history(messages: list, max_turns: int = 20) -> list:
    # 永远保留 system 消息
    system = [m for m in messages if m["role"] == "system"]
    non_system = [m for m in messages if m["role"] != "system"]
    # 保留最近 max_turns 轮的用户+助手消息
    return system + non_system[-max_turns * 2:]
```

**适用**：任务明确、短周期交互（如客服场景）。

**代价**：丢弃超出部分，可能丢失重要上下文。65% 的企业 AI 故障在 2025 年被归因于类似的**上下文漂移（Context Drift）**——每次压缩都可能悄悄丢弃低频但重要的细节，多次压缩后 Agent 只记得一个"消毒过的、通用化的"历史，在边界情况下失效。

**缓解**：将关键决策和约束单独存进结构化笔记（策略三），不依赖历史轮次。

### 策略二：Compaction（增量压缩）

**做法**：不丢弃历史，而是把旧的历史轮次压缩成摘要，保留摘要代替原文。

```
原始历史（2000 token）:
  轮次1: 用户问了 A，模型查了工具，回复了...（300 token）
  轮次2: 用户追问 B，模型分析...（400 token）
  ...

压缩后（400 token 摘要 + 最近 3 轮原文）:
  [摘要] 用户正在排查订单 #12345 的延误问题。
  已确认：物流已发货，预计明天到达。
  用户关心：能否提前取件。
  最近 3 轮原文...
```

**触发时机**：context 使用率超过 70% 时自动触发。

**压缩建议**：
- 保留：决策、结论、用户明确的需求和约束
- 丢弃：工具的原始输出（只保留结论）、模型的中间推理过程
- 保留最近 3-5 轮原文，不要全压缩

**工具输出的压缩比更激进**：工具返回结果通常可以 10:1-20:1 压缩（从原始 JSON 到语义摘要）。

**Anthropic 平台级支持**：Claude 平台提供了服务端的 Compaction API，在对话接近 context 上限时自动压缩较旧的轮次，对应用透明。如果你有严格可观测性要求（需要看到压缩了什么），选择客户端自行实现。

### 策略三：结构化笔记（Structured Note-taking）

**做法**：Agent 在执行过程中，把重要信息主动写进一个持久化的"笔记文件"，而不是依赖 context 里的历史对话来"记忆"。

这是 Anthropic 在研究 Agent（如 Claude 处理 Pokémon 游戏）中观察到的自发行为，后来被工程化成标准模式。

```
# agent_notes.md（Agent 维护，外部存储）

## 任务目标
- 分析竞品价格策略，生成报告

## 已完成
- [x] 爬取竞品 A 的价格数据（2026-07-13，1200 条）
- [x] 爬取竞品 B 的价格数据（2026-07-13，980 条）

## 进行中
- [ ] 竞品 C 的价格数据（已发起请求，等待返回）

## 关键发现
- 竞品 A 在周末有固定 15% 折扣
- 竞品 B 的高端SKU比我们便宜 8%（需要在报告中特别提示）

## 注意事项
- 竞品 C 的 robots.txt 不允许爬取，改用公开 API

## 文件位置
- 数据: /workspace/data/competitors/
- 草稿: /workspace/report_draft_v2.md
```

**工程实现**：Agent 在系统提示里被指定在每次工具调用后更新笔记，在每次新步骤开始时先读取笔记（作为当前 context 的一部分）。

**优势**：
- 信息持久化在上下文之外，不受 context 窗口限制
- 信息密度高（只写关键结论，不含噪音）
- 跨 session 持久（session 重启后，Agent 读笔记就能恢复状态）

**适用**：长时运行的任务（小时级别的编程 Agent、研究 Agent）。

### 策略四：子 Agent 隔离（Multi-Agent 上下文分治）

**做法**：把需要大量上下文探索的子任务交给独立的子 Agent，子 Agent 用几万 token 深度探索，**只把结论（1000-2000 token）**返回给主 Agent。

```
主 Agent context（精简）                  子 Agent context（深度）
┌────────────────────┐                  ┌─────────────────────────────┐
│ 系统提示            │                  │ 具体任务指令                 │
│ 高层任务目标        │                  │ 大量相关文件内容（~50K token）│
│ 子任务委托记录      │ ◄── 结论摘要 ──── │ 搜索结果                    │
│ 其他子任务结果      │                  │ 代码分析                    │
│ ~10K token         │                  │ ~60K token                  │
└────────────────────┘                  └─────────────────────────────┘
```

**适用**：研究型、分析型 Agent，需要并行探索多个子问题。

**来自 Anthropic 工程实践**：主 Agent 保持精简的协调上下文，子 Agent 做深度探索，这是他们在研究型 Agent 系统里发现的最有效的上下文分治方式。

### 四种策略的选型

| 场景 | 推荐策略 |
|---|---|
| 短会话、任务明确 | 滑动窗口 |
| 中等长度、需要全局上下文 | Compaction |
| 长时任务（小时级）、有明确里程碑 | 结构化笔记 |
| 复杂研究、需要并行探索 | 子 Agent 隔离 |
| 超长任务（天级） | 笔记 + 子 Agent 组合 |

## 3.4.6 动态检索：Just-in-Time Context

**旧范式**：在 Agent 启动时，把所有可能用到的信息都预加载进 context。

**新范式**：只存储轻量级的"索引标识符"，在每步推理前，按照当前任务的具体需求，通过工具动态拉取需要的内容。

这个模式 Anthropic 称为"Just-in-Time Context"，在 Claude Code 中有完整的工程实现：

```
旧范式（预加载）：
Agent 启动 ──► 加载所有代码文件（100K token）──► 每步推理带着全量 context

新范式（JIT）：
Agent 启动 ──► 只加载文件树结构（2K token）
   │
   │ 每步推理前
   ├──► 用 grep/find 定位相关文件（工具调用）
   ├──► 读取具体函数（工具调用，精确定位到所需内容）
   └──► 只把这一步需要的内容放进 context
```

**Claude Code 的混合策略**：

```
预加载部分（CLAUDE.md + 项目配置）：
  - 项目说明文档
  - 全局编码规范
  → 小体积、高频使用、适合预加载 + prompt cache

JIT 部分：
  - 具体文件内容（按需读取）
  - 搜索结果（按步骤执行）
  → 内容大、只在特定步骤需要，适合 JIT
```

**工程实现的关键**：

**1. 存标识符，不存内容**

```python
# 差：把所有搜索结果放进 context
search_results = search_web(query)
# 直接把 100 条结果全塞进 context

# 好：存标识符，按需展开
search_index = search_web_index(query)  # 返回 URL + 标题列表（轻量）
# 只在模型决定要看某个结果时，再调用 fetch_url(url) 获取全文
```

**2. 逐层探索（Progressive Disclosure）**

```
文件系统探索示例：

第 1 步：列出顶层目录（50 token）
  /src /tests /docs /config

第 2 步：列出 /src 下的模块（100 token）
  /src/auth /src/api /src/models

第 3 步：读取 /src/auth/__init__.py（500 token）
  # 找到需要的函数后停止展开
```

**不要一次性** `find . -name "*.py" | xargs cat`，这会把整个代码库丢进 context。要像人类工程师一样，先看目录结构，再精准定位。

## 3.4.7 系统提示工程：结构化让模型少犯错

系统提示是 context 里唯一能完全由工程师控制的层，做好它性价比极高。

### 高效结构模板

```markdown
## 角色与使命
你是 [描述角色]，负责 [核心使命]。

## 行为原则
- [最重要的约束 1]
- [最重要的约束 2]
- [何时需要人工确认]

## 工具使用
- [工具名]：[什么情况用，注意事项]
- [工具名]：[什么情况用，注意事项]

## 输出格式
[格式要求或示例]

## 不应做的事（可选，仅在有明确禁止项时）
- [禁止行为 1]
```

**Markdown 标题结构**非常有效：模型对标题格式（`##`、`###`）有较好的结构理解，比无结构的纯文本更容易遵循。

### 几个常见的低效写法

**低效一：规则清单过长**

```markdown
# 差：50 条规则，模型只能记住前几条
- 规则 1...
- 规则 2...
...
- 规则 50...

# 好：把规则归纳为原则，用示例补充
## 核心原则
查工具再回答，不要猜。超过 500 元的退款需要人工确认。

## 示例
用户："我的订单什么时候到？"
错误做法：根据一般配送时间估计告诉用户
正确做法：先调用 get_order_status(order_id)，根据实际状态回复
```

**低效二：把动态内容写死在系统提示里**

```python
# 差：把今天的日期写死在系统提示里
system_prompt = f"""今天是 {datetime.today()}，..."""

# 好：通过工具让模型按需获取动态信息
# 或者只在需要时把动态信息注入 context
```

**低效三：系统提示和工具描述重复**

系统提示写了"不要删除文件"，工具的 delete_file 描述里又写了"仅在用户明确要求时使用"——重复内容浪费 token，还可能造成指令冲突。约束放一个地方，另一个地方引用或省略。

## 3.4.8 Context 质量的量化评估

Context Engineering 的效果需要通过指标来衡量，而不是靠感觉。

### 三个核心指标

| 指标 | 计算方式 | 健康范围 | 异常信号 |
|---|---|---|---|
| **Context 利用率** | 已用 token / 有效容量 × 100% | 60-80% | <60% 过度配置；>80% 风险超限 |
| **信息密度** | 可提取唯一关键事实数 / 总 token | 越高越好 | 低密度说明 context 充斥噪音 |
| **压缩比** | 压缩前 / 压缩后 | 历史 3-5x；工具输出 10-20x | 压缩比太低说明过多冗余 |

### 自动化检测

```python
def check_context_health(messages: list, model_max: int) -> dict:
    used_tokens = count_tokens(messages)
    effective_max = model_max * 0.65  # 有效利用率上限

    utilization = used_tokens / effective_max

    return {
        "utilization": utilization,
        "warning": utilization > 0.8,  # 超过 80% 触发预警
        "suggestion": "compress_history" if utilization > 0.7 else None
    }
```

### 成本影响

用数字说明 context 管理的价值：

一个处理 10,000 次对话/天的客服 Agent：

| 方案 | 日均 context token | 年成本（Claude Sonnet 定价） |
|---|---|---|
| 无管理（历史无限增长） | ~35,000 / 对话 | **~$255,000** |
| 基础压缩（60% 减少） | ~14,000 / 对话 | **~$102,000** |
| 完整 CE 优化 + 缓存 | ~8,000 / 对话 | **~$58,000** |

Context Engineering 做好，年省 $150,000+——这还只是中等规模的单个 Agent。

## 小结

| 维度 | 核心要点 |
|---|---|
| **本质** | Context 是稀缺的注意力资源，不是存储空间 |
| **Context Rot** | 有效利用率只有标称窗口的 60-70%；信息位置影响准确率超 30% |
| **5 层结构** | 系统提示（静态） / 工具（半静态） / 示例 / 历史（动态增长） / 检索（按需） |
| **预算分配** | 系统提示 10-15% / 工具 15-20% / 检索 30-40% / 历史 20-30% / 缓冲 10-15% |
| **历史管理** | 四种策略：滑动窗口 / Compaction / 结构化笔记 / 子 Agent 隔离 |
| **JIT 检索** | 存标识符、逐层探索，不要在 Agent 启动时预加载全量内容 |
| **系统提示** | 结构化（Markdown 标题）、原则而非规则清单、不与工具描述重复 |
| **量化指标** | 利用率 60-80% 健康；信息密度越高越好；做好 CE 可节省 50-75% 成本 |

**下一步**：Context 里的"历史与记忆"需要跨会话持久化时，就进入了 3.5 Memory 系统的领域——从 Context（单次推理的工作记忆）到 Memory（跨会话的长期记忆）。
