# 2.6 重排（Reranking）

> **核心问题**：怎么把检索结果按相关性精排？
> **目标**：能用重排把 Top-K 候选精排到 Top-N，显著提升生成质量。

---

## 一、为什么需要重排：召回与排序解耦

2.4 的漏斗里，检索阶段"宁滥勿缺"地捞回 50–200 条——**保召回率，但排序不一定准**。答案可能确实被捞到了，却排在第 30 位，模型根本看不到。重排就是解决这个：**把召回的一大批候选，用更强的模型重新精排，把真正相关的顶上来。**

这是一个刻意的**两阶段设计（召回 + 排序解耦）**：

| 阶段 | 用什么 | 目标 | 特点 |
|---|---|---|---|
| **召回（2.4）** | Bi-encoder（向量）+ BM25 | 高召回率、快 | 在全库上跑，必须快，排序粗 |
| **重排（本节）** | Cross-encoder / LLM | 高精确率、准 | 只在候选集上跑，可以慢一点，排序精 |

为什么不直接用强模型在全库上排？**因为算不起**——强模型太慢，只能用在缩小后的候选集上。这就是"先粗召回、再精排序"的根本原因。

---

## 二、原理：Cross-encoder 为什么比向量检索更准

理解这一点，就理解了重排的价值来源：

- **Bi-encoder（召回用）**：query 和文档**分别**编码成两个向量，再算距离。文档向量可离线预存，所以快；但 query 和文档**从不"见面"**，交互信息丢失。
- **Cross-encoder（重排用）**：把 **query + 文档拼在一起**送进模型，让它们在每一层充分交互，直接输出一个相关性分数。**更准**，但每个 (query, 文档) 对都要现算一次，**没法预存，所以慢**——只能用在候选集上。

一句话：**Bi-encoder 快而粗（适合海选），Cross-encoder 准而慢（适合精排）**，两者天生互补。

---

## 三、收益与代价：值不值得上

**收益（硬数据）**：

- Cross-encoder 重排通常带来 **+5 ~ +15 NDCG@10**（MTEB/BEIR 基准），**词面难的数据集上能到 +20 以上**。
- 一个直观的对比：Weaviate 基准里，重排把推理型 query 的 NDCG@10 **从 0.13 拉到 0.40（约 3 倍）**——只是把同一批候选重新排序，没换任何检索。

**代价**：

- 重排通常**增加 <200ms 延迟**（top-100 → top-10 约 +120ms；50 篇约 1.5s 视模型而定）。

**结论**：**重排是 RAG 里性价比最高的单点增强之一**。同一批召回结果，仅靠重排序就能大幅提质，且延迟可控。**除非延迟预算极紧，否则生产 RAG 应默认加重排。**

## 四、主流 Reranker 模型选型（2026）

| 模型 | 形态 | 亮点 | 适合 |
|---|---|---|---|
| **Cohere Rerank 3.5 / 4** | 托管 API | 多语言强、长上下文；4 Pro 比 3.5 高 +170 ELO（金融/商业 +400），但慢约 60% | 要省心、要多语言、质量优先 |
| **BGE-reranker-v2-m3** | 开源(Apache 2.0) | **100+ 语言**，中文友好；CPU 约 350ms，GPU 降到约 80ms | 自托管主力，配 BGE-M3 embedding 一套 |
| **Jina Reranker v2** | API/开源 | 速度快、多语言 | 平衡型选择 |
| **ms-marco-MiniLM-L6-v2** | 开源 | **极轻量**：100 对约 50ms，+35% 精度 | 延迟敏感、资源有限 |
| **Voyage rerank-2** | API | 检索质量强（与其 embedding 配套） | 已用 Voyage 生态 |

> **2026 自托管黄金搭配**：**BGE-M3（embedding, 2.3）+ BGE-reranker-v2-m3（rerank）**。同源、多语言、Apache 2.0，中文场景尤其顺手。
> **要省心**：Cohere Rerank（托管、多语言、零基建）。
> **延迟第一**：MiniLM-L6-v2，速度/质量平衡最好。

---

## 五、LLM-as-Reranker：更强但更贵的选项

用大模型本身来判断相关性排序（listwise 让 LLM 直接对候选排序）。

- **优势**：能做**需要推理的相关性判断**——当"答案没被排上来，是因为相关性需要推理才能看出"时，LLM 重排可能是唯一能识别它的方法。比 cross-encoder 再高约 5–8%。
- **代价**：**额外加 4–6 秒延迟，且成本高得多**。
- **什么时候用**：**离线批处理、研究场景、或准确率远比延迟重要的专业领域**。
- ⚠️ **别用于实时交互场景**——4–6 秒对用户不可接受。

> 落地判断：**绝大多数生产场景，标准 bi-encoder + cross-encoder 两阶段就够，质量通常与 LLM 重排相当而简单得多**。LLM 重排是特殊需求下的选项，不是默认。

## 六、托管 vs 自托管

| | 托管（Cohere/Jina/Voyage） | 自托管（BGE/MiniLM） |
|---|---|---|
| 延迟 | 可预测、零基建 | 可控（需自己上 GPU） |
| 数据 | 要出域 | **数据不出域** |
| 成本 | 按调用付费 | 规模化后摊薄、更省 |
| 定制 | 有限 | 可领域微调 |

选型呼应 2.3：**数据敏感/大规模 → 自托管 BGE；要快速上线/省心 → 托管 Cohere。**

---

## 七、落地：标准两阶段管线与 K 值

把 2.4 的漏斗补全：

```
混合检索(2.4)         重排(本节)           送生成(2.7)
召回 top 50~100 ────▶ Cross-encoder 精排 ──▶ 取 top 5，实际送 3~5 条
(bi-encoder+BM25)     到 top 10             (质量优先，控噪声)
```

- **召回 K**：50–100（词面难的场景可到 200），优先召回率。
- **重排后 K**：精排到 top 10，最终送模型 3–5 条。
- **候选别给太多**：候选越多重排越慢；送模型越多噪声越大、越触发 lost-in-the-middle（2.7）。**够用即止。**
- K 的具体值靠 2.10 的评估调，不同数据集最优点不同。

---

## 八、本节小结：重排决策清单

1. **默认加重排**：性价比最高的单点增强（+5~15 NDCG@10，延迟 <200ms），除非延迟预算极紧。
2. **理解两阶段**：bi-encoder 海选（快粗）→ cross-encoder 精排（准慢），互补。
3. **选型**：自托管/中文 → **BGE-reranker-v2-m3**；省心/多语言 → Cohere；延迟第一 → MiniLM-L6-v2。
4. **LLM 重排按需**：只在推理型相关性、离线批处理、准确率压倒延迟时用；实时场景别碰（+4~6s）。
5. **控候选量**：召回 50–100 → 精排 top 10 → 送 3–5 条，够用即止。
6. **和 Contextual Retrieval 叠加**：2.2 提过，Contextual Retrieval + 重排可把检索失败率降到 -67%，两者是叠乘关系。

> 检索三节（2.4 召回 / 2.5 查询 / 2.6 重排）到此收口：**混合检索保召回、查询改写补角度、重排提精度**——这三板斧做扎实，就拿下了"检索是瓶颈"的主战场。
