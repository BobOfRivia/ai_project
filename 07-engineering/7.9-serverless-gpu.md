# 7.9 Serverless GPU 与冷启动治理

> **核心问题**：Modal / RunPod / Replicate / Bedrock Custom Model Import 2026 各占什么位置？冷启动怎么治？AWS Lambda 真的没 GPU 吗？
> **目标**：能根据"流量模式 + 模型大小 + 延迟需求"选 Serverless GPU 平台，并把冷启动从用户感知中隐藏掉。

**展开重点**：
- **2026 Serverless GPU 格局**：
  - **Modal**：**冷启动 <2s**（GPU memory snapshot 加持，部分模型 **10x 加速**）；最低托管 serverless 成本；Python 友好
  - **RunPod**：**最快上手**，社区 templates 多，按秒计费透明
  - **Replicate**：**模型市场最好**，Cog 容器，push 模型即跑，按秒 GPU 计费
  - **Bedrock Custom Model Import**：AWS 体系内最简便；**26B 模型冷启动几分钟**，需 **每 14 分钟 keepalive 探针**
- **AWS Lambda 状态（2026 重要事实）**：
  - **官方明确无 GPU 支持**、无 CUDA、无 GPU 计费、无公开 roadmap commit
  - 2025-09 Lambda serverless GPU 停服，AWS 改推**按需 GPU VM** 与 Bedrock
  - 想"Lambda 风格 LLM"在 AWS 里只能：Bedrock（云 API）/ SageMaker / 自己 EC2 + ASG
- **冷启动来源**（必懂）：
  - **容器拉取**：GB 级 image 下载（quay / ECR）
  - **模型权重 load**：从 S3 / 本地 → GPU 显存
  - **CUDA / 框架初始化**：vLLM / TensorRT 编译 kernels
  - **典型量级**：小模型（7B）几秒、26B 几十秒、70B+ 分钟级
- **冷启动治理策略**（2026 通用 playbook）：
  - **GPU Memory Snapshot**（Modal）：把"权重已 load"的 GPU 状态序列化，启动直接 mmap，**10x 缩短**
  - **预热**（warmup / keepalive）：定时 ping、保留 N 副本始终在线
  - **预测式扩缩**：流量曲线 + 提前调度
  - **Tiered 模型**：默认走小模型（永远在线）+ 必要时唤醒大模型
  - **请求级降级**：冷启动期间走 fallback（云 API）
- **选型决策树**：
  1. 想要极低冷启动 + Python 一行部署 → **Modal**
  2. 想跑社区模型 / 最快上手 → **RunPod**
  3. 想把自家模型作为 API 卖 → **Replicate**
  4. 已在 AWS 栈 → **Bedrock Custom Model Import**（接受冷启动 + keepalive）
  5. 不想动 K8s 但要长期高利用率 → 重新考虑自托管（→ 7.4）
- **成本对比直觉**：
  - Serverless GPU 单 token 成本通常 **比 cloud API 高**，但**比自建 + 低利用率自托管低**
  - 适配"流量 burst + 不稳定"工况，不适配"恒定高 QPS"
- **常见反模式**：
  - 把延迟敏感的用户路径放 serverless 冷启动上：用户感知到几秒"卡"
  - 不开 keepalive → 每次都冷启动
  - 把模型 artifact 打进 image → 每次更新都 GB 级 push
  - 在 Lambda 上"等 GPU 支持"——2026 内 AWS 没承诺过
