# 2.4 检索策略

> **核心问题**：怎么取出最相关的内容？
> **目标**：能搭出符合业务召回质量要求的检索层。

---

## 一、检索是整个 RAG 的胜负手

回顾 2.1 的核心认知：**RAG 失败约 73% 出在检索**。本节就是解决"怎么把对的内容取出来"。

检索按技术分三大类，理解它们的**互补盲区**是本节主线：

| 类型 | 匹配什么 | 代表 | 长处 | 盲区 |
|---|---|---|---|---|
| **稠密检索 Dense** | 语义相似 | 向量检索（2.3 的 embedding） | 懂同义、懂意图、跨措辞 | 精确词（专有名词、ID、版本号）易漏 |
| **稀疏检索 Sparse** | 关键词/词项 | BM25、SPLADE | 精确命中术语、代号、罕见词 | 不懂同义、换个说法就丢 |
| **混合检索 Hybrid** | 两者融合 | Dense + Sparse + RRF | **补齐彼此盲区** | 略增复杂度与延迟 |

一句话结论先给出：**2026 年生产 RAG 的检索层默认就是混合检索（Hybrid）**，不是可选项。下面拆开讲为什么。

---

## 二、稠密检索（Dense / Vector）

就是 2.3 建好的向量检索：把 query 也 embedding，在向量库里用 **ANN 找最近邻**（ANN 到底怎么工作、HNSW/IVF/PQ 索引参数怎么调，见 **[2.4.1 ANN 补充](2.4.1-ann-nearest-neighbor.md)**）。

- **强**：理解语义。"如何重置密码" 能命中 "找回登录凭据的方法"。
- **弱**：**对精确 token 不敏感**。用户搜 "错误码 E1234"、"iPhone 15 Pro"、某个函数名，稠密向量可能把它和一堆"语义相近但不是它"的东西混在一起。

---

## 三、稀疏检索（Sparse：BM25 / SPLADE）

基于词项匹配的传统信息检索，正好补稠密的短板。所谓"稀疏"是指：文本被表示成一个**维度=整个词表、绝大多数分量为 0** 的向量（只有出现/相关的词非零），天然适合**倒排索引**高效检索（倒排索引是什么、BM25 与 SPLADE 怎么接入它，见 3.3）。

### 3.1 BM25：怎么给"词项匹配"打分

BM25（Best Matching 25）是几十年验证的概率排序函数。看公式前，先破一个最容易卡住的误解：

> **BM25 是"逐篇打分函数"，不是一次算出全局排名。** 公式里的 $D$ 是**某一篇**文档——你对**每篇候选各算一次分**，再按分排序取 top-k。所有带 $D$ 的量（词频、长度）都是"当前这一篇"的。

#### 第一步：一次 BM25 查询怎么跑

用一个 3 篇文档的语料 + 查询"重置密码"走一遍：

```
doc1: 如何重置登录密码      doc2: 密码强度规则      doc3: 登录失败排查
```

1. **query 分词 + 归一化**：过**和建索引时同一套**分析器（分词 / 小写 / 去停用词 / 英文词干还原）。
   `"重置密码" → [重置, 密码]`
   ⚠️ 分析器必须和索引端一致，否则 query 的词和索引里的词对不上、直接查不到。
2. **查倒排索引筛候选**（见 3.3）：只捞出**至少含一个 query 词**的文档，其余不碰。
   `重置 → [doc1]`，`密码 → [doc1, doc2]` ⟹ **候选 = {doc1, doc2}**（doc3 一个词都不含，跳过）
3. **对每篇候选各算一次 BM25**（对 query 里每个词求和，$D$ = 当前这篇）：
   ```
   score(doc1) = IDF(重置)·TF项(重置,doc1) + IDF(密码)·TF项(密码,doc1)
   score(doc2) = IDF(密码)·TF项(密码,doc2)      ← doc2 没"重置"，那一项 = 0
   ```
4. **排序取 top-k**：doc1（两词都命中，且"重置"更罕见）> doc2 → 返回 `[doc1, doc2]`。

**所以"对全库每篇都算"是错觉**——倒排索引已经把候选缩到含词的少数文档，BM25 只在这批上逐篇打分。

#### 第二步：打分公式 = 三个直觉的组合

对**当前文档 $D$** 和查询 $Q=\{q_1,\dots,q_m\}$：

$$\text{score}(D,Q)=\sum_{i=1}^{m}\text{IDF}(q_i)\cdot\underbrace{\frac{f(q_i,D)\cdot(k_1+1)}{f(q_i,D)+k_1\cdot\big(1-b+b\cdot\frac{|D|}{\text{avgdl}}\big)}}_{\text{上面的 “TF项”}}$$

$\sum$ 是遍历 **query 里的每个词** $q_i$，各算一份贡献再相加。每份贡献 = **IDF（这个词多值钱）× TF项（这篇里这个词命中得多好）**。三个直觉：

1. **词越罕见越重要（IDF）**：$\text{IDF}(q_i)=\ln\!\Big(\frac{N-n(q_i)+0.5}{n(q_i)+0.5}+1\Big)$，$N$=总文档数、$n(q_i)$=含该词的文档数（**就是该词倒排链的长度**）。"the"到处都是 → IDF 低；"E1234"只在少数文档 → IDF 高，命中它加分多。**这就是 BM25 擅长精确命中罕见术语/代号的根源。**
2. **词频饱和（$k_1$）**：一个词在这篇里出现越多越相关，**但边际递减**——出现 10 次不该是 1 次的 10 倍。分母里的 $f(q_i,D)+k_1(\dots)$ 让 TF 贡献**趋于饱和**。$k_1$（典型 1.2–2.0，默认 ~1.5）控制饱和快慢，越小饱和越早。
3. **长度归一化（$b$）**：长文档天然含更多词、更容易偶然命中，要惩罚。$\frac{|D|}{\text{avgdl}}$ 是"本篇长度 / 全库平均长度"，$b\in[0,1]$（默认 0.75）控制惩罚力度，$b=0$ 关闭、$b=1$ 完全归一化。

> **各量从哪来**：$\text{IDF}$ 来自**全库统计**（词的倒排链有多长）；$f(q_i,D)$、$|D|$ 来自**当前这一篇**（存在倒排链/文档元数据里）；$\text{avgdl}$ 是全库平均长度。这些索引期就备好，查询时只做查表 + 累加，所以**毫秒级**。

> **落地要点**：BM25 **免训练、快、可解释**（能指出是哪个词命中的），是稀疏检索**默认基线**。$k_1,b$ 一般用默认值即可；只有语料长度分布很极端（如混了大量超长文档）时才微调 $b$。ES / OpenSearch / Lucene 全部内置。

### 3.2 SPLADE：用 BERT 学出来的"聪明版 BM25"

**它要解决什么**：BM25 的死穴是**只认字面词**——查"找回登录凭据"匹配不到写着"重置密码"的文档。SPLADE 的目标是：**既保留 BM25"稀疏、可倒排索引、精确"的优点，又让它懂一点同义（词扩展）。**

**核心思路一句话**：把每段文本用一个**"词表维"的稀疏向量**表示——这个向量的每一维对应词表里的一个词，值是"这个词对本文本有多重要"。和 BM25 的词权重是同一类东西，**但权重是 BERT 学出来的，而且会自动补上没出现过的相关词**。

#### 网络结构：BERT + 它自带的 MLM 头（没有新组件）

SPLADE **不发明新网络**，就是"标准 BERT + BERT 预训练自带的 MLM 头"，创新全在输出端的几个变换：

```
输入文本 tokens
      │
 ┌──────────────┐
 │ BERT Encoder │   标准 Transformer（也可用 DistilBERT 等）
 └──────────────┘
      │  每个 token → hidden 向量 h_i
      ▼
 ┌──────────────┐
 │   MLM Head   │   就是预训练"猜被 mask 的词"那个头，直接复用
 └──────────────┘
      │  每个 token i → 一条覆盖【整个词表≈3万维】的 logits
      │              （含义："这个位置像词表里的哪些词"）
      ▼
  w_ij = log(1 + ReLU(logit_ij))     ① ReLU 去负保稀疏 ② log 做饱和
      ▼
  w_j = max over i (w_ij)            ③ 跨所有 token 位置 max 池化
      ▼
  稀疏向量（维度=词表≈3万，绝大多数分量=0）
```

三个输出端变换分别干嘛：

| 变换 | 作用 |
|---|---|
| **ReLU** | 砍掉负 logits → 大量分量归 0，**制造稀疏** |
| **log(1+·) 饱和** | 压制高频词权重爆炸，**防个别词主导**（思想同 BM25 的词频饱和） |
| **max 池化** | 把"每 token × 词表"的矩阵压成一条词表维向量（也有 sum 变体） |

> **结构上要抓住的两点**：① **输出维度 = 词表大小（≈3 万），不是 hidden 维**——所以它是稀疏向量，能像 BM25 一样塞进**倒排索引**检索；这跟 bi-encoder 输出 768/1024 维**稠密**向量本质不同。② **稀疏是训练"逼"出来的**：ReLU 只保证非负，真正让非零项少（几十~几百个）靠训练时的 **FLOPS 正则**（惩罚向量稠密度）。没有它，模型会点亮整个词表、退化成稠密向量、倒排索引失效。

#### 为什么它"懂同义"：隐式词扩展

关键在 MLM 头输出的是**整个词表的分布**——一个 token 位置不仅激活**原词**，还激活 MLM 认为**语义相关的词**。于是"重置密码"这段文本的稀疏向量里，会**自动带上"找回""凭据""登录"等没出现过的词**的权重。这就是**隐式词扩展**：检索时 query"找回登录凭据"就能命中它。匹配方式很简单——**query 稀疏向量与文档稀疏向量做点积**（和 BM25 打分同构，倒排索引高效执行）。

#### 落地注意

- ⚠️ **query 时现算 SPLADE 要跑一次 BERT 前向，加 100–300ms**，对交互式 RAG 不可接受。
- ✅ **做法**：文档端在**索引时预计算**好稀疏向量存起来；线上只对**问题本身**算一次 SPLADE，延迟就可控。
- **选型**：起步用 BM25 足够；对**术语密集领域**（法律、医疗、代码）且已验证 BM25 不够时，再上 SPLADE。

### 3.3 倒排索引：BM25 与 SPLADE 共同的检索底座

BM25 和 SPLADE 的向量都是**几万维、但只有几十~几百个非零**的稀疏向量。要在上亿文档里毫秒级检索，靠的不是暴力比向量，而是**倒排索引（inverted index）**——理解它就懂了稀疏检索为什么快。

#### 正排 vs 倒排：把"文档→词"翻过来

**正排**按文档存（"这篇有哪些词"），查一个词得翻遍全库，$O(N)$，扛不住：

```
doc1 → {重置, 密码, 登录}      doc2 → {密码, 强度}      doc3 → {登录, 失败}
```

**倒排**把它翻过来，按**词**存（"这个词在哪些文档出现"）：

```
密码 → [doc1, doc2]     重置 → [doc1]     登录 → [doc1, doc3]     强度 → [doc2] ...
```

查"密码"就**直接跳到"密码"这一行**拿到 `[doc1, doc2]`，**根本不碰不含它的文档**——这就是"倒排"（inverted）名字的由来，也是它快的根本。

#### 结构：词典 + 倒排链

```
词典(Dictionary)         倒排链(Postings List)
─────────────           ─────────────────────────────────
密码       ───────────►  [ (doc1, 权重, 位置), (doc2, 权重, 位置) ]
登录       ───────────►  [ (doc1, 权重, 位置), (doc3, 权重, 位置) ]
```

- **词典**：所有唯一词的有序表，支持快速定位某个词（B-tree / FST / 哈希）。
- **倒排链**：每个词挂一个列表，记录它出现在**哪些文档**、以及每篇里的**权重**（BM25 存词频 tf、SPLADE 存学出的权重）和可选的**位置**（支持短语查询）。

#### 一次查询怎么执行（"密码 登录"）

```
1. 词典查"密码" → 取倒排链 [doc1, doc2]
2. 词典查"登录" → 取倒排链 [doc1, doc3]
3. 归并两条链（拉链式合并）：doc1 两词都命中 / doc2 只命中密码 / doc3 只命中登录
4. 对候选逐个累加打分：doc1 = score(密码)+score(登录) ← 都命中，分高
5. 排序取 top-k
```

全程只碰倒排链里的**少数候选**，从不遍历全库；链按 doc id 排序，还能用 **WAND / Block-Max WAND** 跳过不可能进 top-k 的文档进一步加速。

#### 【重点】BM25 与 SPLADE 各自怎么接入

**结构完全一样，区别只在"倒排链里存什么权重、以及一篇文档会进入哪些词的链"**：

| | BM25 | SPLADE |
|---|---|---|
| 倒排链里存的权重 | **原始词频 tf**（IDF 由链长 $n(q_i)$ 现算） | **模型学出的词权重** $w_j=\log(1+\text{ReLU}(\cdot))$ |
| 文档进入哪些词的链 | **只进它字面出现的词** | 进字面词 **+ 词扩展的相关词**（原文没有但权重非零） |
| 打分 | 命中词的 BM25 分累加 | 命中词的 query×doc 权重点积累加 |
| 懂同义吗 | 否 | **是**——查询命中扩展词也能召回 |

一句话：**SPLADE = 把 BM25 倒排索引里的"词频"换成"学出来的词权重"，并让每篇文档额外挂进若干语义相关词的倒排链**。检索引擎、数据结构、执行流程都不变，所以它能"懂同义"却仍享受倒排索引的速度。

> ⚠️ 这正是 3.2 强调 **SPLADE 必须保持稀疏** 的原因：若它点亮整个词表，每篇文档就会挂进**几万条**倒排链 → 索引爆炸、链超长、检索退化成全库扫描。FLOPS 正则逼稀疏，就是为了让倒排索引还能高效工作。

> 工程上不用自己实现：**Lucene**（Elasticsearch / OpenSearch 内核）是最成熟的倒排索引实现，BM25 是其默认打分器，SPLADE 也可用它的"稀疏向量 / 词权重"接口落地。

## 四、混合检索（Hybrid）：生产默认

### 4.1 为什么必须混合

BM25 和稠密检索各有**系统性盲区**，融合两者能同时拿到"语义理解 + 精确命中"。硬数据：

- WANDS 电商基准上，调优后的混合检索达 **0.7497 NDCG，比单用 BM25（0.6983）或纯向量（0.6953）高约 7.4%**。
- 延迟代价小：良好调优的栈上，**混合只比纯稠密多 5–20ms**，多数场景完全可接受（sparse 和 dense 可并行查）。

### 4.2 混合的核心难题：两路分数不可比

先想清楚"混合"难在哪：稠密返回的是**余弦相似度**（有界，$[-1,1]$ 或 $[0,1]$），BM25 返回的是**无界的正分数**（可以是 3.2，也可以是 47）。两者**量纲、分布都不同**，直接相加等于让 BM25 主导。融合就是解决"怎么把两个不可比的排序合成一个"。主流两条路：

**融合方式 A：RRF（Reciprocal Rank Fusion）—— 只看排名，不看分数**

彻底回避分数不可比：**扔掉原始分数，只用每路里的名次**。

$$\text{RRF}(d)=\sum_{r\in\text{各路检索}}\frac{1}{k+\text{rank}_r(d)}$$

- $\text{rank}_r(d)$ 是文档 $d$ 在第 $r$ 路里的排名（第 1 名 rank=1）。某路没召回该文档，则那一项记 0。
- **举例**（$k=60$）：文档 X 在稠密路排第 1、BM25 路排第 3 → $\frac{1}{60+1}+\frac{1}{60+3}=0.0164+0.0159=0.0323$。文档 Y 只在 BM25 路排第 1 → $\frac{1}{61}=0.0164$。X 因为**两路都靠前**而胜出——这正是我们想要的"共识"。
- $k$ 是平滑常数：$k$ 越大，头部名次之间的差距被压得越平（第 1 和第 10 差别变小）。**$k=60$ 是跨数据集/领域的通用值**，几乎不用调——这是 RRF 最大的工程优点。
- **原生支持**：Elasticsearch（rrf retriever）、OpenSearch、Weaviate（默认融合）、Qdrant（Fusion.RRF）都内置。

**融合方式 B：加权归一化（Convex Combination / 线性加权）—— 归一化后加权相加**

先把每路分数**归一化到同一量纲**（常用 min-max 归一到 $[0,1]$，或对 BM25 用理论上界归一），再加权：

$$\text{score}(d)=\alpha\cdot\widetilde{s}_{\text{dense}}(d)+(1-\alpha)\cdot\widetilde{s}_{\text{sparse}}(d)$$

- $\alpha\in[0,1]$ 是可调权重（$\alpha=0.5$ 即等权）。$\widetilde{s}$ 表示归一化后的分数。
- **更准但更娇气**：研究（*An Analysis of Fusion Functions*, 2023）显示，**只要有少量带标注的评估集把 $\alpha$ 调好，加权融合在域内/域外通常略胜 RRF**；但它对归一化方式敏感、需要调参和标注数据。

**怎么选**：

| | RRF | 加权归一化 |
|---|---|---|
| 调参 | 几乎不用（$k=60$） | 要调 $\alpha$、选归一化方式 |
| 依赖标注 | 否，即插即用 | 是，需评估集调权重 |
| 精度上限 | 略低 | 调好后略高 |
| 建议 | **默认首选** | 有评估集、要压榨最后几个点时再上 |

> 一句话：**先用 RRF 把系统跑起来**（零调参、生产验证充分）；等你有了 2.10 的评估集、且确认还想再要几个点，再考虑换加权融合调 $\alpha$。

**实现细节（容易忽略）**：融合前每路各取 **top-N（如各 100–200）** 再合并，而不是各取几条；两路可**并行查**（见 4.1，延迟只增 5–20ms）。

### 4.3 落地建议

> **花三小时把 BM25 + 稠密 + RRF 搭起来**，在你真实 query（尤其是用户抱怨过的那些）上测 precision@5。提升通常立竿见影。**对多数团队，"BM25 + Dense + RRF" 是终点，不是过渡。**

## 五、Late Interaction（ColBERT）：介于稠密检索与重排的第三条路

> **进阶选读**：本节是"混合检索 + cross-encoder 重排"这条主流管线之外的**第四条路**，属**进阶/按需**——先把四（Hybrid）落地，确认默认管线不够用了再回看本节。新手可跳到第六节。

2.3 铺垫过多向量表征，这里讲它怎么用在检索。

- **原理**：文档端**索引时**预存 per-token 向量；query 时用 **MaxSim**（query 每个 token 找文档里最相似 token，再求和）打分。它把匹配"延后"到 token 级细粒度做——所以叫 late interaction。
- **定位**：这句话比的是**"算 query–doc 相似度"的机制**（不是流水线岗位）。沿"精度↔速度"看三种机制：

  | 打分机制 | doc 侧表示 | query 时成本 | doc 可离线预计算 |
  |---|---|---|---|
  | Bi-encoder | 1 个向量 | 一次点积（最省） | ✅ |
  | **ColBERT** | 每 token 一个向量 | MaxSim（token 级交互，略重但轻） | ✅ |
  | Cross-encoder | 不单独表示 | 每对 query-doc 跑一遍整个模型（最贵） | ❌ |

  ColBERT 在**"文档侧可离线预存"**这点上和 bi-encoder 同侧（所以**query 时快**），在**"保留 token 级交互"**这点上和 cross-encoder 同侧（所以**准**）——故称"接近 cross-encoder 精度、接近 bi-encoder 速度"。注意：**比的是 query 时刻的实时打分成本**，doc 侧编码对 bi-encoder 和 ColBERT 都是离线免费的；cross-encoder 则因依赖具体 query 而无法预计算。（ColBERT 最初就是作为**检索器**提出的，与 bi-encoder 检索直接对标，这也是拿它和 bi-encoder 比速度的由来。）
- **代价**：存储大（每块上千向量），需 PLAID/MUVERA 等工程手段（见 2.3）。

### 5.1 工业上它到底放在哪（别误解）

⚠️ **常见误解**：以为 ColBERT 是"混合检索 + 重排"整段的替代方案。**不是。** 生产 RAG 的骨架始终是"**混合召回 → 重排 → 生成**"，ColBERT **不改变骨架，它主要插进"重排"那一格**，做 cross-encoder 重排器的**高效替代**。按工业常见度排：

1. **当重排器（最主流）**：混合检索召回 top-100 → **ColBERT 精排** → 送 LLM。卖点是**质量接近 cross-encoder，但延迟低一到两个数量级**（有实测：40 QPS 下 cross-encoder 的 p99.9 >21s，ColBERT p50 仅 ~23ms）。适合**高 QPS、尾延迟敏感**的线上服务。
2. **当第一阶段检索器（较少）**：用 ColBERTv2 + PLAID 索引直接当召回器，替代/增强 dense 那一路。因**存储代价大、运维重**，只在细粒度匹配是核心竞争力时才这么干。
3. **组合叠加（成熟系统）**：混合召回 → ColBERT 快速精排 →（可选）**再对 top-10 加一个小 cross-encoder** 抠最后的语义细节（法律/医疗等长尾场景）。

**和 cross-encoder 不是二选一，而是"质量—速度"光谱上的相邻两档**：

| | 绝对质量 | 速度 | 何时选 |
|---|---|---|---|
| Cross-encoder（2.6） | 最高 | 最慢 | 候选 <100、离线/低 QPS、质量至上 |
| **ColBERT（late interaction）** | 接近 cross-encoder | 毫秒级 | **高 QPS、尾延迟是瓶颈** |
| Bi-encoder / Dense（本节） | 一般 | 最快 | 第一阶段召回 |

### 5.2 什么时候上

**混合检索 + cross-encoder 重排（2.6）已经是绝大多数场景的答案。** 当你**重排质量想要、但 cross-encoder 的延迟/吞吐扛不住**时，用 ColBERT 替换重排那一格；细粒度匹配是产品核心竞争力时，再评估用它做第一阶段检索。**别一上来就上**——它的存储和工程复杂度是实打实的成本。

## 六、元数据过滤（Metadata Filtering）

用 2.2 打好的元数据，在检索前/中缩小范围：

- **先过滤，后检索**：按日期、部门、文档类型、语言等**先框定子集**再做向量搜索，既降噪又提速。
- **典型用途**：只搜"近一年的"、只搜"当前用户有权限看的"、只搜"产品 A 的文档"。
- **权限即过滤**：把权限写进元数据，用过滤在**检索层强制施加**（不能只靠 prompt 提醒模型别看，安全见第 8 章）。

---

## 七、Top-K 权衡：一个漏斗，不是一个数

检索不是一步取 N 条，而是一个**逐级收窄的漏斗**——这也是 2.4→2.6 的串联：

```
混合检索           重排(2.6)          送入 LLM(2.7)
召回 top 50~200 ──▶ 精排到 top 5~20 ──▶ 最终 3~5 条
(宁滥勿缺,          (cross-encoder/     (质量优先,
 保召回率)          ColBERT 精排)        控 token 与噪声)
```

- **召回阶段（本节）**：K 放大些（50–200），**优先召回率**——漏了就再也找不回。
- **精排阶段（2.6）**：把召回的一大批精排，优先精确率。
- **送入阶段（2.7）**：只给模型最相关的 3–5 条，太多反而引入噪声、触发 lost-in-the-middle。

> 经验起点：**召回 top 50 → 重排到 top 5 → 送 3–5 条**。K 具体值靠 2.10 的评估调。

---

## 八、结构化数据检索：Text-to-SQL

不是所有问题都该走"文本相似度"。**精确数值、聚合、过滤类问题**（"上季度华东区销售额是多少""库存低于 100 的 SKU"），文本检索天然不擅长。

- **做法**：用 LLM 把自然语言翻译成 SQL，直接查结构化库（NL→SQL）。
- **何时用**：数据在数据库/表格里、答案要精确计算而非"找相似段落"时。
- **进阶**：可与文本 RAG 组成路由（2.5/2.8）——结构化问题走 SQL，非结构化问题走向量检索。

> 记住 2.1 的决策：**要精确聚合就别硬用向量检索**，Text-to-SQL 才是对的工具。

---

## 九、一个标准生产检索管线

把本节内容串成一条 2026 主流管线：

1. **建双索引**（两条检索路各有对应的底层结构，并行准备）：
   - **稀疏路**：BM25 / SPLADE → **倒排索引**（词→文档链，见 3.3），Lucene/ES/OpenSearch 内建。
   - **稠密路**：embedding 向量 → **ANN 索引**（HNSW / IVF-PQ，见 [2.4.1](2.4.1-ann-nearest-neighbor.md)）。
2. **查询**（结合 2.5 的查询改写）：稠密 + 稀疏并行检索。
3. **融合**：RRF（k=60）合并成一个候选列表（top 50–200）。
4. **（可选）元数据过滤**：日期/权限/类型先框定范围。
5. **重排**（2.6）：cross-encoder 或 ColBERT 精排到 top 5。
6. **送生成**（2.7）：最终 3–5 条 + 引用位。

---

## 十、本节小结：检索决策清单

1. **默认混合检索**：BM25 + 稠密 + RRF（k=60），性价比最高、通常立竿见影。
2. **稀疏选型**：先 BM25；术语密集且 BM25 不够 → SPLADE（文档端预计算）。
3. **别只看一个 Top-K**：召回宽（50–200）→ 重排窄（top 5）→ 送 3–5 条，是漏斗。
4. **元数据先过滤**：降噪提速，权限在检索层强制。
5. **精确聚合走 Text-to-SQL**：别用向量检索硬扛结构化查询。
6. **ColBERT 按需**：细粒度匹配 + 重排是瓶颈时再评估，不是起手式。

> 检索层做扎实，2.5（让 query 更好检索）和 2.6（把召回精排）才有发挥空间。这三节合起来就是"检索质量"的全部战场。
