# 3.7 Multi-Agent 协作

> **核心问题**：你手上一个任务，到底该用**一个** Agent 硬扛，还是拆成**多个** Agent 分工协作？如果用多 Agent——怎么编排（谁指挥谁）、怎么共享状态（它们怎么知道彼此干了什么）、怎么避免上下文爆炸、成本失控和"各干各的最后拼不到一起"？
>
> **目标**：
> - **先想清楚"要不要多 Agent"这个最该谨慎的决策**——通过 Anthropic 与 Cognition 的正面交锋，理解多 Agent 的真实收益与真实代价
> - 掌握主流编排模式：Orchestrator-Worker、Supervisor、Swarm/Handoff、Pipeline、Debate，及其抽象差异
> - 吃透多 Agent 的三大工程难题：**上下文共享、状态一致性、成本与协调**
> - 掌握让多 Agent 真正可靠的工程实践（来自 Anthropic 生产系统的一手经验）
> - 能根据"任务是否可并行、是否强依赖、价值是否配得上 15x 成本"做出单/多 Agent 决策
>
> **边界**：单 Agent 内部的规划模式（ReAct/Plan-Execute 等）属于 **3.6**，本节讲的是**多个 Agent 之间**的分工；Agent 间通信的标准协议（A2A）属于 **3.3**，本节讲的是编排逻辑而非线缆协议；上下文压缩本身的手段属于 **3.4**，本节讲它在多 Agent 场景下为什么变得更棘手。

---

## 3.7.1 先泼一盆冷水：多 Agent 是本章最危险的诱惑

在讲任何编排模式之前，必须先建立一个立场：**"把任务拆给多个 Agent 分工"是一个听起来极其诱人、但绝大多数时候会让你系统变脆的决定。** 这一节和别的技术节不同——它最重要的价值不是教你"怎么搭多 Agent"，而是帮你**判断到底该不该搭**。

为什么说危险？因为多 Agent 天然踩中了 3.1 讲过的可靠性衰减和 3.4 讲过的上下文难题，还额外叠加了"多个不确定系统互相依赖"的新风险。而这一点，业界最顶尖的两家公司给出了**截然相反**的结论——这场交锋，是理解多 Agent 的最佳入口。

### 交锋的一方：Cognition（Devin）——"别搭多 Agent"

2025 年 6 月，Devin 的开发商 Cognition 发了一篇态度鲜明的文章《Don't Build Multi-Agents》。他们的论证建立在**两条原则**上，值得逐字记住：

> **原则一：共享上下文——要共享完整的 agent 轨迹（full agent traces），而不只是单条消息。**（Share context, and share full agent traces, not just individual messages.）
>
> **原则二：行动隐含着决策，而互相冲突的决策会带来糟糕的结果。**（Actions carry implicit decisions, and conflicting decisions carry bad results.）

第二条是精髓。它的意思是：**Agent 每做一个动作，背后都隐含了一堆没说出口的假设和决策。** 当多个子 Agent 并行、各自看不到对方的完整轨迹时，它们会做出互相冲突的隐含假设，最后拼出来一堆自相矛盾的东西。

Cognition 用了一个极其形象的例子——**Flappy Bird 惨案**：把"做一个 Flappy Bird 克隆"拆成两个并行子任务。子 Agent 1 理解偏了，做了个**超级马里奥风格的背景**；子 Agent 2 做了只**画风诡异的鸟**。最后负责整合的 Agent 继承了这两个它无法修复的误解，产出一个四不像。**没有一个子 Agent 是"笨"的——它们各自都合格，但因为看不到彼此的隐含决策，合起来就崩了。**

Cognition 的结论毫不含糊：

> **"在 2025 年，让多个 Agent 协作运行，只会得到脆弱的系统。"**（in 2025, running multiple agents in collaboration only results in fragile systems.）

他们的推荐方案：**单线程线性 Agent**——让上下文保持连续完整。如果任务真的长到有上下文溢出风险，就加一个**基于 LLM 的压缩层**，把行动历史蒸馏成关键决策和事件（他们也坦承这一层"很难做好"）。

### 交锋的另一方：Anthropic——"多 Agent 大幅胜出"

就在 Cognition 发文的**第二天**，Anthropic 发了《How we built our multi-agent research system》，摆出了针锋相对的数据：

> 他们的多 Agent 研究系统（Claude Opus 4 作 lead agent + Claude Sonnet 4 作 subagent），在内部研究评估任务上**比单个 Claude Opus 4 高出 90.2%。**

架构是经典的 **Orchestrator-Worker（编排者-工人）**：lead agent 分析查询、制定策略、并行 spawn 出 3-5 个专职 subagent 去探索不同方面，最后综合它们的发现（还有一个独立的引用核对环节）。关键细节：**每个 subagent 拿到一个自包含的任务描述、一个输出格式、和一个全新的上下文窗口。它不知道其他 subagent 存在，也无法在任务中途与它们协调。**

等一下——这不正是 Cognition 说会崩的那种架构吗？子 Agent 互相看不见？为什么 Anthropic 这边就成了？

### 交锋的真相：他们根本不矛盾，是任务性质不同

这才是本节最重要的洞察，请务必内化：**这两家没有谁对谁错，他们在做完全不同性质的任务，而任务性质决定了多 Agent 是灵药还是毒药。**

| | Cognition（Devin，**编码**） | Anthropic（Research，**研究检索**） |
|---|---|---|
| 任务性质 | **写代码**：各部分强依赖、需要全局一致 | **开放式研究**：多个独立方向可并行探索 |
| 子任务之间 | **高度耦合**——鸟的样式和背景必须协调 | **基本独立**——查"公司A财报"和查"公司B财报"互不影响 |
| 动作的影响 | **写操作**，一个改动影响全局，隐含决策多 | **多为只读检索**，各自搜集信息，冲突少 |
| 上下文 | **必须共享完整轨迹**才能不冲突 | 各 subagent **独立上下文反而是优势**（并行压缩） |
| 结论 | 多 Agent 太脆，**用单线程** | 多 Agent 大幅胜出，**用 Orchestrator-Worker** |

Anthropic 自己也白纸黑字承认了边界，这段话和 Cognition 的立场其实完全一致：

> 多 Agent **不适合**这些情况：**"大多数编码任务的可并行子任务，比研究要少得多"**；需要**所有 Agent 共享同一上下文**、或 Agent 之间有**很多依赖**的系统；以及"LLM Agent 目前还不擅长实时地协调和委派给其他 Agent"。

**所以真正的决策规则不是"多 Agent 好不好"，而是"我的任务长得像 research 还是像 coding"**：
- 像 research（**breadth-first、子任务独立、以只读检索为主、信息量超单个上下文窗口**）→ 多 Agent 可能大幅胜出。
- 像 coding（**子任务强依赖、需要共享上下文、有写操作和大量隐含决策**）→ 老老实实用单 Agent。

## 3.7.2 单 Agent vs 多 Agent 决策框架

把上面的洞察落成一个可操作的判断清单。**在写任何多 Agent 代码之前，逐条过一遍：**

**✅ 倾向多 Agent，当且仅当以下大部分成立：**
1. **任务可以真正并行**——能拆成多个互不依赖的子任务（典型：并行调研多个主题、并行处理多个独立文件）。
2. **信息量超过单个上下文窗口**——单 Agent 塞不下，需要多个独立上下文窗口分头压缩。
3. **子任务之间依赖少**——一个子任务的结果不强烈依赖另一个的中间过程。
4. **任务价值高到能justify成本**——记住下面的经济账：多 Agent 大约烧 **15 倍**于普通聊天的 token。
5. **以只读/信息搜集为主**——冲突的隐含决策少。

**❌ 坚持单 Agent，当以下任一成立：**
1. **子任务强依赖、需要全局一致**（写代码、改一份文档的多个部分）。
2. **需要共享完整上下文**才能不做出冲突决策。
3. **需要 Agent 之间实时协调**——当前 LLM 还不擅长。
4. **任务价值撑不起 15x 的成本**。

> **默认立场（重要）**：**从单 Agent 起步。** 单 Agent 更简单、更可靠、更便宜、更好调试。只有当你**明确撞到"单个上下文窗口装不下、且任务高度可并行"这堵墙**时，才升级到多 Agent。把多 Agent 当成"逼不得已的 scale-out 手段"，而不是"更高级的默认架构"。这是本节最该带走的一句话。

## 3.7.3 多 Agent 编排模式

确实需要多 Agent 了，就要选一种编排结构——**谁来指挥、控制权怎么流转**。主流有五种，从"中心化"到"去中心化"排列：

### 模式一：Orchestrator-Worker（编排者-工人）—— 最主流、最可控

一个 **lead/orchestrator** 居中指挥：它分解任务、把子任务分派给多个 **worker/subagent**、收集结果、综合成最终答案。worker 之间**不直接通信**，都只对 orchestrator 负责。

这就是 Anthropic 研究系统的架构。**它是多 Agent 的默认首选**，因为：
- 控制流清晰，好调试（所有协调都过 orchestrator 这个中心）。
- worker 拿独立上下文窗口，天然并行、天然压缩。
- 适合"一个大任务 → 扇出成多个独立子任务 → 汇总"的模式（fan-out）。

代价：orchestrator 是瓶颈也是单点；worker 看不到彼此，只适合**独立子任务**（回到 3.7.1 的边界）。

### 模式二：Supervisor（主管）—— Orchestrator 的"路由"变体

一个 **supervisor** Agent 管理一组专职 Agent，但它的角色更像**路由器/调度员**：根据当前状态决定"下一步该交给哪个专家 Agent"，可以来回多轮（不像 Orchestrator-Worker 那样偏一次性扇出）。LangGraph 的 supervisor 模式是典型。

- 适合：有一组**明确分工的专家**（研究员、写作者、审校），需要按情况在它们之间反复调度。
- 与 Orchestrator-Worker 的区别更多是**风格连续谱**：Orchestrator 偏"分解-扇出-汇总"，Supervisor 偏"持续路由到下一个专家"。实践中常常混用，不必纠结命名。

### 模式三：Swarm / Handoff（蜂群 / 交接）—— 去中心化

**没有中央协调者。** Agent 之间**直接把控制权交接（handoff）**给彼此——A 觉得该 B 上了，就把控制权连同对话历史一起交给 B。OpenAI 的 Agents SDK 把 **handoff 做成了一等原语**（agent A 可以把控制权连同结构化载荷交给 agent B，并记录这次转移用于回放）。

- 优点：**搭起来轻**，每个 Agent 只需知道"什么时候该交给谁"，不需要一个全局大脑。
- 缺点：**没有任何 Agent 有全局视野**，当交接逻辑产生意外路由时**极难调试**（控制权像击鼓传花，出问题不知道卡在哪一棒）。
- 适合：像客服分流这种"我处理不了就转给对应专家"的场景。

> 历史注记：OpenAI 早期的实验框架 **Swarm** 定义了这个模式（Agent + handoff 两个原语），已于 2025 年 3 月弃用，由 **OpenAI Agents SDK** 取代并把 handoff 正式产品化。

### 模式四：Pipeline（流水线）—— 串行专家链

Agent 像流水线工位一样**串行**：A 的输出是 B 的输入，B 的输出是 C 的输入。本质是把一个复杂任务拆成**固定顺序**的阶段，每阶段一个专职 Agent。

- 适合：**阶段固定、顺序明确**的任务（提取 → 清洗 → 分析 → 写报告）。
- 注意：这其实很接近 3.1 讲的 **Prompt Chaining workflow**——如果流程是写死的，它更像 workflow 而非"自主多 Agent"。**别为一个确定的流水线套上多 Agent 框架的复杂度。**

### 模式五：Debate（辩论）—— 多 Agent 互相批判

多个 Agent 对同一问题**给出不同答案并互相批判、辩论**，最后收敛（或由一个裁判 Agent 定夺）。目的是用"对抗"提升答案质量、减少单一视角的偏见/幻觉。

- 适合：高价值、需要多视角审视的判断（复杂决策、疑难诊断）。
- 代价：**成本高、延迟高**（多个 Agent 跑多轮），且不一定收敛。属于"锦上添花"，不是常规武器。

### 五模式速查

| 模式 | 控制结构 | 好调试度 | 最适合 | 主要风险 |
|---|---|---|---|---|
| **Orchestrator-Worker** | 中心化，扇出汇总 | 高 | 独立子任务并行（research 类） | orchestrator 单点、worker 互盲 |
| **Supervisor** | 中心化，持续路由 | 高 | 专家团队反复调度 | 与前者边界模糊 |
| **Swarm/Handoff** | 去中心化，直接交接 | **低** | 分流转派（客服类） | 无全局视野、路由难调试 |
| **Pipeline** | 串行固定链 | 高 | 阶段固定的流程 | 流程写死时其实是 workflow |
| **Debate** | 对抗收敛 | 中 | 高价值多视角判断 | 成本/延迟高、不保证收敛 |

**默认建议**：需要多 Agent 时，**从 Orchestrator-Worker 起步**——它最可控、最好调试，且匹配"独立子任务并行"这个多 Agent 唯一真正擅长的场景。其余模式在明确匹配其场景时才引入。

## 3.7.4 三大工程难题

多 Agent 的坑集中在三个地方。这三个难题，正是 Cognition 反对、Anthropic 也承认棘手的根源。

### 难题一：上下文共享——多 Agent 的头号杀手

这是 3.7.1 反复强调的核心。Cognition 的两条原则本质上就是在说：**Agent 之间不共享完整轨迹，就会做出冲突的隐含决策。** 而多 Agent 架构（尤其去中心化的）天生难共享上下文——

- 全共享：把每个 Agent 的完整轨迹都塞给其他所有 Agent → **上下文爆炸**（回到 3.4 的 Context Rot），且成本飙升。
- 不共享：各 Agent 独立上下文 → **Flappy Bird 惨案**，隐含决策冲突。

Anthropic 的解法是**回避而非解决**：只在"子任务本就独立"的场景用多 Agent，这样"不共享"不会造成冲突。Cognition 的解法是**单线程 + 压缩层**：干脆不并行，保持一条连续的完整上下文。**没有第三种银弹。** 你要么让子任务真正独立（回避冲突），要么保持单线程（保证共享）。想"既并行又完美共享"，2025 年的技术还做不到。

### 难题二：状态一致性与错误累积

Anthropic 的原话点破了这个难题的本质：**"Agent 是有状态的，错误会累积（Agents are stateful and errors compound）。"**

- **错误级联**：一个 Agent 在早期步骤犯的小错，会通过状态在后续步骤和下游 Agent 里放大，最后酿成不可预测的失败。多个 Agent 叠加，级联风险相乘。
- **同步瓶颈**：Anthropic 坦承他们的 lead agent **同步**执行 subagent——必须等一批 subagent 全部完成才继续。这简化了协调，但代价是任何一个 subagent 拖慢，整个系统就卡住。
- **异步的代价**：异步能提升并行度，但会带来"结果协调、状态一致性、跨 subagent 错误传播"的新难题——**Anthropic 说这些在生产规模上仍未解决。**

**工程应对**：用**持久化执行（durable execution）+ 检查点恢复**——出错时从检查点续跑，而不是整个从头再来。这是多 Agent 生产化的必备基础设施（连着 3.12 的 Harness 设计）。

### 难题三：成本与协调——那笔 15x 的账

多 Agent 最容易被低估的是**成本**。记住 Anthropic 给的这组数字：

- 单个 **Agent** 大约消耗普通聊天 **4 倍** 的 token。
- **多 Agent** 系统大约消耗普通聊天 **15 倍** 的 token。
- 在他们的分析里，**光是 token 用量就解释了 80% 的性能方差**；token + 工具调用次数 + 模型选择这三个因素合起来解释了 **95%**。

这组数字有两个极其重要的落地含义：

1. **多 Agent 的性能提升，很大程度是"用钱买的"**——你堆了 15 倍 token 进去。所以问题永远是：**这个任务的价值，配得上 15 倍成本吗？** 低价值任务上多 Agent，是纯烧钱。
2. **升级模型往往比堆 token 更划算**。Anthropic 的原话：**"升级到 Claude Sonnet 4 带来的性能提升，比在 Claude Sonnet 3.7 上把 token 预算翻倍还大。"** 换句话说，**在考虑"上多 Agent 堆 token"之前，先试试"换个更强的模型"**——可能更便宜也更好。

## 3.7.5 让多 Agent 真正可靠的工程实践

如果你评估下来确实该上多 Agent，Anthropic 的生产经验提炼出这几条最有价值的实践（这些是"多 Agent 能不能落地"的关键，不是可选项）：

1. **显式教会委派（delegation）**。每个 subagent 必须拿到：**一个明确目标、一个输出格式、用什么工具和数据源的指引、清晰的任务边界**。Anthropic 的教训很直接：**指令不够详细，"Agent 就会重复劳动、留下空白、或找不到必要信息"**——这正是 Flappy Bird 惨案的预防针。

2. **让投入规模匹配任务复杂度**。把伸缩规则写进 orchestrator 的 prompt。Anthropic 的具体档位可直接借鉴：**简单事实查找——1 个 Agent，3-10 次工具调用；直接对比——2-4 个 subagent，各 10-15 次调用；复杂研究——10+ 个 subagent。** 别用大炮打蚊子，也别用手枪打坦克。

3. **并行化以换速度**。Anthropic 同时用了两层并行——**并行 spawn subagent + 每个 subagent 内部并行调工具**——把研究时间**砍掉最多 90%**（连着 3.2 的并行工具调用）。

4. **"像你的 Agent 一样思考"来调试**。用 Console 拿真实 prompt 和工具跑模拟，**一步步看 Agent 怎么工作**，失败模式立刻现形（继续搜索明明已经够了、查询词过于冗长、选错工具）。

5. **用分层评估**。小样本（~20 个真实 query）早期就能抓住 80% 的问题；再用 **LLM-as-judge** 规模化打分（事实准确性、引用准确性、完整性、来源质量、工具效率，0.0-1.0）；最后**人工评估抓边缘案例**（他们发现早期 Agent 总是偏爱 SEO 优化的内容农场而非权威来源——这种偏见自动化测不出来）。评估方法论细节归 3.12 和第 6 章。

## 3.7.6 选型决策框架

把全节收敛成一条决策路径：

**第 1 步：默认单 Agent。** 先用单 Agent + 3.6 的规划模式解决。绝大多数任务到这里就够了。

**第 2 步：撞墙了吗？** 只有当你明确遇到**"单个上下文窗口装不下 + 任务高度可并行"**时，才考虑多 Agent。撞的是别的墙（比如只是想要更好的规划）→ 回第 1 步，多 Agent 治不了。

**第 3 步：过任务性质关。** 用 3.7.1 的表自检——**像 research（独立、只读、可并行）还是像 coding（依赖、写操作、需共享）？** 像 coding 就别上多 Agent。

**第 4 步：算经济账。** 这个任务的价值撑得起 **15x token** 吗？先试"换更强的模型"是不是更划算？

**第 5 步：选编排模式。** 默认 **Orchestrator-Worker**；有明确专家团队反复调度用 Supervisor；分流转派用 Swarm/Handoff；固定阶段用 Pipeline（且警惕它其实是 workflow）；高价值多视角判断才用 Debate。

**第 6 步：上可靠性工程。** 显式委派 + 规模匹配 + 并行 + durable execution/检查点 + 分层评估。缺了这些，多 Agent 在生产里活不下来。

## 3.7.7 常见误区与小结

### 常见误区

| 误区 | 为什么错 | 正确做法 |
|---|---|---|
| **"多 Agent 是更高级的架构，应该优先用"** | 多 Agent 更脆、更贵、更难调试；单 Agent 是更好的默认 | 从单 Agent 起步，撞到"装不下+可并行"的墙才升级 |
| **"把任务拆给多个 Agent 就能提速提质"** | 子任务若强依赖，会各做冲突的隐含决策（Flappy Bird 惨案） | 只在子任务**真正独立**时并行 |
| **"Anthropic 证明了多 Agent 更好"** | 那是 **research** 任务（可并行、只读）；同样的架构用在 coding 上会崩 | 先判断任务像 research 还是 coding |
| **"多 Agent 的性能提升是免费的"** | 提升很大程度是 15x token 堆出来的 | 先算经济账；先试升级模型（可能比堆 token 更划算） |
| **"去中心化的 Swarm 更优雅"** | 没有全局视野，意外路由极难调试 | 默认用可控的 Orchestrator-Worker |
| **"给固定流程也上多 Agent 框架"** | 流程写死的其实是 workflow（3.1） | 确定性流程用 workflow，别套多 Agent 复杂度 |

### 小结

- **最重要的一课**：多 Agent 是本章最危险的诱惑。**默认单 Agent**，只在"单上下文装不下 + 任务高度可并行"时才升级。
- **Anthropic vs Cognition 不矛盾**：任务性质决定一切。research（独立、只读、可并行）→ 多 Agent 大胜（+90.2%）；coding（依赖、写操作、需共享）→ 多 Agent 太脆，用单线程。**先判断你的任务像哪个。**
- **五种编排模式**：Orchestrator-Worker（默认首选）、Supervisor（持续路由）、Swarm/Handoff（去中心化、难调试）、Pipeline（固定阶段、警惕它是 workflow）、Debate（高价值多视角）。
- **三大工程难题**：上下文共享（要么子任务真独立、要么单线程，无银弹）、状态一致性（错误累积、同步瓶颈、用 durable execution）、成本协调（**15x token**，先试升级模型）。
- **可靠性实践**：显式委派、规模匹配任务、双层并行、模拟调试、分层评估。
- **一句话纪律**：**能单不多，多则中心化，上则算账。**
