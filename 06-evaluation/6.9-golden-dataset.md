# 6.9 数据集与 Golden Set 工程

> **核心问题**：评估数据集（golden set）怎么从 0 建？怎么用生产 trace + user feedback 让它"活"起来？怎么避免它"过拟合 PR 流程"？
> **目标**：能为应用建立一套"持续演进"的 eval 数据集 —— 有版本、有质量、覆盖业务真实分布、不被反复"训练"成无效。

**展开重点**：
- **数据集是产品，不是 fixture**：
  - 要版本化（Git / DVC / 平台 dataset 管理）
  - 要可追溯（谁 / 何时 / 为什么加 / 修 / 删）
  - 要可演进（业务变了、模型变了，集合要跟着变）
- **冷启动（从 0 → 50 例）**：
  - 列业务里 5–10 个"必须答对"的代表性问题
  - 加入历史 bug、客诉、边界 case
  - 加入 known-hard：超长、含矛盾、需要拒答、含 PII
  - 至少 1–2 例覆盖每条业务路径
- **扩展（50 → 几百 / 几千）**：
  - **生产 trace 回流**：按 sampling / 关键事件（低分 / 用户负反馈 / refusal）反向加入
  - **failure-driven**：每次发现 prod bug，都补一条进 golden set
  - **难例挖掘**：用强 judge 跑 large pool，挑分歧 / 低分例
  - **LLM 生成 + 人审**：让强模型基于业务 spec 生成候选，人审通过率 30–50% 是正常
- **质量控制**：
  - 多人标注 → 测一致性（Cohen's Kappa）
  - 标注规范文档（rubric），定期校准
  - 区分 hard / medium / easy 子集
  - 单独维护"对齐人评"的 judge 校准集（不能与 PR gate 同一批）
- **版本与切分**：
  - **dev / CI gate / hold-out** 三层切分
  - dev 可随时改；CI gate 改要走 review；hold-out 季度更新
  - **Hold-out 是关键**：防止 dataset 被反复回测后失效
- **演进与衰减**：
  - 业务上线新功能 → 加新例
  - 老模型已"100%"通过 → 升级难度 / 退役
  - 季度 audit：跑一次"数据集对当前实际分布的代表性"
- **多模态 / agent 数据集特殊性**：
  - **多模态**：图像 / 音频 / 视频要哈希存档，版权清晰
  - **agent**：不仅存 input/output，还存"期望轨迹"或"期望工具调用集"
- **平台支持**：
  - LangSmith / Braintrust / Langfuse / Phoenix 都有 dataset 管理
  - 关键能力：版本、tag、来源 trace 链回、与 eval 跑结果绑定
