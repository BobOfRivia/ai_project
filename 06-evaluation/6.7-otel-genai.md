# 6.7 OpenTelemetry GenAI 标准

> **核心问题**：`gen_ai.*` semantic conventions 是什么？2026 它怎么"打穿"vendor lock-in？应用工程师该怎么把自己的 trace 接到这一标准？PII 怎么处理？
> **目标**：能在自己应用里输出符合 OTel GenAI 规范的 trace，让 Datadog / Honeycomb / Jaeger / LangSmith / Phoenix 都能直接消费，避免被单一平台绑死。

**展开重点**：
- **标准背景**：
  - **OTel GenAI Special Interest Group** 自 2024-04 起开发
  - **2026 初**：client spans（LLM 调用）退出实验状态、稳定
  - CNCF 背书，事实统一 LLM telemetry
- **核心覆盖范围**（四类）：
  - **Client spans**：LLM 调用 + 检索
  - **Agent spans**：agent 调用 + workflow
  - **Events**：prompt body / completion body
  - **Metrics**：token usage、latency、cost
- **关键属性（`gen_ai.*` 命名空间）**：
  - `gen_ai.system`：openai / anthropic / google / ...
  - `gen_ai.request.model` / `gen_ai.response.model`
  - `gen_ai.usage.input_tokens` / `gen_ai.usage.output_tokens`
  - `gen_ai.request.temperature` / `top_p` / `max_tokens`
  - `gen_ai.response.finish_reasons`
  - `gen_ai.tool.name` / `gen_ai.tool.call.id`
  - `gen_ai.agent.id` / `gen_ai.agent.name`
- **Events / body 处理（PII 关键）**：
  - prompt / completion 的完整文本是 opt-in event，默认不上报
  - 上报必须有"脱敏 / sampling / 仅 staging"策略
  - 高敏感行业（金融、医疗）一般只上元数据 + hash
- **vendor 接入现状**：
  - Datadog、Honeycomb、Jaeger v2、LangSmith、OpenLLMetry 全部映射到 `gen_ai.*`
  - 各家原生 SDK（Langfuse、Helicone、Traceloop）历史上各自定义；OTel GenAI 出现是为收拢这部分碎片
  - MLflow 的 GenAI tracing 也已对齐
- **接入方式**：
  - **OTel SDK + auto-instrumentation**（OpenLLMetry / Traceloop SDK）
  - **平台 SDK 内置 OTel export**：LangSmith / Phoenix / Helicone 都支持
  - **应用层手写 span**：用 OTel SDK 直接打，最灵活、不依赖某家
- **PII / 合规要点**：
  - 默认不打 prompt / completion 内容
  - 打也要走 sampling + 脱敏（regex / Presidio / 自家分类器）
  - 设独立 sensitive trace pipeline，与普通 telemetry 分流存储
  - GDPR / HIPAA 场景：trace 数据存储 / 保留期需符合法规
- **2026 实操建议**：
  - 新项目第一天就接 OTel GenAI
  - 旧项目走"双写"：原平台 SDK + OTel 同时上报，逐步迁移
  - 把"trace 是 OTel 标准"作为采购可观测平台的必要条件
