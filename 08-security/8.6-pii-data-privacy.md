# 8.6 数据隐私与 PII 处理

> **核心问题**：发到 LLM provider 的每个 prompt 都可能包含用户隐私——怎么检测、脱敏、并保留"对话连贯性"？可逆 vs 不可逆 tokenization 怎么选？
> **目标**：能为应用建一条"PII 检测 → 脱敏 → 调 LLM → 反脱敏 → 输出"流水线，覆盖 GDPR / CCPA / HIPAA / 中国《个人信息保护法》。

**展开重点**：
- **为什么 PII 处理是 LLM 应用刚需**：
  - 每次调外部 provider，prompt 全文上云
  - GDPR / CCPA / HIPAA / PIPL 都把这种传输视为"披露"
  - 不做 PII 处理 = 默认违反多个司法管辖区合规要求
- **三层检测架构**（2026 主流）：
  1. **Regex + checksum**：手机号 / 身份证 / 信用卡（Luhn）/ 邮箱——快、稳、覆盖结构化 PII
  2. **NER 模型**：DeBERTa / piranha-PII / Presidio 内置 spaCy——覆盖姓名、地址、组织
  3. **LLM-as-Judge**：小模型审查"是否含敏感个人信息"——兜底语义级
  - 三层并行，任何一层命中即视为 PII
- **主流工具**：
  - **Microsoft Presidio**（开源 Python SDK，2019 起，事实标准）：100+ 现成 recognizer，自定义易扩展
  - **LiteLLM + Presidio**：网关层 PII masking，1 行配置（→ 7.5）
  - **Azure PII Shield**（2026 新）：privacy proxy，per-tenant CMK + WORM 审计
  - **Private AI / Skyflow / Strac**：商业 SaaS，更全 SLA
  - **Grepture / LLM Guard Anonymize**：开源轻量
- **脱敏策略 4 档**：
  | 策略 | 说明 | 适用 |
  |---|---|---|
  | **Redact** | 直接删除或替换为 `[PII]` | 不需要对话连贯的批处理 |
  | **Mask** | 部分掩盖（`13*****1234`） | 客服展示 |
  | **Tokenize (不可逆)** | 替换为随机 token，无映射 | 高敏感、不需还原 |
  | **Tokenize (可逆)** | 加密映射，模型推理后反脱敏 | **多数 LLM 应用默认** |
- **可逆 tokenization 工程**：
  - 用 Presidio Analyzer 检测 → 生成 token (`<PERSON_1>`) → 维护 mapping table（加密存）→ 调 LLM → 输出做反向替换
  - **per-tenant Customer-Managed Key (CMK)**：每租户独立密钥
  - **WORM 审计日志**：Write-Once-Read-Many，监管追溯
- **特殊数据类型**：
  - **健康信息（PHI）**：HIPAA 范围，要 BAA（→ 8.7）
  - **支付卡（PCI-DSS）**：尽量根本不发给 LLM；非要发先 tokenize
  - **儿童数据**：COPPA / GDPR-K，年龄检测 + 父母同意
  - **生物特征**：欧盟视为特殊类别，最高保护级
- **跨地区合规速览**（与 8.9 互补）：
  - **GDPR**（欧盟）：数据主体权利（访问 / 删除 / 携带），违规上限 €20M 或 4% 营收
  - **CCPA / CPRA**（加州）：消费者数据权利，opt-out 销售
  - **PIPL**（中国）：跨境传输需安全评估或 SCC
  - **HIPAA**（美国医疗）：PHI 处理需 BAA
  - **DPDPA**（印度 2023）：本地化趋势
- **RAG / Memory 特别风险**：
  - 把含 PII 的文档塞 vector DB：会跨用户检索到
  - **Memory / scratchpad** 含 PII：多 session 间泄漏
  - 解法：**索引前脱敏 + per-tenant namespace**（→ 2 章 RAG）
- **常见反模式**：
  - 只用 regex，漏掉 NER 才能识别的内容（姓名、地址）
  - 脱敏后忘了反脱敏：用户看到 `<PERSON_1>` 体验崩
  - mapping table 未加密：脱敏等于没做
  - 把 PII 写进 LLM trace 日志（→ 6 章 observability 要先脱敏再 log）
  - "我们已经签了 ZDR"——ZDR 不等于 PII 合规，欧盟仍可能要求脱敏
