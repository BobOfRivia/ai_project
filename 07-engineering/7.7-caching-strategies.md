# 7.7 缓存策略深入

> **核心问题**：KV cache / prompt cache（provider 端）/ semantic cache / exact-match cache 各自适用什么场景？命中率怎么调优？
> **目标**：能为应用设计"三层缓存栈"，把 cache hit rate 当核心 SLO 之一，节约 50-80% token 与首 token 时间。

**展开重点**：
- **四种缓存层次**（自下而上）：
  1. **KV cache**（推理服务内部）：vLLM / SGLang / TensorRT-LLM 自带，**不需应用层管**；SGLang 的 RadixAttention 是前缀复用版
  2. **Prompt cache（provider 端）**：Anthropic / OpenAI / Gemini 都支持；**应用层要做的是 prompt 结构设计**
  3. **Exact-match cache（应用层）**：完全相同输入直接返回缓存输出
  4. **Semantic cache（应用层）**：语义相似就返回；以 embedding + 阈值实现
- **Provider 端 prompt cache 三家对比**：
  | Provider | 折扣 | 起点 | TTL | 控制方式 |
  |---|---|---|---|---|
  | **Anthropic Claude** | **90% off**（5min TTL）/ **更深**（1h）| 任意 | 5min / 1h | 显式 `cache_control` 块 |
  | **OpenAI** | **50% off** | ≥1024 tokens | 自动 | 自动、无代码改动 |
  | **Gemini** | 按存储时长 | 大型 prompt | 自定义 | 显式 cache 对象 |
  - Anthropic write 价 +25%（5min）/ +100%（1h）；高命中率工况净省 **70-90%**
- **设计 cache-friendly prompt 的 5 条**：
  1. **静态前缀放最前**：system prompt + 例子 + 工具描述 → 头部
  2. **动态用户输入放最后**
  3. **稳定 system prompt**：别每次 random 加水印 / 时间戳
  4. **示例集合稳定**：few-shot 例子要尽量复用
  5. **长上下文 RAG**：retrieved chunk 顺序稳定（按 id 排）能提高命中
- **Semantic cache 工程**：
  - **Redis LangCache** / **GPTCache** / **Vercel AI SDK cache** 主流
  - 流程：query → embed → 近邻搜 → 阈值过滤 → 命中返回 / 未命中走模型 + 写入
  - **命中率参考**：高重复客服 / FAQ 工况可到 **~73%**；通用助手 10-30%
  - **关键风险**：相似但语义不同 → 错答。**必须配质量监控 + 离线评估**（→ 6 章）
  - **阈值调优**：先严后松，单 user-level 配相似度阈值
- **Exact-match cache**：
  - hash(normalized_prompt + model + params) → 输出
  - 适合：固定流程的内部任务、批处理、CI eval gate（同 input 不重复跑）
  - 实现简单（Redis SETEX），命中 = 0 cost
- **命中率工程**（核心 SLO）：
  - **看板**：分 cache 类型的命中率（KV / prompt / semantic / exact）
  - **目标**：典型生产应用 prompt cache 命中率 **>60%**，semantic cache 阈值上 **>30%**
  - **失效信号**：模型升级、prompt 改写、tools 变了都会让命中率掉
  - **AB 实验**：上线新 prompt 前先 dry-run 看 cache hit rate 变化
- **常见反模式**：
  - 把动态变量（时间戳、UUID）放在 prompt 最前：**整段 cache 失效**
  - semantic cache 阈值太松：命中但语义错
  - 不监控命中率：以为开了 cache，其实命中率 5%
  - 缓存生成 / agent 工具调用的中间状态：会跨 session 串味
