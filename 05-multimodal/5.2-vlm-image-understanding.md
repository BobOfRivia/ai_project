# 5.2 VLM 调用与图像理解

> **核心问题**：2026 主流 VLM API 怎么调？OCR / 文档智能 / Visual QA / 截图理解 / Computer Use 视觉端各自该选什么？开源 VLM 什么时候自托管？
> **目标**：能为"任意一类图像理解需求"在通用 VLM + 专门 VLM + 开源 VLM 之间选对模型，并把 API 用法、token 计量、降级路径打通。
>
> **与 3.10 的关系**：3.10 讲 Computer Use Agent 的**决策循环与工具**，本节讲它依赖的**视觉理解能力**。

**展开重点**：
- **2026 VLM 格局**：
  - **闭源 / API**：GPT-5 / GPT-4o（MMMU-Pro 59.9%）、Claude（Sonnet 4.6 / Opus 4.7，原生多模态）、Gemini 2.5 Pro（复杂图像推理强）
  - **开源**：Qwen3-VL（阿里）、GLM-4.6V（智谱）、LLaVA-Video-72B、Aria；2026 已逼平 GPT-4o 等闭源
  - **边缘 / 设备端**：轻量 VLM 用于手机、无人机、AR 眼镜
- **典型任务与模型选择**：
  - **OCR / 文档智能**：通用 VLM 已能直接出 Markdown / JSON；高精度场景用 PaddleOCR / Surya / Mistral OCR 专模
  - **图像描述 / 标签**：通用 VLM 通吃，关注一致性时加 prompt 模板与结构化输出（→ 1.4）
  - **Visual QA**：用户问"图里有几个红色物体" → 通用 VLM；高精度计数走专模
  - **截图理解（UI）**：Computer Use 视觉端、Browser Agent；模型需懂 UI 元素与文字
  - **图像分类 / 检测**：传统 CV 仍快、便宜；除非要解释性，否则不必上 VLM
- **API 调用要点**：
  - 图像输入格式（URL / base64 / file id）、分辨率与 token 计费
  - 长图 / 多图 / 视频帧的拼接策略
  - 结构化输出（→ 1.4）；让 VLM 直接输出 JSON / Markdown 减少后处理
  - 流式输出与降级模型路由
- **成本与延迟**：图像 token 远贵于文本；高分辨率会爆 token；常见做法是预处理压缩、分块、缓存（hash → 解析结果）
- **边界**：多模态 RAG / 文档检索系统归 ch2；本节只讲单次调用与理解
