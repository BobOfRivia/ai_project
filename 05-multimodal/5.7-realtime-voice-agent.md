# 5.7 Realtime 语音对话（S2S vs Pipeline）

> **核心问题**：2026 语音 agent 的两条路线 —— Pipeline（STT → LLM → TTS）vs Realtime / S2S（gpt-realtime-2 / Gemini Live / Azure Voice Live）—— 该怎么选？端点检测、打断、工具调用、降级怎么做？
> **目标**：能为对话型产品（IVR、客服、车载、陪伴）在自然度、控制、合规、成本之间选对架构，并把 turn 管理、interruption、tool use 工程到位。
>
> **与 ch3 / ch4 的关系**：本节只讲**语音对话层架构**；agent 的 memory / planning / 工具属于 ch3，编排骨架属于 ch4。

**展开重点**：
- **两种架构对比**：
  - **Pipeline / 级联**：STT（5.5）→ LLM → TTS（5.6）；可换任意组件、可观测、合规友好、成本低；2026 **生产里仍占大头**
  - **Realtime / S2S**：音频→音频，模型内部理解 + 生成；结构性更低延迟、更自然；控制粒度弱、贵
- **2026 Realtime 三强**：
  - **OpenAI gpt-realtime-2**（2026-05-07）：GPT-5 class reasoning；Big Bench Audio 82.8%（前代 65.6%）；adjustable reasoning effort（min/low/medium/high/very high）；context 128k；$32/$64 per M token（约 $0.06 / $0.24 每分钟音频 in/out）
  - **gpt-realtime-translate**：70 入 / 13 出语言，实时对话翻译
  - **gpt-realtime-whisper**：低延迟流式 STT，与 Realtime 配对
  - **Google Gemini Live**：响应快、语言覆盖广
  - **Azure Voice Live API**：企业 / Azure 生态
- **延迟拆解**（Pipeline）：STT 100–500ms + LLM TTFT 200–2000ms + TTS TTFA 200–800ms + 网络 ⇒ 累加易超 1 秒；S2S 一次穿过，但实际差 200–400ms（许多对话场景不可感知）
- **选型矩阵**：
  - **Realtime 占优**：高端对话产品、车载、陪伴、需要情感与自然停顿
  - **Pipeline 占优**：电话 / IVR / 合规需要单独管控转写或合成、需要选 LLM、需要降级到便宜模型、需要可观测每一步 token
- **工程要点**：
  - **VAD / 端点检测**：服务端 VAD（Deepgram Flux 强）vs 客户端 VAD vs S2S 模型自管
  - **Barge-in / 打断**：用户打断时立即停 TTS、清空缓冲；Realtime 模型内置；Pipeline 需自己实现
  - **Tool use**：gpt-realtime 与 Gemini Live 都支持函数调用；Pipeline 自然支持（在 LLM 一层）
  - **降级**：Realtime → Pipeline 兜底（Realtime 故障 / 高峰 / 成本敏感时切换）
  - **可观测**：Pipeline 每段都可记录；Realtime 需要请求 server 端 transcripts
- **常见组合**：
  - Pipeline 主线：Deepgram Nova-3 + Flux（STT）→ GPT-4o-mini / Claude Haiku（LLM）→ Cartesia Sonic 4（TTS）
  - Realtime 主线：gpt-realtime-2 + 函数调用（工具）
  - 全栈：Deepgram / Cartesia / ElevenLabs / Azure 自家全栈
