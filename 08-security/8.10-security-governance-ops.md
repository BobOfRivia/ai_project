# 8.10 安全治理与运营

> **核心问题**：除了部署期控件，怎么把 AI 安全跑成"持续运营"？事件响应、SOC 集成、模型 / Prompt 审计、漂移监控怎么落地？
> **目标**：能为团队设计"AI 安全运营 runbook"：日常监控、事件响应、定期红队、合规审计、岗位职责，让安全不只是发布前的一道关。

**展开重点**：
- **AI Security Ops 与传统 SOC 的差异**：
  - 浪费 / 损失单位从"ms / 请求"变成"**$ / token**"和"**信任损失**"（→ 7.8 / 7.10）
  - 攻击向量不在网络层，而在**自然语言层 + 工具调用层**
  - 检测信号是**异常 prompt 模式 / 异常工具调用 / 异常输出**，不是 syscall / 端口扫描
- **日常监控指标**（与 6.10 / 7.10 共面板）：
  - **Guardrail block rate**：突涨 = 攻击尝试 / 突跌 = 审查器挂了
  - **Refusal rate**：拒答率，按类型拆分
  - **Tool use anomaly**：单 user 工具调用频次 / 类型异常
  - **Cross-tenant signals**：跨租户 prompt 模式相似度（共谋检测）
  - **Output PII leakage rate**：输出含 PII 比率
  - **Red Team CI pass rate**：每天 / 每 PR 趋势
- **事件分级与响应**（Incident severity）：
  | 级别 | 触发 | 响应 SLA |
  |---|---|---|
  | **P0** | 数据泄漏、有害内容大规模、监管事件 | < 30 min on-call、立即拦截路径 |
  | **P1** | 工具滥用、单租户大规模拒答 | < 2h |
  | **P2** | 单点注入成功、PII 泄漏单例 | < 24h |
  | **P3** | Guardrails 误拦率上升 | 下个 sprint |
- **事件响应 Runbook（基本动作）**：
  - **遏制**：关 feature flag / 降级到安全 prompt / 切断工具
  - **调查**：trace 回放（→ 6 章 LangSmith / Langfuse）、日志关联
  - **取证**：保留 prompt / output / tool calls（注意 PII 合规存）
  - **修复**：Prompt / Guardrails / 工具权限调整
  - **回归**：Promptfoo / Garak 加测试用例（→ 8.8）
  - **复盘**：blameless postmortem、更新威胁模型（→ 8.1）
- **模型与 Prompt 审计**：
  - 每个生产 prompt 有版本号 + diff 历史（Git）
  - 每次模型切换有 risk 评估（含安全 + 合规）
  - **Model Card** / **System Card** 内部维护：能力、限制、训练数据、已知风险
  - 高风险场景（EU AI Act 高风险 / NIST 高影响）需 **Conformity Assessment** 文档
- **漂移与回归监测**：
  - **Drift signals**：模型升级后 Guardrails 命中率变、拒答率变、PII 检出变
  - **A/B**：新 prompt / 新模型先 1% 流量
  - 定期重跑离线 eval + Red Team（每周 / 每月）
- **岗位与责任**（小团队也要明确）：
  - **AI Application Engineer**：应用层控件 + Guardrails + PII pipeline
  - **Security Eng / AppSec**：威胁建模 + 红队 + 事件响应
  - **DPO / Privacy / Legal**：合规、DPIA、跨境
  - **Product**：feature 风险 review、用户告知
  - 小团队用 **RACI**（Responsible / Accountable / Consulted / Informed）矩阵明确
- **与传统 SOC / SIEM 集成**：
  - LLM trace 日志（OTel GenAI semconv）转发到 SIEM（Splunk / Datadog / Elastic）
  - 关键事件触发 PagerDuty / Opsgenie
  - 与 IAM / Audit 系统联动（谁 / 何时 / 触发了 LLM）
- **持续合规**：
  - **EU AI Act 高风险**：登记、定期审计、CE 标识
  - **HIPAA**：年度审计、BAA 续签
  - **SOC 2 / ISO 27001 / ISO 42001 (AI 管理体系)**：审计员越来越懂 AI，准备 evidence
  - **训练数据 / Prompt / 审计日志**保留期与销毁策略
- **常见反模式**：
  - 把 AI 安全只当"上线前 Red Team"，没运营机制
  - 事件响应没 runbook：第一次出事人手乱
  - Guardrails 命中没人看，告警淹没
  - 安全只有工程负责，法务 / 产品不参与
  - 模型 / Prompt 改了不留版本：复盘无据
  - 合规审计前临时补文档，平时不留 evidence
