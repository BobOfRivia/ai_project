# 2.1 RAG 基础与决策框架

> **核心问题**：RAG 是什么？何时用（vs 长上下文 / 微调）？整体架构？
> **目标**：能在一个新场景下判断"该不该用 RAG、用什么形态的 RAG"。

---

## 一、RAG 是什么，解决什么问题

**RAG（Retrieval-Augmented Generation，检索增强生成）**：在模型生成回答之前，先从外部知识源检索出相关内容，把它拼进 prompt 作为上下文，让模型"看着资料回答"。

它要解决的是大模型的三个结构性短板：

| 短板 | 具体表现 | RAG 如何缓解 |
|---|---|---|
| **知识截止** | 模型只知道训练截止前的世界，不知道昨天发生的事 | 检索层实时接入最新数据 |
| **不懂私有知识** | 模型没见过你公司的文档、工单、代码库 | 把私有语料索引进检索层 |
| **幻觉 & 不可溯源** | 一本正经地编造，无法核对出处 | 答案挂靠检索到的原文，可引用、可审计 |

一句话记忆：**RAG = 把"开卷考试"的能力加给模型**。模型负责"理解题目 + 组织答案"，检索层负责"把对的那几页书翻到它面前"。

到 2026 年，RAG 已经从"技巧"变成了**任何需要基于私有/实时数据回答的 AI 应用的默认架构**——几乎每个企业知识库问答、客服助手、文档助手底层都是它。

---

## 二、整体架构：两条流水线，三个阶段

RAG 系统在物理上分成**离线**和**在线**两条流水线，逻辑上是三个阶段：

```
【离线 · Indexing 索引】只在数据变化时跑
  原始资料 → 加载解析 → 切片(Chunking) → 向量化(Embedding) → 写入向量库/索引
                                                                │
                                                                ▼
【在线 · 每次提问都跑】                                      ┌──────────┐
  用户提问 →(2.5 查询理解/改写)→ 检索(2.4) → 重排(2.6) ──────▶│ 索引/向量库│
                                    │                     └──────────┘
                                    ▼
              Retrieval 检索出 Top-K 相关片段
                                    │
                                    ▼
【Generation 生成】
  拼装上下文(2.7) → 塞进 Prompt → LLM 生成 → 带引用的答案
```

| 阶段 | 时机 | 干什么 | 对应本章小节 |
|---|---|---|---|
| **Indexing 索引** | 离线，数据变了才跑 | 把原始资料变成可快速语义检索的形态 | 2.2 文档处理、2.3 向量化 |
| **Retrieval 检索** | 在线，每次提问 | 找出与问题最相关的片段 | 2.4 检索、2.5 查询理解、2.6 重排 |
| **Generation 生成** | 在线，每次提问 | 用检索到的上下文合成答案 | 2.7 上下文组装与生成 |

**理解这张图，就理解了整章的骨架**——后面每一节都是在把这三个阶段里的某个环节做深、做对。

### 一条贯穿全章的关键认知：检索是瓶颈，不是生成

2026 年的行业共识：**当 RAG 出错，约 73% 的失败点在"检索"、而不是"生成"**。一个周末教程搭出来的朴素 RAG，检索出错率大约在 40%——模型会拿着**错误的文档**，一本正经地给出结构漂亮但内容错误的答案。

推论很直接：**LLM 已经很擅长"根据正确上下文合成答案"，难的是把正确的上下文喂给它**。所以本章的篇幅重心在 2.2–2.6（怎么把对的内容检索出来），而不是 2.7（怎么让模型说话）。

## 三、RAG 的三个成熟度层次（认知地图）

不要一上来就上复杂架构。RAG 有清晰的演进阶梯，**从下往上，复杂度和成本递增，只在下一层顶不住时才往上走**：

| 层次 | 形态 | 特征 | 什么时候用 |
|---|---|---|---|
| **Naive RAG** | 索引一次 → 检索即生成 | 最简单，也是正确的起点 | 语料干净、问题直接的场景 |
| **Advanced RAG** | 加查询改写、混合检索、重排等增强 | 针对朴素 RAG 的检索质量瓶颈逐点补强 | 检索质量不够、但流程仍确定 |
| **Modular / Agentic RAG** | 查询路由、多跳检索、自我反思纠错 | 按查询复杂度动态编排检索策略 | 复杂多源、多跳推理场景 |

- **Naive** 的细节在 2.2–2.4、2.7；
- **Advanced** 的增强手段散布在 2.5（查询改写）、2.4（混合检索）、2.6（重排）；
- **Modular/Agentic** 的完整形态集中在 2.8（Self-RAG / CRAG / RAPTOR / GraphRAG / Adaptive / Agentic RAG）。

> **落地原则**：绝大多数生产系统，"Naive RAG + 混合检索 + 重排 + 好的切片"就能解决 80% 的问题。上 Agentic RAG 之前，先确认前面几层的检索质量已经压榨到位。

## 四、决策框架：该不该用 RAG，用什么形态

这是本节最有价值的部分。面对"要让模型用上某批知识"的需求，2026 年有**三条路线**，不是非 RAG 不可。

### 4.1 三种知识增强路线

| 路线 | 一句话 | 知识放在哪 |
|---|---|---|
| **长上下文直塞** | 知识不多，全部塞进 prompt（可配 Prompt Cache 降本提速） | 每次请求的上下文里 |
| **RAG** | 用时才检索相关片段 | 外部向量库/索引 |
| **微调（Fine-tuning）** | 把知识/行为烤进模型权重 | 模型参数里 |

> **关于 "CAG"（Cache-Augmented Generation）**：近来常被当成"第四条路线"，但它本质就是**长上下文直塞 + Prompt/KV Cache**——把整批知识预载进缓存复用，省掉检索步。它的机制属于**推理优化**（KV cache 复用），和 Prompt Caching 是一回事（详见 [第 1 章 1.5 性能与成本] 与 [第 7 章 工程化]），**不是一种独立的检索架构**。目前它更像新概念/工程技巧，尚未成为经过大规模生产验证的成熟架构，且受硬约束（见 4.3）。所以本章把它归入"长上下文直塞"这条路线，不单列展开。

### 4.2 核心分工原则

一句话记住行业共识：

> **易变的知识放检索（RAG），稳定的行为放微调，别指望一个工具干两件事。**

- **RAG / 长上下文**：适合"**知识**"——会更新、要溯源、按需取用（产品手册、法规、工单、最新数据）。
- **微调**：适合"**行为/风格/格式**"——固定的语气、领域话术、输出结构、决策倾向；或让小模型在单一窄任务上逼近大模型（2026 微调最强的商业理由是**成本/延迟**，不是塞知识）。

对本项目：用户已具备算法背景、**微调默认不纳入应用层学习脉络**，所以下面重点讲 RAG vs 长上下文 的取舍。

### 4.3 RAG vs 长上下文：按语料规模的经验分界

现代模型上下文窗口很大，"能不能不做 RAG，直接全塞进去"是 2026 必答题。经验分界（数量级参考，非硬阈值）：

| 语料规模 | 推荐 | 理由 |
|---|---|---|
| **< ~200K tokens** | 长上下文直塞 + Prompt Cache | 常常比搭一套检索基建**更快更便宜**，别过度工程 |
| **~200K – 1.5M tokens** | 长上下文（+缓存）**或** RAG，看新鲜度/成本 | 塞得下但接近上限时对冷门细节会"注意力衰减" |
| **> ~1.5M tokens** | **RAG** | 语料太大装不下，必须按需检索 |

规模只是第一刀。**语料"塞得下"时，选长上下文还是 RAG，看这几个维度**：

| 维度 | 偏向长上下文（±缓存） | 偏向 RAG |
|---|---|---|
| 更新频率 | 稳定、少变 | 高频更新（要"新鲜度"） |
| 成本形态 | 一次预载、反复复用（缓存摊薄） | 按查询付检索/重排成本 |
| 引用可审计 | 弱（模型自己从大上下文里找） | **强需要引用溯源** |
| 查询分布 | 广泛、覆盖全语料 | 长尾、局部命中 |
| 多租户 | 共享同一份知识 | **每租户隔离**的知识 |

一句话：**长上下文赢在"小、稳、共享"；RAG 赢在"大、新、按租户隔离、要溯源"。**

> 补充认知：**长上下文没有杀死 RAG**。近年基准显示没有普适赢家，2026 的最佳实践往往是**混合**——事实用检索、风格/策略用微调、小而稳的共享知识直接塞长上下文（配缓存）。

### 4.4 什么时候"不该上 RAG"

反过来同样重要，避免过度工程：

- 知识小到能塞进上下文 → 直接长上下文 / CAG。
- 需求是固定风格/格式，而非知识 → 微调或纯 Prompt。
- 需要的是**精确数值/聚合查询**（"上季度华东区销售额"）→ 走 **Text-to-SQL 直连结构化库**（见 2.4），文本相似度检索反而不精确。
- 纯闲聊/创意生成、不依赖事实 → 不需要检索。

## 五、主流框架选型

不用自己从零造轮子。2026 年成熟的选择就四类，各有明确定位：

| 框架 | 定位 | 最适合 | 特点 |
|---|---|---|---|
| **LlamaIndex** | 数据/检索框架 | **文档密集型 RAG**（PDF、知识库、结构化数据） | 最认真对待"检索质量"，摄取/索引/查询引擎最强 |
| **LangChain / LangGraph** | 编排框架 | 检索只是其中一步的**Agent/多步工作流** | 灵活，擅长"检索 + 推理 + 调工具"的编排 |
| **Haystack**（deepset） | 企业级生产框架 | **强监管场景**（金融、医疗、法律、政府） | 模块化 pipeline、评估工具强、有商业支持 |
| **Dify** | 低代码/无代码平台 | 快速搭建、快速验证 | 上手最快，适合原型和轻量业务 |

**主流生产团队的组合拳**（多来源印证）：

> **用 LlamaIndex 做摄取与检索，用 LangGraph 做编排与 Agent。**

选型建议：
- 只做**知识库问答** → LlamaIndex 起步足够。
- 检索只是复杂流程的一环、要多步/多工具 → LangChain/LangGraph（编排见第 4 章）。
- 强监管、要审计和评估纪律 → Haystack。
- 要最快出 demo / 交给非工程同学维护 → Dify。

## 六、本节小结：一张决策清单

面对一个新场景，按顺序问自己：

1. **需要的是知识还是行为？** 行为/风格 → 微调或 Prompt；知识 → 继续。
2. **知识是精确结构化的吗？** 要精确数值/聚合 → Text-to-SQL；非结构化文本 → 继续。
3. **语料多大？**
   - < 200K tokens、稳定共享 → 长上下文 + Prompt Cache，先别做 RAG。
   - 大 / 高频更新 / 要溯源 / 多租户 → **RAG**。（中间 200K–1.5M 看新鲜度与成本二选一）
4. **上哪一层 RAG？** 先 Naive + 混合检索 + 重排；不够再上 Advanced；仍不够、且是复杂多跳 → Agentic（2.8）。
5. **用什么框架？** 文档问答 LlamaIndex；复杂编排 LangGraph；强监管 Haystack；快速验证 Dify。

> 记住贯穿全章的那句话：**检索是瓶颈**。决定用 RAG 之后，你 70% 的工程精力应该花在"把对的内容检索出来"上（2.2–2.6）。
