# 5.8 视频理解

> **核心问题**：长视频该怎么"喂"给模型？均匀抽帧 / shot-aware / adaptive / generative 怎么选？Gemini 长上下文直接吃整段 vs Twelve Labs 专模 vs 自建抽帧 + VLM 哪条路？
> **目标**：能为"视频搜索 / 内容审核 / 摘要 / QA / 屏幕录制理解"在精度、延迟、成本、可扩展之间选对路径。

**展开重点**：
- **三条主流路径**：
  - **抽帧 + 通用 VLM**（最常见）：均匀 / 关键帧抽样 → 拼接 → GPT-5 / Claude / Gemini 输入；上手快，缺点是长视频 token 爆炸
  - **原生长视频 VLM**：Gemini 长上下文支持几小时视频；LLaVA-Video-72B（LongVideoBench 66.8）；Aria（HourVideo 39.2，超 Gemini-1.5-pro）；适合需要时间推理的场景
  - **专门视频理解平台**：**Twelve Labs**（Marengo 嵌入 + Pegasus VLM），AWS Bedrock 上架；适合企业批量视频搜索 / 元数据生成；注意 Marengo 2.7 已于 2026-03-30 下线，需用新版本
- **Frame Sampling 工程**：
  - **均匀抽样**：等间隔，最简单；长视频效率低
  - **Shot-aware**（如 InfoShot）：先按镜头切分再选关键帧；语义连贯
  - **Adaptive**（如 VideoBrain）：模型学会按需采样；少 30–40% 帧仍涨 3.5–9%
  - **Generative**（如 GenS）：模型生成"该看哪几帧"；2026 给 GPT-4o / Gemini 都能加 buff
- **典型应用范式**：
  - **视频搜索 / 语义检索**：Twelve Labs Marengo 嵌入 + 向量库；或开源 video embedding + 自建
  - **视频摘要 / 章节切分**：Pegasus / Gemini；output Markdown + 时间戳
  - **内容审核**：抽帧 + VLM；或视频原生模型直接判
  - **屏幕录制理解**：UI 元素 + OCR + 时序操作；常配合 Computer Use（→ 3.10）
  - **创作辅助**：Twelve Labs Rodeo（2026-06）等 application-layer 工具直接把视频理解嵌入剪辑流
- **成本与延迟**：视频是 token 大户；预处理（降采样、关键帧、ASR 提取语音）能省 10× 以上；Twelve Labs 把嵌入和理解都做成离线 index，查询变廉
- **边界**：视频内容 RAG / 语义检索系统归 ch2；本节聚焦"模型如何理解一段视频"
