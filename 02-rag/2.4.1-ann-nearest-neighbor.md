# 2.4.1 ANN：向量库怎么"找最近邻"（补充）

> **定位**：2.4 正文里稠密检索一句话带过——"在向量库里用 **ANN** 找最近邻"。这一步是向量检索的**引擎**，决定了它的**速度、召回率、内存成本**。本篇讲清楚 ANN（近似最近邻）到底怎么工作，帮你看懂向量库的索引参数（HNSW 的 `M`/`efSearch`、IVF 的 `nprobe`…）在调什么、该怎么权衡。属工程/算法交叉，应用工程师**建索引和调优时用得上**。
> **核心问题**：几百万上亿个向量里，怎么在毫秒内找到和 query 最近的几个？为什么要"近似"？

---

## 一、为什么不能精确算，要"近似"

最直接的做法是**暴力精确检索（Flat / brute-force）**：query 向量和库里**每一个**向量都算一次距离，排序取 top-k。结果绝对准确，但——

- 复杂度 $O(N\cdot d)$：$N$ 是向量数、$d$ 是维度。100 万条 × 1024 维，每次查询要算 10 亿次乘加，**几百毫秒到秒级**，扛不住线上 QPS。
- 高维还有"**维度灾难**"：维度一高，暴力扫描的剪枝手段几乎都失效。

于是用 **ANN（Approximate Nearest Neighbor，近似最近邻）**：**牺牲一点点召回率，换几个数量级的速度**。它不保证找到"绝对最近的"，但能高概率找到"足够近的"——检索场景完全够用（本来 embedding 就有噪声，差第 8 近还是第 10 近无所谓）。

> **一句话**：ANN = 用"高概率对"换"快几百倍"。衡量它好坏的核心指标就是 **Recall@k**（找回的 top-k 里有多少是真正的 top-k）vs **延迟** vs **内存**——这是一个三角权衡，下面所有算法都在这个三角里选位置。

---

## 二、三种主流思路（+ 一种压缩术）

ANN 算法按"怎么缩小搜索范围"分三类，另有一类是"怎么把向量压小"：

| 思路 | 代表 | 一句话 | 长处 | 短处 |
|---|---|---|---|---|
| **图（Graph）** | **HNSW** | 建一张"邻居图"，从入口点贪心跳向 query | **召回/延迟俱佳**，事实标准 | 内存大、构建慢 |
| **聚类（Cluster）** | **IVF** | 先 k-means 分桶，只搜最近的几个桶 | 省内存、构建快 | 桶边界处易漏，召回略低 |
| **哈希（Hash）** | LSH | 相似向量哈希到同一桶 | 插入快、有理论保证 | 高维召回一般，实战少用 |
| **压缩（量化）** | **PQ** | 把向量压成短码，省内存+加速距离计算 | 内存降 10–100× | 有精度损失，常需重排补偿 |

实战里最常见的是 **HNSW**（追求召回、内存够）和 **IVF-PQ**（超大规模、内存吃紧）。下面分别拆开。

---

## 三、HNSW：邻居图 + 分层跳转（生产默认）

**HNSW（Hierarchical Navigable Small World，分层可导航小世界）** 是目前向量库的事实标准（Qdrant、Weaviate、Milvus、pgvector、FAISS、ES 都用）。

### 3.1 直觉：像跳表，但是图

- **底层**：把所有向量连成一张图——每个向量是节点，和它**语义相近的若干向量**连边（邻居）。相近的向量在图上"挨着"。
- **分层**：在底层之上再叠几层**越来越稀疏**的图（类比**跳表 skip list**）。上层节点少、边"跨度大"，用来**快速粗定位**；下层节点全、边"跨度小"，用来**精细定位**。每个节点出现在第几层是概率决定的（越高层越稀有）。
- **检索**：从**最上层的入口点**出发，在当前层里**贪心地跳向离 query 更近的邻居**，跳不动了就下沉一层继续，直到最底层收敛到 query 附近，返回最近的 k 个。整个过程是 $O(\log N)$——这就是它快的原因。

```
Layer 2   ●───────────────●          稀疏、大跨度：粗定位
              ╲          ╱
Layer 1   ●────●─────●────●          中等
           ╲  ╱ ╲   ╱ ╲  ╱
Layer 0  ●─●─●─●─●─●─●─●─●─●─●        全部节点、密集：精定位
                    ▲
                  query 落点附近收敛
```

### 3.2 三个关键参数（建索引/调优时要懂）

| 参数 | 管什么 | 调大 → | 典型值 |
|---|---|---|---|
| **M** | 每个节点的最大邻居数 | 召回↑、内存↑、构建慢 | 16–48（高维/高召回取大） |
| **efConstruction** | 建图时每步考察的候选数 | 图质量↑、构建慢 | 100–400 |
| **efSearch**（ef） | 查询时考察的候选数 | 召回↑、延迟↑ | 按目标召回调，几十~几百 |

- **`M` 和 `efConstruction` 建索引时定，改动要重建**——想清楚再建。`M` 直接决定内存（每个向量多存 `M` 条边）。
- **`efSearch` 查询时可动态调**：**这是你线上调"召回 vs 延迟"的旋钮**。要更高召回就调大 `efSearch`，用延迟换。
- **`efSearch` 必须 ≥ 你要的 k**，否则连候选都不够。

> **落地**：绝大多数场景 **HNSW + 默认参数**（M=16、efConstruction=200）起步，然后**只调 `efSearch`** 对着评估集（2.10）压到目标召回。内存不够再考虑 IVF-PQ 或给 HNSW 叠量化。

---

## 四、IVF：先分桶，只搜几个桶

**IVF（Inverted File，倒排文件）** 走聚类思路：

1. **建索引**：用 **k-means** 把所有向量聚成 `nlist` 个簇（每个簇一个中心，构成 **Voronoi 划分**）。每个向量归到最近的簇。
2. **检索**：query 先找**最近的 `nprobe` 个簇中心**，**只在这几个簇里**做精确检索，其余簇整块跳过。

- **`nlist`**：簇的总数（如 $\sqrt{N}$ 量级）。建索引时定。
- **`nprobe`**：查询时搜几个簇——**这是 IVF 的召回/延迟旋钮**。`nprobe=1` 最快但可能漏（query 恰好落在簇边界、真正最近邻在隔壁簇）；`nprobe` 调大 → 召回↑、延迟↑；`nprobe=nlist` 退化成暴力精确。

**优缺点**：构建比 HNSW 快、内存省；但**簇边界处天然会漏**，同等召回下延迟通常不如 HNSW。适合"数据自然成簇 + 内存敏感"的场景，且常和 PQ 搭配（下节）。

---

## 五、PQ：把向量压小（量化压缩）

**PQ（Product Quantization，乘积量化）** 不是"怎么缩小搜索范围"，而是"**怎么把每个向量压小**"——省内存、加速距离计算，常叠在 IVF 上（**IVF-PQ**）。

**思路**：把一个 $d$ 维向量**切成 $m$ 段子向量**，每段单独跑 k-means 学一本"码本"（如各 256 个中心）。原向量就用"每段最近的中心编号"表示——$m$ 个字节（每段 1 字节存 0–255）就够了。

- **举例**：1024 维 × float32 = 4096 字节/向量；切 128 段、每段 1 字节 → **128 字节/向量**，**内存降 32 倍**。
- 距离计算也变快：query 各段到 256 个中心的距离**预计算成查找表**，算文档距离时只需**查表相加**，不用逐维乘。
- **代价**：有损压缩，**精度下降**。所以实战常用 **IVF-PQ 粗筛 + 用原始向量对候选精算距离重排**（rerank）补回精度。

> **什么时候上 PQ**：**内存装不下全量原始向量**时（上亿级、或本地/边缘部署）。内存够就别上——HNSW 存原始向量召回更高更省心。

---

## 六、距离度量：别配错

ANN 找的是"最近"，但"近"按哪种距离算，必须和你的 embedding **训练时一致**：

- **余弦相似度（cosine）**：最常用；只看方向不看长度。多数 embedding（BGE、E5、Qwen3-Embedding…）按它训练。
- **内积 / 点积（dot / IP）**：向量已归一化时，等价于余弦。有些模型（如部分 DPR）用它。
- **欧氏距离（L2）**：看绝对位置，图像等场景用。

> ⚠️ **配错度量 = 召回崩盘**。看 embedding 模型卡说明它用哪种，向量库索引就配哪种。归一化后用 cosine 是最稳的默认。

---

## 七、和"过滤"的关系（工程坑）

2.4 第六节讲了**元数据过滤**（只搜近一年的、有权限的…）。它和 ANN 有个著名冲突：

- **朴素做法（后过滤 post-filter）**：先 ANN 取 top-k，再按元数据筛——可能筛完只剩两三条，**召回被打穿**。
- **过滤内嵌（filtered ANN）**：现代向量库（Qdrant、Weaviate、Milvus）支持**在图/桶遍历时就跳过不满足条件的节点**，兼顾召回与过滤。**选型时确认你的库支持"filtered search"**，别自己在外面 post-filter。

---

## 八、选型速查

```
内存够 + 要高召回     → HNSW（默认，只调 efSearch）
超大规模 / 内存吃紧   → IVF-PQ（nprobe 调召回 + 原向量重排补精度）
数据频繁增删          → 关注库对增量索引的支持（HNSW 增删相对麻烦）
需要过滤              → 选支持 filtered ANN 的库，别 post-filter
```

对应用工程师的**核心 takeaways**：

1. **默认 HNSW**，把 **`efSearch`** 当"召回 vs 延迟"旋钮，对着评估集调——这是你 90% 会用到的操作。
2. **`M`/`efConstruction`/`nlist` 是建索引参数**，改动要重建，想清楚再建。
3. **内存不够才上量化（PQ/SQ）**，且配重排补精度。
4. **距离度量必须和 embedding 训练一致**，cosine 是最稳默认。
5. **过滤要用库原生的 filtered search**，别在结果外面 post-filter 打穿召回。
6. ANN 是**近似**的：召回不会 100%，这是设计使然——用 2.10 的评估量化到"够用"即可，别追求极限召回而牺牲全部延迟。

> 一句话收束：**ANN = 用可控的近似换数量级的速度**。你不需要自己实现它，但要看懂向量库那几个索引参数在这个"召回—延迟—内存"三角里替你做了什么选择，并按业务把旋钮拧到对的位置。
