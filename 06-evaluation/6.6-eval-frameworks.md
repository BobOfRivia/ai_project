# 6.6 评估框架对比与选型

> **核心问题**：Promptfoo / DeepEval / RAGAS / Inspect AI / MLflow Evaluate 在 2026 各占什么位置？什么时候单选、什么时候组合？
> **目标**：在动手写第一个 eval 之前，能根据团队语言栈、CI 工具、垂域类型选对框架，避免一年后整套迁移。

**展开重点**：
- **五大框架定位**（2026）：
  - **Promptfoo**：YAML / CLI 驱动；强在**多模型 / 多 prompt 对比 + red-team / 对抗测试**；含 40+ red-team 插件；典型场景：跨 provider sweep、安全测试
  - **DeepEval**：**pytest 风格**；14+ 内置指标；最适合 Python-native 团队把质量门禁塞进 pytest CI；像写单测一样写 eval
  - **RAGAS**：**RAG 专模指标库**（faithfulness / context precision / recall / utilization / noise sensitivity）；不是框架是库；嵌入到 Promptfoo / DeepEval / 平台里用
  - **Inspect AI**：Anthropic 系，task-decorator 风格；强在 **agent / capability eval at scale**；公开 benchmark（GAIA、SWE-bench）的事实标准
  - **MLflow Evaluate**：MLflow 套件自带；适合已经在 MLflow 上做实验追踪的团队
- **其他选择**：
  - **DeepChecks**：持续验证 + drift detection
  - **UpTrain**：notebook 风格，开发期友好
  - **TruLens**：与 LangChain 紧耦合，trace + eval 一体
- **风格分类**（2026）：
  - **pytest 风格**：DeepEval / DeepChecks
  - **YAML 风格**：Promptfoo
  - **notebook 风格**：UpTrain
  - **task-decorator 风格**：Inspect AI / RAGAS
  - **平台-bundled**：MLflow Evaluate / LangSmith Evaluations / Braintrust Eval
- **生产组合（2026 最常见两套并跑）**：
  - **Promptfoo + DeepEval**：Promptfoo 做 red-team / 对抗 + 跨模型对比；DeepEval 做 metric CI gate
  - **DeepEval + RAGAS**：RAG 应用，RAGAS 指标在 DeepEval 跑
  - **Inspect AI + 平台**：agent / capability eval 走 Inspect，trace 落 LangSmith / Phoenix
- **选型决策树**：
  1. 主要做 RAG 评估？→ **RAGAS**（嵌入到下面框架）
  2. 团队 Python + 想塞 pytest？→ **DeepEval**
  3. 想跨 provider sweep + red-team？→ **Promptfoo**
  4. 主要做 agent / 复杂 capability？→ **Inspect AI**
  5. 已在 MLflow 上？→ **MLflow Evaluate**
- **注意**：框架是 **execution layer**；platform layer（LangSmith / Braintrust 等）会**把它们集成**，落生产时常常通过 platform 间接调用框架
