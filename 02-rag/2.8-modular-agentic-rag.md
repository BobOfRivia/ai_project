# 2.8 Modular / Agentic RAG 架构

> **核心问题**：Self-RAG / CRAG / RAPTOR / GraphRAG / Adaptive RAG / Agentic RAG / Long RAG 各自的适用场景与权衡？
> **目标**：能识别这些架构的适用场景，知道工程代价。

> **先消歧（重要）**：本节讲的是三范式（Naive → Advanced → Modular/Agentic，见 [2.1 第三节](./2.1-rag-fundamentals.md)）里的**第三层 Modular/Agentic**——带反思纠错、图谱、Agent 编排的**架构模式**。口语里常把它们统称"高级 RAG"，但**本书语境里的 "Advanced RAG" 是另一回事**：它专指范式②那层"查询改写 + 混合检索 + 重排 + 上下文优化"的**增强手段，已拆进 2.4–2.7**，不在本节。本节只讲范式③的 **Modular/Agentic** 架构模式。

---

## 一、先立一条铁律：别在检索不牢时上这些重型架构

2.2–2.7 走通了 Naive/Advanced RAG 主链路。本节这些 Modular/Agentic 架构是**在主链路顶不住某类问题时的针对性升级**，不是"更高级所以更好"。

> **2026 年最贵、最常见的错误：在底层检索还不扎实时，就急着上 agentic 编排。**"一个在弱检索器上反复循环的 agent，只是花更多钱、更精致地给出错误答案。"

所以进本节前先确认：**混合检索 + 重排 + 好切片（2.2/2.4/2.6）已经压榨到位了吗？** 是，再往下看。

---

## 二、认知地图：三条主线 + 一个特例

七个架构不要死记，按**它们解决的问题**归成三类：

| 主线 | 解决什么问题 | 架构 |
|---|---|---|
| **① 反思纠错** | 检索到的东西不对 / 答案没证据支撑 | Self-RAG、CRAG |
| **② 结构化组织** | 单块检索答不了"跨文档关系"和"全局汇总" | RAPTOR、GraphRAG |
| **③ 路由调度** | 不同复杂度的问题该走不同管线 | Adaptive RAG、Agentic RAG |
| （特例） | 上下文够长，干脆少切多塞 | Long RAG |

---

## 三、主线一：反思纠错

核心思想：**给 RAG 加一个"自我检查"环节**，不盲目相信一次检索的结果。

### 3.1 CRAG（Corrective RAG，纠错式）—— 最实用

- **怎么做**：加一个**评估器模型**给检索结果打分。分数太低 → 不信内部库，**触发 web 搜索或改写查询**去别处找。
- **为什么最实用**：**换掉底座 LLM 依然有效**（评估器和主模型解耦），是**最贴近企业落地**的自纠错方案。
- **成熟度**：✅ **LangGraph 的 CRAG 教程是 2025 年被复现最多的 RAG 模式**，工程资料丰富，可放心上。
- **适合**：知识库覆盖不全、需要"库里没有就去外部找"的场景。

### 3.2 Self-RAG（自我反思）

- **怎么做**：模型在生成过程中自己发信号——**要不要检索、检索到的相不相关、自己的答案有没有被证据支撑**。
- ⚠️ **成熟度提醒**：**原版 Self-RAG 需要用特殊反思 token 微调模型**（偏研究、落地门槛高）。生产里的做法是**用现成 LLM 近似**：把每步包成显式评估 prompt（"这段相关吗？"），用**结构化输出**（JSON `is_relevant: true/false`）来 gate 流程。LlamaIndex 有 `SelfRAGQueryEngine` 干净地实现了这个模式。
- **适合**：对答案可靠性要求高、愿意多花推理成本换质量的场景。

> 一句话对比：**CRAG 更工程化、即插即用；Self-RAG 思想更完整但原版偏研究，生产用近似版**。多数团队先上 CRAG。

## 四、主线二：结构化组织

核心思想：**改变"知识的组织方式"，让检索能回答单块层面答不了的问题**（跨文档关系、全局汇总）。

### 4.1 RAPTOR（递归树摘要）

- **怎么做**：对文档块**递归地聚类 + 摘要**，建成一棵树——叶子是原始块，父节点是子节点的摘要，一路汇总到根。检索时可**多粒度**命中（要细节走叶子、要概览走上层摘要）。
- **收益**：QuALITY 基准 **+20 个百分点**（GPT-4）。
- **适合**：长文档、需要"既能答细节又能答全局概览"的场景。比 GraphRAG 轻，是"想要层次化能力但不想上知识图谱"的中间选择。

### 4.2 GraphRAG（知识图谱 RAG）—— 强大但要算清成本

- **怎么做**：不再只切块 embedding，而是**抽取实体和关系建成知识图谱**，再对图做社区摘要 / 图遍历检索。
- **收益（关系型/多跳问题）**：
  - 多跳任务准确率 **86%，而向量 RAG 只有 32%**。
  - 查询涉及实体 **超过 5 个时向量 RAG 准确率趋近 0，GraphRAG 在 10+ 实体下仍稳定**。
- ⚠️ **成本诚实**（这是关键决策点）：
  - **建索引比向量 RAG 贵 10–40 倍**：同一批语料向量库 <$5，GraphRAG 实体抽取 + 社区摘要要 $50–200；1 万文档的知识库，跑第一条查询前就是四位数的索引成本。
  - 好消息：成本在快速下降。**LightRAG**（扁平图、简化抽取）能以约 **1/100 的索引成本拿到 GraphRAG 70–90% 的质量**；微软还出了更省的 **LazyGraphRAG**。
- **什么时候值得**：**只有当答案依赖"实体/事件之间的关系"、而非单纯词面匹配时**。甜区大致是 **>1k 文档、<1M tokens**；再大 LightRAG 更划算。
- **落地现实**：**多数团队生产里仍跑向量 RAG，切到 GraphRAG 的都是有明确关系推理需求**。别默认升级。

## 五、主线三：路由调度

核心思想：**不是所有问题都走同一条管线**，按复杂度动态调度，平衡成本与质量。

### 5.1 Adaptive RAG（复杂度分级路由）

- 已在 **2.5 第三节**展开：小分类器把 query 分为直答 / 单跳 / 多跳，路由到不同成本的管线（延迟 -35%、成本 -28%、准确率 +8%）。
- 这里定位：它是**连接"简单管线"和下面"agentic 重管线"的调度层**——简单问题别浪费 agent。

### 5.2 Agentic RAG —— 五种可落地的生产模式

**一句话**：不再走固定管线，而是让一个 Agent 在检索这件事上做决策——**要不要检索、检索几次、查哪个源、怎么改写、够了没**。下面五种模式覆盖了绝大多数生产系统（已有 ServiceNow P1 事件管理、Workday HR 等真实部署）；它们**不是互斥的**，真实系统常是"Router 兜底 + 复杂 query 触发 ReAct / Plan-Execute"的组合。

> **边界**：这里只讲"把 Agent 用在检索上"的**检索切面**——控制流、停机条件、护栏、成本。Agent 本体机制（ReAct 循环、Plan-Execute、多 Agent 协作、工具调用协议）在**第 3、4 章**，不在此重复。

先给对照表（由轻到重排序）：

| 模式 | 核心动作 | 额外成本 | 成熟度 | 什么时候上 |
|---|---|---|---|---|
| **① Router** | 一次决策，选源/选管线 | +200–400ms，无循环 | ✅ 最成熟 | 有 ≥2 个知识源、问题类型可判别 |
| **② ReAct（迭代反思）** | retrieve→critique→改写重检 | 3–10× token、2–5× 延迟 | ✅ 成熟 | 单次检索答不全的开放式问题 |
| **③ Plan-and-Execute** | 先拆子问题再并行检索 | 多路检索，延迟方差小 | ✅ 成熟 | 明确多跳/比较/多部分问题 |
| **④ Multi-agent** | 多专精 agent + 编排 | 五种里最重 | 🟡 按需 | 知识分散多域、单 agent 顾不过来 |
| **⑤ Self-RAG** | 把"该不该信"内化为评估步 | 每步多一次判断 | 🟡 原版偏研究 | 对 faithfulness 要求高 |

#### ① Router（路由 Agent）—— 80% 团队的第一步，也是最该先上的

- **怎么做**：一个轻分类器（小模型 / 规则+embedding）看 query，判断走哪条路——便宜的向量直查、BM25、SQL、web 搜索，或干脆直答不检索。**只做一次决策，不迭代**。
- **控制流 / 数据**：单次调用，典型 **+200–400ms**。经验分布是 **classic/简单路径吃掉 70–85% 流量，agentic 重路径只留给 15–30%**——这条路由本身就是后面所有重模式的**成本闸门**。
- **判别信号**：query 长度、多跳标志词（"相比…""在…背景下"）、涉及实体数、用户角色 / 价值敏感度。**Router 本质是把 [2.5 查询理解](./2.5-query-understanding.md) 的多源路由 + 复杂度分级"架构化"成一个显式节点**。
- **LangGraph 落地**：一个 `route` 节点 + `conditional_edges` 分流到不同子图，是最小可用的 agentic 形态。
- **代价 / 风险**：几乎无循环开销；主要风险是**错分**（把多跳问题误判成简单直查）——所以路由分类器要进评估集**单独盯准确率**。

#### ② ReAct / 迭代反思检索 —— agentic 的核心循环

- **怎么做**：`retrieve → 读结果 → 结构化 critique → 改写查询再检索 or 收敛`。关键是 critique **必须结构化**，例如：
  ```json
  { "answered": ["公司成立年份"], "missing": ["2025 年营收"],
    "next_query": "X 2025 revenue", "confidence": 0.62, "should_continue": true }
  ```
- **三件套停机条件**（缺一不可）：
  - **置信阈值** 0.75–0.85（注意"置信度虚高"，阈值要拿 ground truth 校准）；
  - **硬迭代上限** 3–7 轮、**生产推荐 2–3**——反思的质量增益几乎全在第 1–3 轮，**超过 5 轮极少划算**；
  - **token 预算** 20–40k + **wall-clock** 30–60s 兜底。
- **头号护栏**：**没有结构化 critique，agent 只会原地重复同一次检索**（"spams the same retrieval"）——必须逼它**指名"缺了什么"**，而不是笼统"再多找点"。
- **代价**：**3–10× token、2–5× 延迟**——这正是为什么 Router 必须先把它挡在 15–30% 流量内。
- **何时**：单次检索答不全、需要"发现缺口→补检索"的开放式 / 研究型问题。

#### ③ Plan-and-Execute / 查询分解 —— 多跳问题的主力

- **怎么做**：planner **先**把复杂问题拆成 **2–6 个可独立检索的子问题**（一棵浅树），**并行**检索每个子问题 → 汇总 → 有子问题失败则重规划 / 重试。
- **硬约束**：
  - 子问题必须**可独立检索**（树内**无前向引用**——A 的检索不能依赖 B 的结果，那种得退回 ReAct 的串行循环）；
  - **树深最多 1 层嵌套、≤6 个叶**，否则检索量指数爆炸。
- **vs ReAct**：**先规划后执行 → 可并行、延迟方差小**（3 个子问题并行 2s，串行要 6s）；ReAct 更灵活但逐步串行、方差大。**能一次拆清楚的用 Plan-Execute，得边查边调整的用 ReAct。**
- **何时**：明确的多跳、比较（"A 和 B 在 C 上的差异"）、多部分问题。

#### ④ Multi-agent Retrieval —— 知识分散在多域时才上

- **怎么做**：一个 orchestrator / supervisor 协调**多个专精子 agent**，每个负责一个域或数据源（内部库 / 邮件 / web / SQL），各自检索后由编排层汇总。
- **落地红利**：每个 agent 隔离在**自己的 collection**——天然是**多租户 / 权限的硬隔离边界**。
- **代价**：五种里**最重**（编排开销 + 多路 LLM 调用），也最容易在协调层出问题。
- **何时**：**只有当"知识很少在同一个 collection 里"、单个 agent 无法胜任所有检索**时才上。**别默认升级**——很多"多 agent"需求，其实一个带多工具的 ReAct agent 就够了。

#### ⑤ Self-RAG（生产近似版）

- 机制已在 **3.2** 讲过：把"要不要检索 / 检索到的相不相关 / 答案有没有证据"内化为 agent 的**评估步**。作为 agentic 模式，它更像给上面几种循环**加一道质量闸**。
- 原版靠 **control token 微调**（KILT faithfulness **+13 点**），落地门槛高；生产里用**现成 LLM + 结构化输出**近似（见 3.2）。
- **何时**：对 **faithfulness / 可审计性**要求高、愿多花一次判断成本的场景。

#### 横切护栏：让 agentic RAG 在生产里不失控（落地关键）

这几条比"选哪个模式"更决定成败：

- **三层预算，硬编码**：迭代上限 **2–3** + token 预算 **20–40k** + wall-clock **30–60s**——任一触顶即收敛返回。
- **成本头号杠杆 = 复杂度路由**：用 Router 把 agentic 路径压到 **20–30% 流量**；planning / evaluation 等辅助步**用小模型**（如 Claude Haiku）；对结构相似的 query **缓存分解结构**；**prompt caching** 命中 70%+ 可省 **40–60%** 成本。
- **必盯信号 = loop termination rate**：大量 query 打到 MAX_ITER，说明**要么检索覆盖不足、要么相关性阈值没校准**，是最灵敏的健康度指标。此外盯工具选择准确率、分解质量、P50/P90/P99 延迟。
- **观测**：每次 retrieve / reflect / tool-call 打一条 **OpenTelemetry span**（时间戳、token、决策理由）。
- **评估**：从真实用户 query 建 **100–200 条 golden set**，**按轮打分** + 跟每 query 成本，**每周看趋势**而非绝对值（RAGAS 四件套细节见 [2.10](./2.10-rag-engineering.md)）。

#### 组合与选型（务实版）

- **默认骨架**：Router 兜底 + 简单 query 走 classic RAG + 复杂 query 触发 ReAct 或 Plan-Execute；Self-RAG 作为质量闸叠在生成前；Multi-agent 仅当知识确实跨多域。
- **Build vs Buy**：需要专有检索逻辑 / 私有化 / 自定义编排 → 自己搭 **LangGraph**（stateful 多 agent、`conditional_edges` 天然映射重试循环）或 **LlamaIndex Workflows**（文档 ingest 与 query 紧耦合时更顺手）；用例落在标准模式、要几天内上线 → 直接上托管平台。
- ⚠️ **重申铁律**：这五种都建立在**检索已牢**（混合检索 + 重排 + 好切片，2.2/2.4/2.6）之上。**在弱检索器上套 agent，只是更贵、更精致地给出错答案。**

## 六、特例：Long RAG（大块 + 长上下文）

- **怎么做**：上下文窗口够大时，**少切、切大块**，直接把大段内容塞给长上下文模型，减少切碎带来的语义割裂。
- **权衡**：省了精细切片的麻烦，但要注意 **lost-in-the-middle（2.7）** 和成本；本质是"用模型的长上下文能力换检索精细度"。
- **和 2.1 的呼应**：这是"RAG vs 长上下文"光谱上偏长上下文的一端，语料能塞下时可考虑。

## 七、选型决策：什么时候上哪个

按"先便宜后贵、先简单后复杂"的顺序判断：

| 症状 | 优先考虑 | 别急着上 |
|---|---|---|
| 主链路够用 | **不上这些架构** | 所有 |
| 库里常没有、要去外部找 | **CRAG** | GraphRAG |
| 要答案强可靠、可接受慢 | Self-RAG（近似版） | Agentic |
| 长文档要"细节+全局概览" | **RAPTOR** | GraphRAG |
| 答案依赖实体间关系/多跳、且预算够 | GraphRAG（或先试 LightRAG） | — |
| 问题复杂度差异大、要控成本 | **Adaptive 路由** | — |
| 有多个知识源、要按类型分流 | Agentic-**Router**（最先上） | 重循环模式 |
| 明确多跳/比较、可先拆子问题 | Agentic-**Plan-Execute** | — |
| 开放式、要边查边补 | Agentic-**ReAct 迭代反思** | — |
| 知识跨多域、单 agent 顾不过来 | Agentic-**Multi-agent** | 别默认升级 |
| 语料塞得下、想省切片 | Long RAG | — |

**贯穿原则**：**每上一层都要能说出它修复的具体问题和愿付的代价**（成本/延迟/复杂度）——这正是我们知识整理的"成熟度分级、权重匹配"原则在架构选型上的应用。

## 八、本节小结

1. **铁律**：检索不牢别上这些重型架构，尤其别过早上 agentic。
2. **三条主线**：反思纠错（CRAG/Self-RAG）、结构化组织（RAPTOR/GraphRAG）、路由调度（Adaptive/Agentic）。
3. **最实用起点**：**CRAG**（LangGraph 教程成熟、换底座仍有效）。
4. **GraphRAG 算清成本**：贵 10–40 倍，只为关系/多跳问题；先评估 LightRAG。
5. **RAPTOR 是轻量层次化选择**：想要全局概览但不想上图谱。
6. **Agentic 是终点不是起点**：门槛是"检索已牢 + 问题确实复杂"。五种落地模式由轻到重——**Router（先上、当成本闸门）→ ReAct 迭代反思 → Plan-Execute 分解 → Multi-agent → Self-RAG 质量闸**；比选模式更决定成败的是**横切护栏**（三层预算、复杂度路由压到 20–30% 流量、盯 loop termination rate）。
7. **Agent 机制细节**看第 3/4 章，本节只讲"用在检索上"的切面。
