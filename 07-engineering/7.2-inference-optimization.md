# 7.2 推理优化技术总览

> **核心问题**：FP8 / PagedAttention / continuous batching / speculative decoding / Flash Attention 3 这些"算法侧"技术，应用工程师到底要懂多少？
> **目标**：能在选模型 / 选推理框架 / 评 cloud API 时，看懂"为什么这家比那家快/便宜"，避免被 marketing 数字误导。

**展开重点**：
- **2026 推理优化共识栈**：FP8 + Flash Attention 3 + continuous batching + speculative decoding 组合，比朴素 FP16 静态 batching **5-8x cost-efficiency**（H100）
- **核心技术速览**（应用工程师视角）：
  - **量化（Quantization）**：
    - **FP8** 已是 Hopper（H100/H200）质量/性能黄金线
    - INT8 / INT4 老一代 GPU 用，质量损失需要评估
    - 你看到"models 在 FP8 上跑"= 同模型成本砍半起步
  - **PagedAttention（vLLM 原创）**：
    - KV cache 浪费从 60-80% 降到 <4%
    - 是 vLLM 性能领先的根本原因
  - **Continuous batching**：
    - 旧静态 batching：等批次满了才推
    - 连续 batching：请求随时进出，吞吐 **+23x**
    - 几乎所有现代推理服务（vLLM / SGLang / TGI）都默认开
  - **Speculative decoding**：
    - 小 draft 模型快速生成候选，大模型只做验证
    - 提速 **2-3.6x**（TensorRT-LLM 已 GA）
    - 对延迟敏感场景必开
  - **Flash Attention 3**：
    - H100 上重写的 attention kernel
    - 与 FP8 / 长上下文协同时收益最大
  - **KV cache 压缩前沿**：
    - vLLM FP8 KV cache（GA）：2x 压缩、零损
    - **TurboQuant**（Google, ICLR 2026）：6x 压缩、attention 8x 加速
    - **CommVQ**（2-bit）：KV cache size -87.5%；2026 上半年开始进生产
    - 让 128k context 在单卡 4090 上跑得动
  - **RadixAttention（SGLang）**：
    - 共享前缀缓存复用（agent / 多轮 / RAG 场景）
    - 同 prompt 前缀的吞吐显著超 vLLM
- **应用工程师要带走的 3 件事**：
  1. **选模型 / 选 provider 时，问 FP8 / continuous batching / 推测解码是否支持**
  2. **agent / RAG / 共享前缀场景**，SGLang / Anthropic prompt cache 比纯 vLLM 单点强
  3. **不要重复造轮子**：上述全部由 vLLM / SGLang / cloud provider 内置，应用层只做"选型 + 配置"
- **常见误区**：
  - 看 throughput benchmark（请求/秒）而不看 latency（首 token / 全 token）
  - 把训练侧的 LoRA / 微调与推理侧的 quantization 混为一谈
  - 用 INT4 跑生产业务、没做质量回归（→ 6 章）
