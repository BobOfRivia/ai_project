# 7.8 限流、并发、容灾、流式

> **核心问题**：怎么防 429 雪崩、retry 风暴、单 provider 故障？流式响应下并发模型有什么不同？
> **目标**：把"3 原语 + bulkhead + 流式架构"做成默认骨架，让 LLM 应用面对 provider 抖动、流量峰值、上下游故障都不挂。

**展开重点**：
- **2026 共识 3 原语**（"传统 SRE 适配 LLM——浪费单位从 ms/请求变成 $/token"）：
  - **Token bucket per identity**：每用户 / 租户 / API key 一个桶；防滥用 + 公平
  - **Circuit breaker per pattern**：按 provider × model × endpoint 维度断路；半开探针恢复
  - **Fallback chain per route**：每条业务路由配一条降级链
- **限流（Rate limiting）层次**：
  - **Gateway 层**：第一道防线，按 identity / model 限
  - **Provider 层**：OpenAI / Anthropic 自己的 TPM / RPM 上限
  - **应用层**：业务粒度限流（"单用户每分钟最多 3 次 deep research"）
  - **Cost 层**（2026 新）：按 token 或 $ 限，超预算拒绝（→ 7.10）
- **Retry 策略**（2026 必知）：
  - **指数退避 + jitter**：避免同步重试雪崩
  - **可重试 vs 不可重试**：429 / 5xx / timeout 可重试；4xx 业务错误别重试
  - **总预算上限**：每请求最多 N 次重试 + 总时长 ≤ 业务 SLA
  - **Retry storm 防护**：上游 retry × 下游 retry = 平方级负载（LangGraph multi-agent 是高发场景）
- **Circuit Breaker 实现**：
  - 状态：Closed → Open → Half-open
  - 触发：错误率 > 阈值 OR 连续 N 次失败
  - 半开：放 1-N 个探针请求看是否恢复
  - 库：Resilience4j（Java）、Polly（.NET）、PyBreaker（Python）、`resilient-llm`
- **Fallback chain（降级链）**：
  - **同 provider 内**：opus → sonnet → haiku
  - **跨 provider**：Claude → GPT-4.1 → Gemini → 本地小模型
  - **降级到非 LLM**：模板 / 规则回退 / "稍后再试"
  - **缓存兜底**：命中缓存优先于 fallback chain
- **Bulkhead（舱壁）**：
  - 不同租户 / 不同业务 / 不同优先级分开**独立并发池**
  - 防止"一个 P0 长任务把所有 worker 占满"
- **流式（Streaming SSE）并发**：
  - SSE / WebSocket 连接是长连接，并发数 ≠ QPS
  - 反向代理需配长连接超时（nginx `proxy_read_timeout`）
  - 取消传播：用户断开 → 必须取消上游 provider 流（节约 token）
  - 流式过程中 partial 失败要决定：截断 / 重发 / 续传
- **Timeout 分层**：
  - **首 token timeout**（TTFT）：3-10s 量级
  - **整体 timeout**：与业务相关，复杂 agent 可达 60min（Anthropic Messages API 默认）
  - **client / proxy / provider** 三层 timeout 都要量、不要冲突
- **常见反模式**：
  - 只 retry 不限流 → 雪崩
  - 限流但没 fallback → 大量用户被拒
  - SSE 不传播 cancel → 用户离开了，上游还在烧 token
  - 配置 circuit breaker 但**从未触发演练**
