# 7.10 FinOps 与成本治理

> **核心问题**：怎么让团队 LLM 月支出不爆？哪个功能在烧钱？预算超限时是 alert 还是真的拦截？
> **目标**：把"四杠杆（tag / count / aggregate / enforce）"做成治理骨架，让成本和质量（→ 6 章）一起进入工程化流程。

**展开重点**：
- **2026 LLM FinOps 共识：四杠杆**（必备顺序）：
  1. **Tag every call**：每次 LLM 调用打上 `tenant / feature / user / env` 标签
  2. **Count tokens authoritatively**：用 provider 返回的 usage，不是估算
  3. **Aggregate per feature**：把 cost 汇总到产品功能 / 团队维度
  4. **Enforce per-feature budget**：超预算 → 降级 / 拒绝 / 告警
- **Visibility vs Enforcement**（决策起点）：
  - **Visibility**：看得见但拦不住（仪表盘 / 月报）
  - **Enforcement**：在 gateway 层真拦截（"超预算请求拒绝"）
  - 起步先 visibility、跑 1-2 个月后再上 enforcement
- **Chargeback vs Showback**：
  - **Showback**：每团队"看到"自己花了多少，预算仍在中央
  - **Chargeback**：真把成本计到团队 P&L
  - 大公司多走 showback → chargeback 演进
- **实施层次**：
  - **Gateway 层（Portkey / LiteLLM）**：API key per feature；预算 enforcement
  - **应用层**：标签透传（header / OTel attribute → 6.7 GenAI semconv）
  - **数据层**：FinOps 平台聚合（Finout / Amnic / CloudZero / Bifrost）
- **监控指标**（与 6.10 互补）：
  - **$ / request / feature**：单位成本
  - **Token in / out 比例**：异常膨胀信号
  - **Cache hit rate**（→ 7.7）：失效会让成本暴涨
  - **Routing 分布**：cheap vs frontier 比例漂移
  - **Anomaly detection**：单功能日支出突增 → 自动告警
- **预算 enforcement 模式**：
  - **硬上限**：超额直接 429 / 降级
  - **软上限**：超额告警 + 自动切换便宜模型
  - **配额池**：每用户 / 每天 token 池
  - **优先级**：付费用户优先、免费用户先降级
- **常见反模式**：
  - 月底看账单大吃一惊（无 per-feature 归因）
  - "我们设置了预算告警"——实际上从未触发拦截
  - 把 cache hit rate 和成本割裂监控
  - reasoning / thinking 模型的"思考 token"没算（Claude / GPT-5 思考 token 计费）
- **运营节奏**（与 6.10 / Org runbook 对齐）：
  - **周度**：分功能 cost 趋势
  - **月度**：模型路由分布 + cache hit rate 复盘
  - **季度**：预算重定 + 自托管 breakeven 重算（→ 7.4）
- **平台对照**：
  - **gateway 层 enforcement**：Portkey / LiteLLM 自带
  - **可视化 / 归因**：Finout / Amnic / CloudZero / Bifrost / Portkey 内置 dashboard
  - **trace + cost 一体**：LangSmith / Braintrust / Langfuse / Phoenix（与 6.8 重叠）
