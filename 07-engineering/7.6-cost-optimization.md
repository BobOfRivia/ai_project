# 7.6 成本优化体系

> **核心问题**：要砍 LLM 月支出，应该按什么顺序、组合什么手段？为什么 caching 是 ROI 第一？
> **目标**：能为团队画出"成本优化优先级矩阵"，每个杠杆量化预期收益与代价，组合落地 50-80% 月支出下降。

**展开重点**：
- **2026 共识："五大杠杆"按 ROI 排序**：
  1. **Prompt caching（provider 端）**：Anthropic 90% off / OpenAI 50% off，**单点 55-82%**，几乎无代价
  2. **Model routing（智能路由）**：简单任务走小模型，**60-80% 成本下降**
  3. **Semantic caching（应用端）**：Redis LangCache 高重复场景 **~73% 命中**
  4. **Batching**：异步、可容忍延迟的任务批量发，**单价砍半**（OpenAI Batch API 50% off）
  5. **Context 压缩 / 缩短**：删冗余、摘要、合理 chunk（前置约束 → 节约后续每一次）
- **杠杆组合 = 乘性效益**：
  - 仅 caching：-60%
  - + routing：-80%
  - + semantic cache：-87%
  - + batching（适用场景）：-93%
  - 真实工况累计 **50-80%** 下降是常态
- **决策原则**（2026 实操）：
  - **从 caching 开始**：最小改动、最大回报
  - **再做 routing**：要先建评估（→ 6 章）保证路由后质量不掉
  - **semantic cache 要谨慎**：相似但语义不同会导致错答，需要质量监控
  - **batching 只对可异步场景**：用户感知任务别用
- **路由策略**（细化 7.5 Gateway 路由能力）：
  - **任务类型路由**：分类、抽取 → 小模型；推理、代码 → 大模型
  - **难度路由**：先用小模型，质量低再 escalate 大模型（cascade）
  - **延迟路由**：低延迟需求走 Haiku / Flash，慢任务走 Opus
  - **成本上限路由**：单请求超预算 → 强制降级或拒绝
- **Token 节约**（应用层）：
  - 删除多余的 system prompt 反复内容
  - JSON schema 用 description 收紧字段，避免模型 verbose
  - **结构化输出**比"用 markdown 写一段"省 30-50% 输出 token
  - tool / function description 精简（避免每次几千 token 描述工具）
- **常见陷阱**：
  - 只优化输入 token、忽略输出：输出常贵 4-5x（"thinking" 模式更贵）
  - 路由没做 fallback：小模型挂了无路可走
  - 上线后没做 token 归因（→ 7.10）：不知道是哪个功能在烧钱
  - Reasoning / thinking 模型的隐藏 token 没算（Claude / GPT-5 思考 token 也付费）
- **指标看板**（与 6.10 / 7.10 互补）：
  - $ / request、$ / task（端到端）
  - token in / out 比、cache hit rate
  - 路由分布（cheap vs frontier 比例）
  - 月趋势 + per-feature 拆分
