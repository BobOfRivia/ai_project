# 6.4 RAG 评估

> **核心问题**：RAG 应用怎么评？端到端答案质量 vs 组件级（retriever / reranker / generator）该怎么拆？RAGAS / TruLens / DeepEval 在 RAG 上怎么选？
> **目标**：能为一个 RAG 系统给出"组件级 + 端到端"双层评估方案，并定位"是检索差还是生成差"，避免"答案错了瞎调 prompt"。
>
> **与 ch2 的关系**：ch2 讲 RAG 系统本体（chunking / embedding / retrieval / rerank），本节只讲怎么评。

**展开重点**：
- **RAG 评估的两层结构**：
  - **组件级**：retriever 是否取到正确 chunk？reranker 排序是否合理？generator 是否忠于上下文？
  - **端到端**：用户问题 → 最终答案 的整体质量
  - **诊断价值**：只看端到端无法定位问题；只看组件不能反映用户感受
- **RAGAS 核心指标**（2026 最深的 RAG 指标库）：
  - **Faithfulness**：答案是否基于检索内容，不编造（生成端）
  - **Answer Relevancy**：答案是否回答了用户问题（生成端）
  - **Context Precision**：检索到的 chunks 中相关的占比（检索端）
  - **Context Recall**：相关 chunks 是否都被检索到（检索端，需要 ground truth）
  - **Context Utilization**：生成时利用了多少 context（生成端）
  - **Noise Sensitivity**：噪声 chunk 对答案的扰动（生成端鲁棒性）
- **端到端指标补充**：
  - Answer Correctness（用 reference 或 judge）
  - Helpfulness（judge）
  - Citation accuracy（如果生成带引用）
- **评估流水线建议**：
  1. 构 golden set：问题 + 期望答案 + 期望 chunk ID（如果可标）
  2. 运行 RAG → 收集：检索结果 + 生成答案
  3. 组件级：context precision/recall / faithfulness
  4. 端到端：answer correctness / helpfulness
  5. 失败分桶：检索错？检索对但生成丢？生成幻觉？
- **框架选择**：
  - **RAGAS**：指标最全、专注 RAG；Python 库；推荐做组件级评估主力
  - **DeepEval**：pytest 风格，含 RAG 指标子集；适合做 CI 集成
  - **TruLens**：trace + 评估一体；适合 dev 阶段反复迭代
  - **平台层**：LangSmith / Langfuse / Arize 都集成了 RAGAS 风格指标
- **常见陷阱**：
  - Context Recall 没 ground truth 不能算（要么人标，要么用强模型预生成 oracle）
  - Faithfulness 用便宜 judge 容易误判（轻量 judge 不擅判"细微编造"）
  - 把 chunk 当 ground truth 时，chunk 切分一变，整套数据集就过时
