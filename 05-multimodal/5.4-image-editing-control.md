# 5.4 图像编辑与可控生成

> **核心问题**：从"一次性生成"到"指令式编辑 + 多轮迭代"的转变怎么落地？Flux Kontext / GPT Image 2 edit / ControlNet 各自的取舍是什么？
> **目标**：能为"修一张图 / 换主体 / 保持角色一致 / 风格迁移 / 局部 inpainting"等业务，选对编辑范式与具体 API。

**展开重点**：
- **三类编辑范式**：
  - **指令式编辑（instruction-based）**：自然语言描述要改什么 —— Flux Kontext、GPT Image 2 edit。2026 主流方向
  - **掩码 inpainting / outpainting**：用户画 mask + 描述新内容 —— SD inpaint 节点、Flux Kontext mask、Stability API
  - **条件控制（ControlNet 系）**：用边缘、姿态、深度、线稿等额外 condition 引导 SD 系列模型
- **Flux Kontext（Black Forest Labs，2025 起，2026 成熟）**：
  - 三档：**dev**（开源权重）、**Pro**（仅 API，$0.08 / 图）、**Max**（企业）
  - 12B MMDiT，内置图像-到-图像，多轮编辑保持上下文与高频细节
  - 子-10 秒延迟（对比 GPT Image 30+ 秒），适合"创意迭代流"
- **GPT Image 2 edit**：
  - 与生成共享一套 API；强项是文字渲染、复杂指令、跨多元素一致性
  - 延迟较高（30+ 秒），更适合"一步到位"而非快速迭代
- **ControlNet 生态（仍是 SD 系列主导）**：
  - 边缘 / 深度 / 姿态 / Canny / 涂鸦
  - 自托管（ComfyUI / diffusers）或走聚合器
  - 适合需要精确空间控制的场景（产品图、UI 模拟、设计稿）
- **多轮一致性**：
  - 角色 / 风格一致 → reference image + 指令式编辑；或用 IP-Adapter / LoRA（自托管）
  - "同一角色多场景"是 2026 普遍需求，普遍解法是 Flux Kontext 多轮 + reference image
- **接入与成本**：
  - 创意迭代流 → Kontext Pro（快、贵但单图便宜）
  - 一次性精修 → GPT Image 2（慢、跟随指令好）
  - 精确控制 → SD + ControlNet 自托管 / 聚合器
