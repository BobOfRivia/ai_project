# 3.12（一）Harness 工程：Agent 的"运行时骨架"

> **本文定位**：这是 3.12「Agent 工程化」三支柱（**Harness / 评估 / 可观测**）里的**第一支柱**。评估与可观测另开文件。之所以把 Harness 单拎出来讲透，是因为它是 2026 年**最被低估的工程杠杆**——同一个模型，换一套 harness，能从榜单第 30 名跳到第 5 名。而理解它最好的切片，是 Pydantic AI v2 在 2026 年 6 月押下的那个赌注：**harness-first design**。
>
> **核心问题**：一个"Agent"到底是什么？如果把模型抽走，剩下的那一坨东西——工具执行、上下文管理、状态持久化、护栏、反馈循环——它有没有名字、有没有工程方法论？为什么说"你评估的从来不是模型，而是 harness + 模型的组合"？以及——**Pydantic AI 说自己是 harness-first，这四个字到底与众不同在哪，还是只是又一个营销词？**
>
> **目标**：
> - 建立一个精确的定义：**Agent = Model + Harness**，说清 harness 的 8 个组成，以及"harness 工程"为什么是继 prompt engineering、context engineering 之后的**第三层工程学科**
> - 用硬证据理解 harness 是杠杆（换 harness 不换模型的性能跃迁），并接受一个反直觉结论：**"一个 agent 好不好"这个问题，本身是问错的**
> - 把前面 3.2–3.11 讲过的碎片（工具/上下文/记忆/规划/护栏）**收拢成一个统一的系统视角**——这正是 harness 工程区别于单点技巧的地方
> - **吃透 Pydantic AI 的 harness-first**：capability 原语打包了什么、hooks 为什么是灵魂、"小而稳的 core + 快速迭代的 Harness 库"的分层，以及它和 LangGraph / CrewAI / OpenAI SDK 在哲学上的根本不同
> - 想清楚这套哲学**什么时候对你有利、什么时候是负担**
>
> **边界**：**框架怎么选**（LangGraph vs CrewAI vs Pydantic AI…）属于 **3.8**，本文不重复选型，而是借 Pydantic AI 讲清 harness 这个**概念**；**评估**（轨迹评估 / LLM-as-judge / 每完成任务成本）和**可观测**（tracing / replay / OTel）是 3.12 的另两支柱，本文只在它们作为 harness 的组成时点到；**沙箱/护栏的安全细节**在 **3.11**，**上下文管理策略**在 **3.4**，**记忆生命周期**在 **3.5**——本文讲的是"如何把它们当作一个 harness 系统统一设计"。

## 一、什么是 harness——先把这个词钉死

"harness" 直译是**马具**。一匹马本身很有力气，但没有马具——缰绳、挽具、辕——你没法把这股力气导向任何有用的农活。这个比喻是整个概念的钥匙：

> **模型是马，harness 是马具。模型提供"推理能力"这股原始力气，harness 把它导向"在真实世界里完成任务"。**

放到工程语言里，一个精确的定义（Databricks / Firecrawl 等多方共识）：

> **AI agent harness = 包裹在 LLM 外面、让它能"采取行动"而不只是"回答问题"的全部软件基础设施——模型本身除外。**

于是有了这一节最该记住的等式：

> ### Agent = Model + Harness

模型负责**推理**，harness 负责**运转**。你平时说的"我做了一个 agent"，其实是"我为某个模型配了一套 harness"。

### harness 的 8 个组成

把"模型之外的那一坨"拆开，业界收敛出的清单大致是这 8 块（你会发现每一块前面某节都单独讲过——这不是巧合，后面会回收这个点）：

| # | 组成 | 干什么 | 本书归属 |
|---|---|---|---|
| 1 | **系统提示 / 指令** | 定义行为、规则、角色 | 1.3 / 3.1 |
| 2 | **工具与执行** | 模型能调用的 API、代码运行器、数据库 | 3.2 |
| 3 | **沙箱** | 隔离环境，防止危险动作影响真实世界 | 3.11（护栏视角）/ [3.12 沙箱深挖](./3.12-harness-sandbox.md)（基础设施视角） |
| 4 | **持久化存储** | 文件系统，让工作能跨会话延续 | 3.5 |
| 5 | **记忆管理** | 上下文裁剪 + 跨会话历史保留 | 3.4 / 3.5 |
| 6 | **反馈循环** | 自我验证、错误纠正 | 3.6（Reflexion 等） |
| 7 | **护栏** | 权限控制、人工审批关卡 | 3.11 / 3.9 |
| 8 | **可观测** | 日志、追踪，用于调试与合规 | 3.12（另文） |

这张表本身就藏着一个重要认知：**你前面学的东西，其实一直在学 harness 的零件。** 3.2 学工具、3.4 学上下文、3.5 学记忆、3.6 学反馈、3.11 学护栏——它们不是彼此无关的技巧，而是**同一台机器的不同零件**。harness 工程，就是**把这些零件当作一个系统来设计**的那门学问。这是它和"单点技巧"的根本区别。

### 一个关键的模型：LLM 即"新操作系统"

有一个更有野心的比喻值得记住：**把 LLM 当成新的操作系统内核，把 harness 当成围绕它的操作系统。** 内核（模型）负责"思考"，但一个能用的系统还需要文件系统（持久化）、进程调度（agent loop）、权限系统（护栏）、系统调用（工具）、日志（可观测）。你不会拿一个裸内核去办公——你用的是内核 + 操作系统。**Agent 也一样：没人部署一个裸模型，大家部署的都是 harness + 模型。**

## 二、为什么 harness 是 2026 最被低估的杠杆

概念清楚了，但你可能还没感到它的分量。下面是让我改变看法的三组硬证据。

**证据一：换 harness，不换模型，排名从第 30 跳到第 5。** 在一个公开的编码 agent 基准上，**仅仅替换 harness、保持底层模型不变**，就把一个 coding agent 从榜单第 30 名推到了第 5 名。同一个大脑，换一副马具，判若两人。

**证据二：同一个模型，准确率从 36% 到 53%。** Databricks 报告：把 GPT-4.5 配上他们为企业文档任务定制的 harness（OfficeQA Pro），准确率从 **36.10% 提到 52.63%**——**几乎砍掉一半错误率，模型一个字没换。**

**证据三：三个人，harness engineering，百万行代码。** OpenAI 记录过一个三人工程团队用"harness engineering"的方法，产出了百万行级代码库，人均 **3.5 个 PR/天**。他们优化的不是模型，是包裹模型的那套系统。

这三组数字合起来指向一个**反直觉但极其重要的结论**：

> **"这个 agent 好不好？"是一个问错了的问题。** 你永远是在评估 **harness + model 的组合**，两者在实践中不可分离。同一个模型配不同 harness 是两个不同的 agent；同一套 harness 配不同模型也是。Anthropic 说得很直白——评估一个 agent，就是在评估 harness 和 model 这个**不可分的整体**。

这个结论直接影响你怎么花工程时间。很多团队一遇到 agent 效果不好，第一反应是"换个更强的模型"或"再调调 prompt"。但上面的证据说明：**当模型已经够强时，剩下的性能空间大部分躺在 harness 里**——在你怎么管上下文、怎么设计工具返回、怎么做反馈循环、怎么持久化状态里。这就是为什么会有这句话：

> **prompt engineering 优化"输入的措辞"，context engineering 优化"模型每一步看到什么信息"，而 harness engineering 优化"模型周围的整个系统架构"。工程的价值，正在从前两者向第三者迁移。**

这是一条清晰的演进线：**Prompt Engineering → Context Engineering → Harness Engineering**。不是替代，是**外扩**——关注点从"这一句话怎么写"，扩到"这一步喂什么上下文"，再扩到"整台机器怎么搭"。

## 三、harness 工程的几根主梁（统一视角）

前面说 harness 工程 = 把零件当系统设计。这一节挑几根**最能体现"系统性"**的主梁细讲——重点不是复述每个零件（那在各自章节），而是讲**它们作为一个整体互相咬合**的地方。

### 主梁 1：agent loop 本身，就是最小的 harness

最朴素的 harness 是什么？就是那个循环：

```
while 没完成:
    response = model.request(history + instructions + tools)
    if response 里有工具调用:
        result = 执行工具()
        history.append(result)
    else:
        return response
```

**这个 15 行的循环，已经是一个 harness 了。** 它决定了"模型输出工具调用 → 谁去执行 → 结果怎么塞回上下文 → 什么时候停"。3.6 讲的 ReAct / Plan-Execute，本质都是**这个循环的不同形状**。所有更重的 harness，都是在这根主梁上加东西。记住这点，你就不会被框架的复杂度吓住——**再花哨的 agent，剥到底都是这个循环。**

### 主梁 2：上下文 / 记忆管理——harness 里最赚的一块

3.4 / 3.5 讲过手段（裁剪、摘要、卸载到文件、分层记忆）。从 harness 视角要补的一句是：**这些手段不是可选的优化，而是长时运行 agent 的生存条件。** 因为 agent 每走一步都往上下文追加"思考+动作+观察"，不管理就必然撞上 3.4 的 Context Rot。所以成熟 harness 都内建：滑动窗口裁剪、LLM 摘要、大输出截断/外溢到文件、token/迭代上限告警。**证据二里那 16 个百分点，很大一块就来自这里。**

### 主梁 3：durable execution（可持久执行）——长任务的地基 ⭐

这是 harness 工程里**最"基础设施"、也最容易被初学者漏掉**的一根梁，值得重点讲。

问题场景：一个 agent 任务要跑 20 分钟、调 50 次工具、中间还要等人工审批。跑到第 30 步时，API 抽风超时，或者你的进程被重启了。**没有 durable execution，整个任务从头再来**——前 29 步的 token 和时间全白烧，人工审批还得再来一遍。

durable execution 的解法：**把 agent 的执行过程做成可持久、可恢复的**。每一步的状态（历史、进度、检查点）落盘，进程挂了/重启了，能从断点继续，而不是从头。它让 agent 能：
- 扛住短暂的 API 失败、应用错误、进程重启；
- 支撑长时运行、异步、以及**人在环路（HITL）**的工作流——审批可以等几小时甚至几天，agent 状态一直"冻"在那儿，人回来了接着跑。

这就是为什么生产级框架（LangGraph 的 checkpointing、Pydantic AI 的 durable execution + Temporal 集成、LlamaIndex Workflows）都把它当一等公民。**它是"能 demo"和"能上生产"之间的那道坎。** 3.12 的可观测支柱和它是一对：durable execution 让你能"恢复"，可观测让你能"回放看哪一步错了"。

### 主梁 4 & 5：反馈循环 + 护栏——把 3.6 和 3.11 咬进 loop

- **反馈循环**（3.6 的 Reflexion 思想）：让 agent 在 loop 里自我验证、发现错误就重试。harness 视角的关键是——**在哪一步插入验证、验证失败怎么回退**，这是循环结构的一部分，不是外挂。
- **护栏**（3.11）：沙箱、最小权限、高危动作人工审批。harness 视角的关键是——**护栏是 loop 里的拦截点**，在"执行工具"那一步之前插入检查。3.11 的"四层护栏"本质就是在 harness 的不同层埋闸门。

**把这五根梁放一起看，你就懂了 harness 工程的精髓**：不是"我装了记忆、装了护栏、装了反馈"这种清单式堆砌，而是**这些东西如何在同一个 loop 里协同**——上下文管理喂什么给下一步、护栏在哪拦、失败了 durable execution 从哪恢复、反馈循环怎么触发重试。**系统性，就是这门学科的名字。**

## 四、Pydantic AI 的 "harness-first"——到底与众不同在哪

现在进入你最想问的部分。Pydantic AI 在 2026 年 6 月 23 日发布 v2.0 stable，官方发布说明的原话是——**它"lean into a harness-first design with capabilities as a core primitive"（转向以 harness 为先的设计，以 capability 作为核心原语）**。

这四个字不是营销。要看懂它的与众不同，得先看清**别人是怎么做的**。

### 先看对立面：绝大多数框架是 "framework-first"

主流框架的心智模型是——**先给你一个 agent 抽象，harness 能力挂在它上面**：

- **CrewAI**：先给你"角色/crew"抽象，你围着角色组织；
- **OpenAI SDK**：先给你"handoff"抽象，你围着分流组织；
- **LangGraph**：先给你"图/节点/边"抽象，你围着状态图组织；

在这些框架里，**"agent" 是主角，记忆/护栏/沙箱这些 harness 能力是"集成""插件""middleware"**——它们挂在框架预设的抽象外面。你要的能力如果框架没预设好接口，就得改框架或绕过它。

### Pydantic AI v2 的反转：几乎没有 agent 抽象，harness 就是框架本体

Pydantic AI v2 做了一个反转：**core 里几乎没有"重"的 agent 抽象，只保留一个最小的、类型安全的 loop + 一套 capability/hooks API。然后——记忆、护栏、上下文管理、文件访问、代码执行、子 agent 编排，全部都是 capability。**

换句话说：

> **别人是"框架优先，harness 是框架的插件"；Pydantic AI 是"harness 优先，框架只是一个跑 capability 的最小 loop"。harness 不是框架的附加物——harness 就是框架本身。** 这才是 "harness-first" 的字面意思。

### 技术内核之一：capability 原语——一个能"打包一切"的单元

v2 的核心原语叫 **capability**。一个 capability 把四样东西**打包成一个可组合的单元**：

> **capability = 指令（instructions）+ 工具（tools）+ 生命周期钩子（hooks）+ 模型设置（model settings）**

官方原话：它让"memory、guardrails、coding toolkit 这样的扩展，能通过**一个概念**触达 agent 的每一层"。这是关键——在 v2 之前，这些配置是分散在好几个小抽象里的碎片；v2 **把整层碾成一个东西让你 compose（组合）**。

用起来就是往 agent 上挂一个 capability 列表：

```python
from pydantic_ai import Agent
from pydantic_ai_harness import CodeMode, SlidingWindow, Guardrails

agent = Agent(
    'anthropic:claude-opus-4-8',
    capabilities=[
        CodeMode(),          # 沙箱化 Python 执行
        SlidingWindow(...),  # 上下文裁剪
        Guardrails(...),     # 输入输出校验
    ],
)
```

你的 agent = 一个模型 + 一串 capability。**这不就是本文第一节那个等式 `Agent = Model + Harness` 的代码直译吗？** capability 就是 harness 的可组合零件。

### 技术内核之二：hooks——才是真正的灵魂 ⭐

capability 里最有分量的，是 **hooks**。官方对 hook 的定义一针见血：

> hooks 是能在**每一步（every step）读取并改写模型看到的一切**的函数——**包括它的工具、它的指令、它的消息历史**。

停下来体会这句话的分量。它意味着：**每一个 capability，都能深入到 agent loop 的每一步内部，动态改写"模型这一拍到底看到什么"**——加/减工具、改写指令、裁剪历史、注入上下文。

这和"framework-first + middleware"的做法有本质区别：

| | framework-first 的 middleware | Pydantic AI 的 hooks |
|---|---|---|
| 触达深度 | 通常只在 loop 的**边界**（请求前/响应后）看到输入输出 | 在**每一步内部**，能读改工具/指令/历史 |
| 扩展方式 | 受框架预设的扩展点限制 | 任何 capability 都能挂 hook，**无需改框架** |
| 心智负担 | "我的能力得适配框架的抽象" | "我的能力就是 loop 每一步的一段拦截逻辑" |

官方的话说得很直白：capability 和 hooks API 就是**"你给 agent 配 harness 的方式——一捆工具、生命周期钩子、指令、模型设置，在不改框架的前提下扩展 agent 能做什么"**。**"不改框架"这四个字，是 harness-first 相对 framework-first 的核心卖点。**

### 技术内核之三：小而稳的 core + 快速迭代的 Harness 库

v2 还做了一个刻意的分层（和 LangGraph 的分层栈形似，哲学不同）：

```
Pydantic AI Harness（快速迭代）
   memory / guardrails / context 管理 / 文件系统 / code mode / 子agent 编排…
        ↑ 都是 capability，建在
Pydantic AI core（小而稳）
   agent loop + providers + capability/hooks API + 少数需要深度 provider 支持的能力
        ↑ loop 底层是
pydantic-graph（typed 状态机：BaseNode / Graph）
```

- **core 小而稳**：只留 loop、provider 接入、capability/hooks API。provider 支持还改成了 opt-in——**用什么装什么**，不再一股脑拖下所有后端。
- **Harness 库快速跑**：memory、guardrails、context 管理、文件系统、code mode（用 Monty 做沙箱 Python）、子 agent 委派、planning、cost 预算、secret 脱敏……都在这层，可以频繁演进而不动摇 core。

对比 LangGraph 的分层栈（runtime → create_agent → DeepAgents）：两者都做"底层稳、上层快"的分层，但 **LangGraph 底层是"运行时（执行/状态/durability）"，Pydantic AI 底层是"类型安全的 loop + capability/hooks 协议"**。前者的地基是**执行模型**，后者的地基是**组合协议 + 类型系统**。这正好呼应各自的灵魂：LangGraph 卖"状态化控制流"，Pydantic AI 卖"类型安全 + 可组合的 harness"。

### 源码级视角：这台 loop 长什么样

为落到实处，看一眼它 run loop 的真实骨架（据其源码/架构）：

1. 历史 + 指令 → 转成 `ModelRequestParameters`；
2. 经 `Model.request()` / `Model.request_stream()` 发给 LLM；
3. 若返回里有 `ToolCallPart`，执行注册的 Python 工具函数；
4. 结果 append 回历史，循环继续；
5. 最终响应用 **Pydantic 对着你声明的结果类型做校验**。

底层由 **`pydantic-graph`**（一个 typed 图 / 状态机库，导出 `BaseNode` / `Graph`）驱动这个 loop。**类型安全贯穿始终**——依赖是 typed 的、输出是 typed 的、工具调用是被校验的。这就是它 "FastAPI-style"、"the Pydantic way" 的由来：**把 Pydantic 在数据校验领域干的事，搬到 agent 上——让 agent 的每一个边界都有类型保证。**

### 一个必须澄清的对照：DeepAgents 是不是也 "harness-first"？

学到这里很自然会问：LangChain 的 **DeepAgents** 官方标语就是 "the batteries-included agent harness"，它是不是也 harness-first？这个对照能让 "harness-first" 这个词精确起来，值得专门讲。

**先说一个会让人意外的事实：两者的组合机制几乎同构。** 你可能以为 Pydantic AI 的独特之处是"把 harness 能力做成可插拔的拦截器"，但 DeepAgents 也这么干，只是换名叫 **middleware（中间件）**：

| | Pydantic AI | DeepAgents / LangChain |
|---|---|---|
| 组合原语 | **capability + hooks**（每步读改工具/指令/历史） | **middleware**（生命周期各阶段拦截并修改执行） |
| 记忆 | Memory capability | `MemoryMiddleware` |
| 上下文压缩 | SlidingWindow capability | `SummarizationMiddleware` |
| 人工审批 | Guardrails capability | `HumanInTheLoopMiddleware` |
| 子 agent | 子 agent capability | `SubAgentMiddleware` |

**一个 capability ≈ 一个 middleware**，都是"挂在 loop 每一步、能改写模型看到什么"的可组合拦截器。所以"harness 能力被做成可组合零件"这点两家都成立，**区分不了它们**。这本身是个信号：**"每步可插拔的拦截器"正在成为这个时代 harness 工程的通用模式，无关框架哲学。**

既然机制一样，"harness-first" 讲的其实是另外两件事：

**区别一：谁在底座——"first" 落在哪。** DeepAgents 的 `create_deep_agent` 最终返回一个 `CompiledStateGraph`，即**编译到 LangGraph 运行时之上**的一层。回到分层栈——

```
DeepAgents        ← 一副"装配好的 harness"（顶层产品）
   ↑ 建在
LangChain create_agent + middleware
   ↑ 编译到
LangGraph（runtime：执行/状态/durability）← 整个栈的"first/底座"
```

对 LangChain 这一栈，**最底、最基础的是运行时**，harness 是堆在运行时之上的顶层——所以这一栈是 **runtime-first**，DeepAgents 是它上面那顶 harness 帽子。而 Pydantic AI 反过来：**它的 capability/hooks 组合 API 本身就是底座，底下再没有更重的东西**（core 只是最小 typed loop）。harness 组合**就是框架的脊柱**——这才是 "harness-**first**" 里 "first" 的字面意思：harness 组合是第一位的地基，不是盖在地基上的一层。

**区别二：给你一副 harness，还是给你造 harness 的原语**（正好是 3.8 那根"现成 vs 自组"的轴）：
- **DeepAgents = 有主见、开箱即用的 harness**——给你一个**特定形状**（"deep agent" 模式：规划工具 + 文件系统 + 子 agent + skills，灵感来自 Claude Code / Deep Research）。你**消费**一副装配好的 harness，顺带继承它的形状。
- **Pydantic AI = 无主见、自己拼的 harness 基底**——给你**原语**（capability/hooks）加一个零件库，**没有默认形状**，你**作者**一副属于自己的 harness。

一个是"这儿有副马具，骑上就走"；一个是"这儿有做马具的皮革和扣件，自己缝一副合身的"。

**结论**，用一句话钉死：

> **DeepAgents 是"一副 harness"（名词——装配好的产品，你消费它）；Pydantic AI 是 "harness-first"（形容词——以 harness 组合为地基的框架，你用它造 harness）。前者是 harness 工程的产出，后者是 harness 工程的基底。**

所以与其问 "DeepAgents 是不是也 harness-first"，不如说 **DeepAgents 是 "harness-most"**（最完整的现成 harness）；而 "first" 这个词，精确地属于把 harness 组合放在框架最底层的 Pydantic AI。**两者不冲突——一个在栈顶交付 harness，一个在栈底把 harness 变成框架本体。**

## 五、这套哲学，什么时候对你有利，什么时候是负担

批判地看，harness-first 不是免费的午餐。

**对你有利，当**：
- 你是**严肃的 Python 团队**，重视类型安全、可测试、可维护——capability + typed loop 让 harness 的每一块都可组合、可单测、有类型边界；
- 你想**自己攒一套 harness**，按需拼零件，而不是接受某个框架的预设剧本——hooks 的"每步可改写 + 不改框架"给了你极大的自由度；
- 你的场景是 **extraction / 分类 / 路由 / 轻量助手**这类"结构化输出很重要"的偏线性工作流——类型安全在这里价值最大。

**是负担，当**：
- 你要的是**开箱即用的、有主见的多 agent 剧本**（"给我个 researcher+writer 的 crew"）——那 CrewAI 的 framework-first 更省心，Pydantic AI 会让你觉得"什么都要自己组合"；
- 你的主约束是**复杂状态化控制流 + durable 长任务**——LangGraph 的运行时地基更对口（回顾 3.8：**按主约束选，不按功能清单选**）；
- 你在乎**生态成熟度**——v2 是 2026 年 6 月才 stable 的，harness-first 这套抽象还年轻，capability 生态还在长。

一句话收口这一节：**harness-first 是"把 harness 工程做成框架的一等公民"这个理念最彻底的一次落地。它对"想自己掌控 harness、且珍视类型安全"的团队是绝配；对"想要现成剧本"的团队则显得什么都得自己拼。** 这和 3.8 的结论完全一致——**没有最好的框架，只有对上你主约束的框架。**

## 六、小结

- **Agent = Model + Harness。** harness 是"模型之外让它能行动的全部基础设施"，8 大组成（指令/工具/沙箱/持久化/记忆/反馈/护栏/可观测）恰好就是你前面各章学的零件——**harness 工程就是把这些零件当一个系统统一设计。**
- **harness 是 2026 最被低估的杠杆**：换 harness 不换模型，能让 coding agent 从第 30 名到第 5 名、让 GPT-4.5 准确率从 36% 到 53%。由此得到反直觉结论——**"这个 agent 好不好"问错了，你永远在评估 harness+model 的不可分组合。**
- **工程价值在迁移**：Prompt Engineering（措辞）→ Context Engineering（每步看什么）→ **Harness Engineering（整台机器怎么搭）**。
- **durable execution 是长任务的地基**，是"能 demo"和"能上生产"之间的坎，和可观测（回放）、HITL（冻结等待）一对。
- **Pydantic AI 的 harness-first**：别人是 framework-first（agent 抽象是主角，harness 是插件），它反转成 **harness-first（core 只是最小 typed loop，harness 能力=一等公民 capability）**。技术内核三样：**capability 原语**（打包 指令+工具+hooks+模型设置）、**hooks**（每步可读改工具/指令/历史，不改框架就扩展）、**小而稳 core + 快速 Harness 库**的分层。灵魂是**类型安全贯穿 + 可组合**。
- **什么时候选它**：珍视类型安全、想自组 harness、偏结构化线性工作流 → 绝配；要现成多 agent 剧本 → CrewAI；要 durable 复杂状态流 → LangGraph。**按主约束选（呼应 3.8）。**
