# 7.1 工程化与部署基础与决策框架

> **核心问题**：从"能跑的 demo"到"上线 prod"差在哪？应用工程师该用什么视角拆解 LLM 工程化？SLO / SLA 怎么定？
> **目标**：建立 LLM 工程化的"4 层视角"心智，知道在每一层要决策什么、和传统 web 工程的差异在哪。

**展开重点**：
- **demo → prod 的 5 类断层**（2026 共识）：
  - **可用性**：单点 provider 抖动 / 429 / 5xx 直接挂业务
  - **延迟**：P50 看着 OK，P99 灾难；流式首 token 才是真感知
  - **成本**：token 计量、上下文膨胀、retry 倍数效应
  - **质量**：模型更新静默回退（→ 6 章评估）
  - **合规**：数据出境 / PII / 日志保留期
- **4 层视角**（贯穿全章）：
  - **推理层**：模型本身怎么跑（vLLM / SGLang / 云 API）→ 7.2 / 7.3
  - **网关层**：多 provider 接入、路由、限流、fallback → 7.5 / 7.8
  - **应用层**：缓存、批处理、流式 UI、重试策略 → 7.6 / 7.7 / 7.8
  - **治理层**：观测、成本归因、预算、合规 → 7.10
- **LLM 应用 SLO 三件套**：
  - **质量 SLO**：golden set 通过率 / judge 分（与第 6 章对齐）
  - **可用性 SLO**：成功率（5xx / refusal / timeout 全算失败）
  - **延迟 SLO**：**首 token TTFT** + **每 token 速率** + **端到端 P99**
  - **成本 SLO**（2026 新增）：单请求 / 单任务 token 上限；每月预算
- **关键工程化原则**：
  - **No single provider**：从第一天起多 provider，否则锁死
  - **流式优先**：UI 体感 ≠ 总延迟，首 token < 1s 是 2026 基线
  - **可观测先行**：trace 不全等于没上线（→ 6.7 OTel GenAI）
  - **成本不是事后看，是设计输入**：路由 / cache / 上下文压缩在 prompt 设计阶段就要想
- **常见反模式**：
  - 只用一个 provider，靠它的 SDK retry，不做应用层 fallback
  - 把 LLM 调用塞进同步请求路径，不考虑 timeout
  - 上线时 latency P99 没量过
  - 成本只看月度账单、没做 per-feature 归因（→ 7.10）
