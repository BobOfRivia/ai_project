# 3.2 工具工程（Agent 视角）

> **核心问题**：怎么为 Agent 设计出"模型用得对、出错能恢复、上规模不崩"的工具？
>
> **目标**：掌握工具描述质量、粒度设计、数量上限、幂等性、错误信息设计、生产级重试/熔断模式的工程要点。

---

## 3.2.1 先建立正确认知：工具工程不等于 API 设计

许多工程师第一次给 Agent 写工具时，会自然而然地按"为人类开发者设计 API"的方式来做：返回完整的 JSON 结构、暴露底层的技术细节、用技术命名约定……这是一个根本性的错误方向。

**API 设计的读者是代码，工具设计的读者是 LLM。**

| | 传统 API 设计 | Agent 工具设计 |
|---|---|---|
| **消费者** | 代码（确定性解析） | LLM（概率性推理） |
| **输入验证** | 强类型 + 运行时报错 | 模糊描述 + 模型理解 |
| **返回值** | 完整、精确、机器可读 | 精简、语义清晰、便于推理 |
| **错误信息** | 状态码 + 技术描述 | 自然语言 + 下一步建议 |
| **命名** | 技术惯例（camelCase、REST路径） | 语义直白（动词+名词，人类可理解） |
| **文档** | Swagger / JSDoc | 嵌入 schema 的描述字段，是 prompt 的一部分 |

Anthropic 在自己的工程博客里明确说过：工具描述和规格（spec）需要跟 prompt 一样认真做 prompt engineering——**工具描述不是注释，是 prompt 的一部分**。

这是整个 3.2 的基础认知。

---

## 3.2.2 工具描述：最被低估的工程工作

工具的 `description` 字段直接决定 Agent 能不能在正确时机选对工具、用对参数。这是工具工程中投入产出比最高的地方。

### 一个真实案例说明重要性

Anthropic 在推出 Claude 的 web search 工具后发现：Claude 会无缘无故在搜索词后面追加"2025"，导致搜索结果质量下降、性能下滑。

原因不是模型坏了，而是工具描述没有说清楚"不要修改用户原始搜索词"。一句话的描述优化，直接修复了这个生产问题。

### 好的工具描述包含哪些要素

**基础四要素**：

```python
{
    "name": "search_customer_orders",
    "description": """
    按客户 ID 查询该客户的历史订单列表。

    使用场景：当用户询问某个客户的订单状态、历史购买记录、或需要根据
    订单信息做后续操作（如退款、修改）时使用本工具。

    注意：
    - customer_id 必须是系统内部 ID（数字格式），不是邮箱或姓名
    - 如果只知道邮箱，请先用 find_customer_by_email 工具获取 ID
    - 返回最近 90 天的订单，更早的订单请使用 search_order_archive 工具

    示例调用：{"customer_id": 12345, "status_filter": "pending"}
    """,
    "parameters": {
        "customer_id": {
            "type": "integer",
            "description": "客户的系统内部 ID（整数），可从 find_customer 工具的返回结果中获取"
        },
        "status_filter": {
            "type": "string",
            "enum": ["all", "pending", "shipped", "delivered", "cancelled"],
            "description": "按订单状态过滤。不确定时使用 'all'",
            "default": "all"
        }
    }
}
```

**关键要素拆解**：

| 要素 | 作用 | 如果缺失 |
|---|---|---|
| **使用场景** | 告诉模型什么时候选这个工具 | 模型在不该用时调用，或在该用时不调用 |
| **参数说明** | 说清楚每个参数的语义和格式 | 模型传错格式（字符串 vs 整数）或语义错误 |
| **限制和边界** | 明确工具的能力边界 | 模型用一个工具做它做不到的事，反复失败 |
| **与其他工具的关系** | 说明工具之间的依赖/区别 | 模型混淆相似功能的工具，或跳过必要的前置工具 |
| **示例调用** | 给模型一个锚点 | 模型在格式上犯低级错误 |

### 参数描述的细节原则

**1. 不要依赖模型"推断"格式**

```python
# 差：模型会猜日期格式，产生各种写法
"description": "查询开始日期"

# 好：明确告知
"description": "查询开始日期，格式 YYYY-MM-DD，例如 '2026-01-15'"
```

**2. 用人类语言命名，不用技术缩写**

```python
# 差
"cust_id", "ts", "cfg", "max_res"

# 好
"customer_id", "timestamp", "configuration", "max_results"
```

**3. enum 类型要加语义说明**

```python
# 差：模型不知道选哪个
"enum": ["v1", "v2", "v3"]

# 好
"enum": ["basic", "standard", "premium"],
"description": "报告详细程度：basic=仅关键指标，standard=含趋势图，premium=含原始数据导出"
```

**4. 可选参数要给出默认行为描述**

```python
"description": "分页大小，默认 20。大型数据集建议设为 50，不超过 100（超过会自动截断）"
```

### 工具描述的迭代方式

工具描述的质量通过**评估（Eval）**来验证，而不是靠感觉。具体做法：

1. 跑一批测试任务，观察模型是否在该用工具时正确选择、参数是否正确
2. 统计"参数错误率"——频繁出现的参数错误通常是描述不清导致的
3. 修改描述，重新跑评估，对比指标变化

Anthropic 的实践经验：**迭代工具描述的效果有时超过换更好的模型**。

### 工具描述不够：系统提示必须跟工具共同设计

这是一个容易被忽视的整体性问题。工具描述解决"模型理解工具本身"，但仅此不够——**模型还需要知道在什么上下文下该用这个工具，用完结果该怎么处理**。这部分靠系统提示（system prompt）来补全。

LangChain 工程师 Nuno 在工程实践中总结道：

> "在我所见的几乎所有生产 Agent 中，你需要定制系统提示，甚至架构的其他部分来适配你提供的工具。"

具体来说，系统提示需要与工具配套解决以下问题：

| 系统提示需要补充的内容 | 示例 |
|---|---|
| **工具调用的优先级** | "如果用户提问涉及订单，优先用 search_orders，不要直接回答" |
| **工具结果的处理方式** | "search_orders 返回的结果要按时间倒序展示给用户" |
| **工具失败时的降级策略** | "如果 get_inventory 返回错误，告知用户稍后重试，不要猜测库存" |
| **工具调用的约束** | "不要在用户没有明确授权的情况下调用 send_email" |

工具 + 系统提示是一个**整体**，分开设计、分开测试会留下行为漏洞。在实际工程中，修改工具描述后，要同步检查系统提示是否需要调整。

## 3.2.3 工具粒度：多细才合适？

工具粒度（granularity）是工具工程中最难拿捏的决策之一，没有唯一答案，但有清晰的权衡逻辑。

### 粒度谱系

```
粗粒度 ←───────────────────────────────────────────► 细粒度
   │                                                      │
单工具覆盖整个子系统                              每个原子操作一个工具
   │                                                      │
例：process_order(action, params)            例：create_order()
   包含：创建/修改/取消/查询                      update_order()
                                                  cancel_order()
                                                  get_order()
```

### 粗粒度工具的问题

```python
# 看起来"简洁"，实际上是灾难
def manage_database(operation: str, table: str, data: dict):
    """执行数据库操作"""
    if operation == "select":
        return db.select(table, data)
    elif operation == "insert":
        return db.insert(table, data)
    elif operation == "delete":
        return db.delete(table, data)  # 危险！
```

问题所在：
- 模型需要"猜"正确的 `operation` 字符串
- 无法在工具层面区分只读和写操作的权限
- 一个工具失败，无法区分是哪个分支出的问题
- 无法对 `delete` 操作单独加确认步骤

### 细粒度工具的问题

把所有操作都拆成独立工具会导致两个问题：

**1. 上下文膨胀**：每个工具的 schema 描述都要传给模型，工具越多，消耗 token 越多，也越可能超出上下文窗口。

**2. 步骤碎片化**：原本可以在一次工具调用里完成的事，被拆成 5 次，Agent Loop 来回 5 轮，延迟 × 5，成本 × 5。

### 粒度设计的实用原则

**原则一：操作语义是分水岭**

同一语义的操作合并，不同语义的操作分开：
- `create_order` 和 `update_order` 语义不同，分开
- 但查询订单的"按ID查"和"按状态查"可以合并为 `search_orders(filter)` 因为语义相同

**原则二：权限边界决定粒度**

读操作和写操作分开，高风险操作（删除、外发消息、扣款）单独成工具，便于在工具层面加权限控制和审计日志。

**原则三：组合操作应封装**

如果某个业务场景总是需要"先查，再写"，考虑把它封装成一个工具，减少循环轮次。

```python
# 不好：Agent 需要两步
get_account_balance(account_id)  # 第一步
transfer_money(from_id, to_id, amount)  # 第二步（可能余额不足）

# 好：封装成一个原子操作
transfer_money_with_balance_check(from_id, to_id, amount)
# 工具内部处理余额检查，返回清晰的成功/失败+原因
```

**原则四：复杂数据操作用 wrapper 而非直接暴露 API**

直接把 REST API 暴露给 Agent 是生产事故的常见来源：

```python
# 危险：直接暴露底层 API，模型需要处理分页、错误码、字段过滤
def call_crm_api(endpoint: str, method: str, params: dict) -> dict:
    return requests.request(method, f"{CRM_BASE}/{endpoint}", json=params).json()

# 安全：封装成面向任务的工具
def get_customer_info(customer_id: int) -> CustomerInfo:
    """
    只返回 Agent 需要的字段（姓名、邮箱、状态），
    内部处理分页、认证、重试，
    把 API 错误码映射为语义清晰的错误消息
    """
    raw = crm_client.get(f"/customers/{customer_id}")
    return CustomerInfo(
        name=raw["profile"]["full_name"],
        email=raw["contact"]["primary_email"],
        status=raw["account"]["status_code"]  # 内部映射为 "active"/"suspended"
    )
```

## 3.2.4 工具数量与精度：一个必须正视的上限

这是工具工程中最容易被忽视的硬约束，有明确的实验数据支撑。

### 数字说话

来自 2024-2025 年的研究数据：

| 工具数量 | 主流模型的工具选择准确率 |
|---|---|
| ~10 个 | 90%+ |
| ~50 个 | 84% - 95% |
| ~200 个 | 41% - 83%（差异巨大） |
| ~500 个 | 大多数模型接近崩溃 |
| 740+ 个 | 接近随机猜测 |

更关键的是：**精度下降不是线性的**，而是在某个阈值处断崖式下跌。超过 10-15 个相似功能的工具时，模型就开始出现"选择困难"。

还有一个来自生产实践的更严峻数字：LangChain 工程师 Nuno 引用的内部数据显示，即便在**针对工具集量身定制了 system prompt 和架构**的前提下，模型在调用正确工具时的失败率仍约为 **50%**。这意味着工具工程不是一次性配置完成的事，而是需要持续迭代、持续评估的工程工作。

### 为什么会这样？

**"迷失在中间"（Lost in the Middle）效应**：

当工具列表很长时，排在列表中间位置（40%-60% 位置）的工具被选中的概率远低于排在开头和结尾的工具。这是 LLM 处理长序列时的注意力分布问题，不是模型智力问题，是结构性限制。

**认知超载**：

模型在推理"用哪个工具"时，需要同时理解所有工具的语义差异。工具越多，这个推理任务越难，越容易出错——就像人类面对 200 个选项时比面对 10 个选项更容易选错。

### 如何控制工具数量

**策略一：功能合并**

把相似的工具合并，用参数区分操作：
```python
# 原本 4 个工具
search_orders_by_id(id)
search_orders_by_customer(customer_id)
search_orders_by_date(start, end)
search_orders_by_status(status)

# 合并为 1 个
search_orders(id=None, customer_id=None, date_range=None, status=None)
```

**策略二：动态工具加载**（见 3.2.5，这是规模化的关键解法）

**策略三：工具分层**

把工具按使用频率和任务相关性分层：
- 核心工具（始终加载，5-10 个）
- 场景工具（按对话意图按需加载）
- 高级工具（用户明确请求时才加载）

## 3.2.5 动态工具加载：规模化的核心解法

当你的系统有几十上百个工具时，"一次性把所有工具传给模型"是不可行的。动态工具加载是解决这个问题的工程方案。

### 基本思路

把 RAG 的逻辑用到工具上：不是把所有工具的 schema 都塞进 context，而是**在每一步推理前，根据当前任务意图检索最相关的工具子集**，只把这些工具传给模型。

```
用户意图 / 当前任务
      │
      ▼
[工具检索层]
  对所有工具的名称+描述建立向量索引
  用当前步骤的语义检索 Top-K 相关工具
      │
      ▼
[模型推理]
  只看到 5-10 个相关工具
  工具选择准确率维持高位
```

### 性能数据

来自 2025 年的研究（Instruction-Tool Retrieval, ITR）：
- 相比"静态加载全部工具"，动态加载把每步上下文 token 降低 **95%**
- 工具选择准确率提升 **32%**（相对值）
- 端到端 API 成本降低 **70%**

一个更直观的数字对比：

| | 静态加载 | 动态加载 |
|---|---|---|
| 工具数量（全量） | 80 个 | 80 个 |
| 每步传给模型的工具 | 80 个 | ~5 个 |
| 10 步任务的工具 schema token | 80 × 10 = 800 次传输 | 5 × 10 = 50 次传输 |
| Token 节省 | — | **约 94%** |

### 实现方式

**最简单的实现**：把工具的名称和描述向量化，存入向量数据库（或内存索引），每步推理前用当前步骤的指令做相似度检索：

```python
import numpy as np
from typing import List

class DynamicToolRegistry:
    def __init__(self, tools: List[Tool], embedding_fn):
        self.tools = tools
        self.embeddings = {
            t.name: embedding_fn(f"{t.name}: {t.description}")
            for t in tools
        }

    def get_relevant_tools(self, current_task: str, top_k: int = 8) -> List[Tool]:
        task_embedding = self.embedding_fn(current_task)
        scores = {
            name: cosine_similarity(task_embedding, emb)
            for name, emb in self.embeddings.items()
        }
        top_names = sorted(scores, key=scores.get, reverse=True)[:top_k]
        return [t for t in self.tools if t.name in top_names]
```

**注意事项**：
- 保留 2-3 个始终加载的"安全工具"（如 `ask_for_clarification`、`mark_task_complete`）
- top_k 一般设 5-10，不要超过 15
- 建议在检索结果里加一个置信度阈值，低于阈值的工具不传入

## 3.2.6 工具输出设计：返回值决定推理质量

工具的输出（tool result）是下一步 LLM 推理的输入。**输出设计直接决定 Agent 能不能从工具结果中推理出正确的下一步**。

### 原则：只返回 Agent 推理所需要的信息

不要把底层 API 的完整响应直接丢给模型：

```python
# 差：原始 API 响应，包含 Agent 不需要的技术字段
{
    "_id": "507f1f77bcf86cd799439011",
    "created_at": 1703001600,
    "updated_at": 1703088000,
    "status_code": 200,
    "meta": {"version": "v2.3", "region": "us-east-1"},
    "data": {
        "customer": {
            "profile": {"full_name": "张三", "age": 35},
            "contact": {
                "primary_email": "zhang@example.com",
                "backup_email": null,
                "phone_numbers": [{"type": "mobile", "number": "13812345678"}]
            },
            "account": {"status_code": 1, "tier": "gold", "points": 1250}
        }
    }
}

# 好：只返回 Agent 推理所需的字段，用语义字段名
{
    "name": "张三",
    "email": "zhang@example.com",
    "phone": "13812345678",
    "account_status": "active",  # 把 status_code: 1 映射为语义值
    "membership_tier": "gold",
    "reward_points": 1250
}
```

**为什么这样设计**：
- 减少模型需要"读懂"的信息量，减少出错概率
- 减少无效 token 消耗
- 把技术 ID 转换为语义值（`status_code: 1` → `"active"`），让模型不用做类型转换

### 原则：ID vs 人类可读字段

工具输出中，优先返回**人类可读字段**而非技术 ID：

```python
# 差：模型看到 ID，不知道该怎么向用户解释
{"assigned_agent_id": "usr_abc123", "department_id": 42}

# 好：语义清晰，模型可以直接用于推理和回复
{"assigned_agent": "李晓明", "department": "技术支持部"}
```

当 ID 本身是必要的（下一步工具调用需要用到），才保留 ID，并在字段名上明确说明：

```python
{"agent_name": "李晓明", "agent_id": "usr_abc123"}
# agent_id 在后续的 reassign_ticket(agent_id=...) 中需要用到
```

### 原则：大量数据的截断与提示

当结果集很大时，不要直接返回全部数据。应该：
1. 返回最相关的 N 条（N 通常是 10-20）
2. 明确告知有多少条被截断
3. 提供进一步查询的参数提示

```python
{
    "orders": [...],  # 前 10 条
    "total_count": 247,
    "shown": 10,
    "note": "共 247 条订单，显示最新 10 条。如需更多，请用 page=2 参数继续查询"
}
```

## 3.2.7 错误信息设计：Agent 的"自我恢复能力"来自这里

工具的错误消息（error message）是 Agent Loop 中的"观察"（Observation）。**错误信息的质量决定 Agent 能不能从错误中恢复，还是会陷入死循环**。

### 错误信息的黄金标准：告知"发生了什么"+"为什么"+"下一步能做什么"

```python
# 差：技术报错，模型无法推理如何恢复
raise Exception("Permission denied: EACCES")

# 差：稍好一点，但还不够
return {"error": "insufficient_permissions"}

# 好：富语义错误，模型可以根据这个信息决定下一步
return {
    "success": False,
    "error_type": "insufficient_permissions",
    "message": "无法写入 /etc/config，当前进程权限不足。",
    "suggestion": "可以写入 /tmp/agent_workspace 目录，或请求用户授权后重试",
    "retry_possible": False
}
```

**模型收到第二种错误消息**：知道发生了什么，但不知道怎么办，可能无效重试。

**模型收到第三种错误消息**：知道发生了什么，知道有替代路径，能做出有效决策。

### 错误分类与对应处理提示

工具应该把错误分类，并针对不同类型给出不同的恢复建议：

| 错误类型 | 示例 | 推荐的错误消息策略 |
|---|---|---|
| **参数错误** | 传了字符串但期望整数 | 说明期望的格式，提供示例 |
| **权限不足** | 没有写权限 | 说明需要什么权限，或建议替代操作 |
| **资源不存在** | 找不到该 ID 的记录 | 建议检查 ID 或搜索相关内容 |
| **速率限制** | API 限流 | 告知等待时间，`retry_after_seconds` |
| **外部服务不可用** | 第三方 API 超时 | 说明服务状态，建议稍后重试或降级 |
| **业务逻辑错误** | 余额不足、订单不可取消 | 说明业务约束，建议替代方案 |

### 一个结构化错误响应的模板

```python
class ToolError:
    error_type: str          # 错误分类，用于程序处理
    message: str             # 人类可读的错误描述
    suggestion: str | None   # Agent 下一步可以做什么
    retry_possible: bool     # 是否值得重试
    retry_after_seconds: int | None  # 如果可重试，建议等多久
    context: dict | None     # 额外上下文（如：实际值 vs 期望值）
```

## 3.2.8 幂等性：让工具可以安全重试

**幂等性（Idempotency）**：同一个操作执行一次和执行多次，产生相同的结果。

这在 Agent 工程中至关重要，原因是：Agent 会重试。网络抖动、模型幻觉、工具超时……各种原因都可能触发重试。如果工具不是幂等的，重试就可能造成重复扣款、重复发邮件、重复创建记录等严重问题。

### 哪些工具需要重点考虑幂等性

```
只读操作 ─────────────────────────────────────── 天然幂等
  get_order()、search_customers()、read_file()

写操作（部分幂等）────────────────────────────── 需要设计
  update_order_status()  ← 设置固定值，调多次结果相同
  set_configuration()    ← 覆盖写，幂等

写操作（非幂等）────────────────────────────────── 必须处理
  create_order()         ← 每次调用都创建一个新订单
  send_email()           ← 每次调用都发一封邮件
  charge_payment()       ← 每次调用都扣款
```

### 幂等键（Idempotency Key）模式

对于非幂等的写操作，使用幂等键是标准解法：

```python
def send_notification(
    user_id: int,
    message: str,
    idempotency_key: str  # Agent 在重试时传相同的 key
) -> dict:
    """
    发送通知。如果 idempotency_key 已经被使用过，
    直接返回第一次的结果，不重复发送。

    idempotency_key: 本次操作的唯一标识，建议使用任务ID+操作序号，
    如 "task_abc123_step_3"
    """
    existing = cache.get(f"notification:{idempotency_key}")
    if existing:
        return {**existing, "was_duplicate": True}

    result = notification_service.send(user_id, message)
    cache.set(f"notification:{idempotency_key}", result, ttl=3600)
    return result
```

**工具描述中要说明如何生成幂等键**：

```python
"description": """
发送通知给用户。本工具是幂等的——使用相同的 idempotency_key
多次调用只会发一次通知。

idempotency_key 的生成规则：使用格式 "{task_id}_{step_number}"，
例如当前任务 ID 是 "task_001"，这是第 3 步，则填写 "task_001_3"。
重试时使用完全相同的 key。
"""
```

### 高影响操作的额外防护

对于不可撤销且影响大的操作（发送大量邮件、执行支付、发布内容），仅靠幂等性不够，还应该加：

1. **确认工具**：在执行前先调用一个"预览/确认"工具，让 Agent 或人类确认再执行
2. **频率上限**：工具内部限制在某时间窗口内的最大调用次数
3. **审计日志**：每次调用都记录，便于事后核查

## 3.2.9 生产工程模式：超时、重试、熔断

工具调用外部服务时，必须面对网络抖动、第三方 API 不稳定、超时等现实问题。这部分讲的是工具层面的容错工程，不是 Agent 层面的——工具自己要健壮，不能把外部错误直接抛给 Agent 处理。

### 超时（Timeout）

对任何外部调用都要设超时。不设超时的工具可能让 Agent Loop 永远卡住。

三种超时都要设：

```python
def call_external_api(params: dict) -> dict:
    session = requests.Session()
    adapter = HTTPAdapter(
        max_retries=Retry(total=0)  # 重试在外层处理
    )
    session.mount("https://", adapter)

    response = session.post(
        url=API_ENDPOINT,
        json=params,
        timeout=(
            5,    # 连接超时：5 秒内必须建立 TCP 连接
            30    # 读取超时：30 秒内必须收到完整响应
        )
    )
```

**超时时长参考**：
- 简单 API 查询：5-10 秒
- 数据库查询：10-30 秒
- 文件处理、代码执行：60-120 秒
- 整个工具的最长容忍时间：建议不超过 180 秒，否则应该考虑异步化

### 重试（Retry）

不是所有错误都值得重试。分清楚"可重试错误"和"不可重试错误"：

```python
RETRYABLE_ERRORS = {
    429,  # Rate limit，等一会再试
    500,  # 服务器临时错误
    502,  # Bad Gateway
    503,  # Service Unavailable
    504,  # Gateway Timeout
}

NON_RETRYABLE_ERRORS = {
    400,  # 参数错误，重试没用
    401,  # 未授权，需要修复认证
    403,  # 禁止访问，权限问题
    404,  # 资源不存在，重试没用
    422,  # 业务逻辑错误
}
```

重试策略用**指数退避 + 随机抖动**，避免多个 Agent 同时重试造成"雪崩"：

```python
import time
import random

def retry_with_backoff(fn, max_attempts=3, base_delay=1.0):
    for attempt in range(max_attempts):
        try:
            return fn()
        except RetryableError as e:
            if attempt == max_attempts - 1:
                raise
            # 指数退避：1s, 2s, 4s... 加随机抖动
            delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
            time.sleep(delay)
```

**多 Agent 系统的重要警告**：在 Multi-Agent 系统（如 LangGraph）中，**不要在工具节点内部自动重试**。原因是：如果 5 个并行 Worker Agent 各自重试 3 次，一次失败就变成 15 次请求，很容易触发级联故障（"重试风暴"）。应该在编排层面统一处理重试逻辑。

### 熔断器（Circuit Breaker）

当外部服务持续失败时，继续重试只是在浪费时间和金钱。熔断器在检测到持续失败后，直接"断路"——停止向该服务发请求，直到它恢复。

```
熔断器状态机：

  [Closed（正常）]  ──失败次数超阈值──►  [Open（断路）]
        ▲                                       │
        │                              等待冷却时间（如 60s）
   测试请求成功                                 │
        │                                       ▼
        └──────────────────────  [Half-Open（试探）]
                                   发一个测试请求
```

一个真实的生产故障案例：一个定时运行的 Pipeline Agent 在夜间进入重试循环，从晚上 11 点到早上 7 点一直在重试失败的 API 调用，产生了 **$437 的 API 账单**。熔断器可以在第一个小时就切断这个循环。

## 3.2.10 工具工程检查清单

在把工具交给 Agent 使用之前，过一遍这个清单：

**描述质量**
- [ ] `description` 说清楚了这个工具的使用场景（什么时候用）
- [ ] 每个参数都有明确的格式要求（不依赖模型推断）
- [ ] 说明了与其他相似工具的区别
- [ ] 有使用示例（至少一个典型调用）
- [ ] 说明了工具的能力边界（做不到什么）

**输出设计**
- [ ] 返回字段使用语义命名，不是技术 ID
- [ ] 技术状态码已映射为人类可读的值
- [ ] 大量数据有截断提示，告知总数和查询更多的方式
- [ ] 成功和失败都有统一的响应结构

**错误处理**
- [ ] 错误信息说明了"发生了什么"
- [ ] 错误信息说明了"为什么"（业务原因）
- [ ] 错误信息提供了"下一步建议"
- [ ] 区分了可重试和不可重试的错误

**安全性**
- [ ] 写操作实现了幂等性（或有幂等键机制）
- [ ] 高风险操作（删除、外发、扣款）有额外的确认机制
- [ ] 所有外部调用都设置了超时
- [ ] 有重试逻辑，且配置了退避策略

**规模化**
- [ ] 了解该工具在全量工具集中的位置（是否可能被动态加载）
- [ ] 工具粒度合理（不过细、不过粗）
- [ ] 工具名称语义清晰，不与其他工具混淆

## 小结

| 维度 | 核心要点 |
|---|---|
| 设计视角 | 工具的读者是 LLM，不是代码；以语义为中心，不以技术为中心 |
| 描述质量 | 工具描述是 prompt 的一部分，需要 prompt engineering 的严肃对待 |
| 粒度 | 按操作语义和权限边界切分，封装复杂序列为原子工具 |
| 工具数量 | 超过 10-15 个相似工具，精度开始下降；规模化靠动态加载 |
| 输出设计 | 只返回 Agent 推理需要的字段，技术值映射为语义值 |
| 错误信息 | 富语义错误（发生了什么 + 为什么 + 下一步）是 Agent 自我恢复的基础 |
| 幂等性 | 写操作必须考虑，用幂等键处理非天然幂等的操作 |
| 生产可靠性 | 超时 + 指数退避重试 + 熔断器，Multi-Agent 场景慎用节点内重试 |
| 并行调用 | Code-as-Orchestrator：Agent 写代码并行执行工具，中间结果留在沙箱，只把精简结果传回 context |
