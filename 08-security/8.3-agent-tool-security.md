# 8.3 Agent 与工具安全

> **核心问题**：Agent 调工具时，怎么防 indirect prompt injection 通过工具返回 / MCP metadata 投毒？沙箱怎么搭？
> **目标**：能为 Agent 应用设计"最小权限 + 沙箱 + 工具校验 + 审计 + Blast radius"的安全模型，并理解 MCP 工具中毒为什么是 2026 Agent 头号漏洞类型。

**展开重点**：
- **2026 现实**：
  - Agent 时代 **indirect prompt injection** 比 direct 危险得多——攻击不在用户输入里
  - **MCP 工具中毒（Tool Poisoning）**：2026 单次披露涉 **~200,000 暴露 MCP 实例**（IDE / 内部工具 / 云服务）
  - Claude Code / Cursor 等 Coding Agent 受影响最直接（能读代码、改文件、跑 shell）
- **Agent 攻击面（5 个入口）**：
  1. **用户输入**（direct injection）
  2. **工具返回**（外部 API 返回里藏指令——最常见）
  3. **工具描述 / metadata**（MCP server 里的 description / parameter 描述被改）
  4. **RAG 内容 / 浏览到的网页**
  5. **共享 memory / scratchpad**（多 Agent 跨实例污染）
- **MCP 工具中毒典型路径**：
  - 攻击者在 MCP server 的 `description` 字段嵌入隐藏指令（unicode / 注释 / 长描述末尾）
  - Agent 读取 tool spec 时把指令当系统级提示
  - 后果：调用其他工具时把秘密、token、文件泄漏给攻击者控制的 endpoint
- **防御核心：缩小爆炸半径**（NVIDIA 2026 Sandboxing 指南）：
  - **最小权限**：每个工具的权限**单独定义**（read-only / write-restricted / network-restricted）
  - **沙箱**：高风险动作放隔离环境（gVisor / Firecracker / 临时 container）
  - **网络分段**：Agent 工具调用走 egress proxy，白名单域名
  - **不可逆动作要 HITL**：发邮件、转账、删数据库——人类确认
- **工具治理（Engineering controls）**：
  - **Tool registry**：所有工具集中注册、签名、版本控制
  - **Schema sanitization**：MCP description 在 Agent 读之前先**剥离 unicode 异常、长度限制、HTML 实体清洗**
  - **Output validation**：工具返回必须符合 JSON schema，超长 / 异常 unicode 触发告警
  - **审计每个 action**：tool_name / params / result / 谁触发 / 是否经过 HITL
- **多 Agent 特别风险**：
  - **Memory / 共享 state 投毒**：Agent A 被注入 → 写入共享 memory → Agent B 读到指令
  - **Retry storm 与权限放大**（→ 7.8）：注入 + 重试 = 同一个恶意动作执行 N 次
  - **委托链路追溯**：A 让 B 执行某动作，要保留 chain-of-trust 审计
- **Coding Agent / 工程类 Agent 特别提示**：
  - 受信任的代码库里也可能藏注入（注释里的 prompt、commit message）
  - **Codespace / devcontainer 隔离**比直接在主机上跑安全得多
  - 给 PR review Agent 限制 write 权限，只能 comment
- **常见反模式**：
  - 一个 Agent 一把"超级 API key"调所有 provider：被注入即全失守
  - 工具返回不校验直接喂给下一步推理
  - "默认信任内部工具"：内部工具被同事注入也是注入
  - 没区分 read 工具和 write 工具的审批门槛
