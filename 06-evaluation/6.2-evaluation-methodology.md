# 6.2 评估方法论

> **核心问题**：四类 scorer —— **基于参考答案 / LLM-as-Judge / 人评 / 代码断言** —— 各自适合什么场景？怎么混用？怎么避免"看似在评其实在自我安慰"？
> **目标**：拿到任何一个评估需求，能立刻决定用哪类 scorer、需要多少数据、需要多少 judge / 人评对齐。

**展开重点**：
- **四类 scorer**：
  - **Reference-based**（有标准答案）：exact match、F1、BLEU、ROUGE、embedding similarity、BERTScore；适合**有明确正确答案**的任务（分类、抽取、翻译、QA）
  - **LLM-as-Judge**（用模型评模型）：rubric-based scoring、pairwise comparison、reasoning trace；适合**开放式文本**（摘要、对话、创意）；详见 6.3
  - **Human eval**（人评）：内部专家 / 众包 / 用户反馈；是 ground truth，但贵、慢；用来**对齐 judge**
  - **Code-based**（程序化断言）：JSON schema 是否合法、是否调对了工具、是否含禁词、长度、URL 可访问性；**快、便宜、零偏差**；优先用
- **选择决策树**：
  1. 能写程序断言吗？→ 用 code-based
  2. 有唯一正确答案吗？→ 用 reference-based
  3. 是开放式 / 主观？→ 用 LLM-as-Judge + 月度人评对齐
  4. 高风险 / 监管 / 重大决策？→ 必须 human eval（哪怕只是少量样本）
- **常见组合模式**：
  - **金字塔**：code-based 跑 100%（最快）→ judge 跑 100%（较快）→ human 跑 5–10% 抽样（对齐 judge）
  - **多 scorer 并跑**：同一例同时拿 reference + judge + code 评，看分歧
  - **质量 + 安全双轨**：质量用 judge，安全用 keyword + code 断言（不依赖 judge）
- **关键工程选择**：
  - **scorer 输出形态**：boolean / 0–1 / 1–5 / pairwise 胜负 / reasoning trace
  - **样本量**：50–200 是冷启动起步；几千例进入"稳定回归"；A/B 见到统计显著要看效应大小
  - **置信区间 / 显著性**：评估两个版本时要看 CI 是否重叠，不能只看均值
- **常见误区**：
  - 只看平均分，不看分布与失败例
  - judge 不对齐人评直接用
  - reference-based 用在开放式任务（语义对、字符串不同 → 误判失败）
  - code-based 能写偏不写，反而花钱让 judge 干
