# 7.5 LLM Gateway 与多 provider 编排

> **核心问题**：LiteLLM / Portkey / OpenRouter / Helicone / Kong AI Gateway 2026 各占什么位置？什么团队该选谁？什么时候要自建 gateway？
> **目标**：能为团队选定一款 LLM Gateway 并把"统一 API + 路由 + 限流 + fallback + 观测"四合一打通。

**展开重点**：
- **为什么需要 Gateway**（2026 必备组件）：
  - **统一 OpenAI 兼容接口**：业务代码无锁定
  - **路由**：根据任务 / 成本 / 延迟选模型（与 7.6 路由策略协同）
  - **Fallback / 重试 / 断路**：一家挂不影响业务
  - **限流 / 配额**：保护下游 + 控成本
  - **观测**：trace、cost、token 计量统一出口
  - **安全**：API key 收口、PII 拦截、guardrails
- **三档对比**（2026）：
  - **LiteLLM**（**开源 Python proxy**）：
    - 100+ 模型统一 OpenAI 接口
    - 路由 / fallback / 限流 / 配额都有
    - P95 ~8ms 延迟开销
    - 自托管 / 免费、社区大；企业版有
  - **Portkey**（**企业 AI Gateway**）：
    - 延迟开销 **<1ms**
    - 语义缓存 / guardrails / 高级观测内置
    - SaaS 或 self-host
    - 强 SLA 场景首选
  - **OpenRouter**（**SaaS 模型市场**）：
    - 200+ 模型、单 API key 即可
    - 自动 fallback 透明做掉
    - 延迟开销 **100-150ms**（外部 SaaS 本身成本）
    - **不支持显式 rate limiting**
    - 适合"快速接入 / 多模型尝鲜"
- **第二梯队**：
  - **Helicone**：极简接入（改 BASE_URL 即可）+ 基础 gateway，trace 强
  - **Kong AI Gateway**：已有 Kong 栈的团队自然延伸；多协议
  - **TrueFoundry / Lynkr**：企业新势力
- **选型决策树**：
  1. 想自托管 + 开源 → **LiteLLM**
  2. 企业、强 SLA、要语义缓存 + guardrails → **Portkey**
  3. 想尝鲜大量模型、不在意 latency 开销 → **OpenRouter**
  4. 已经在 Kong 上 → **Kong AI Gateway**
  5. 只想要 trace、最小改动 → **Helicone**
- **自建 vs 用现成**：
  - 现成方案已覆盖 90% 需求，**第一年不要自建**
  - 自建的合理理由：私有协议 / 极致延迟（<1ms 之内还要更低）/ 强定制 guardrails
- **Gateway 集成路径**：
  - **应用 SDK 直连 Gateway**（推荐）：代码里 base_url 指向 gateway
  - **侧车（sidecar）**：服务网格场景
  - **集中代理**：所有出站 LLM 流量经一个集群
- **常见反模式**：
  - 直接在业务代码里 import openai / anthropic SDK，**不经 gateway**：换 provider 要改全部代码
  - 用 OpenRouter 但开发期没量延迟开销，上线发现 100ms 顶不住
  - Gateway 配了 fallback 但**从未演练**：真断那天才发现没生效
