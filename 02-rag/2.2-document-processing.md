# 2.2 文档处理与切片

> **核心问题**：怎么把原始资料变成可检索单元？
> **目标**：能为不同格式（PDF / Word / HTML / Markdown / 代码）选对处理管线。

---

## 一、为什么切片是 RAG 的隐形胜负手

2.1 讲过"检索是瓶颈"。而**检索的上限，在文档进向量库之前就被切片方式锁死了**——检索再强，也只能召回你当初切出来的那些块。切错了，后面全错：

- 切太碎 → 一个完整论点被拆成半句，检索到的片段缺上下文，模型看不懂。
- 切太粗 → 一个块里混了好几个主题，向量被"平均"稀释，命中率下降，还浪费 token。
- 不管结构乱切 → 把表格从中间劈开、把标题和正文分家。

一句话：**切片（Chunking）是整个 RAG 里投入产出比最高的环节之一**，值得优先打磨。

---

## 二、文档处理管线全景

从"一堆原始文件"到"可检索单元"，标准管线是四步：

```
原始资料 → ① 加载 Loading → ② 解析 Parsing → ③ 切片 Chunking → ④ 附元数据 → 进 2.3 向量化
 (PDF/Word/         (读进来)      (还原版面结构:     (切成检索单元)   (来源/页码/
  HTML/代码…)                    标题/表格/段落)                    时间/权限…)
```

- **①加载**：把各种格式读进内存。框架里就是各种 `Loader`（LangChain / LlamaIndex 都内置几十种）。
- **②解析**：难点在 PDF、扫描件、复杂版面——要还原出标题层级、表格、多栏、图文关系。**这一步是脏活，也是质量分水岭。**
- **③切片**：把解析后的内容切成大小合适、语义完整的块。
- **④元数据**：给每个块打上来源、页码、章节、时间、权限等标签，为 2.4 的过滤和 2.7 的溯源铺路。

> 纯文本/Markdown 场景②几乎可以跳过；PDF/扫描件场景②往往是最花功夫的地方。

---

## 三、版面解析（Parsing）：两大范式与工具选型

复杂 PDF 是绝大多数 RAG 项目的第一个大坑。2026 年解析工具分成**两条技术路线**：

| 范式 | 原理 | 代表工具 | 特点 |
|---|---|---|---|
| **启发式/版面模型** | 用规则 + 训练好的版面检测模型识别结构 | **Unstructured、Docling** | 快、可自托管、可控；极复杂版面偶尔失手 |
| **VLM / Agentic 解析** | 用视觉大模型"像人一样看渲染后的页面" | **LlamaParse、Reducto** | 复杂版面/图表最强；有 API 成本、数据要出域 |

### 主流工具速查（2026）

| 工具 | 开源 | 最适合 | 一句话 |
|---|---|---|---|
| **Unstructured** | ✅ | 通用、格式杂（30+ 格式） | 最全能的开源起点，自带多种切片策略 |
| **Docling** | ✅ (Apache 2.0) | 表格、复杂版面，要自托管 | 开源里表格/版面综合最强之一 |
| **Marker** | ✅ | 学术论文、参考文献 | 论文场景好，建议上 GPU |
| **LlamaParse** | ❌ (API) | 视觉复杂文档、用 LlamaIndex 栈 | 复杂版面输出的 Markdown 最干净 |
| **pdfplumber** | ✅ | 规整的电子版 PDF、精确取表格坐标 | 轻量、可编程控制，非扫描件够用 |

### 选型建议（落地导向）

- **数据敏感、要自托管** → Docling 或 Marker。
- **已经在用 LlamaIndex、且量不大（<1000 页/天）、文档非敏感** → LlamaParse，免费额度 + 复杂版面精度难被打败。
- **格式五花八门、要一个统一入口** → Unstructured。
- **只是规整电子 PDF、想完全可控** → pdfplumber 就够，别上重工具。

> **落地提醒**：解析质量直接决定切片质量。宁可在解析上多花时间产出干净的 Markdown（保留标题层级、表格结构），也不要让脏解析污染后面每一步。扫描件还需先过 OCR（或直接走 2.9 的 ColPali 视觉路线跳过 OCR）。

## 四、切片策略：四种主流方式

这是本节核心。按"结构感知能力"从弱到强：

| 策略 | 怎么切 | 优点 | 缺点 | 适用 |
|---|---|---|---|---|
| **Fixed-size 固定大小** | 按固定 token 数硬切 | 最简单、可预测 | 常从句子/论点中间切断 | 基线，或结构极均匀的文本 |
| **Recursive 递归** | 按分隔符层级（段落→句子→词）递归切，尽量不破坏边界 | 兼顾结构与成本，**通用默认首选** | 对深层结构仍不完美 | 文档、文章、书籍等绝大多数场景 |
| **Semantic 语义** | 逐句 embedding，相邻句语义"断层"处才切 | 语义最完整 | 要给每句算 embedding，**成本高** | 主题跳跃频繁、且预算充足时 |
| **Document-aware 结构感知** | 按文档自身结构切（Markdown 标题、代码函数、HTML 标签、表格） | 尊重原生结构，块最"干净" | 依赖解析质量、需按格式定制 | 结构化强的格式（MD/代码/HTML） |

### 关键数据（帮你决策，别盲目上语义切片）

- Chroma 研究：**递归切片在 400 tokens 时召回约 85–90%，语义切片约 91–92%**——只多 2–3 个百分点，代价却是给全库每句话算 embedding。
- 2026 年 1 月的系统分析：**句子级切片在 ~5000 tokens 内就能匹配语义切片的效果，成本只是零头**。

**结论**：**递归切片（Recursive）是绝大多数场景的正确默认**，只在明确证明质量不够时才升级到语义切片。别一上来就上最贵的。

### 实战组合：按格式路由

生产系统常按文件类型分流，而不是一套切法走天下：

- **PDF/长文档** → 页级或递归 + 结构感知
- **网页/文章** → 递归切片
- **代码** → 代码感知分隔符（按函数/类边界）
- **Markdown** → 按标题层级切

## 五、chunk size 与 overlap 的权衡

两个最常调的旋钮：

**Chunk size（块大小）**

- 太小：语义不完整，检索到碎片。
- 太大：稀释向量、混入噪声、浪费上下文预算。
- **经验起点：递归切片 400–512 tokens。**
- ⚠️ **Context Cliff（上下文悬崖）**：2026 年 1 月分析发现，块过大（约 **2500 tokens** 附近）时回答质量会明显下降。块不是越大越好。

**Overlap（重叠）**

- 相邻块保留一小段重叠，避免关键句正好被切在边界、两个块都缺它。
- **经验起点：块大小的 10–20%。**

> 起手式：**Recursive、400–512 tokens、10–15% overlap**——先用这个基线跑起来，再根据评估（2.10）调。

## 六、高级切片：让每个块"自带上下文"

朴素切片有个通病：**一个块被单独取出来后，丢失了它在原文中的位置感**（"它说的'该产品'是哪个产品？"）。两个已在生产验证的高价值技术专治这个：

### 6.1 Contextual Retrieval（Anthropic）

**做法**：在给每个块做 embedding 和 BM25 索引**之前**，用 LLM 为它生成一段 50–100 token 的上下文摘要（"这段出自 XX 年报第 3 季度营收章节，讲的是……"），**预置到块的开头**再入库。

**效果（Anthropic 官方数据）**：

- 单独用可把 **检索失败率降低 49%**；
- **叠加重排（2.6）后降低 67%**。

**为什么 2026 才流行**：给每个块都调一次 LLM 本来很贵，但靠 **Prompt Caching**（把整篇原文档载入缓存一次、后续块复用），成本被压到可接受，才真正可落地。

> 这是目前"低成本、高收益"的切片增强首选。数据来源见文末 Anthropic Engineering 博客。

### 6.2 Small-to-Big / Parent-Document 检索

**核心矛盾**：检索时想要**小块**（向量精准、命中率高），生成时又想要**大块**（上下文完整、模型看得懂）。

**做法**：**用小块做检索，命中后回溯它所属的父文档/大块喂给模型。**

```
索引：大文档 → 切成父块 → 父块再切成子块（子块带 parent_id）
检索：拿用户问题去匹配【子块】(精准)
生成：命中子块后，取出它的【父块】(完整) 交给 LLM
```

这样"检索精度"和"生成上下文完整度"两头都要到了，是 2026 生产系统的常见默认之一，LangChain/LlamaIndex 都有现成实现（Parent Document Retriever）。

---

## 七、别忘了元数据（Metadata）

切片时顺手给每个块打标签，后面两处会用上：

- **来源、页码、章节** → 2.7 的**引用溯源**靠它。
- **时间、部门、权限、语言、文档类型** → 2.4 的**元数据过滤**和多租户隔离靠它。

原则：**能在入库时结构化记录的元数据，一定要记**，不要等到检索时抓瞎。

---

## 八、本节小结：切片决策清单

1. **先解析干净**：复杂 PDF 选对工具（自托管 Docling/Marker、托管 LlamaParse、规整 PDF pdfplumber），产出保留结构的 Markdown。
2. **默认递归切片**：400–512 tokens、10–15% overlap；别盲目上语义切片（只多 2–3% 召回）。
3. **按格式路由**：代码走函数边界、Markdown 走标题、网页走递归。
4. **注意上下文悬崖**：块别超过 ~2500 tokens。
5. **加两个增强**（性价比高）：Contextual Retrieval（-49%~-67% 检索失败）+ Small-to-Big（检索精度与上下文完整兼得）。
6. **打好元数据**：来源/页码/时间/权限，为过滤和溯源铺路。

> 记住：**切片的质量上限 = 检索的质量上限**。这一步做扎实，后面每一节都省力。
