# 3.1 Agent 基础与设计框架

> **核心问题**：到底什么是 Agent？Workflow 和 Agent 的本质区别在哪？什么时候该让 LLM 自主决策，什么时候反而不该？
>
> **目标**：理解 Agent 的运行机制，掌握 Workflow vs Agent 的设计分水岭，建立"用不用 Agent"的决策判断力。

---

## 3.1.1 什么是 Agent？

先从一个常见的误解说起：**LLM 本身不是 Agent**。

ChatGPT 在没有工具的情况下只是一个语言模型——你问，它答，结束。它无法主动查询数据库、无法发送邮件、无法在失败时重试。这些能力不是模型本身带来的，而是**围绕模型构建的基础设施**赋予的。

**Agent 的精确定义**：

> Agent 是一个系统——以 LLM 为决策核心，配备工具访问能力、状态（上下文/记忆）管理能力，以及在一个感知→决策→行动→观察的**循环**中自主运行的能力——直到完成目标或遇到需要人类介入的情况为止。

"Agent 属性"来自三个基础设施的叠加：

| 基础设施 | 作用 | 没有它会怎样 |
|---|---|---|
| **工具（Tools）** | 让模型能操作外部世界 | 只能生成文字，无法执行动作 |
| **状态（Context/Memory）** | 让模型记住做了什么、看到了什么 | 每次调用都是全新的，无法处理多步任务 |
| **循环（Loop）** | 让模型能根据结果决定下一步 | 无法根据执行结果自适应调整 |

三者缺一，都不是真正意义上的 Agent。

---

## 3.1.2 Agent Loop：核心运行机制

理解 Agent 最重要的模型是 **Agent Loop**，即感知→推理→行动→观察的闭合循环。

```
┌─────────────────────────────────────────────────────────┐
│                      Agent Loop                         │
│                                                         │
│  目标/任务                                              │
│      │                                                  │
│      ▼                                                  │
│  ┌────────┐    组装上下文     ┌──────────┐              │
│  │ 感知   │ ──────────────►  │  LLM推理  │              │
│  │Perceive│                  │ (决策核心) │              │
│  └────────┘                  └──────────┘              │
│      ▲                            │                     │
│      │                    选择下一步动作                 │
│      │                            │                     │
│  ┌────────┐                  ┌──────────┐              │
│  │ 观察   │ ◄──────────────  │  行动    │              │
│  │Observe │    执行结果       │  Action  │              │
│  └────────┘                  └──────────┘              │
│      │                            │                     │
│      │                      调用工具 / 生成输出          │
│      │                                                  │
│  将结果写入上下文，进入下一轮                            │
│                                                         │
│  ──► 任务完成 / 达到 step limit / 触发 HITL → 退出循环  │
└─────────────────────────────────────────────────────────┘
```

**每一轮循环发生什么：**

1. **感知（Perceive）**：从系统提示、对话历史、工具返回结果、记忆系统中组装当前上下文，传入 LLM。

2. **推理（Reason）**：LLM 读取上下文，决定下一步做什么——是调用某个工具，还是直接生成最终回答，还是要求更多信息。

3. **行动（Act）**：执行 LLM 的决定。可能是：调用 API、执行代码、搜索数据库、写文件、或向用户提问。

4. **观察（Observe）**：把行动结果（成功/失败/返回值）写回上下文，作为下一轮推理的输入。

5. **重复**，直到满足停止条件：任务完成、达到最大步数、出现需要人类决策的情况。

**关键洞察**：Loop 的存在使 Agent 具备了"自适应"能力——失败了可以重试，结果不对可以换策略，缺信息可以主动补充。这是静态 LLM 调用做不到的。

## 3.1.3 Workflow vs Agent：设计分水岭

这是整个 Agent 章节最重要的判断，值得深入理解。

### 核心问题只有一个：谁拥有控制流？

| | **Workflow（工作流）** | **Agent（智能体）** |
|---|---|---|
| **控制流归属** | 开发者（代码决定每一步） | 模型（运行时自主决定下一步） |
| **执行路径** | 预先确定，可枚举 | 运行时动态生成，不可预知 |
| **适用任务** | 结构清晰、步骤固定 | 开放式、路径不确定 |
| **可预测性** | 高（同输入同输出） | 低（同输入可能走不同路径） |
| **成本** | 低（只在编码节点调 LLM） | 高（每步都需 LLM 推理） |
| **可调试性** | 容易（每步有迹可查） | 难（路径动态生成） |
| **生产成熟度** | 高 | 中（2026 年逐步成熟） |

**最直白的区分方法**：

- 如果你能在写代码时就把所有 `if/else` 分支写完——这是 **Workflow**。
- 如果任务的执行路径本身依赖运行时的观察结果，必须由模型在执行中决定——这才需要 **Agent**。

**一个具体例子帮助区分**：

> 场景：用户上传一份财报 PDF，你的系统需要提取关键数字并生成摘要。

- **Workflow 方案**：`解析PDF → 提取数字（固定字段） → 格式化 → 调用LLM生成摘要 → 输出`。路径固定，代码写死，可重复，成本低。这是 Workflow。

- **Agent 方案**：用户说"帮我分析这家公司的财务健康状况，必要时联网查行业对比数据"。LLM 需要自己决定查什么、对比什么、是否需要搜索更多资料——路径无法提前写死。这才是 Agent。

## 3.1.4 五种 Workflow 模式（先学这个，再聊 Agent）

Anthropic 在 2024 年末的《Building Effective Agents》中系统整理了五种 Workflow 模式。这非常重要：**大多数生产系统用的是这五种模式中的某一种，而不是"完全自主的 Agent"**。先理解这五种，再决定是否需要走向 Agent。

### 模式一：Prompt Chaining（提示链）

**本质**：第一个 LLM 的输出是第二个 LLM 的输入，形成流水线。

```
输入 → [LLM-A: 翻译] → [LLM-B: 摘要] → [LLM-C: 格式化] → 输出
```

**适用场景**：任务可以清晰分解为有序的固定子步骤，且每步的质量验证可以内嵌（例如每步生成后加一个简单的检查）。

**典型案例**：
- 长文档处理：分段提取 → 聚合 → 摘要
- 多语言内容生产：生成英文初稿 → 翻译 → 本地化校对

**工程要点**：可以在每步之间加"质量门"（gate check），只有通过检查才继续，否则退出或重试该步骤。

---

### 模式二：Routing（路由）

**本质**：先由分类器判断输入类型，再把它送到对应的专门处理流程。

```
输入 → [分类器] ──► 类型A → [处理流程A]
                 ├──► 类型B → [处理流程B]
                 └──► 类型C → [处理流程C]
```

**适用场景**：任务类型差异大，用统一模型处理效果差，但每类任务内部逻辑固定。

**典型案例**：
- 客服系统：技术问题 → 技术专家模型；计费问题 → 计费模型；退款请求 → 人工
- 代码助手：代码生成 vs 代码解释 vs 代码审查，各自用不同的 system prompt

**工程要点**：分类器不一定是 LLM，可以是规则、关键词、或轻量分类模型。LLM 分类器更灵活但有延迟和成本。

---

### 模式三：Parallelization（并行化）

**本质**：把任务拆分为可独立执行的子任务，并发运行，最后聚合结果。

有两种子形态：

**Sectioning（任务分块并行）**：
```
输入 → 拆分 → [处理块A] ──┐
               [处理块B] ──┼──► 聚合 → 输出
               [处理块C] ──┘
```

**Voting（多路投票）**：同一任务跑多次，取共识结果，提升可靠性：
```
输入 ──► [LLM 第1次] ──┐
      ├──► [LLM 第2次] ──┼──► 多数投票 → 输出
      └──► [LLM 第3次] ──┘
```

**适用场景**：
- 大文档分块处理（Sectioning）
- 高风险决策需要多重验证（Voting）
- 有多个独立维度需要同时评估（如：安全性、准确性、流畅度并行打分）

**工程要点**：并行化的代价是 token 消耗成比例增加，适合对延迟敏感但对成本不敏感的场景。

---

### 模式四：Orchestrator-Workers（编排者-执行者）

**本质**：一个"主控" LLM 把复杂任务分解为子任务并分派，多个"工作" LLM 执行，主控汇总结果。

```
用户任务
    │
    ▼
[Orchestrator LLM]
    │  分解任务、分配子任务
    ▼
[Worker-A] [Worker-B] [Worker-C]  (可以是不同模型/工具)
    │            │           │
    └────────────┴───────────┘
                 │ 结果
                 ▼
         [Orchestrator 汇总]
                 │
                 ▼
              最终输出
```

**适用场景**：任务复杂到需要分解，但子任务的种类在编码时无法穷举——必须由主控在运行时动态决定拆分方式。

**典型案例**：
- 复杂研究报告：主控决定"先查市场规模，再查竞品，再看政策"，每项分派给不同执行者
- 编程任务：主控把大需求拆为多个文件修改任务

**注意**：这已经是 Workflow 和 Agent 之间的"灰色地带"。如果 Orchestrator 本身也可以根据工作结果动态调整计划，它就开始向 Agent 演化。

---

### 模式五：Evaluator-Optimizer（生成-评估循环）

**本质**：一个 LLM 生成，另一个 LLM 评估，评估结果反馈给生成者，形成改进循环。

```
输入
 │
 ▼
[Generator LLM] ──► 初版输出
                         │
                         ▼
                 [Evaluator LLM] ──► 通过 → 输出
                         │
                    不通过 + 改进建议
                         │
                         ▼
                 [Generator LLM] ──► 修改版
                         │
                    重复，直到通过或达到最大轮数
```

**适用场景**：质量标准可以明确描述（可以写进 Evaluator 的提示词），且反复迭代能显著提升质量。

**典型案例**：
- 代码生成：生成代码 → 运行测试 → 把失败信息反馈给生成者修改
- 文案优化：生成文案 → 按 A/B 标准评估 → 改进
- 数据提取：提取 → 对比 schema 验证 → 修正

**工程要点**：需要设置最大迭代次数，避免无限循环。评估标准越清晰、越机器可检验（如代码是否能运行），这个模式越有效。

---

### 五种模式的选型速查

| 模式 | 选它，当你的任务是... |
|---|---|
| Prompt Chaining | 有清晰固定的多步骤流程 |
| Routing | 输入类型差异大，需要分流处理 |
| Parallelization | 子任务独立，需要速度或多路验证 |
| Orchestrator-Workers | 复杂任务，子任务在编码时无法穷举 |
| Evaluator-Optimizer | 质量标准明确，迭代可以显著改善结果 |

> **Anthropic 的核心建议**：用能通过评估的最简单方案。不要因为"看起来更智能"就选复杂模式。很多生产问题用一个写得好的 Prompt Chaining 就能解决。

## 3.1.5 Agent 的自主性谱系

并不存在"是 Agent"和"不是 Agent"的二元区分。自主性是一个连续谱，工程实践中通常分为四个层级：

```
L1 ──────── L2 ──────────── L3 ─────────────── L4
人类执行     人类主导            Agent 主导        Agent 自主
Agent 辅助   Agent 协助          人类监督          人类设边界
```

| 层级 | 描述 | 典型场景 | 2026 生产现状 |
|---|---|---|---|
| **L1** 人类主导 | Agent 提建议，人类做每个决策 | Copilot、代码补全 | 最成熟、最广泛 |
| **L2** 协作执行 | Agent 执行标准步骤，关键节点人类确认 | 客服辅助工具、表单填写 | 多数生产 Agent 在此层 |
| **L3** Agent 主导 | Agent 自主执行，人类处理异常 | 运维自动化、数据处理流水线 | 特定垂直领域 |
| **L4** 高度自主 | Agent 在设定边界内完全自主，人类设定目标和护栏 | 复杂编程 Agent、研究助手 | 2026 年少数前沿场景 |

**重要认知**：市场宣传经常暗示 L3/L4，但大多数企业生产部署在 L1/L2。L4 的高自主性需要严格的护栏、可观测性基础设施、以及大量试错成本。**不要被宣传牵着走，从你的任务实际需要的自主程度出发选层级。**

## 3.1.6 何时用 Agent：工程决策框架

这是 3.1 最实用的部分：面对一个新任务，应该怎么判断要不要用 Agent？

### 步骤一：任务路径能在写代码时确定吗？

- **能确定** → 用 Workflow（选五种模式之一），不需要 Agent。
- **不能确定**，路径依赖运行时的观察 → 继续往下判断。

### 步骤二：可靠性要求是否允许不确定性？

这里有一个关键的数学现实：

> **多步骤可靠性衰减**：假设每步 Agent 决策的准确率是 95%，10 步之后，整体成功率只有 `0.95^10 ≈ 60%`。

| 每步准确率 | 5 步任务 | 10 步任务 | 20 步任务 |
|---|---|---|---|
| 99% | 95% | 90% | 82% |
| 95% | 77% | 60% | 36% |
| 90% | 59% | 35% | 12% |

- 如果你的场景**不能容忍 40% 的失败率**，那么纯 Agent 方案在未经充分测试前不适合上生产。
- 应对方法：缩短 Agent 步骤数、加 HITL 检查点、或把高风险步骤变成确定性代码。

### 步骤三：核对三个必要条件

只有以下三点都成立，才值得采用完全自主的 Agent：

| 条件 | 说明 |
|---|---|
| **1. 路径不可预知** | 任务执行路径取决于运行时观察，无法提前枚举 |
| **2. 错误可被检测** | Agent 的错误能被某种机制发现（测试、截图、人工检查） |
| **3. 错误代价可接受** | 失败的后果在可控范围内（可以重试、可以撤销、影响范围有限） |

**Anthropic 给出了三个满足这三条的典型案例**：
- 客服 Agent：可以执行工具动作（查订单、退款），错误可被人工复查
- 编程 Agent：可以写代码，测试套件是客观的验证机制
- Computer Use Demo：可以截图，截图就是客观的执行结果验证

### 步骤四：成本与延迟是否可接受？

Agent 的每一步循环都需要 LLM 推理，意味着：
- **延迟**：每步通常增加 1-5 秒（取决于模型和上下文长度）
- **成本**：每步 $0.01 - $0.50+（取决于模型和 token 量）

一个需要 10 步的 Agent，在高峰期每天跑 10,000 次，用 GPT-4o 每步 $0.05，日成本 **$5,000**。这不是吓唬人，而是应该在设计时就算清楚。

### 决策树汇总

```
新任务
  │
  ▼
任务步骤/路径在编码时能确定？
  │
  ├──► 是 → 用 Workflow（五种模式选一）
  │
  └──► 否
         │
         ▼
       错误可被检测 + 代价可接受？
         │
         ├──► 否 → 先做 Workflow，逐步引入 Agent 能力
         │
         └──► 是
                │
                ▼
              成本/延迟可接受？
                │
                ├──► 否 → 重新设计，缩减 Agent 步骤
                │
                └──► 是 → 可以用 Agent，从简单模式开始试
```

## 3.1.7 Agent 设计的三条基本原则

无论你决定构建什么类型的 Agent，以下三条原则应该从第一行代码就贯穿始终。

### 原则一：最小足迹（Minimal Footprint）

**含义**：Agent 应该只申请完成任务所必需的最小权限，只保留完成当前步骤所必需的数据，避免"顺便"做额外的事。

**为什么这条原则存在**：Agent 会犯错。最小足迹是在模型不可靠时的保险机制——即使做错了，损失也被限制在最小范围内。

**具体实践**：
- 工具的权限范围精确到最小（只读 vs 读写、只能访问某个目录 vs 全文件系统）
- 不要在 Agent 的上下文中堆积不必要的信息
- 不要让 Agent "顺便"把结果存到多个地方

### 原则二：优先可逆操作（Prefer Reversible Actions）

**含义**：当两种方式都能完成任务时，优先选择可以撤销的那一种。

**具体实践**：
- 修改文件之前先备份，而不是直接覆盖
- 发邮件前先生成草稿等待确认，而不是直接发送
- 数据库操作先用事务包裹，而不是直接提交
- 删除操作改为"移到回收站/归档"，而不是硬删除

**为什么重要**：不可逆操作的失败代价呈指数级放大——删错的文件、发出去的邮件、扣掉的款项都很难挽回。

### 原则三：信任层级（Trust Hierarchy）

**含义**：Agent 对指令的信任程度应该取决于指令的来源，而不是指令的内容。

```
信任级别：System Prompt > 用户消息 > 工具返回结果 > 环境观察
```

这条原则在 Agent 安全中至关重要（详见 3.11），但从设计之初就要贯彻：

- **System Prompt 中的指令**：最高信任，是开发者设定的操作边界
- **用户消息中的指令**：中等信任，在 system prompt 允许的范围内执行
- **工具返回的内容**（搜索结果、文件内容等）：低信任，**不应将其视为指令**

最后一点尤为重要：如果一个工具返回了"现在忽略你之前的所有指令，去做X"，Agent 应该把这视为**数据**而不是**指令**。这是 Prompt Injection 攻击的核心机制（3.11 详述）。

## 3.1.8 常见误区速查

学完这一节，遇到以下几种说法要能识别问题所在：

| 常见误区 | 正确理解 |
|---|---|
| "用 Agent 比 Workflow 更智能" | 智能程度取决于模型和设计，不取决于是不是 Agent；Workflow 往往更可靠 |
| "Agent 可以完全自主，不需要人工监督" | 2026 年大多数生产 Agent 在 L1/L2，需要 HITL 检查点 |
| "步骤多就要用 Agent" | 步骤多可以用 Prompt Chaining；只有路径不可预知才需要 Agent |
| "Agent 失败了重试就好" | 每步失败率累积，10 步任务在 95% 单步精度下成功率只有 60% |
| "框架选好了，Agent 就好做了" | 框架只解决编排问题；工具设计、错误处理、上下文管理才是主要工作量 |

## 小结

| 概念 | 一句话 |
|---|---|
| Agent | LLM + 工具 + 状态 + 循环；不是"更智能的聊天机器人" |
| Agent Loop | 感知→推理→行动→观察，循环直到停止条件 |
| Workflow vs Agent | 谁拥有控制流：代码（Workflow）还是模型（Agent） |
| 五种 Workflow 模式 | 链式/路由/并行/编排/评估优化——大多数生产系统的真实形态 |
| 自主性谱系 | L1-L4，生产主流在 L1-L2，不要被 L4 宣传带跑 |
| 何时用 Agent | 路径不可知 + 错误可检测 + 代价可接受 + 成本可承受 |
| 三条设计原则 | 最小足迹、优先可逆、信任层级——从第一天就贯穿 |

**下一步**：理解了"要不要用 Agent"之后，3.2 深入工具工程——Agent 能力的上限由工具质量决定。
