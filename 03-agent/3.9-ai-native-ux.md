# 3.9 AI Native UX：为自主软件设计交互

> **核心问题**：当软件从"你点一下、它执行一步"变成"你说一句、它自己跑一串、有时还替你做决定"时，交互该怎么设计？
>
> **一句话结论**：Agent UX 不是"给聊天框做美化"，而是一门解决**信任、控制、可观测**三大新问题的独立学科。传统 GUI 的心智——"每个操作都是用户主动发起、结果确定、即时可见"——在 Agent 场景里全部失效。你要设计的是**一个会自己行动、可能出错、需要被监督的协作者**的交互语言。

---

## 3.9.0 先厘清：为什么这是一门新学科，而不是"UI 换皮"

工程师最容易犯的错误，是把 Agent 产品当成"多了个 AI 后端的传统 App"来做交互。这个假设会让你在三个地方翻车。

**传统软件 UX 的三条隐含假设，在 Agent 里全部不成立：**

| 传统假设 | Agent 现实 | 带来的新 UX 问题 |
|---|---|---|
| **操作即时完成**（点击→结果，几百毫秒） | 一个任务可能跑几十秒到几十分钟，中间调很多次工具 | 用户盯着空白屏幕会以为卡死 → **必须让"正在进行"可见** |
| **结果确定可复现**（同样输入→同样输出） | 概率性输出，同一句话两次结果可能不同，还可能出错 | 用户无法预判 Agent 会做什么 → **必须让"打算做什么"先可见、可否决** |
| **用户主动发起每一步** | Agent 自己决定下一步调哪个工具、要不要行动 | 用户交出了控制权 → **必须重建"我仍在掌控"的感觉，否则不敢用** |

这三条推导出 Agent UX 的**三大支柱**，本节全部围绕它们展开：

- **可观测（Observability）**——让用户看见 Agent 在做什么、做到哪、为什么这么做。对应 3.9.2 流式 + 3.9.4 Generative UI。
- **可控（Control）**——让用户能在合适的节点介入、审批、否决、回退。对应 3.9.3 HITL + 3.9.5 信任模式。
- **可信（Trust）**——让用户在"授权 Agent 自主"和"事事亲自确认"之间找到匹配风险的平衡点，并在出错时不崩盘。对应 3.9.5 + 3.9.6。

> **一句话记住**：传统 UX 设计的是"用户如何操作软件"；Agent UX 设计的是"用户如何**监督一个替自己干活的下属**"。后者的核心不是按钮好不好看，而是**授权边界、汇报节奏、出错兜底**。

## 3.9.1 统一坐标：三种交互范式（Chat / Collaborative / Ambient）

在动手设计任何一个 Agent 产品前，先回答一个根问题：**用户和 Agent 是什么协作关系？** LangChain 的 Harrison Chase 在《UX for Agents》系列里给出了目前最实用的三分法，核心区分维度是**并发性（concurrency）**——即"用户和 Agent 是否同时在干活"。

| 范式 | 用户与 Agent 的关系 | 典型形态 | 代表产品 |
|---|---|---|---|
| **Chat（对话式）** | 一问一答，**轮流**。用户发起，等 Agent 回，再发起 | 聊天框、streaming 回复 | ChatGPT、Claude、客服 bot |
| **Collaborative（协作式）** | **同时**在同一个产物上工作，互相接力、互相"喂料" | 共享文档/画布/代码，人和 Agent 都能改 | Cursor / Copilot（人写几行、AI 补几行）、协同白板、Google Docs 式 AI 编辑 |
| **Ambient（环境/后台式）** | Agent 在**后台持续跑**，用户去干别的，只在需要时被叫回来 | 后台监控、事件触发、Agent Inbox | 邮件分诊 Agent、监控告警处理、Devin 式长任务 |

**这三者不是"谁更先进"，而是匹配不同任务性质**（这一点和 3.7 多 Agent "任务性质决定架构"是同一条方法论）：

- **Chat** 适合**探索性、需要人频繁掌舵**的任务（你还不确定要什么，边聊边收敛）。它的优点是控制感最强、门槛最低；缺点是**人被绑在回路里**，Agent 一次只能替你干一件事，你的产出被你打字速度锁死。
- **Collaborative** 适合**有明确共享产物、人机各有所长**的任务（写代码、写文档）。人负责判断和方向，Agent 负责填充和加速。关键 UX 难点是**粒度**——要显示的是 token 级别还是段落/函数级别的变化。
- **Ambient** 是 Chase 最看好、也最难做对的范式。它的杀手锏是**打破"一次一个 Agent"的天花板**：Agent 在后台跑，你可以同时有一队 Agent 替你干活，人的能力被真正放大。代价是**信任门槛陡增**——你没盯着它，凭什么信它？所以 Ambient 范式的成败，几乎全押在 3.9.3 的 Human-in-the-Loop 设计上。

> **落地建议**：绝大多数团队的第一个 Agent 产品应该从 **Chat** 起步（控制感强、容错高、用户熟悉），在验证了 Agent 的可靠性后，再针对高频、可信度已建立的场景，把某些流程"下沉"成 Collaborative 或 Ambient。**不要一上来就做全自动后台 Agent**——用户对一个还不信任的东西，不会允许它在背后自己乱跑。

## 3.9.2 流式输出：Agent 的刚需，不是锦上添花

在传统 App 里，loading spinner 是可选的礼貌；在 Agent 里，**流式输出是生死线**。原因很简单：Agent 任务动辄几十秒起步，如果这段时间屏幕是静止的，用户 100% 会认为它死了、然后刷新或放弃。流式的本质作用**不是"更快"（首字延迟并没有变短），而是"让等待变得可信"**——用户看到进展，就愿意等。

### 两种流式，别只做一半

工程上有两个层次的"流"，很多团队只做了第一个，导致 Agent 的中间过程仍是黑盒：

| 流的类型 | 流的是什么 | 解决什么问题 | 类比 |
|---|---|---|---|
| **流 Token（token streaming）** | LLM 逐字生成的最终回复文本 | 消除"最后一步的等待焦虑" | 打字机效果 |
| **流 Step（step / event streaming）** | Agent 每一步的**动作和结果**：调了哪个工具、传了什么参数、返回了什么 | 消除"整个中间过程的黑盒焦虑" | 快递物流轨迹 |

- 只做 token 流：用户在 Agent 调用 5 个工具、思考 40 秒的全程只看到一个转圈，直到最后才吐字。**中间的 40 秒仍然是黑盒。**
- 加上 step 流：用户实时看到"🔍 正在搜索订单库 → ✅ 找到 3 条 → 🧮 正在计算退款金额 → ✍️ 正在起草回复"。**等待被转化成了信任**——每一步都在证明 Agent 走在正确的路上。

> **LangGraph / Vercel AI SDK 的对应能力**：LangGraph 原生支持流 agent progress（每个节点执行后推状态）和流 LLM tokens 两种模式，可组合使用；Vercel AI SDK 的 `message.parts` 数组把文本片段和工具调用片段（`tool-*`，带 `state: loading/output-available/error`）统一成一个可流式渲染的结构，前端按 part 类型逐块渲染。**这就是 step 流在框架层的落地形态。**

### 思考过程可视化（Reasoning / Thinking）

推理模型（3.6 讲过）带来一个新的 UX 维度：**要不要、以及如何展示 Agent 的"思考"**。

- **展示的价值**：思考过程是最强的信任信号和最好的进度指示。用户读到"我在权衡两个方案：A 更快但有风险，B 更稳……"会显著提升对结果的信任，也天然填满了等待时间。
- **展示的代价与设计**：原始思考往往冗长、跳跃、包含自我否定，直接全量展示会让用户困惑甚至降低信任。主流做法是**摘要式展示 + 可折叠**——默认显示精炼后的思考要点（Anthropic 的 extended thinking 就用 summarization 在几乎不加延迟的前提下保留关键思路），用户想深挖再展开完整版。Claude Code 里 `Tab` 键切换 thinking 显示、标注 thinking budget，就是这种"默认克制、按需展开"的体现。
- **交错思考（interleaved thinking）**：Claude 4.x 支持在**工具调用之间**思考——调完工具、拿到结果后再想一步再决定下一步。UX 上这意味着思考不是开头一大段，而是**穿插在 step 流里**（Think → Act → Think → Act）。前端渲染时要把"思考块"和"动作块"交替编排，而不是假设思考只在最前面。

> **落地要点**：thinking 可视化默认**折叠 + 摘要**，不要把原始 CoT 糊用户一脸；把它当成"可选的深度"，而非默认的噪音。

## 3.9.3 Human-in-the-Loop：从"in the loop"到"on the loop"

这是 Agent UX 里最关键、也最能决定产品成败的一节。核心命题：**Agent 越自主，越需要设计好"人如何介入"。** 完全不介入的全自动 Agent 在今天几乎没有能安全上线的高风险场景；而介入设计得笨（事事弹窗确认），又会让 Agent 沦为"带更多点击步骤的普通软件"。

### 一个关键的范式跃迁：in-the-loop → on-the-loop

| | **Human-in-the-loop（人在回路中）** | **Human-on-the-loop（人在回路上）** |
|---|---|---|
| 人的位置 | 卡在执行路径上，**每一步都得等人点确认** | 站在回路之上**监督**，Agent 自己跑，人按需干预 |
| 节奏 | Agent 走一步停一步 | Agent 连续跑，只在关键点或异常时叫人 |
| 适合 | 高风险、低信任的早期阶段 | 已建立信任、需要吞吐的成熟阶段 |
| 代表 | 每个工具调用前弹窗审批 | Devin：你观察全过程，随时可倒回某个 checkpoint 修正 |

产品的成熟路径，往往就是**从 in-the-loop 逐步移向 on-the-loop**——随着信任积累，把审批点一个个撤掉。这与 3.9.5 的"渐进式自主"是同一件事的两个视角。

### Ambient / 后台 Agent 的三种介入模式（notify / question / review）

LangChain 从大量 Ambient Agent 实践里提炼出**三种、且仅三种**人机交互时机。做后台 Agent，你的交互设计基本就是这三张牌的组合：

| 模式 | 触发时机 | Agent 说什么 | 用户做什么 | 例子 |
|---|---|---|---|---|
| **Notify（通知）** | 发生了值得你知道、但**不需要你决策**的事 | "我处理了这封邮件 / 这件事有异常" | 知悉即可，可选择性介入 | 分诊 Agent 把一封邮件标记为垃圾并告诉你 |
| **Question（提问）** | Agent **被卡住**，缺一个只有你能给的信息 | "这个会议要不要参加？我需要你拍板才能回复" | 提供缺失信息，解锁 Agent | 邮件助手不知道你日程冲不冲突 |
| **Review（审批）** | Agent **想执行一个有后果的动作**，需你放行 | "我起草了这封回复，发不发？" | 批准 / 编辑 / 拒绝 | 发邮件、下单、改数据库前 |

> **为什么"仅三种"很有用**：它给了你一个**穷举清单**。设计后台 Agent 时逐一问：哪些事该 notify、哪些该 question、哪些必须 review？凡是没归入这三类的自主动作，就是你默认让 Agent 全自动了——**这个决定必须是显式的、经过风险评估的**，而不是漏设计。

### Show → Correct → Resume：可介入的三步骨架

无论哪种范式，"人能介入"落到工程上都是同一个三步骨架，它也是 LangGraph 这类框架用 **状态检查点（checkpoint）** 支撑的核心能力：

1. **Show（可见）**：Agent 的每一步都可观测（回到 3.9.2 的 step 流）。看不见就无从干预。
2. **Correct（可纠正）**：用户能在中途暂停并修正某一步。修正有三种粒度——
   - 直接**替换**这一步的工具调用/参数（我来手动填对的）
   - 让 Agent **带着我的指引重新预测**这一步（你再想想，提示是……）
   - **更新指令后继续**，甚至不暴露之前那步错了（悄悄纠偏）
3. **Resume（可续跑）**：从修正点**继续**执行，而不是从头再来。

这套骨架的技术底座是**可持久化、可回溯的 Agent 状态**（durable execution / checkpointing，3.12 会深入）——没有它，"暂停-修正-恢复"就无从谈起。**这是 UX 需求倒逼架构选型的典型例子**：你想要"可倒回 checkpoint"的 UX，就必须在架构上选带状态持久化的框架。

### 反转视角：Agent Inbox

Chase 提出一个精妙的思路反转。我们习惯把界面想成"人给 Agent 派活的地方"；但对 Ambient Agent，更该做一个**"Agent 向人求助的地方"**——一个**收件箱**，把所有需要人 notify/question/review 的事项，像邮件/工单一样按优先级列出，附带上下文元数据。

这个"Agent Inbox"心智非常适合**一人监督多 Agent**的未来形态：你不是同时盯 10 个聊天窗口，而是打开一个收件箱，批量处理"待你拍板的 10 件事"。它把 Ambient 范式"人的注意力如何分配给一队后台 Agent"这个难题，收敛成了一个所有人都会用的交互隐喻。

## 3.9.4 Generative UI：让 Agent 生成界面，而不是吐一堆文字

纯文字聊天是 Agent UX 的**最低配**，很多时候是糟糕配。当 Agent 查到一趟航班、算出一张报表、拉出一个订单，用一大段文字描述"航班 CA1234，08:30 起飞……"远不如**直接渲染一张航班卡片**——带选座按钮、改签入口。**Generative UI（生成式界面）** 就是让 Agent 的输出从"文本"升级为"可交互的富组件"。

### 核心机制：Tool → Component 映射

Generative UI 落地时，**并不是让 LLM 生成 HTML/CSS**（不可控、不安全、还慢）。主流且可靠的做法是**工具调用结果 → 预定义组件的映射**：

1. 你预先写好一批**受信任的 React 组件**（`<Weather>`、`<FlightCard>`、`<OrderTable>`……）。
2. Agent 调用某个工具（如 `getWeather`），返回**结构化数据**（JSON）。
3. 前端拿到这个 tool part，按类型**路由到对应组件**渲染，而不是打印 JSON 或文字。

以 Vercel AI SDK 为例，`message.parts` 里的工具片段长这样 `{ type: "tool-getWeather", state: "output-available", output: {...} }`，前端渲染逻辑就是一个条件路由：

```tsx
if (part.type === "tool-getWeather" && part.state === "output-available") {
  return <Weather data={part.output} />;      // 成功 → 渲染富组件
}
// loading / error 状态 → 回退到通用的工具进度/错误 UI
```

工具一多，就升级成**组件注册表（registry）**：一张 `toolType → Component` 的映射表，动态解析。这就是"**component grammar（组件语法）**"的工程形态——**你定义好积木和拼装规则，Agent 在运行时按任务挑选、组装界面**，而不是你预先画死每一个屏幕。

### 分级理解 Generative UI 的"生成"程度

"生成式界面"是个光谱，从弱到强：

| 级别 | 谁决定用什么组件 | 灵活性 | 可控性/安全性 | 落地成熟度 |
|---|---|---|---|---|
| **L1 工具结果套模板** | 开发者写死 tool→component 映射 | 低 | 高 | ✅ 成熟，生产首选 |
| **L2 Agent 选组件** | Agent 从组件库里挑用哪个、怎么排布 | 中 | 中 | 🟡 渐成熟 |
| **L3 Agent 生成组件** | LLM 现场生成界面代码 | 高 | 低（安全/一致性风险大） | 🔴 实验性，慎用生产 |

> **落地建议**：**生产环境优先 L1，谨慎试 L2，别碰 L3。** L1 的"套模板"听起来不炫，但它可控、安全、可测、可维护，覆盖了绝大多数真实需求。让 LLM 现场生成 UI 代码（L3）在 demo 里很酷，但一致性、可访问性、XSS 安全全是坑，目前不具备生产可靠性。**"生成式"的价值在于运行时按数据装配预制积木，不在于让模型写前端代码。**

### 流式 Generative UI 与内嵌的 HITL

Generative UI 与前面两节天然咬合：

- **渐进渲染**：多步任务里，每个工具返回后组件就地出现（先出航班卡，再出酒店卡……），而不是等全部跑完才刷一屏。这就是 step 流（3.9.2）在**视觉层**的体现。
- **把审批 UI 内嵌进对话流**：3.9.3 的 review 不必是丑陋的浏览器弹窗，而可以是对话流里一张**带"确认发送 / 编辑 / 取消"按钮的卡片**。HITL 从"打断式弹窗"变成"对话里的一个可交互节点"——这是体验上的巨大差别，也是 Generative UI 对控制感的关键贡献。

## 3.9.5 信任与控制：六个经过验证的实操模式

前面讲了"怎么让用户看见、怎么让用户介入"。这一节是**把信任做成可复用的模式**。综合 Smashing Magazine 的实践总结，有六个覆盖"行动前/行动中/行动后/高歧义时"全生命周期的模式。它们不是理论，每个都带可量化的成功指标——这正是"可落地"的标志。

| # | 模式 | 时机 | 做什么 | 成功指标（参考） |
|---|---|---|---|---|
| 1 | **Intent Preview（意图预览）** | 行动前 | 用大白话列出"我打算分几步做什么"，给「执行/改计划/我自己来」三个选择 | 无需编辑就被接受 >85% |
| 2 | **Autonomy Dial（自主度旋钮）** | 配置 | 让用户**按任务类型**调 Agent 的自主档位：只建议 / 提方案待审 / 执行前确认 / 全自动+事后通知 | 监控档位反复横跳率（高=信任不稳） |
| 3 | **Explainable Rationale（可解释理由）** | 行动中 | 主动回答"为什么这么做"，且**锚定用户说过的话/先例**：「因为你说过 X，所以我做了 Y」 | "行为不明"类工单 <5% |
| 4 | **Confidence Signal（置信信号）** | 行动中 | 亮出自己的把握程度（百分比/图标/范围声明），**引导用户重点审查低置信的决定** | 置信度与用户接受率相关性 >0.8 |
| 5 | **Action Audit & Undo（操作台账+撤销）** | 行动后 | 按时间列出所有 Agent 动作，状态清晰，提供**限时撤销**并标明哪些不可逆 | 回滚率 <5%（过高就该关掉该自动化）|
| 6 | **Escalation Pathway（升级通道）** | 高歧义 | 拿不准时**升级求助而非瞎猜**：意图不清 / 多个等价选项 / 高风险场景 | 升级频率 5–15%，升级后完成率 >90% |

**贯穿这六个模式的一条主线，是"摩擦要匹配风险"（confidence calibration）**：

- 低风险、高置信的动作（把邮件归档）→ 零摩擦，全自动 + 事后 notify。
- 高风险、不可逆的动作（转账、删库、发对外邮件）→ 高摩擦，强制 Intent Preview + Review 审批。
- **最糟的设计是摩擦和风险错配**：让用户为每次无害归档点确认（摩擦过高，用户烦到关掉 Agent），或让 Agent 不经确认就转账（摩擦过低，一次事故毁掉全部信任）。

### 渐进式自主（Progressive Autonomy）：信任是长出来的，不是一次给的

六模式如何随时间演进，就是**渐进式自主**——**不要一上来就给 Agent 最高权限**，而是分阶段随信任升级：

| 阶段 | 自主档位 | 主打模式 | 用户获得 |
|---|---|---|---|
| **Phase 1** | 只建议/提方案 | Intent Preview + Action Audit | 完全掌控，建立信任 |
| **Phase 2** | 执行前确认 | Autonomy Dial + Explainable Rationale | 按自己节奏放权 |
| **Phase 3** | 全自动+事后通知 | Confidence Signal + Escalation Pathway | 主动委托，被真正放大 |

这与 3.9.3 的 in-the-loop → on-the-loop 是**同一条曲线的两种说法**：产品旅程就是"逐步撤掉审批点"的过程，而每一次撤，都必须由**积累起来的可靠性数据**来支撑，而不是拍脑袋。

## 3.9.6 出错时的 UX：失败可读、可撤销、可修复

Agent 一定会出错——这不是 bug，是概率系统的属性。一个成熟 Agent 产品和玩具的区别，往往不在"顺利时多惊艳"，而在"**出错时如何收场**"。这里有三个必须做对的设计。

**1. 失败可读（Failure Legibility）**——错误要被**解释**，不能被隐藏，也不能甩一句"出错了"。好的失败消息包含三段：承认错误 + 说明纠正动作 + 给出人工支持入口。**把错误藏起来（假装成功、或含糊其辞）是信任的头号杀手**——用户宁可要一个诚实报错的 Agent，也不要一个悄悄搞砸的。

**2. 可撤销 + 操作回执（Undo & Receipt）**——每个有后果的动作都应产生一张"回执"：改了什么、在哪改的、用了什么权限、附 diff 或确认、以及**一个回滚钩子**。心理学依据很实在：**"知道能轻松撤销"本身就会鼓励用户放心委托**——安全网让人敢走钢丝。反过来，不可逆的动作必须在事前用最高摩擦拦住（Intent Preview + Review）。

**3. 修复与补偿（Repair & Redress）**——当错误已经造成影响，道歉要有结构：①明确承认（"我错误地转出了资金"）②说明即时补救（"已冲正，$250 已退回"）③提供超出 Agent 本身的支持通道（人工客服链接）。这里有个反直觉但被反复验证的现象——**服务补救悖论（service recovery paradox）**：一次被漂亮处理的失败，建立的忠诚度可能**高于**从头到尾的完美执行。**所以别把出错当纯粹的减分项，它也是赢得深度信任的机会**——前提是你认真设计了收场。

> **一句话**：Agent 产品的信任，不是在"零故障"里建立的（做不到），而是在"故障被诚实、可控、可修复地处理"里建立的。

## 3.9.7 决策框架：给你的场景选交互范式与自主档位

把本节收敛成一套可执行的决策顺序。做一个新 Agent 功能时，依次回答：

**第 1 步：选交互范式（3.9.1）**
- 任务探索性强、需人频繁掌舵、还没验证可靠性 → **Chat**（默认起点）
- 有明确共享产物、人机各擅其长（写码/写文） → **Collaborative**
- 高频、可后台跑、信任已建立、想让一人监督多任务 → **Ambient + Agent Inbox**

**第 2 步：定可观测层级（3.9.2 / 3.9.4）**
- 至少做到 **step 流**（别只做 token 流留一个黑盒中间态）
- 输出是结构化数据（航班/订单/报表）→ 上 **Generative UI（L1 套模板）**
- 用了推理模型 → thinking **默认折叠+摘要**展示

**第 3 步：设介入点（3.9.3）**
- 用 **notify / question / review** 三张牌**穷举**所有需要人参与的时机
- 凡是没归进这三类的自主动作，都是你在显式地"授权全自动"——逐一做风险评估
- 架构上确保 **checkpoint / 可持久化状态**，以支撑 show-correct-resume

**第 4 步：让摩擦匹配风险（3.9.5）**
- 给每个动作评风险×可逆性，据此挂上对应的信任模式（预览/确认/台账/撤销/升级）
- **不可逆 + 高风险 = 强制审批**；**可逆 + 低风险 = 全自动 + 事后 notify**

**第 5 步：设计失败收场（3.9.6）**
- 失败可读、关键动作可撤销、高影响错误有 repair & redress 脚本

**第 6 步：规划自主度演进（3.9.5）**
- 从 Phase 1（只建议）起步，用可靠性数据驱动逐步撤审批点，走向 on-the-loop

## 3.9.8 常见误区 · 小结 · 延伸阅读

### 六个常见误区

| 误区 | 为什么错 | 正确做法 |
|---|---|---|
| "Agent UX = 把聊天框做好看" | 忽略了可观测/可控/可信三大新问题 | 按信任-控制-可观测三支柱系统设计（3.9.0） |
| "有 loading 转圈就够了" | 几十秒黑盒中间态照样让人以为死机 | 做 **step 流**，让每一步可见（3.9.2） |
| "全自动才叫先进" | 无介入设计的高风险 Agent 没人敢用、也不该上线 | 设计好 notify/question/review 和渐进自主（3.9.3/3.9.5） |
| "每步都弹窗确认才安全" | 摩擦过高，用户烦到弃用，反而不安全 | **摩擦匹配风险**，低风险动作零确认（3.9.5） |
| "让 LLM 现场生成 UI 代码最灵活" | 一致性/可访问性/安全全是坑，不可生产 | Generative UI 用 **L1 tool→组件套模板**（3.9.4） |
| "出错就含糊带过别惊动用户" | 隐藏错误是信任头号杀手 | 失败可读 + 可撤销 + repair & redress（3.9.6） |

### 小结

- Agent UX 是解决**信任 / 控制 / 可观测**的独立学科，不是 UI 换皮。
- 先按**并发关系**选范式：**Chat**（轮流）/ **Collaborative**（同时）/ **Ambient**（后台），多数产品从 Chat 起步。
- **流式是刚需**：至少做 step 流，把等待转化成信任；thinking 默认折叠+摘要。
- **HITL 是成败手**：用 **notify/question/review** 穷举介入点，用 **show-correct-resume** 落地介入，产品旅程是从 **in-the-loop 走向 on-the-loop**；Agent Inbox 是一人管多 Agent 的关键隐喻。
- **Generative UI** 用 **tool→component 套模板（L1）** 落地，把富交互和审批 UI 内嵌进对话流，别让模型现场写前端。
- **信任靠六模式 + 渐进自主**做出来，核心律是**摩擦匹配风险**。
- **出错必然发生**，产品成熟度取决于**失败收场**：可读、可撤销、可修复（善用服务补救悖论）。
