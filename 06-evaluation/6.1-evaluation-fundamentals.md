# 6.1 评估与可观测基础与决策框架

> **核心问题**：评估（evaluation）与可观测（observability）的边界与衔接在哪里？2026 的"评估栈"包含哪些层？为什么"先建评估，再调 prompt"是 2026 共识？
> **目标**：能为一个 LLM 应用画出"评估栈分层图"，并按业务阶段（开发 / 上线前 / 上线后）选对工具与方法。

**展开重点**：
- **核心区分**：
  - **评估（offline）**：固定数据集 + 期望输出 / 评分函数；目的是版本之间可比较；是 unit test 的 LLM 化
  - **可观测（online）**：生产里发生的 trace / 指标 / 日志；目的是发现真实分布问题；含 drift、refusal、retry、cost
  - **CI 闸门**：评估 + Git PR + 阻塞合并；2026 已是事实标准
- **2026 评估栈分层**：
  1. **trace 层**（OTel GenAI `gen_ai.*`）：所有 LLM / agent / 工具调用的结构化记录
  2. **dataset 层**：golden set + 用户反馈滚动扩充
  3. **scorer 层**：reference / judge / human / code-based 四类
  4. **execution 层**：Promptfoo / DeepEval / RAGAS / Inspect AI 等框架运行
  5. **platform 层**：LangSmith / Langfuse / Arize / Braintrust 汇总 + 看板 + 告警
  6. **gate 层**：CI 集成（PR 触发 / 报告 / 阻塞）
- **2026 工程共识**：
  - **"先建评估，再调 prompt"**：没有 eval 的迭代是盲改
  - **trace ≠ eval**：trace 告诉你"发生了什么"，eval 告诉你"好不好"
  - **online + offline 双轨**：少一边都不安全
  - **每个 PR 都跑 eval**：质量回归不允许合并
  - **eval 数据集是产品**：要版本化、可追溯、可演进
- **按业务阶段选工具**：
  - **PoC 阶段**：人评 + 少量 spot check；先别花太多精力
  - **MVP / 上线前**：golden set（50–500 例）+ judge + CI 闸门
  - **生产 / 规模化**：online tracing + drift + A/B + 用户反馈回流 → dataset 滚动
- **常见误区**：
  - 用 benchmark 分数代替业务评估
  - judge 不做对齐就上生产
  - 只看离线不看线上、只看线上不防回归
  - golden set 一年不动，已经"过拟合"PR 流程
